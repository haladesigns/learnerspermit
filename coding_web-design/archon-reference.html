Archon — The Open-Source Harness Builder

Archon is a workflow engine for AI coding agents. Define your entire development process — planning, implementation, validation, review, PR creation — as YAML workflows that run deterministically across any codebase. Like what Dockerfiles did for infrastructure and GitHub Actions did for CI/CD, Archon does for AI coding.

Core Properties
🔁
Repeatable — same workflow, same sequence, every time. Eliminates model-mood variance.
⚙️
Deterministic steps — validation, tests, context curation are enforced by the harness, not left to the agent.
🪢
Multi-session orchestration — chains many Claude Code sessions across a single workflow with full log visibility.
⚡
Parallel execution — run multiple workflows simultaneously via git worktree isolation, one branch per run.
🧩
Per-node model selection — use Haiku for classification, Sonnet for implementation. Token-efficient by design.
🤝
Human-in-the-loop — pause at any node for your approval before proceeding.
The Problem It Solves
😤
AI shepherding — you're still manually running skills → review → implement → test in sequence, remembering what comes next.
🎲
Non-determinism — when you ask an agent to "fix this bug", it might skip planning, forget tests, or write a bad PR description. Every run differs.
📉
6.7% PR acceptance rate — baseline for unstructured AI coding. Harnesses like the RLEF loop can push this to ~70%.
💸
Monolithic context — planning and implementation in the same session creates bias. Archon enforces separate sessions per phase.

🔑 Key Analogy

Think n8n, but for software development. Archon wraps above Claude Code and Codex the same way GitHub Actions wraps above your scripts — defining the structure, sequence, and gates so the agent just fills in the intelligence at each step.

📦 V1 vs V2 — Important Distinction

The original Python-based Archon (RAG + task management MCP server) is fully preserved on the archive/v1-task-management-rag branch. The new Archon is a TypeScript workflow engine — completely different product, no migration path. Don't confuse them.

The Evolution: Prompt → Context → Harness

Three generations of working with AI. Each one unlocks a larger scope of reliable work.

Three Engineering Paradigms
1️⃣
Prompt Engineering (2022–2024) — craft the single best prompt to get the best one-shot output. Scope: one turn.
2️⃣
Context Engineering (2024) — curate perfect context for a single agent session so it can handle a larger set of work. Scope: one session.
3️⃣
Harness Engineering (now) — orchestrate many coding agent sessions through a structured harness. Scope: entire SDLC.
What a Harness Is
🧱
The tooling, prompting, and chaining of different coding agent sessions to elevate a single LLM's capabilities.
🔧
Hybrid architecture — some nodes are AI prompts (creative, reasoning), some are deterministic commands (test runs, context curation). Both types live in the workflow.
🎯
Two node types: prompt (sent to an agent session) and command (deterministic CLI/bash action).
📂
Workflows live in .archon/workflows/, commands in .archon/commands/. Commit them — your whole team runs the same process.
The Hybrid Secret
🤖
AI nodes — planning, implementation, investigation, code review. The agent fills in the intelligence here.
⚙️
Deterministic nodes — test runners, context curation scripts, branch creation. You don't want the agent deciding whether to run tests.
🔐
Human approval gates — pause any node in the workflow to review before proceeding. Optionally loop on feedback until satisfied.
💡
This is exactly what Stripe Minions does — 1,300 AI-only PRs/week via enforced context curation + validation at specific steps.
Key Harness Principles
🔀
Plan and implement in separate sessions. Doing both in one context window introduces bias. Always use a fresh context for implementation after planning.
🌿
Git worktree isolation. Every workflow run gets its own isolated worktree and branch. Run 5 fixes simultaneously without conflicts.
💰
Per-node model selection. Use Haiku for classification/routing (cheap, fast). Use Sonnet for implementation and review (smarter). Never burn Opus tokens on trivial steps.
🪄
Skills are the bridge. Install the Archon skill into your target repo so the agent knows how and when to invoke the Archon CLI automatically.
🔄
Retry loops are first-class. If tests fail, the workflow can automatically retry fixing and re-validating before escalating or stopping.
📊 Why This Works — The Numbers

Anthropic's own Claude Code source code is 40% harness code — agent teams, sub-agent features, and orchestration logic. That's the signal for how central harness engineering has become.

Parallel Nodes Within a Single Workflow
⚡
Not just parallel workflows — individual nodes inside a single workflow can run in parallel. Example: 4 research agents investigating different concerns simultaneously before a synthesis node merges results.
🔗
Dependency gates — parallel nodes declare dependencies. The synthesis node waits until all 4 research nodes are done before firing. The DAG structure handles this automatically.
🌿
Separate from parallel workflows — running 4 workflows in parallel means 4 isolated worktrees working on 4 different tasks. Parallel nodes within a workflow means multiple agents collaborating on the same task in the same run.
💡
Use intra-workflow parallelism for research/investigation phases where different focus areas can be covered simultaneously, then synthesized into a single artifact.
Archon vs Existing Harnesses (BMAD, GSD, SpecKit, RLEF)
🧱
Those are harnesses. Archon is a harness builder. BMAD, GSD, SpecKit, and the RLEF loop are opinionated workflows. Archon lets you build any of them — or your own — as a reusable YAML workflow.
✂️
They require changing how you work. Adopting BMAD means changing your SDLC to fit BMAD's opinions. Archon lets you encode your existing process and enforce it deterministically.
🔁
Use them as inspiration, not replacement. Point Archon at the GSD or BMAD repo. Have it analyze the phases and translate them into an Archon workflow. You get the ideas without the lock-in.
🏢
Enterprise-friendly. If your team already has a defined SDLC, Archon wraps it rather than replacing it — making adoption far easier than asking a whole team to adopt BMAD.

Installation & Setup

Get Archon running in under 5 minutes. The setup wizard handles prerequisites, credentials, platform integrations, and skill installation in one guided flow.

Full Setup (Recommended)
1
Clone the Archon repo
Clone into a local directory separate from your projects. 
git clone https://github.com/coleam00/Archon
cd Archon
bun install
2
Open Claude Code inside the Archon repo
The Archon skill lives here and guides the entire setup. 
claude
💡 Simply say: set up archon — that's all. The skill loads automatically.
3
Register your first target project
Specify the local path to the codebase you want to run workflows on. Archon auto-registers any new repo the first time it runs a workflow there. ⚠ Enter your project repo path, NOT the Archon repo itself.
4
Select platforms
Choose which interfaces to enable: CLI (default), GitHub, Slack, Telegram, Discord. Claude opens a separate terminal for credential entry — your API keys never pass through the coding agent.
5
Configure database and AI assistant
Choose SQLite (easy) or PostgreSQL. Select Claude as the AI assistant. Authenticate via your Anthropic subscription — allowed for local use via the Claude Agent SDK. 💡 Codex support is nearly complete. Poe agent SDK and OpenCode coming later.
6
Install the Archon skill into your target repo
Say yes when asked. Paste your target repo path. This copies the Archon skill so any Claude Code session there knows how to invoke the Archon CLI. ⚠ Without this step, the agent won't know to load the workflow on your behalf.
7
Return to the first Claude session and say "done"
Archon verifies all credentials, lists all default workflows, and runs a test workflow to confirm the CLI is working end-to-end.

2
Copy the Archon skill to your target repo manually
Find it at .archon/skills/ in the Archon repo. Copy to your project root.

Start the dashboard from the Archon repo by asking Claude Code to spin up the front end and back end, or run bun run dev manually.

Setup Gotchas
⚠️
Wrong repo location: Always open Claude Code in your target project repo, not in the Archon repo, when running workflows.
⚠️
Missing skill: Without the Archon skill copied into your project, the agent won't know to invoke the Archon CLI when you ask it to fix issues.
⚠️
VPS / headless setup: The credential wizard tries to open a new terminal automatically. On remote servers, you may need to open a new session manually and run archon setup.
⚠️
V1 confusion: The old Python Archon (RAG/MCP tool) is archived on archive/v1-task-management-rag. There is no migration path to V2.

⚡ Running Workflows in Parallel

From Claude Code in your project, say: "Use archon to fix GitHub issues 5, 7, 8, 9, 10, and 11" — Archon invokes the CLI 6 times simultaneously. Each run gets its own git worktree. Monitor all of them from the web UI or ask Claude for a status update. Use /loop to auto-check every N minutes.

💰 Token Efficiency — Real Numbers

Running 4 parallel fix-GitHub-issue workflows (each including investigation, implementation, validation, and PR creation) consumed roughly 20% of a 5-hour Claude limit. The key lever: use Haiku for classification and routing nodes, Sonnet for implementation and review, and reserve Opus only for the most complex implementation nodes. A well-tuned harness on Sonnet can outperform unstructured Opus usage while costing significantly less.

Building Custom Workflows

Any development process — GSD, BMAD, BEADS, your own SDLC — can be encoded as an Archon workflow. Defined once, run forever, reusable across all projects.

Using the Workflow Builder
1
Open Claude Code in the Archon repo (not your project repo).
Say: "Use the workflow builder workflow to help me make an Archon workflow." 💡 The skill loads automatically, asks you to describe the workflow, then generates the full YAML.
2
Describe your workflow — paste a repo URL, describe a framework (BEADS, BMAD, etc.), or explain your custom SDLC steps.
3
Archon generates the YAML — a full workflow file with description, provider, default model, and all nodes defined. 💡 The generated workflow appears in the web UI as a visual DAG immediately.
4
Workflow is immediately runnable on any registered codebase. No additional setup.
YAML Workflow Anatomy
description: "Use this workflow when the user asks to fix a GitHub issue."
# ^ This is what Claude reads first to decide IF to run this workflow

provider: claude
default_model: sonnet

nodes:
  - name: classify
    model: haiku          # override per node — cheap for routing
    prompt: |
      Classify this issue as bug or feature: {issue}

  - name: investigate
    command: archon-web-research  # reference a commands/ markdown file
    session: new             # fresh context window

  - name: implement
    session: new             # separate from planning — removes bias
    prompt: "Implement the fix..."

  - name: validate
    type: deterministic      # runs shell command, not AI
    command: "bun test"
    on_failure: retry        # loop on test failure

  - name: human_approval
    type: gate               # pause for human review

  - name: open_pr
    type: deterministic
    command: archon-pr-skill
Session Control
🆕
session: new — start a fresh Claude Code session for this node. Use for implementation after planning.
➡️
session: continue (default) — continue the same conversation. Use for tightly related steps.
💰
New sessions = clean context = no token bloat from earlier steps. Use strategically for token efficiency.
Overriding Defaults
📁
Default workflows are bundled in the CLI. Override any by placing a same-named YAML in .archon/workflows/ in your repo.
📝
Commands are markdown prompts in .archon/commands/. Same override rule — your repo file wins over the bundled default.
👥
Commit your custom workflows and commands — the whole team runs the same process.
✓ Good Custom Workflow Candidates
✅Your team's specific code review checklist baked into a review node.
✅Importing a framework (BMAD, GSD) as a structured SDLC harness.
✅BEADS-style persistent memory: exploration → decompose → implement loop with progress tracking.
✅Domain-specific workflows (data migrations, infra changes, security audits).
✅Any multi-step process you currently run manually and wish was automatic.
✗ Don't Put These in Workflows
❌One-off tasks that don't repeat — just use Claude Code directly.
❌Steps that inherently require human judgment at every decision — use human gates or stay manual.
❌Overly granular prompts that fragment naturally flowing work into too many context switches.
❌Credentials or secrets inline in YAML — use environment variables or the credential wizard.
🛠️ Workflow Builder Shortcut

Rather than writing YAML by hand, use the meta-workflow: "Use the workflow builder workflow to create a workflow that does X". Describe the repo or framework you want to model. Archon researches it, decomposes the steps, and writes the full YAML for you.

Context Persistence Between Nodes
🗂️
The artifact directory — each workflow execution has a dedicated artifact dir. Nodes can write markdown/JSON files here (e.g. a plan, a research summary) for downstream nodes to read.
🔀
Session: continue — pass context by continuing the same Claude Code conversation. The agent retains full history from the prior node. Use for tightly coupled, sequential steps.
🆕
Session: new + artifact read — start a fresh context window (removes planning bias) but prompt the agent to read the artifact from the prior node. This is the recommended pattern for plan → implement transitions.
📋
Parallel node outputs — when multiple nodes run in parallel, each writes its output to a named artifact. A subsequent synthesis node reads all of them and combines into a single document before the workflow continues.
# Plan node writes artifact
- name: plan
  session: new
  output: artifacts/plan.md

# Implement node reads it in a fresh session (no bias)
- name: implement
  session: new
  command: archon-fix-issue   # prompt references {artifact_path}

Key Design Decisions
🌿
Git worktree isolation per run. Every workflow execution creates its own branch and worktree. Parallel runs never conflict. Branch naming is handled automatically.
📋
YAML as the workflow definition format. Declarative, version-controllable, diff-able. The same philosophy as GitHub Actions or Docker Compose.
🔍
Description-first routing. The workflow's description field is what the coding agent reads to decide which workflow to invoke — not the filename. Write it like a skill trigger.
🔌
Skill-based agent bridge. The Archon skill (ARCHON.md) teaches the coding agent how and when to call the CLI. Without it, the agent won't self-invoke workflows.
📊
Full observability. Every node, tool call, and session is logged. View as a DAG in the web UI or stream logs via Claude Code. The web UI is a direct visualization of the YAML graph.
Adding New Coding Agents
🔌
Interface-first architecture. Every adapter and every coding agent implements a clean generic interface. Adding a new agent means following the existing Claude or Codex pattern — the codebase is designed so a coding agent can nearly one-shot it.
🔑
SDK requirement. Archon calls agents programmatically via their SDK (not headless CLI mode). Supported today: Claude Agent SDK, Codex SDK. Planned: Pi agent SDK, OpenCode. Gemini CLI has no SDK yet — use Pi SDK with Gemini as a workaround.
💡
GitHub Copilot does have a TypeScript SDK — a community PR adding support would likely one-shot given the existing patterns. Same for AMP.
🌐
Use any model through Claude Code. You can configure Claude Code to use Ollama, MiniMax, GLM, or other providers via its provider config. Archon inherits this — you're not locked to Anthropic's models even when using Claude Code as the agent.

Patterns & Operational Gotchas

Hard-won lessons from real Archon usage — workflow iteration, token management, the Dark Factory concept, and common traps to avoid.

Workflow First-Run Reality Check
⚠️
First runs won't be perfect. Even auto-generated workflows need iteration. Watch the human approval gates closely — if the workflow pauses without surfacing the artifact path or right context, that's a prompting gap to fix.
⚠️
Approval gates should surface artifacts explicitly. Add a line to your pause-step prompt: "Tell the user the exact file path of the artifact before asking for approval." Otherwise you're left asking for it manually.
⚠️
Open questions may not be for you. A planning node's "open questions" might be things the agent should research, not things for you to answer. In your prompts: "If questions require web research, answer them yourself. Only ask the human for strategic decisions."
⚠️
Inline prompts get unwieldy. Auto-generated workflows put all prompts inline in YAML. For production use, extract long prompts to .archon/commands/ markdown files — keeps the YAML readable and prompts independently editable.
⚠️
No model CLI flag yet. You can't override the model at runtime without editing YAML. Workaround: ask your coding agent to update the YAML before running and revert it after.
Token Management Patterns
💰
Haiku for routing/classification, Sonnet for implementation/review. Reserve Opus for the single most complex implementation node if needed — and only that node.
📊
Real benchmark: 4 parallel full fix+validate workflows consumed ~20% of a 5-hour Anthropic limit. That's roughly 20 complete fix-issue cycles per 5h window with efficient model selection.
🔄
Harness on Sonnet beats bare Opus. A well-structured workflow with Sonnet produces better PR acceptance rates than unstructured single-session Opus. Structure matters more than model tier.
🚫
No token budget cap per node yet — open issue. Workaround: if a node runs excessively long, ask your agent to check workflow logs and abandon/restart.
⚡
Use /loop for self-healing parallel runs. "Every 10 minutes, check on the workflows and restart any that have failed." Keeps parallel runs healthy without manual babysitting.
Workflow Patterns That Work
📂
GitHub issues as task management. File everything — bugs, features, refactors — as GitHub issues. Git log + GitHub issues = the long-term memory your agents use to understand the codebase history and what's coming next.
🤝
Second brain + Archon = delegation at scale. Add the Archon skill to your second brain repo. Your second brain learns how you work over time and can autonomously decide which workflows to invoke for a given task.
📐
Translate, don't replicate, other harnesses. Point Archon at the GSD or BMAD repo and say "take inspiration from this." You get the philosophy without the lock-in or the complexity tax.
🔗
Ask questions before building. Prompt: "Ask me questions to make sure we're aligned on what this workflow does before you create it." Reduces assumptions and cuts iteration cycles significantly.
🛠️
Bring your existing skills, commands, and MCPs. Linear MCP, Confluence skill, Playwright validation scripts — all work as nodes. Archon doesn't replace your context engineering stack; it packages it.
The Dark Factory Pattern
🏭
What it is: a codebase where AI handles 100% of writing, reviewing, and merging. Humans only provide issues. No human approval gates — PRs merge automatically. Originally a manufacturing concept (robots run factories with the lights off), now applied to AI-run codebases.
🔁
Archon as the harness: define workflows for issue triage, implementation, validation, PR review, and release. Archon enforces the entire lifecycle deterministically — AI writes, AI reviews, AI merges.
🧪
Precedent: StrongDM built an internal dark factory pushing thousands of lines to production without human review. Stripe Minions is a partial version — humans still review final PRs before merging.
⚠️
Requires comprehensive automated testing. Without human review, your test suite is the only reliability gate. Use browser automation (Playwright CLI) for end-to-end validation on every merge.
💡
Smaller models become viable when the harness is strong enough. MiniMax M2.7, Gemma 4, and other cheaper models can drive a dark factory effectively — the workflow structure compensates for model capability gaps.
Workflow Resumption & Interruption
⏸️
No "pause all" button yet. Ask your coding agent to interrupt and stop specific workflows by workflow ID. Resume: archon workflow resume <id> or ask your agent to do it.
🔁
Resume behavior: retries the last interrupted node from scratch, then continues from there. Agent retains workflow IDs in the same session — useful for batch restart after a rate limit pause.
🔮
Subworkflow nesting (one workflow calling another) is planned but not yet supported. Chain workflows manually: ask your agent to run B after A completes, or combine them into a single larger workflow.
