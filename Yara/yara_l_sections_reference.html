YARA-L 2.0

Google Chronicle's detection language. Rules match against the Unified Data Model (UDM) — a normalised schema applied to all ingested log sources. Two modes: UDM Search (ad-hoc queries) and Detection Engine (persistent, time-windowed rules that fire alerts).

Key Concepts
🔍
events: is a WHERE clause. Every line is AND. All must be true. Eliminates non-matching events permanently.
🪣
match: is a GROUP BY + time window. Forms buckets of events per unique key value per window.
📊
outcome: runs aggregate functions per bucket. Results surface in alert detail for triage.
⚡
condition: evaluates once per bucket. A per-bucket gate — not a global check across all buckets.
Full Rule Structure
rule example_rule {
  meta:
    author      = "analyst"
    severity    = "HIGH"      // LOW | MEDIUM | HIGH | CRITICAL
    description = "Detects repeated failed logins"
    tactic      = "TA0006"   // MITRE tactic
    technique   = "T1110"    // MITRE technique

  events:
    $e.metadata.log_type       = "USER_LOGIN"
    $e.security_result.action  = "BLOCK"
    $user = $e.principal.user.userid    // bind placeholder

  match:
    $user over 10m               // group by user, 10-min windows

  outcome:
    $hits = count($e.metadata.id)  // surfaces in alert

  condition:
    #e > 5                          // fire if bucket has >5 events
}
💡 Always include severity in meta:

Omitting meta: with at least severity means alerts will have no triage priority in Chronicle's alert pipeline — analysts see them with no severity label. Always include severity = "HIGH" (or LOW / MEDIUM / CRITICAL) at minimum.

Detection Rule — events:
Required
events: (required)
Single event
events:
  $e.metadata.log_type = "PROCESS_LAUNCH"
  $host = $e.principal.asset.hostname
  $host = "dc01.corp.local"
  re.regex($e.target.process.command_line,
    `powershell.*-enc`) nocase
Multi-event (correlation)
events:
  // event 1 — network scan
  $scan.metadata.log_type = "NETWORK_SCAN"
  $ip = $scan.principal.ip

  // event 2 — login from same IP
  $login.metadata.log_type      = "USER_LOGIN"
  $login.security_result.action = "ALLOW"
  $ip = $login.principal.ip   // shared var = join key
Use different event variables ($scan, $login) to correlate multiple event types. A shared placeholder variable ($ip) is the join key — Chronicle enforces they share the same value. Every line is AND. Placeholder vars bound here are available in match:, outcome:, and condition:.
Detection Rule — match:
Required / Optional
match: (optional — omit for per-event alerts)
Structure
match:
  $var1, $var2 over 10m

match:
  $host over 1h

match:
  $ip over 30m after $scan   // anchor window to event
Window durations
Xm (minutes) Xh (hours) Xd (days)
Variables listed here must be plain placeholders ($hostname) — never dotted field paths. They become the GROUP BY key. Chronicle evaluates the condition once per unique value of these vars per window.
after keyword
after $event anchors the window start to a specific event's timestamp. Used for ordered sequence detection — "B must happen after A within the window". Each triggering event creates its own independent window.
Composite key (multi-field grouping)
match: cannot reference dotted paths. Bind fields to placeholders in events: first, then use strings.concat() for a single combined key if needed: 
events:
  $user = $e.principal.user.userid
  $host = $e.target.asset.hostname
  $key  = strings.concat($user, ":", $host)
match: $key over 1h

Outcome variables surface in the alert detail — analysts see computed values without re-running queries. They can also be used in condition: via $var > N. Arithmetic on outcome vars is supported ($risk = max($score) + 10).
count vs #e: #e (the count shorthand in condition:) only drives the logic — it doesn't surface in the alert. If you also want the count visible in the alert for triage, define it as an outcome variable. The two are independent.
Detection Rule — condition:
Required
condition: (required in Detection Engine)
Patterns
// fire on every matching event
condition:
  $e

// fire when count exceeds threshold
condition:
  #e > 10

// fire on outcome variable threshold
condition:
  $total_bytes > 500000000

// fire when both event types present
condition:
  $scan and $login

// fire when first present, second absent
condition:
  $e and not $e2

// combined
condition:
  #e > 5 and $total_bytes > 1000000

// OR threshold — lateral movement OR exfil
condition:
  $scan and $login
  and ($hosts_hit > 2 or $total_bytes > 500000)
Count shorthand
#e is the count of matched events for variable $e. Equivalent to defining $hits = count($e.metadata.id) in outcome and testing $hits > N — just shorter. Does not surface in the alert detail.
Logical operators
andornot
condition: is a per-bucket gate, not a global one. You can have 100 buckets and fire exactly 1 alert, depending on whether each bucket individually crosses the threshold.
re.capture — underused but powerful. Use it whenever you need to group or filter on a part of a field value (subdomain, path segment, username prefix). RE2 only — no lookaheads. Pattern must contain exactly one capture group.
strings.concat — solves the match: dotted-path constraint. When you need a composite group-by key, bind fields to placeholders then concat into one variable.
timestamp.* — always pass an explicit IANA timezone string. Omitting it defaults to UTC and after-hours rules will misfire for non-UTC environments.
arrays.contains vs = — Chronicle's implicit array expansion means = already checks all elements for most repeated fields. Use arrays.contains() when a field doesn't expand implicitly, or when intent needs to be explicit.
re.regex(field, `pattern`) [nocase] Returns true if field matches the regex. Backtick delimiters — no escaping needed for most chars. RE2 syntax (no lookaheads).
UDM Search — encoded PowerShell
events:
  $e.metadata.log_type = "PROCESS_LAUNCH"
  re.regex($e.target.process.command_line,
    `powershell.*((-|\/)(e|en|enc|enco|encod|encoded))`) nocase
UDM Search — suspicious URI paths
events:
  $e.metadata.log_type = "NETWORK_HTTP"
  re.regex($e.target.url,
    `\/(wp-admin|phpmyadmin|\.env|\.git|admin)`) nocase
Detection Rule — beacon-like user-agent frequency
rule suspicious_useragent {
  meta: severity = "MEDIUM"
  events:
    $e.metadata.log_type = "NETWORK_HTTP"
    re.regex($e.network.http.user_agent,
      `^(curl|wget|python-requests|go-http)`) nocase
    $ip = $e.principal.ip
  match: $ip over 1h
  outcome: $hits = count($e.metadata.id)
  condition: #e > 20
}
nocase is a keyword after the closing paren, not a flag inside the pattern. Anchors ^ $, character classes, and quantifiers all work normally.
re.capture(field, `pattern`) Extracts the first capture group. Use to pull a sub-value for binding or comparison when the field doesn't exist directly in UDM.
UDM Search — extract and match domain from URL
events:
  $e.metadata.log_type = "NETWORK_HTTP"
  $domain = re.capture($e.target.url, `https?://([^/]+)`)
  $domain = "evil.example.com"
Detection Rule — group by extracted subdomain
rule dns_subdomain_spray {
  meta: severity = "HIGH"
  events:
    $e.metadata.log_type = "NETWORK_DNS"
    $subdomain = re.capture(
      $e.network.dns.questions.name, `^([^.]+)\..+`)
  match: $subdomain over 10m
  outcome: $src_count = count_distinct($e.principal.ip)
  condition: $src_count > 50  // same subdomain hit from 50+ IPs
}
Pattern must contain exactly one capture group (...). If there is no match, the variable is unset and the event is excluded. This is your tool for deriving a grouping key that doesn't exist as a direct UDM field. RE2 only — no lookaheads, no lookbehinds.
strings.concat(a, b, ...) Concatenates two or more string values. Main use: build composite group-by keys when match: can't take dotted field paths.
Detection Rule — composite key (user + host) in match
rule lateral_movement_by_pair {
  meta: severity = "HIGH"
  events:
    $e.metadata.log_type = "USER_LOGIN"
    $e.security_result.action = "ALLOW"
    $user = $e.principal.user.userid
    $host = $e.target.asset.hostname
    $pair = strings.concat($user, ":", $host)
  match: $pair over 1h
  outcome: $src_count = count_distinct($e.principal.ip)
  condition: $src_count > 3  // same user+host from 3+ source IPs
}
You can't write match: $e.principal.user.userid, $e.target.hostname over 1h — dotted paths are not allowed in match:. Bind each field to a placeholder, concat into one key, group on that.
strings.to_lower(field) Lowercases a string. Use before comparison when source data casing is inconsistent — cheaper than regex+nocase for fixed-string matches.
UDM Search — case-insensitive user match without regex
events:
  $e.metadata.log_type = "USER_LOGIN"
  strings.to_lower($e.principal.user.userid) = "jsmith"
Detection Rule — normalise before grouping
rule repeated_admin_login {
  meta: severity = "MEDIUM"
  events:
    $e.metadata.log_type = "USER_LOGIN"
    $user_lc = strings.to_lower($e.principal.user.userid)
    re.regex($user_lc, `^(admin|administrator|root)$`)
  match: $user_lc over 30m
  condition: #e > 5
}
Prefer strings.to_lower + exact match over re.regex + nocase when doing a fixed-string comparison — it's cheaper at scale and intent is clearer.
net.ip_in_range_cidr(ip_field, "cidr") Returns true if the IP falls within the given CIDR. Works with IPv4 and IPv6. Stack multiple NOT calls to exclude all trusted ranges.
UDM Search — internal src to external dest
events:
  $e.metadata.log_type = "NETWORK_HTTP"
  net.ip_in_range_cidr($e.principal.ip, "10.0.0.0/8")
  NOT net.ip_in_range_cidr($e.target.ip, "10.0.0.0/8")
  NOT net.ip_in_range_cidr($e.target.ip, "172.16.0.0/12")
  NOT net.ip_in_range_cidr($e.target.ip, "192.168.0.0/16")
Detection Rule — data leaving a sensitive subnet
rule sensitive_subnet_egress {
  meta: severity = "CRITICAL"
  events:
    $e.metadata.log_type = "NETWORK_CONNECTION"
    net.ip_in_range_cidr($e.principal.ip, "10.10.50.0/24")
    NOT net.ip_in_range_cidr($e.target.ip, "10.0.0.0/8")
    $e.network.sent_bytes > 0
    $src = $e.principal.ip
  match: $src over 1h
  outcome: $total = sum($e.network.sent_bytes)
  condition: $total > 100000000  // 100 MB out of sensitive subnet
}
Detection Rule — login from outside known-good ranges
rule login_outside_hq_ranges {
  meta: severity = "MEDIUM"
  events:
    $e.metadata.log_type = "USER_LOGIN"
    $e.security_result.action = "ALLOW"
    NOT net.ip_in_range_cidr($e.principal.ip, "203.0.113.0/24")
    NOT net.ip_in_range_cidr($e.principal.ip, "198.51.100.0/24")
    NOT net.ip_in_range_cidr($e.principal.ip, "10.0.0.0/8")
  condition: $e
}
NOT net.ip_in_range_cidr(...) is your main tool for "everything outside these known-good ranges". Stack multiple NOT calls to exclude all trusted CIDRs before asserting the match.
timestamp.get_hour(ts, "tz") Extracts the hour (0–23) from a timestamp in the given IANA timezone. Use for business-hours detection.
Detection Rule — logins outside business hours
rule after_hours_login {
  meta: severity = "MEDIUM"
  events:
    $e.metadata.log_type = "USER_LOGIN"
    $e.security_result.action = "ALLOW"
    $hour = timestamp.get_hour(
      $e.metadata.event_timestamp, "Africa/Nairobi")
    $hour > 19   // after 7pm local time
  condition: $e
}
timestamp.get_day_of_week(ts, "tz") Returns 1 (Sunday) through 7 (Saturday). Combine with get_hour for full business-hours coverage.
Detection Rule — weekend admin activity
rule weekend_admin_activity {
  meta: severity = "HIGH"
  events:
    $e.metadata.log_type = "USER_LOGIN"
    $e.principal.user.userid = "svc-admin"
    $dow = timestamp.get_day_of_week(
      $e.metadata.event_timestamp, "Africa/Nairobi")
    $dow = 1 or $dow = 7  // 1=Sun, 7=Sat
  condition: $e
}
Detection Rule — ordered sequence with timestamp comparison
rule recon_before_exfil {
  meta: severity = "CRITICAL"
  events:
    $scan.metadata.log_type  = "NETWORK_SCAN"
    $exfil.metadata.log_type = "NETWORK_HTTP"
    $exfil.network.sent_bytes > 10000000
    $ip = $scan.principal.ip
    $ip = $exfil.principal.ip
    $exfil.metadata.event_timestamp.seconds >
      $scan.metadata.event_timestamp.seconds
  match: $ip over 2h after $scan
  condition: $scan and $exfil
}
Timezone must be an IANA name: "Africa/Nairobi", "Europe/London", "America/New_York", "UTC" etc. Omitting it or using "UTC" when analysts work in another timezone will produce confusing alert times.
math.abs(n) Absolute value. Use to detect deviation in either direction from a known baseline.
Detection Rule — response size deviation from baseline
rule anomalous_response_size {
  meta: severity = "MEDIUM"
  events:
    $e.metadata.log_type = "NETWORK_HTTP"
    $e.target.url = "/api/export"
    $deviation = math.abs(
      $e.network.received_bytes - 50000)  // 50 KB baseline
    $deviation > 200000                   // 200 KB off baseline
  condition: $e
}
math.log(n) Natural logarithm. Rarely used as a filter — mainly for compressing large numeric ranges into a readable score in outcome.
Detection Rule — log-scaled score surfaced in alert
rule high_volume_dns {
  meta: severity = "MEDIUM"
  events:
    $e.metadata.log_type = "NETWORK_DNS"
    $host = $e.principal.asset.hostname
  match: $host over 10m
  outcome:
    $count     = count($e.metadata.id)
    $log_score = math.log(count($e.metadata.id))
  condition: #e > 500
}
math.log is useful for surfacing a compressed score in the alert — log(1000) ≈ 6.9, log(10000) ≈ 9.2 — so analysts see signal magnitude without raw counts that vary wildly. Rarely used as a filter condition itself.
arrays.contains(array_field, value) Returns true if any element in a repeated/array UDM field equals the value. Use when you need explicit membership testing.
UDM Search — user in a specific group
events:
  $e.metadata.log_type = "USER_LOGIN"
  arrays.contains(
    $e.principal.user.group_identifiers, "domain admins")
Detection Rule — privileged group deleting sensitive files
rule admin_group_file_delete {
  meta: severity = "HIGH"
  events:
    $e.metadata.log_type = "FILE_DELETION"
    arrays.contains(
      $e.principal.user.group_identifiers, "domain admins")
    re.regex($e.target.file.full_path,
      `\\(finance|legal|hr)\\`) nocase

}
For most repeated UDM fields (like email_addresses, dns.questions.name) Chronicle's implicit array expansion means = already checks all elements. Use arrays.contains explicitly for fields that don't expand implicitly, or when intent needs to be clear. When in doubt, try = first.

Parser RuleSection order is strict
Detection Engine rules must always follow this exact order — Chronicle's parser will reject anything else:

meta → events → match → outcome → condition

You can omit optional sections, but the sections you do include must appear in this sequence.
Required sectionsOnly events: and condition: are mandatory in Detection Engine
meta, match, and outcome are all optional. But omitting meta with at least severity means your alerts will have no triage priority in Chronicle's alert pipeline — analysts will see them with no severity label. Always include at minimum:

meta:
  severity = "HIGH"  // LOW | MEDIUM | HIGH | CRITICAL
UDM SearchUDM Search is just the events: block — nothing else
No rule {} wrapper. No condition:. No meta:. The entire query is a bare events: block. Time scoping is done via the UI date picker, not in the query syntax. 
events:
  $e.metadata.log_type = "NETWORK_HTTP"
  $e.principal.ip = "10.1.2.3"
Array fieldsChronicle auto-expands repeated fields — no indexing needed
For repeated/array UDM fields like $e.principal.user.email_addresses or $e.network.dns.questions.name, Chronicle automatically checks if any element in the array satisfies your condition. You do not need to index into them.

// This checks ALL email addresses, not just index 0
$e.principal.user.email_addresses = "user@corp.com"
Use arrays.contains() explicitly only when a field doesn't support implicit expansion or when intent needs to be unambiguous.
match: constraintmatch: only accepts plain placeholder variables — never dotted paths
This is the most common parse error. You cannot use a field path directly in match:.

// WRONG — parser error
match:
  $e.principal.asset.hostname over 1h

// CORRECT — bind first, then match on the placeholder
events:
  $host = $e.principal.asset.hostname
match:
  $host over 1h
match: + concatCan't group by two fields directly — use strings.concat() for a composite key
match: takes a variable, not a field expression. If you want to group by two fields together, bind each to a placeholder and concat them into a single key.

// WRONG
match: $e.principal.user.userid, $e.target.hostname over 1h

// CORRECT
events:
  $user = $e.principal.user.userid
  $host = $e.target.asset.hostname
  $key  = strings.concat($user, ":", $host)
match: $key over 1h
Timestamp functionstimestamp.* functions require an explicit IANA timezone
If you omit the timezone argument, Chronicle defaults to UTC. After-hours and weekend rules will misfire for any environment not in UTC. Always pass the IANA name for your environment:

timestamp.get_hour($e.metadata.event_timestamp, "Africa/Nairobi")
timestamp.get_day_of_week($e.metadata.event_timestamp, "Europe/London")
Other valid examples: "America/New_York", "Asia/Dubai", "UTC".
re.capturere.capture is underused — it's the right tool for sub-field grouping
When you need to group or filter on a part of a field value — a subdomain, a path segment, a username prefix — re.capture is your tool. The extracted value can be used as a match: group-by key or constrained directly.

Limitation: RE2 only — no lookaheads, no lookbehinds. The pattern must contain exactly one capture group. 
// extract leftmost DNS label and group on it
$sub = re.capture($e.network.dns.questions.name, `^([^.]+)\..+`)
match: $sub over 10m
