Sub-Agents

A sub-agent is a separate Claude worker — its own conversation, its own context window — that handles one focused task and reports a summary back to your main session. Instead of one agent doing everything and filling up memory, you have a team of specialists.

Why Sub-Agents?
🧹
Keeps context clean. An agent that explores 50 files doesn't dump all that output into your main chat — you get the summary only.
🔒
Safety through tool restrictions. A read-only reviewer literally cannot edit your files — enforced at the tool level, not just by asking nicely.
💸
Route by cost. Simple file exploration → haiku. Deep reasoning → sonnet. You control spend per agent.
⚡
Parallelism. Claude Code can spin up sub-agents automatically for tasks like "write 20 blog posts" — splitting work across workers simultaneously.

Foreground vs. Background
⏸️
Foreground (default). Blocks your main conversation until the sub-agent finishes.
▶️
Background. Runs while you keep working. Set background: true in the agent file, or press Ctrl+B to send a running task to the background.
📋
Use /tasks to view all running background sub-agents and their status.
One Rule to Remember
🚫
Sub-agents cannot create other sub-agents. Always one level deep. For multi-step chains, launch each sub-agent from your main conversation, passing results forward.
🔗
Chain pattern: Agent A finishes → you get summary → launch Agent B with Agent A's output as input.
💡 Key Takeaway

99% of the time Claude Code spins up sub-agents automatically. Custom sub-agents are for when you have a very specific, reusable specialist — like a "skill on steroids" with its own tools and instructions.

Step-by-Step: Creating Your First Sub-Agent
1
Create the agent file
Save a Markdown file at .claude/agents/your-agent-name.md in your project. 💡 Use lowercase letters and hyphens for the filename.
2
Add frontmatter + instructions
The top block (between --- lines) configures the agent. Everything below it is the agent's system prompt. 
---
name: code-reviewer
description: Expert code review specialist. Reviews code for quality,
  security, and maintainability. Use proactively after writing
  or modifying code.
tools: Read, Grep, Glob, Bash
model: sonnet
---

Your agent's instructions go here as normal Markdown.
3
No restart needed
Claude Code picks up new agent files immediately. You can also use the interactive builder instead of writing the file manually. 
# Interactive agent manager
/agents  → Create new agent → Generate with Claude
💡 "Generate with Claude" lets you describe the agent in plain English and it writes the config for you.
4
Use it in conversation
Reference the agent by name — Claude delegates automatically. 
"Use the code-reviewer agent to review the changes I just made."
"Run the code-reviewer and test-runner subagents in parallel."
5
Iterate on the instructions
Treat the agent file like any important project file — version control it, refine the prompt, improve the checklist over time. 💡 Test on a simple task first before relying on the agent in a real workflow.
Full Frontmatter Reference
---
name:            agent-name          # lowercase, hyphens only
description:     When Claude should use this agent   # Claude reads this to decide when to delegate
tools:           Read, Grep, Glob, Bash    # explicit list — omit = inherit ALL (usually wrong)
disallowedTools: Write, Edit              # tools to explicitly block
model:           sonnet                   # sonnet | opus | haiku | inherit
permissionMode:  default                  # default | acceptEdits | plan
maxTurns:        25                       # max steps before stopping
background:      false                    # true = always runs in background
memory:          project                  # enables persistent memory across sessions
---

Example 1: Codebase Researcher
🎯
Use case: Understand how a system is built before making changes — without dumping 50 file reads into your main conversation.
🔑
Why Haiku? Exploration involves many file reads, not deep reasoning. Fast + cheap for this use case. maxTurns: 40 to allow thorough exploration.
🔒
Why plan mode? Hard guarantee: the agent cannot modify files. Not just "please don't" — physically impossible.
---
name:           codebase-researcher
description:    Deep codebase research specialist. Use when you need to understand
                how a system is implemented before making changes.
tools:          Read, Grep, Glob, Bash
model:          haiku
permissionMode: plan
maxTurns:       40
---

You are a codebase research specialist. Produce a structured report:
1. Start broad — use Glob to understand project structure
2. Use Grep to search for key terms and patterns
3. Read the most relevant files for detail
4. Trace data flow start to finish

Output sections: Summary | Key Files | Architecture | Patterns | Issues | Recommendations
Example Prompt

"Use the codebase-researcher subagent to investigate how our authentication system works end-to-end. I need to understand the full flow before adding OAuth support."

Example 2: Test Runner
🎯
Use case: Run tests after changes, but only surface failures — not the wall of passing-test output.
▶️
Why background? Tests can take minutes. background: true lets you keep working while they run.
🧠
Why Sonnet? Analyzing test failures requires reasoning about code logic — worth the extra capability.
---
name:       test-runner
description: Test execution specialist. Use after making code changes to run
             tests and provide fix recommendations. Use proactively.
tools:      Bash, Read, Grep, Glob
model:      sonnet
background: true
---

For each failing test report: File | Error | Root Cause | Suggested Fix
Do NOT include passing test output. Only failures + final pass/fail summary.
Example Prompt

"I just refactored the user service. Use the test-runner subagent to run the user-related tests and tell me what broke."

Example 3: Smart Reviewer (with Memory)
🧠
memory: project — the agent maintains a memory file that persists across sessions. Each review builds on what it learned from the last.
📚
After a few reviews it builds up project-specific knowledge: naming conventions, intentional patterns (empty catch blocks that are "fire-and-forget"), types that are acceptable in legacy dirs, etc.
👍
The "Positive Notes" section calls out good patterns — making reviews feel less adversarial.
---
name:       smart-reviewer
description: Adaptive code reviewer that learns project patterns over time.
             Use after any significant code changes.
tools:      Read, Grep, Glob, Bash
model:      sonnet
memory:     project
---

BEFORE REVIEWING: Read agent memory for known project patterns.
Don't flag things previously learned to be intentional.

Output: Review Summary | Critical | Warnings | Suggestions | Positive Notes

AFTER REVIEWING: Update memory with new patterns discovered.
💡 What memory accumulates over time

Project Conventions (repo pattern for DB access, error response format, validation middleware) · Known Patterns not to flag (intentional empty catch blocks, any types in legacy dirs during migration) · Preferred code style specifics per module

✓ Good Practices
✅Always explicitly list tools: — never rely on the default (inherits everything).
✅Write a specific, detailed description: — Claude uses it to decide when to delegate.
✅Use permissionMode: plan for any research/exploration agent — hard read-only guarantee.
✅Use Haiku for file-read-heavy exploration; Sonnet for reasoning-heavy tasks.
✅Test sub-agents on a simple task before relying on them in real workflows.
✅Version control your .claude/agents/ files — they're part of the project.
✅Use isolation: worktree for agents that make experimental code changes.
✗ Avoid These
❌Omitting tools: — sub-agent inherits ALL tools including destructive ones.
❌Vague descriptions like "Helps with code" — Claude won't know when to delegate.
❌Expecting a sub-agent to create its own sub-agents — one level deep only.
❌Background sub-agents that ask clarifying questions — they can't interact mid-task.
❌Using prompts alone to prevent destructive commands — use hooks for hard guarantees.
❌Using Opus for every agent — route simple tasks to Haiku/Sonnet to control cost.
Gotchas
⚠️
Omitting tools = inheriting everything. If you don't specify tools:, the sub-agent gets ALL tools — including Edit, Write, and Bash. Always be explicit.
⚠️
Description quality matters enormously. This is what Claude reads to decide whether to delegate. Vague = missed delegations. Be specific about when and why to use the agent.
⚠️
Background agents can't ask questions. Design their prompts to be entirely self-sufficient — include all needed context upfront.
⚠️
Hooks enforce, prompts suggest. Prompts say "please don't do X." Hooks make X physically impossible. Use hooks if you need a real guarantee on destructive commands.
⚠️
One level of delegation only. Sub-agents cannot create sub-agents. Chain them from your main conversation by passing results from Agent A into Agent B's instructions.
