The Memory System

Every new Claude Code session starts with an empty context window — no memory of yesterday's breakthroughs or your team's conventions. The memory system solves this by loading persistent notes at session start, turning Claude from a generic assistant into a project-aware collaborator.

Two Types of Memory
✍️
CLAUDE.md — You write this. Instructions, rules, preferences. Loaded every session. Committed to git.
🤖
Auto Memory — Claude writes this. Learnings, patterns, discoveries it makes while working with you. Stored in a separate folder on your machine.
📂
Auto Memory location: a folder outside your project directory, stored under ~/.claude/projects/ and linked to your project path.
The 200-Line Limit
200
Lines auto-loaded from MEMORY.md
silent
Lines beyond 200 are silently ignored — no warning
Keep MEMORY.md as a concise index. Move detailed notes into topic files (debugging.md, patterns.md) that Claude loads on demand.
CLAUDE.md — You Write
Instructions & rules: "Use pnpm, not npm."
Fully loaded every session start
Committed to git — shared with team
Best for: "Always do X" directives
MEMORY.md — Claude Writes
Learnings Claude discovers while working
First 200 lines loaded at session start
Not in git — personal to your machine
Best for: "I discovered Y about this project"
💡 Key Insight

A well-maintained memory system means Claude already knows your build commands, conventions, deployment pipeline, and codebase gotchas before you type a single word.

Three Memory Layers

Understanding which layer to use is the key to effective memory management. Each layer has a different author, scope, and loading behavior.

Layer 1 — CLAUDE.md Files (You Write These)

Layer 2 — Auto Memory (Claude Writes These)
📋
MEMORY.md — A concise summary index. First 200 lines are loaded into every session. References topic files.
📄
Topic files (e.g. debugging.md, patterns.md) — Detailed notes loaded only when Claude needs them. No size limit matters here.
📁
Location: a folder outside your project, managed by Claude Code. View it with /memory or navigate there manually.
Layer 3 — Session Memory (Background Summaries)
🔄
Claude continuously saves summaries of what you worked on, decisions made, and open questions — in the background, automatically.
🔁
Used for continuity across sessions. Fully automatic — you don't manage this layer.

Setting Up Your Memory System

You don't have to configure much — auto memory runs by default. The key is learning when to nudge it and how to keep it organized.

Step-by-Step Setup
1
Verify auto memory is enabled
Auto memory is on by default. Confirm it and explore your memory files. 
/memory   # opens memory selector — shows files + on/off toggle
2
Let Claude build memories naturally
Just start working. As Claude discovers your build commands, test conventions, and code quirks, it saves notes automatically. You'll see it writing to the memory directory mid-session. 💡 Nothing to configure here — just observe it happening.
3
Explicitly save important discoveries
When you solve something tricky or make a key decision, tell Claude directly using natural language — no special syntax needed. 
"Remember that the payment service expects amounts in cents, not dollars."

"Save to memory that the API tests require a local Redis instance on port 6379."
⚠ Say "remember that..." or "save to memory..." — these go to MEMORY.md. The # prefix goes to CLAUDE.md instead.
4
Review and organize after a few sessions
After 3–5 sessions, open /memory and audit what Claude has saved. Look for: 
• Duplicate entries to consolidate
• Outdated info to delete
• Learnings that are really instructions → move to CLAUDE.md
• Detailed entries that should become topic files
5
Create topic files for detailed knowledge
Once MEMORY.md approaches 100 lines, extract detailed notes into topic files. 
"Move all debugging-related memories into a separate debugging.md topic file
and update MEMORY.md to reference it."
💡 MEMORY.md becomes an index. Topic files hold the detail. Claude reads them on demand.
6
Prune regularly (every 2–4 weeks)
Remove entries about fixed bugs, temporary workarounds, and outdated notes from before a refactor. ⚠ Stale memory causes confusion — it can contradict your current CLAUDE.md. Keep it accurate.

🗂 Organize by Topic, Not by Date

Chronological logs ("Session 2026-01-15: learned X") just become a growing mess. Group entries by topic — Payment Service, Testing, Deployment — so outdated entries are easy to spot and prune.

# Option 2: environment variable
export CLAUDE_CODE_DISABLE_AUTO_MEMORY=1
Gotchas — The Traps That Catch People
⚠️
Silent 200-line cutoff: Content past line 200 in MEMORY.md is silently ignored — no warning, no error. You won't know it's being skipped. Keep MEMORY.md lean; use topic files for the rest.
⚠️
CLAUDE.md vs Auto Memory conflicts: If MEMORY.md says "use npm" but CLAUDE.md says "use pnpm," Claude gets confused. Periodically audit auto memory for contradictions with your CLAUDE.md rules.
⚠️
MEMORY.md doesn't travel with you: It lives on your local machine, not in git. Switch machines and your auto memory stays behind. For critical knowledge, put it in CLAUDE.md (which is in git).
⚠️
Topic files are NOT loaded at startup: They're loaded on demand. Make sure MEMORY.md clearly indexes what each topic file contains — otherwise Claude won't know to look for them.
⚠️
The # shortcut goes to CLAUDE.md, not MEMORY.md: When you prefix a message with #, that note goes into your CLAUDE.md instruction file. To add to auto memory, say "remember that..." or "save to memory that..."
⚠️
Imports only work in CLAUDE.md files: The @path/to/file syntax works in CLAUDE.md and its variants — not in MEMORY.md or auto memory topic files.
✓ Save This
✅Build & test commands that work for your project
✅Decisions and the reasoning behind them
✅Debugging solutions to problems that took time
✅Environment quirks — required services, port numbers
✅Project-specific vocabulary and domain terms
✅Important file locations
✗ Don't Save This
❌Generic advice ("write clean code") — wastes context
❌Secrets or API keys — use environment variables
❌Things Claude can figure out from reading your code
❌One-off instructions — those belong in your prompt
❌Long documentation — link with @path/to/file instead
❌To-do lists — memory is for knowledge, not task tracking
✅ Promotion Pattern — When to Move Memory to CLAUDE.md

Watch for auto memory entries that are actually instructions, not learnings. "The team prefers named exports" or "PRs must include a Test Plan" are conventions — promote them to CLAUDE.md where they carry more weight. Keep MEMORY.md for genuine discoveries: "the flaky test is caused by a race condition."
