Debugging with Claude Code

When things break, Claude Code is your best debugging partner — but it has one requirement: it needs to see what went wrong. Vague prompts get vague answers. Full context gets precise fixes on the first try.

Core Principle
📋
Context is everything. Always give Claude the full error output, the file in question, and what you expected to happen.
🔄
Most bugs fix in ≤ 3 exchanges when you provide full context from the very first message.
🖼️
No error message? Use text descriptions or screenshots for UI bugs that don't produce explicit errors.
🔁
Claude is good at its own loop — errors it encounters in-session it can see, recognize, and fix automatically.
Prompt Quality
The difference a good prompt makes:
❌ Bad
"The API route is broken, can you fix it?"
✅ Good
"POST /api/apply returns 500. Full terminal output: [paste]. Worked yesterday. Added Resend in last commit. .env.local has RESEND_API_KEY set."
Good prompts tell Claude what changed, what you expected, and give it everything to find the root cause.
💡 Key Takeaway

Debugging isn't about typing the right magic words — it's about showing Claude exactly what broke. The more real output you paste, the faster you get a fix.

The 4-Step Debugging Loop
1
Reproduce
Can you trigger the bug reliably? Run the failing command, visit the broken page, or run the test. Copy the exact output — not a summary, not a screenshot of half the terminal. 💡 Pipe output directly: npm run build 2>&1 | claude -p "Fix these errors"
2
Isolate
Give Claude the full error output and let it trace through your code. It will read files, follow imports, and find the root cause. For multi-file bugs, use Plan Mode first. 💡 Plan Mode: press Shift+Tab (×2)Shift+Tab (×2), or say "Let's plan first — don't make any changes yet."
3
Fix
Claude proposes a change. You review it. If it looks right, let it apply the edit. Claude creates automatic checkpoints before changes — you can always rewind. ⚠ If the fix looks wrong, press Esc to interrupt and clarify the correct behavior before proceeding.
4
Validate
Run the command again. If the error is gone, you're done. If a new error appears, go back to Step 1 — Claude already has context from this round and can fix faster. 💡 Ask Claude to re-run the command explicitly: "Run npm run build again and confirm it passes."
When to Use Plan Mode
🗂️
Bug could be in multiple files (frontend, API route, database query)
👥
You're debugging someone else's code
🔗
A fix in one file might break another file
🔍
You want to understand root cause before patching the symptom
⚠ When to Reset

If you've been going back and forth for more than 2–3 correction cycles, stop. Type /clear and write a single comprehensive prompt with everything you've learned. A clean session with a precise prompt almost always outperforms a long session with accumulated failed attempts.

SCENARIO 1
Build Error After Adding Supabase
▼
1
Capture & pipe the error
Send all output (including errors) directly to Claude: 
npm run build 2>&1 | claude -p "Fix these TypeScript build errors"
Or from inside a Claude Code session: "Run npm run build and fix any errors you find."
2
Common causes after adding Supabase
  • Missing env vars at build time — Claude will suggest adding variables or handling missing values gracefully
  • Type mismatches — Supabase data may not match component expectations; Claude will add types or suggest the Supabase CLI
3
Validate
"Run npm run build again and confirm it passes." 💡 If new errors appear, Claude already has context and can fix them faster.
SCENARIO 2
API Route Returns 500 With No Useful Error
▼
1
Add visibility first
Terminal shows only POST /api/apply 500 in 234ms — no details. Tell Claude: 
"POST /api/apply returns 500 with no stack trace. Read
app/api/apply/route.ts and check for issues. I suspect
the try/catch is swallowing the error."
2
Claude fixes error swallowing
It will find the catch block returning a generic message without logging the real error. Claude adds console.error to surface the actual exception. ⚠ This is a common Next.js pattern — errors in API routes get swallowed by default.
3
Debug the real error now visible
Hit the endpoint again. Example: Error: Resend API key is missing — could be a wrong env var name, or you need to restart the dev server after adding new env vars.
SCENARIO 3
Intermittent Failure (JSON Parse Error)
▼
1
Reproduce and capture the raw response
"Run the carousel generation and show me the full response from the Anthropic API. I need to see what it returns when it fails." 💡 You may discover the API sometimes wraps JSON in markdown code fences, breaking the parser.
2
Isolate the parsing code
"Read the route file and find where we parse the response. Is there handling for code fences?"
3
Fix and stress-test
Claude applies a robust parsing function that strips code fences before parsing. Then validate: 
"Run the carousel generation 3 times and confirm
it succeeds each time."
Piping Errors to Claude Code
🔧
Build errors: npm run build 2>&1 | claude -p "Fix these build errors"
🧪
Test failures: npm run test 2>&1 | claude -p "Fix these failing tests"
✂️
Last 100 lines: npm run dev 2>&1 | tail -100 | claude -p "What is causing the error?"
⚡
Reusable shortcut — add to your shell config: 
ask() { "$@" 2>&1 | claude -p "Analyze this output and suggest fixes."; }
# Then use:
ask npm run build
ask npx tsc --noEmit

5 Reusable Debugging Patterns
1
The Step-by-Step Logger
Can't figure out where a bug starts? Ask Claude to add logging at each key point: 
"The Supabase query returns an empty array but I have published posts.
Add console.log statements to app/blog/page.tsx to show:
1. The exact query being executed
2. The raw response from Supabase
3. Any error object."
💡 Very effective at surfacing RLS (Row Level Security) policy blocks — a common Supabase issue.
2
The Before/After Comparison
Something worked before a change and broke after: 
"The blog page worked before my last commit. I added tag filtering
and now clicking a tag shows no posts. Compare current
app/blog/page.tsx with the previous version
(use git diff HEAD~1 app/blog/page.tsx) and find what broke."
3
The Environment Check
Works locally, fails in production: 
"Everything works in dev but Vercel returns 'Failed to fetch'.
Check: 1. Are env vars correct for production?
2. Are Supabase URL/key set in Vercel's env vars?
Read Supabase client files and .env.local to diagnose."
4
The Subagent Investigator
For bugs that require reading many files, delegate to a subagent: 
"Use a subagent to investigate why the carousel PDF generation
times out after 30 seconds. Have it trace the full path from
the API route through the Google Docs integration and report back."
💡 Subagent runs in its own conversation memory. Your main session stays clean.
5
The Fresh Start
After 2–3 failed correction cycles: 
/clear
# Then write ONE comprehensive prompt with:
# - What you're trying to do
# - What you've tried
# - All relevant error output
# - What changed before the bug appeared
⚠ Don't carry failed attempts forward — context pollution makes Claude less effective, not more.
✓ Do This
✅Include the complete error + stack trace, not a summary.
✅Use /clear after 2–3 failed cycles and restart with a clean prompt.
✅Run /cost periodically and /compact at ~70% capacity.
✅Restart npm run dev after any .env.local change.
✅Use Plan Mode before letting Claude touch multi-file bugs.
✅Add a # Debugging section to CLAUDE.md for persistent rules.
✗ Avoid This
❌Vague prompts like "it's broken" or "fix the API."
❌Continuing a long broken session instead of using /clear.
❌Letting Claude fix test files to match broken implementation (fix the code, not the test).
❌Assuming env vars are live without restarting the dev server.
❌Debugging without first ensuring errors are properly logged.
❌Spending hours on the same problem — know when to change approach.
Known Gotchas
⚠️
The Fix Loop: Claude fixes file A → breaks file B → fixes file B → breaks file A. Press Esc to interrupt and clarify: "Stop. The correct behavior is X. Fix the implementation to match the test, not the other way around."
⚠️
Swallowed Errors in Next.js: Before debugging the real issue, always ensure errors are being logged. Prompt: "Check all API routes in app/api/ and make sure every catch block logs the error with console.error."
⚠️
Full Conversation Memory: Long debugging sessions fill up memory. Claude starts "forgetting" earlier instructions. Run /cost to check, /compact at 70%, and /clear between unrelated tasks.
⚠️
Env Vars After Changes: After adding, renaming, or changing any value in .env.local, you must restart npm run dev. Next.js does not hot-reload env var changes. This catches everyone at least once.
⚠️
Supabase Empty Results: If Supabase queries return empty arrays despite having data, check Row Level Security (RLS) policies first — they silently block reads rather than throwing errors.
