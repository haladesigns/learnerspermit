playwright-cli

Token-efficient browser automation for Claude Code. Runs headless or headed, supports parallel sub-agents, and uses an accessibility-tree snapshot saved to disk — not dumped into context — for dramatically lower token usage than MCP or the Chrome extension.

What is Playwright?
🎭
Browser automation framework from Microsoft, open source and free. It lets you control a real browser — clicking, typing, navigating, filling forms — programmatically instead of by hand.
🌐
Supports Chromium, Firefox, and WebKit (Safari engine) on Windows, macOS, and Linux. Most Claude Code workflows use Chromium.
🧪
Originally a testing tool — end-to-end UI tests, form validation, regression checks. But it can do anything a person can do in a browser: shopping, scraping, logging in, filling out forms.
🤝
Playwright + Claude Code = Claude gets hands. Instead of only reading and writing code, it can now actually operate your app like a user would.
Headed vs Headless
👁️
Headed — a real browser window opens on your screen. You can watch every click and keystroke happen in real time. Useful for debugging or demos. Slower.
👻
Headless — the browser runs silently in the background with no visible window. Same browser, same behavior, just invisible. Faster, lighter on your machine.
⚙️
playwright-cli is headless by default. You won't see anything happen unless you explicitly ask for headed mode. This is intentional — headless is better for automated, parallel, or repeated tasks.
💡
To watch it run: tell Claude "make it headed" or add --headed to the command. To go back to invisible: just don't say anything — headless is the default.
Why CLI over the alternatives
💰
~90k fewer tokens vs Playwright MCP for the same task — CLI saves the accessibility tree to disk and sends only a summary to Claude.
📸
No screenshot flood — Claude in Chrome takes screenshots to understand pages. Every image costs tokens. The CLI reads the accessibility tree instead.
⚡
Parallel sub-agents — spawn 3+ agents testing simultaneously. Claude in Chrome is one tab at a time only.
👻
Headless by default — browsers run invisibly in the background, less drag on your machine. Add --headed when you want to watch.

The Accessibility Tree — what Playwright actually reads
♿
It's a real web standard, not a Playwright invention. Every browser builds an accessibility tree for every page — it's the technology that lets blind and visually impaired users operate websites via screen readers. Playwright simply taps into what's already there.
🗺️
It maps the entire page as structured data — every button, input, link, heading, and interactive element, described by role and name, with no images. Think of it as a text-only blueprint of the UI that anyone (or any agent) can navigate without seeing it.
🤖
This is how Claude "sees" a page — not by looking at a screenshot, but by reading this structured tree. That's why it can find and interact with elements precisely, even in headless mode with no visible browser.
📦
The tree is large. A typical page's accessibility tree can be substantial — every element, its role, label, state, and relationship to other elements. This is what makes naive implementations expensive.
⚠ MCP approach
Gets the full accessibility tree and dumps the entire thing into Claude's context on every single step. Repeated every action. ~90k extra tokens for a typical task.
✓ CLI approach
Gets the same full tree, saves it to disk at .playwright-cli/, then sends Claude only a concise summary — the refs and context it actually needs to act.
💡 Practical upside: because Claude reads the accessibility tree rather than screenshots, it works equally well headed or headless — and you can inspect the raw tree yourself in .playwright-cli/ if Claude is struggling to find an element.
💡 Mental Model

Think of playwright-cli as giving Claude Code hands to operate a browser, at the lowest possible token cost. Anything requiring a browser — testing, scraping, e-commerce automation, auth flows — is in scope.

Three-step installation
1
Install the Playwright CLI globally 
npm install -g @playwright/cli@latest
# verify it works
playwright-cli --help
💡 Or use locally: npx playwright-cli --help
2
Install the browser engine 
# Chromium (recommended for most tasks)
npx playwright install chromium

# Or install all browsers + system deps
npx playwright install --with-deps
⚠ Use a different engine? Check playwright install --help for webkit, firefox options.
3
Install the Claude Code skill 
playwright-cli install --skills
💡 This installs the Microsoft-authored skill into your Claude Code project. You can edit it like any skill — it's a living document. Use the skill creator to audit or improve it.
🤖 Let Claude Do It

Alternatively, open Claude Code, paste the GitHub repo URL and say "install everything I need for playwright-cli." It will run all three steps for you.

How a session works
🟢
Session starts when Claude opens a browser. Cookies and storage state are kept in memory for the duration of the session.
🔄
Commands chain — each CLI call outputs a snapshot of the current page state, which Claude uses to decide the next action. No screenshots needed.
💾
Persist a session with --persistent: cookies and localStorage survive browser close and can be reloaded next run. Useful for staying logged in.
🏷️
Named sessions let you run separate browser instances for different projects simultaneously: -s=project-name. Claude Code can be pointed at a specific session via PLAYWRIGHT_CLI_SESSION=name.
🔴
Session ends when the browser closes — state is lost unless you used --persistent or explicitly saved with state-save.

Targeting Elements — how Claude finds things on the page
🏷️
Element refs from snapshots — Claude runs snapshot, gets refs like e15, then uses click e15. Fast and precise.
♿
Role selectors — click "role=button[name=Submit]" — most stable method, matches directly against the accessibility tree. Prefer this when prompting Claude.
🎯
CSS selectors — click "#main > button.submit" — works but can break if markup changes.
🔗
Chained selectors — click "#footer >> role=button[name=Submit]" — scope a role selector inside a CSS container for precision.

The UI Testing Workflow

The primary use case from the video: run parallel agents testing a UI from multiple angles simultaneously — happy path, edge cases, validation — while you watch or let it run headless.

Single Headed Test (watch it work)
1
Start your dev server (Claude Code can do this for you too) 💡 Claude has full visibility into your project's page structure — tests are more accurate than against unknown URLs.
2
Ask Claude in plain language 
Use the playwright-cli tool to test the form submission.
Single agent. Make it headed so I can watch.
Claude loads the skill, inspects the project, opens the browser, fills the form, and returns results + screenshots.
3
Browser closes automatically after the test — results and screenshots delivered in Claude's response.
Three Parallel Agents (the demo)
1
Describe your angles — ask Claude what it recommends: 
What are the best ways to stress-test this form?
2
Launch parallel sub-agents 
Spin up 3 parallel sub-agents using the playwright-cli skill.
Test the form from three angles: happy path, edge cases, validation.
⚠ Default is headless — you won't see the browsers. Add "make them headed" if you want to watch all three.
3
Review consolidated results — each agent reports independently, Claude summarizes findings. 💡 This entire workflow can be packaged into a skill — see the Workflow tab for skill creation.
Turning a Workflow into a Skill
1
Articulate the workflow until Claude can't get it wrong — describe agents, test angles, success criteria fully.
2
Open the skill creator 
I want to turn this workflow into its own meta skill:
[paste full workflow description]
3
Invoke it with one line from now on 
Use the form-tester skill.
Three parallel agents spawn automatically — no re-describing needed. 💡 The skill creator also lets you run evals and measure improvement vs the plain-language version.
🧠 The Mindset

Ask: "Can I standardize this?" If yes → ask: "Can I turn that standard into a skill?" Repeated workflows always deserve a skill. You should never describe the same multi-agent process twice.

Key Gotchas
⚠️
Headless by default: you will NOT see any browser window unless you explicitly say "headed" or add --headed to the command. Don't wonder why nothing popped up.
⚠️
CLI ≠ full Playwright test suite: playwright-cli is for coding agents. For full test framework features (retries, reporters, sharding), you still want npx playwright test with a config file.
⚠️
Skills are not locked: the Microsoft-provided skill is a starting point. If Claude keeps getting something wrong, edit the skill rather than fighting it with prompts.
⚠️
Snapshots live in .playwright-cli/: after each command, the full accessibility tree YAML is saved here. You can inspect it if Claude is struggling to find an element.
⚠️
Internal pages are more reliable: testing your own project is easier than testing external URLs — Claude has full code context about your page structure, reducing accessibility tree mismatches.
