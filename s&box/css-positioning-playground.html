Which ancestor does absolute use?
The target is always position: absolute; top: 12px; left: 12px;. What changes is which ancestor's position is non-static — and that's the box those numbers measure against.
the rule An absolute element measures its top/left/right/bottom against its nearest positioned ancestor. "Positioned" means anything except static. If there is no positioned ancestor, it falls back to the root (in s&box, the screen panel).

Flexbox — the s&box workhorse
display: flex turns a container's children into flex items. They line up along a main axis (the direction) and a cross axis (the other one). Two properties do 90% of the work.
s&box note Flexbox is how almost every s&box HUD is built. display: flex, flex-direction, justify-content, align-items, and gap all work as expected. flex-wrap and flex: 1 (grow shorthand) too. CSS Grid is not supported — flex is your only layout primitive (??: as of my last check).

Putting it together: a HUD
This is a stripped-down version of what a real s&box ScreenPanel looks like. Each element is position: absolute with two of top/right/bottom/left set — that's the standard "pin to a corner" idiom.
crosshair — top:50%; left:50%; transform:translate(-50%,-50%)
health — bottom:16px; left:16px
ammo — bottom:16px; right:16px
prompt — top:60%; left:50%; transform:translateX(-50%)
the centering trick To center an element of unknown size: set its position 50% along the axis, then pull it back by half its own size with transform: translate(-50%, -50%). This is the standard idiom and it works in s&box.
pointer-events For HUDs you usually want pointer-events: none on the root (so clicks pass through to the game) and pointer-events: all on specific clickable children. That's an s&box-specific gotcha, not a positioning concept — but it bites everyone once.
Drag, click, observe. The whole point: positioning is geometry, not vibes.
