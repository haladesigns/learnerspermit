01
Simple vs Easy
The meta-concept behind every Clojure design decision

Rich Hickey — Clojure's creator — draws a sharp distinction between two words that programmers use interchangeably. Getting this distinction right changes how you think about every language choice.

Simple

Objective. Not tangled together. A thing that does one job, has one role, is not complected (braided) with other concerns. Simplicity can be measured. It doesn't depend on who you are.

Easy

Subjective. Familiar or nearby. Something is easy for you if you already know it, if it's close to tools you've used before. Easy is relative — what's easy for one person is hard for another.

Clojure deliberately chose simple over easy. Parentheses feel unfamiliar (not easy) but there's only one syntax rule (simple). Immutability feels inconvenient at first (not easy) but eliminates an entire class of bugs (simple — no entangled state). The payoff compounds: simple things remain understandable as programs grow. Easy-but-complex things become unmaintainable.

The key word: complect Hickey uses the verb "to complect" — to interweave or braid things together that should be separate. Objects in OO languages complect state, identity, and value into one thing. Functions complect in Clojure are called out and eliminated. The goal is always: separate concerns that are logically separate.
Python comparison Python is designed to be easy — familiar syntax, gentle learning curve, lots of batteries. It's also more complex in the Hickey sense: mutable by default, methods tied to objects, state mixed with identity. This isn't a criticism of Python — it's a different trade-off. Clojure trades early ease for long-term simplicity.
02
S-expressions, Forms & cons
The syntax, the evaluable unit, and the primitive everything is built from

These three ideas explain the shape of every line of Clojure you have ever written. They are not abstract — they are the concrete mechanics of how the language reads and evaluates code.

S-expression (symbolic expression)

The universal syntax unit of Lisp languages. An s-expression is either an atom (a single value: a number, string, keyword, symbol) or a list (a parenthesised sequence of s-expressions). Every piece of Clojure code is an s-expression. There is no other syntax.

Form

Any s-expression that Clojure can evaluate. A special form is one the compiler handles directly (def, if, let, fn). A macro form is expanded before evaluation. A function call form evaluates the first element as a function and applies it to the rest. Every form returns a value.

The rule for reading a list form is always the same: the first element is what gets called, everything after is an argument. This single rule covers 100% of Clojure syntax — arithmetic, conditionals, function definitions, everything.

; Atoms — self-evaluating s-expressions
42           ; => 42    (number)
"hello"      ; => "hello" (string)
:name        ; => :name  (keyword)
true         ; => true   (boolean)

; List forms — (operator arg1 arg2 ...)
(+ 1 2)             ; function call — calls + with 1 and 2
(if true "yes" "no"); special form  — if is handled by the compiler
(def x 42)          ; special form  — binds symbol x to 42
(println "hi")      ; function call — calls println with "hi"

; Nested s-expressions — each sub-expression evaluates first
(+ (* 2 3) (- 10 4))   ; => 12   (* 2 3)=6, (- 10 4)=6, (+ 6 6)=12

; The shape never changes — operator always comes first
(> 10 5)     ; not  10 > 5
(not true)   ; not  !true
(str "a" "b"); not  "a" + "b"
Python comparison Python has multiple syntax forms: infix operators (1 + 2), function calls (f(x)), subscripts (d[k]), attribute access (obj.method()), statements (if, for), comprehensions. Each has its own rules. Clojure has one: (operator args...). The uniformity feels strange at first and pays off forever — there are no special cases to memorise.

Special forms

Built-in forms the compiler handles directly — they cannot be redefined. There are fewer than 20: def, if, do, let, fn, loop, recur, quote, try, throw, new, .. Everything else — when, cond, defn, ->> — is built on top of these using macros.

Macro forms

Forms that are transformed into other forms before evaluation. defn expands to def + fn. when expands to if + do. ->> rearranges nested calls. The macro runs at compile time, receiving unevaluated code as data and returning new code to evaluate.

cons — the primitive beneath all sequences

cons (short for "construct") is the fundamental building block of Lisp. It takes a value and a sequence and returns a new sequence with the value prepended. A Lisp list is nothing more than a chain of cons cells — each cell holds one value and a reference to the next cell. The last cell points to nil.

A list as a chain of cons cells:
The list '(1 2 3) in memory:

[ 1 | ─────]──►[ 2 | ─────]──►[ 3 | ─────]──► nil
  ^               ^               ^
  head            head            head
  (first)         (second)        (third)

Each box is one cons cell: a value + a pointer to the next cell.
(first '(1 2 3))  => 1          (the head of the first cell)
(rest  '(1 2 3))  => (2 3)      (the tail — everything after)
(cons 0 '(1 2 3)) => (0 1 2 3)  (a new cell prepended)

first, rest, and cons are the three primitives that define the entire sequence abstraction. Every sequence function — map, filter, reduce, take, drop — is ultimately built from these three operations. This is why the sequence abstraction is universal: if something can do first, rest, and cons, every sequence function works on it automatically.

; cons builds sequences
(cons 1 '(2 3))      ; => (1 2 3)   — prepend to a list
(cons 1 [2 3])       ; => (1 2 3)   — prepend to a vector (returns seq)
(cons 1 nil)         ; => (1)        — cons onto nil = single-element list

; Decompose with first and rest
(first '(1 2 3))     ; => 1
(rest  '(1 2 3))     ; => (2 3)
(first (rest '(1 2 3))) ; => 2      — second element

; The recursive pattern: process head, recurse on tail
; (This is how map, filter etc. work internally)
(defn my-count [coll]
  (if (seq coll)
    (inc (my-count (rest coll)))
    0))

(my-count '(1 2 3))  ; => 3
Why this matters Understanding cons explains why conj adds to different ends for lists vs vectors (lists are cons-based — prepending is O(1); vectors are trees — appending is O(1)). It explains why rest always returns a seq regardless of the input type. And it connects Clojure directly to McCarthy's original 1958 Lisp — car/cdr/cons are the same three primitives, just renamed to first/rest/cons.
03
Values, Identity & State
The most important conceptual split in all of Clojure

Object-oriented languages fuse three distinct concepts into one mutable object. Clojure separates them deliberately. Understanding the difference is the single biggest step toward thinking in Clojure.

Value

An immutable piece of information. 42. "hello". {:name "Ada"}. A value never changes — it just is. The number 42 is always 42. Two values are equal if they contain the same information.

Identity

A logical entity that takes on different values over time. "Ada's score" is an identity. Right now it holds the value 95. After a game it might hold 112. The identity is stable; the values it holds change.

State

The value of an identity at a specific point in time. "Ada's score at 3pm was 95." State is always a pairing of identity + value + time. Yesterday's state and today's state are different, but both are valid values.

Time

Clojure models time explicitly. An identity moves through a succession of values — like a river that is always "the river" even as the water flowing through it changes. Old values are never destroyed.

In Python, a list scores = [95, 88] fuses all three: the name scores (identity), the current content (value), and the ability to change in place (time) are all bundled into one mutable object. When you do scores.append(72), the old value is gone. You cannot ask "what was scores before?"

Clojure's model — an identity across time:
scores = [95, 88]
→ t1
scores = [95, 88, 72]
→ t2
scores = [95, 88, 72, 60]
Both old values still exist. Any thread that grabbed a reference at t1 still sees [95, 88].
The OO problem In OO languages, identity and value live in the same memory location. The only way to "change" an object is to mutate that memory. Everyone pointing to it sees the change — whether they expected it or not. Clojure separates them: values are immutable, identities are explicit references that can be updated to point at a new value.
Python comparison x = [1, 2, 3] — x is a name (identity) pointing at a mutable list (value). When you do x.append(4), the list itself changes. Any other variable pointing to the same list sees the change too. This is mutation — and it's the default in Python. In Clojure, no value ever changes. To "update" something, you create a new value.
04
Immutability
The default is safe — mutation is the opt-in exception

In Clojure, every value is immutable by default. Calling assoc, conj, dissoc, or any other "change" function does not modify the original. It returns a new value. The original is unchanged and always safe to share.

Python — mutation is the default
person = {"name": "Ada", "age": 36}
person["age"] = 37    # mutates in place
# person is now {"name": "Ada", "age": 37}
# the old value {age: 36} is gone forever

numbers = [1, 2, 3]
numbers.append(4)     # mutates the list
# any other variable pointing to
# numbers now sees [1, 2, 3, 4]
Clojure — new values, original untouched
(def person {:name "Ada" :age 36})
(def older (assoc person :age 37))
; person  => {:name "Ada" :age 36}  unchanged
; older   => {:name "Ada" :age 37}  new map

(def numbers [1 2 3])
(def more (conj numbers 4))
; numbers => [1 2 3]   unchanged
; more    => [1 2 3 4] new vector

This has cascading benefits that compound throughout a codebase:

Benefits of immutability Free thread safety. Immutable values can be shared across threads with zero synchronization. No locks, no mutexes, no concurrent modification exceptions — because nothing can change.

Simpler reasoning. A function that receives a map will always see the same map. No defensive copying. No wondering if something mutated it in another branch.

Free equality. Two values are equal if they contain the same data. Since values don't change, equality checks are stable.

History for free. Keep a reference to an old value and you have undo. Keep a sequence of old values and you have a complete audit trail.
Common misconception Immutability does not mean "you can never change anything." It means values don't change — but you can always create a new, updated value and point an identity at it. The distinction is between mutating a value (forbidden) and updating a reference (allowed, explicit, controlled). Atoms, refs, and agents are how you do the latter.
05
Persistent Data Structures & Structural Sharing
How immutability stays fast — the implementation that makes it practical

The most common objection to immutability is performance: "If every update creates a new copy, won't that be impossibly slow and waste enormous memory?" The answer is no — because Clojure's data structures don't actually copy. They share structure.

Surface level — what you need to know When you call (conj my-vector 4), Clojure does not copy the entire vector. The new vector shares most of its memory with the original. Only the changed parts are new. This is safe because immutable values can never be modified — sharing is always safe. The result is that "copying" is nearly free.

This property is called persistence — not in the sense of disk storage, but in the sense that every old version of a data structure persists in memory, accessible and correct. "Persistent" means: an operation on a data structure produces a new version while the old version remains intact and accessible.

Mechanistic detail — how it actually works:

Clojure's vectors are implemented as 32-way trees (a tree where each node can have up to 32 children). When you update one element, Clojure walks down the tree to that element and copies only the nodes on the path from root to that element. With a branching factor of 32, a vector of 1 million elements has a tree depth of only ~4 levels (32⁴ ≈ 1,000,000). Updating one element copies at most 4 nodes — not 1,000,000.

Structural sharing — updating element at index 2 in a vector of 8:
                    root
                   /    \
               node-A    node-B            <-- shared, unchanged
              /  |  \    /  |  \
           [0] [1] NEW [3] [4] [5]         <-- NEW replaces old [2]
            |   |    |   |   |   |
           "a" "b" "X" "d" "e" "f"

Only 2 nodes copied (path from root to element).
The rest of the tree — everything in grey — is SHARED with the old vector.
Old vector [0][1][2][3][4][5] still exists, unchanged, pointing to original node.

For maps and sets, Clojure uses a more sophisticated structure called a Hash Array Mapped Trie (HAMT) — invented by Phil Bagwell and adapted by Rich Hickey. A HAMT combines the best properties of a hash table and a tree: near-constant lookup time, efficient memory use, and structural sharing. The branching factor is still 32. Updating a map of 1 million entries still copies only ~4 nodes.

Python — copying is expensive
import copy
original = {"a": 1, "b": 2, "c": 3}

# Shallow copy — new dict but same values
modified = {**original, "b": 99}
# Creates a completely new dict object
# Memory: O(n) — proportional to dict size

# Deep copy — even more expensive
deep = copy.deepcopy(original)
# Must traverse entire structure
Clojure — structural sharing
(def original {:a 1 :b 2 :c 3})
(def modified (assoc original :b 99))

; original => {:a 1 :b 2 :c 3}  unchanged
; modified => {:a 1 :b 99 :c 3} new map
; But :a and :c are SHARED in memory
; Only :b and the path to it are new
; Memory: O(log32 n) — nearly constant!
Why "persistent"? The term comes from a 1986 computer science paper: "Making Data Structures Persistent" by Driscoll, Sarnak, Sleator, and Tarjan. A persistent data structure is one that preserves all previous versions of itself when modified. Clojure's collections are all persistent. Every version of every collection you've ever created is still valid and accessible — at essentially zero cost.
06
The Sequence Abstraction
One interface that works on everything — the source of Clojure's composability

In most languages, you need different functions for different data structures. Python has list.append() but dict.update(). You iterate over a list differently from how you iterate over a dict. Clojure takes a different path: one unified interface called seq works on everything.

The seq abstraction is built on three operations. If something can do these three things, it's a sequence and all sequence functions work on it:

OperationWhat it doesWorks on
(first coll)Returns the first element, or nil if emptyvectors, lists, maps, sets, strings, ranges, files…
(rest coll)Returns a seq of everything after the firstsame — all seqable things
(cons x coll)Returns a new seq with x prependedsame

Because map, filter, reduce, take, drop, and every other sequence function is built on these three primitives, they all work on any collection automatically:

; The SAME map function works on all of these:
(map inc [1 2 3])          ; => (2 3 4)   vector
(map inc '(1 2 3))         ; => (2 3 4)   list
(map str "hello")          ; => ("h" "e" "l" "l" "o")  string!
(map first {:a 1 :b 2})    ; => (:a :b)   map keys
(map inc (range 5))        ; => (1 2 3 4 5)  lazy range

; filter works the same way
(filter even? [1 2 3 4 5]) ; => (2 4)
(filter even? (range 10))  ; => (0 2 4 6 8)
(filter :active [{:name "Ada" :active true}
                 {:name "Alan" :active false}])
; => ({:name "Ada" :active true})
Python comparison Python has made progress here — map() and filter() work on iterables generally. But Clojure goes further: maps (dicts) are seqable as sequences of key-value pairs, so you can map over a map the same way you'd map over a list. There's no dict.items() call needed. The abstraction is more complete and more uniform.
Programming to abstractions This is a design principle: write functions that work against the interface (seq), not a specific type (vector, list, string). A function that accepts "anything seqable" is immediately more useful than one that only accepts vectors. This is why Clojure code is so composable — functions written for one purpose tend to work in many more contexts.
07
Everything is an Expression
No statements — every form returns a value, always

In Python, there are statements (things that do something but don't return values — like if blocks, for loops, assignments) and expressions (things that evaluate to a value — like 1 + 2 or "hello".upper()). Clojure has no statements. Everything is an expression. Everything returns a value.

Python — statements don't return values
# if is a STATEMENT — doesn't return a value
if x > 0:
    label = "positive"
else:
    label = "other"

# Workaround: ternary expression
label = "positive" if x > 0 else "other"

# for loop is a STATEMENT
for i in range(5):
    print(i)
# returns None implicitly
Clojure — every form is an expression
; if IS an expression — returns a value
(def label (if (> x 0) "positive" "other"))

; let IS an expression
(def result
  (let [x 10
        y 20]
    (+ x y)))  ; => 30

; Everything composes naturally
(str "Score: " (if (> score 90) "A" "B"))

Prefix notation is the other side of this coin. In Clojure, the function always comes first. This is called prefix (or Polish) notation:

; Python infix:   1 + 2 + 3 + 4
; Clojure prefix: (+ 1 2 3 4)  — same rule, n arguments

; Python:  10 > 5
; Clojure: (> 10 5)  — same form as any other function call

; Python:  "hello"[1]
; Clojure: (get "hello" 1)  — or (nth "hello" 1)

; The rule never changes:
; (function arg1 arg2 arg3 ...)
; Whether it's arithmetic, comparison, or your own function
Why this matters When everything is an expression and the syntax is uniform, code becomes data. You can pass expressions around, generate them, transform them. There are no special cases where "this kind of thing works differently." The uniformity is what makes macros possible and what makes the code so composable.

The parentheses are not a cost — they're the payoff of this uniformity. Every pair of parens is a function call. The first thing inside is always the function. This single rule covers 100% of Clojure syntax.

08
Code as Data / Homoiconicity
Code and data share the same structure — the foundation of macros

This is one of the most distinctive and powerful properties of Lisp languages. The word homoiconicity means "same representation" — code and data are represented the same way. In Clojure, a function call is a list. A list is a data structure. Therefore a function call is a data structure that you can manipulate with ordinary Clojure functions.

; This is a function call:
(+ 1 2 3)

; This is a LIST of data:
'(+ 1 2 3)

; They look identical — the quote just says "treat this as data,
; don't evaluate it." Remove the quote and it evaluates.

; So code IS data. Data CAN BE code.
; The quote (') is the only difference.

(def my-code '(+ 1 2 3))  ; a list — just data
(first my-code)             ; => +  (the symbol +)
(rest  my-code)             ; => (1 2 3)
(eval  my-code)             ; => 6  (now execute it as code)

This property is what makes macros possible. A macro is a function that receives unevaluated code (as data structures — lists, symbols, vectors) and transforms it into new code (other lists, symbols, vectors) before evaluation. The macro runs at compile time, manipulating your code the same way any other function manipulates data.

; The threading macros you already use are macros
; ->> is not a function — it's a macro that rewrites code

; You write:
(->> [1 2 3 4 5]
     (filter even?)
     (map #(* % 10))
     (reduce +))

; The macro rewrites it to:
(reduce + (map #(* % 10) (filter even? [1 2 3 4 5])))

; These are identical at runtime. The macro ran at compile time,
; received your ->> form as a data structure, rearranged it,
; and handed back the nested version to be compiled.
The practical upshot In Python, if you want a new language feature — say, a custom loop construct, or a new control flow form — you cannot add it. You're limited to what the language provides. In Clojure, you can add new syntax as a library. when, cond, for, ->>, and, or — these are all macros written in Clojure itself. The language can grow new features from userland.
Python comparison Python has decorators and metaclasses as metaprogramming tools, but they operate at a different level and with much less power. Python code cannot easily manipulate other Python code as data structures — the AST is accessible but not the primary representation. In Clojure, the primary representation of code is already a data structure. That's the essential difference.
09
Lazy Evaluation
Computation is deferred until results are actually needed

Most sequence functions in Clojure return lazy sequences — descriptions of a computation that hasn't happened yet. Nothing is computed until something actually consumes the values. This makes it possible to work with infinite sequences, build processing pipelines without intermediate memory allocation, and compose large data transformations efficiently.

; (range) is an INFINITE sequence — no upper bound
; This line itself costs almost nothing
(def all-numbers (range))

; Nothing is computed until you consume:
(take 5 all-numbers)   ; => (0 1 2 3 4)   only 5 values computed

; Pipeline over potentially huge data — no intermediate collections
(->> (range 1000000)       ; infinite lazy range
     (filter even?)         ; lazy — nothing computed yet
     (map #(* % %))         ; lazy — still nothing computed
     (take 5))              ; NOW computation happens, for 5 values only
; => (0 4 16 36 64)
; Only 10 numbers were ever examined (the first 5 evens: 0,2,4,6,8)
Surface level — what you need to know Lazy sequences are like a recipe, not a meal. Writing the recipe costs nothing. Cooking happens only when you eat. take, first, doall, into, and reduce are the "eat" operations — they force realization. Until then, the sequence is just a description of what to compute.

Mechanistic detail: A lazy sequence is a linked structure where each node holds either a computed value or a thunk — a zero-argument function that, when called, produces the next value and another thunk. The sequence realizes itself incrementally as elements are consumed. Once realized, the value is cached so the thunk runs at most once per element.

Python — generators are similar but different
import itertools

# Python generators are lazy
def naturals():
    n = 0
    while True:
        yield n
        n += 1

nums = naturals()  # lazy, like Clojure
list(itertools.islice(nums, 5))
# => [0, 1, 2, 3, 4]

# But: generators are stateful and one-shot
# You cannot go back or share them safely
Clojure — lazy seqs are values
(def naturals (range))  ; lazy, infinite

(take 5 naturals)  ; => (0 1 2 3 4)
(take 5 naturals)  ; => (0 1 2 3 4) AGAIN — safe!

; Clojure lazy seqs are IMMUTABLE VALUES
; You can share them, pass them to multiple
; functions, and each gets the same result.
; No stateful iterator that "runs out."
The key gotcha — don't mix laziness with side effects Because lazy sequences compute on demand, side effects inside them fire at unpredictable times — or not at all. (map println [1 2 3]) may print nothing if nothing consumes the result. Use doseq for side-effecting loops, or run!. Use doall to force a lazy seq when you need effects to happen immediately.

Managed References
Mutation is not forbidden — it is explicit, controlled, and coordinated

Clojure doesn't pretend the world is stateless. Programs need to model change — a counter that increments, a cache that fills, a user's session that updates. Clojure handles this by making mutation explicit, controlled, and limited to specific reference types. You cannot accidentally mutate a value — you must explicitly use a reference type and a coordinated update operation.

There are four reference types, each designed for a different concurrency scenario:

Type Coordination When to use Update function
atom Uncoordinated, synchronous Single independent piece of state. The default choice. swap! reset!
ref Coordinated (STM), synchronous Multiple pieces of state that must change together consistently. alter commute
agent Uncoordinated, asynchronous State that changes as a result of actions dispatched to it. send send-off
var Per-thread Dynamic binding — different values in different threads. binding set! 
; ATOM — the everyday state container
(def counter (atom 0))

@counter           ; => 0   (deref to read current value)
(swap! counter inc)        ; applies inc to current value atomically
@counter           ; => 1
(swap! counter + 10)       ; applies + with 10
@counter           ; => 11

; The key: swap! takes a PURE FUNCTION.
; It may be retried if another thread updates concurrently.
; Never use swap! with functions that have side effects.

; REFS — coordinated updates across multiple values
(def account-a (ref 1000))
(def account-b (ref 500))

(dosync                           ; transaction — all or nothing
  (alter account-a - 200)
  (alter account-b + 200))
; Either both update or neither does. No inconsistent state.
The model: identity + value + time A reference type (atom, ref, agent) is an identity. At any point in time it holds an immutable value. swap! doesn't change the value — it atomically updates the reference to point at a new value. Anyone holding a reference to the old value (from before the swap) still sees the old immutable value. This is concept 02 — Values vs Identity — made concrete.
Python comparison Python's approach to shared mutable state is threading.Lock, queue.Queue, or multiprocessing.Value. These are manual — you are responsible for acquiring and releasing locks correctly. Forget a lock and you have a race condition. Clojure's reference types build the coordination semantics into the type itself: atoms use compare-and-swap internally, refs use Software Transactional Memory. You cannot do it wrong by accident.
How swap! works internally (compare-and-swap loop):
1. Read current value from atom:          old = @counter  (= 5)
2. Apply your function to old value:       new = (inc 5)  (= 6)
3. Compare-and-swap atomically:
   If atom still holds 'old', set it to 'new'. Return new.
   If atom was changed by another thread: RETRY from step 1.

This is why your swap! function must be pure — it may run 2, 5, 10 times
if there is heavy contention. Side effects in the function would fire multiple times.

Namespaces
How Clojure organises, isolates, and connects code

Every Clojure source file starts with (ns my.app.core). A namespace is a named container for vars — it's how Clojure keeps names from colliding and how code in one file refers to code in another. Understanding namespaces is essential because you encounter them in every session, every error message, and every require call.

A namespace is a mapping

A namespace maps symbols to vars. When you write (def x 42), Clojure creates a var named x in the current namespace. The fully qualified name is my.ns/x. Any other namespace that wants to use it must refer to it by that full name, or import it.

Namespaces map to files

The namespace name mirrors the file path: my.app.core lives in src/my/app/core.clj. Dots become directory separators, hyphens become underscores in the filename. This convention is enforced — violating it causes load errors.

; Every file begins with ns — this is a macro form
(ns my.app.core
  (:require [clojure.string :as str]       ; load and alias
            [clojure.set    :as set]        ; load and alias
            [my.app.utils   :refer [helper-fn]]))  ; load and refer specific var

; After this:
(str/upper-case "hello")   ; => "HELLO"   — via alias
(set/union #{1} #{2})      ; => #{1 2}    — via alias
(helper-fn data)           ; => ...        — referred directly, no prefix needed

; Fully qualified names always work without require:
(clojure.string/upper-case "hello")  ; => "HELLO"

; The current namespace
*ns*           ; => #object[clojure.lang.Namespace "my.app.core"]
(ns-name *ns*) ; => my.app.core
:require vs :refer vs :as These three options do different things and are worth knowing precisely. :as str loads the namespace and gives it an alias — you access its vars with the str/ prefix. :refer [fn1 fn2] pulls specific vars into your namespace so you can call them without a prefix. :refer :all pulls everything in (avoid this — it pollutes your namespace and makes it hard to know where names come from). The idiomatic choice is always :as with a short alias.

  (:require [clojure.string :as str]

Keywords and namespaces

Keywords can be namespaced too, using the :: prefix or an explicit namespace. ::name is shorthand for :current.ns/name. Namespaced keywords are the idiomatic way to add context to data, especially when maps from different domains might share key names.

; Plain keyword — no namespace
:name            ; => :name

; Auto-namespaced keyword (expands to current ns)
::name           ; => :my.app.core/name  (in ns my.app.core)

; Explicitly namespaced keyword
:user/name       ; => :user/name
:db/id           ; => :db/id

; Useful for distinguishing domains in the same map
{:user/name "Ada"
 :db/id     42
 :role/type :engineer}

; Access with the full keyword
(:user/name m)   ; => "Ada"
Python comparison Python modules and Clojure namespaces serve similar purposes — organising code and avoiding name collisions. The key difference: Python's import system is flexible (you can import anything, anywhere, in any order). Clojure's ns declaration is strict and front-loaded — the entire dependency structure of a file is declared at the top. This is the Simple vs Easy trade-off again: stricter upfront, easier to reason about as the codebase grows.
