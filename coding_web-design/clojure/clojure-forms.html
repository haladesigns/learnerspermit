The Four Meanings of "Form"
Why the word feels confusing — it means four related things

Documentation, tutorials, and error messages use the word form in four distinct (but overlapping) ways. Separating them mentally is the first step to making Clojure syntax feel less arbitrary. Every piece of syntax on this page fits into one of these four buckets.

1. Form = evaluable expression

Anything the compiler can evaluate to a value. A number is a form. A string is a form. (+ 1 2) is a form. {:a 1} is a form. When the REPL docs say "evaluate the next form", they mean this.

2. Special form = compiler primitive

A specific, short list of forms the compiler handles directly — if, let, fn, def, quote, ., etc. They cannot be redefined or built from other forms. Everything else in the language is built on top of them.

3. Binding form = the left-hand side

The shape used to name things. In (let [x 1] ...), the x is a binding form. In (defn f [a b] ...), the vector [a b] is a binding form. Destructuring patterns are binding forms too.

4. Reader form = textual syntax

How characters on the page map to data structures. 'x, @x, #{...}, #(...), ^:tag — these are read as specific data, before any evaluation happens. The reader runs first; the compiler runs on what the reader produced.

The layering matters. When Clojure processes a file, three phases run in order:

The three phases — read, macroexpand, evaluate:
    Source text              Data structures            Values
     (characters)             (lists, maps, ...)

    "(when x (println x))"    ────►  (when x (println x))   ────►  prints x, returns nil
          │                         │                      │
          │                         │                      │
       READER                  MACROEXPANDER           EVALUATOR
    (reader forms)           (macro & special forms)   (produces values)
    ' @ ^ # :: etc.          when → if + do           run compiled code

This explains why the same symbol can do different things. ^ is a reader macro — it attaches metadata to the next form at read time, and never appears as a function. if is a special form — it's compiled directly. when is a macro form — it expands into if before compilation. All three look like normal syntax, but they fire at different phases.

The mental model that unlocks everything Reader macros (phase 1) produce data. The data is either a special form (compiled directly) or a macro call (expanded, then compiled) or a function call (compiled as a call). There are no other categories. Once you can tell which phase a piece of syntax belongs to, nothing in Clojure looks mysterious.
Python comparison Python has nothing analogous to phase 1. Python's parser produces an AST directly from text, with no user-accessible intermediate. Clojure's reader is user-accessible: you can call (read-string "(+ 1 2)") and get back the list (+ 1 2) as data. That's why you can write macros in Clojure but not in Python.

Lists vs Vectors — why both exist

Python has one general-purpose linear container (list). Clojure has two, and they look almost identical — but they play completely different roles. Mixing them up is a common source of confusion.

List — (...) — for code
; A list evaluates by CALLING the first element:
(+ 1 2)          ; => 3  — the list IS a function call
(println "hi")   ; => nil — same

; To write a list as DATA (not a call), quote it:
'(1 2 3)         ; => (1 2 3) — just data, no call
(list 1 2 3)     ; => (1 2 3) — same, built programmatically

; Without the quote, Clojure tries to call 1:
(1 2 3)          ; ERROR — 1 isn't a function
Vector — [...] — for data
; A vector is just data — no call attempt:
[1 2 3]          ; => [1 2 3]  — self-evaluating
[inc dec +]      ; => [#'inc #'dec #'+] — three fns

; Vectors are used for:
;   - collection literals       [1 2 3]
;   - binding forms             (let [x 1] ...)
;   - function parameters       (defn f [a b] ...)
;   - anything "structural"
The rule Lists are for code. Vectors are for data. When you see [...], expect values. When you see (...), expect a call. This is why defn's parameter list uses [...] — the parameters are data, not a call. It's why let's bindings use [...] — pairs of data.

Clojure inherits Java's numeric types and adds arbitrary-precision and ratios. Every number below is a distinct literal form:

To force floating-point, use (/ 1.0 3) or (double 1/3). Python 3 has fractions.Fraction but it's not the default.

Special Forms
The ~20 primitives the compiler handles directly

Special forms are the compiler's primitives — the forms that cannot be built from other forms and cannot be redefined. If you understand these, you understand 100% of Clojure's evaluation rules, because everything else is built on top of them. when is a macro expanding to if + do. defn is a macro expanding to def + fn. and, or, cond, for, doseq, ->, ->> — all macros. The special forms are the bedrock beneath.

What "the compiler handles directly" actually means

When the compiler sees a list form like (something arg1 arg2), it runs a decision tree to figure out what kind of thing something is. Three categories, three different code paths:

The compiler's decision when it sees a list form:
                       (something arg1 arg2)
                                │
                                ▼
                Is "something" a special form?
                    ├── YES ──► compile it directly
                    │           (hard-coded compiler logic
                    │            for that exact symbol)
                    │
                    └── NO  ──► Is it a macro?
                                ├── YES ──► expand it,
                                │           then re-compile
                                │           the result
                                │
                                └── NO  ──► it's a function call.
                                            evaluate all args,
                                            then invoke.

"Handles directly" means the compiler has hard-coded logic for that specific symbol. When it sees if, there is literally a branch in the compiler's source that says "this is if — emit a conditional jump." It does not look if up as a function or expand it as a macro. The symbol is recognised and dedicated bytecode-emitting code runs.

Why special forms have to exist Functions follow a fixed rule: every argument is evaluated before the function is called. But if can't work that way — it has to skip evaluating the not-taken branch. (if true (println "yes") (println "no")) only prints "yes". A regular function couldn't do that, because both println calls would run before if ever saw them. The same is true for quote (its argument must arrive unevaluated), for fn (its body must not run when the fn form is evaluated), for let (its bindings have to be staged correctly). These behaviours require the compiler to handle them specially. There's no way to write them as functions.

Three concrete consequences flow from this:

Cannot be redefined

You can shadow inc with your own function. You cannot redefine if. The compiler doesn't look it up — it just recognises the symbol. There's no var to override.

Cannot be passed as values

You can write (map inc [1 2 3]). You cannot write (map if ...). if isn't a value; it's a syntactic construct. There's no function object behind it to pass around.

Can break evaluation rules

Special forms decide when (or whether) to evaluate their operands. if evaluates one branch. quote evaluates none. and/or short-circuit — that's why they had to be macros, ultimately resting on if.

Are a closed list

You can write new functions and new macros forever. You cannot write a new special form — that would require modifying the compiler. The list of ~20 is fixed by the language.

Python comparison Python has the same split, just hidden. if, for, while, def, return, lambda, import are keywords baked into the parser — Python's compiler "handles them directly" in exactly this sense. You can't redefine them or pass them as values. The difference is that Python has no macros, so the middle category in the diagram doesn't exist. Clojure exposes the layering: macros bridge the gap, letting you build new control-flow constructs on top of the small set of true primitives.

Why knowing the list helps When you see a form you don't recognise, a useful question is: "is this a special form, a macro, or a function?" Special forms are on the list above. Macros you can ask the REPL: (macroexpand '(when x y)) → (if x (do y)). Functions you can inspect: (source +). The categorisation tells you what rules apply to argument evaluation — functions always evaluate all args; macros and special forms do not.

A note on arity — what fn actually dispatches on

The word arity is just the math term for "how many arguments a function takes" — a binary function has arity 2, a ternary function arity 3, a function with no arguments arity 0. Clojure leans on the term because that's exactly what it dispatches on — nothing more, nothing less.

A single function can define multiple arities, each with its own parameter list and body. When the function is called, Clojure picks the matching arity by counting the arguments. This is the same idea as method overloading in Java/C#, with one key restriction: Clojure dispatches only on count, never on type.

; Multi-arity fn — each clause is wrapped in its own (...)
(defn greet
  ([]              (greet "world"))            ; 0-arg
  ([name]          (greet "Hello" name))       ; 1-arg
  ([greeting name] (str greeting ", " name)))  ; 2-arg

(greet)              ;=> "Hello, world"
(greet "Ada")        ;=> "Hello, Ada"
(greet "Hi" "Ada")   ;=> "Hi, Ada"

; Variadic arity with & — collects extras into a seq
(defn flexible
  ([]          "no args")
  ([x]         "one arg")
  ([x & more]  (str "many: " x " + " (count more) " more")))

(flexible)           ;=> "no args"
(flexible 1)         ;=> "one arg"
(flexible 1 2 3 4)   ;=> "many: 1 + 3 more"

; A common idiom — lower arities call higher ones with defaults:
(defn connect
  ([host]      (connect host 80))
  ([host port] (connect host port {:timeout 5000}))
  ([host port opts] (str "connecting to " host ":" port " with " opts)))
Comparison — Java overloading, Python defaults Java overloads on count and types: process(int) and process(String) are distinct overloads. Clojure cannot do this — only count matters. Python handles "different shapes" via default arguments: def greet(name="world", greeting="Hello"): .... Clojure handles it by writing multiple arities that delegate to each other — the example above. Both express the same intent; the syntax is different.
If you need type-based dispatch defn can't do it — only arity. For dispatch on argument type or value, Clojure has dedicated tools: defmulti + defmethod (multimethods, dispatching on the result of any function you choose) and defprotocol (protocols, dispatching on the type of the first argument). Those are out of scope for this page but worth knowing as the answer when "I need to pick behavior based on what's passed in" comes up.

Binding Forms & Destructuring
The pattern-matching syntax on the left side of any binding

A binding form is the position in any Clojure construct where you write a name to introduce. The simplest binding form is just a symbol — like x in (let [x 5] ...). But Clojure lets you write a shape in that position instead of a single name. The shape is a template that pulls the incoming value apart and routes the pieces into multiple names at once. This is called destructuring.

The mental model — a shaped catcher A binding form is a template of names you set up in advance, ready to catch a value when it arrives. The shape of the template is the shape it expects to receive. [a b c] expects something sequential with three elements. {:keys [name age]} expects a map with :name and :age keys. When the value arrives, each piece falls into its matching slot, and the names become locally available in the body that follows.
A binding form as a catcher — sequential:
     Binding form:    [a       b       c]
                         │       │       │
                         ▼       ▼       ▼
                      (catches)(catches)(catches)
                         ▲       ▲       ▲
                         │       │       │
     Incoming value:   [10      20      30]

     Result: a→10, b→20, c→30
A binding form as a catcher — associative (by key):
     Binding form:   {name :name,  age :age}
                        │              │
                        ▼              ▼
                    (catches the    (catches the
                     :name slot)     :age slot)
                        ▲              ▲
                        │              │
     Incoming value:  {:name "Ada"  :age 36}

     Result: name→"Ada", age→36

Where the value comes from depends on the construct. In (let [shape value] ...) it's the expression on the right. In (defn f [shape] ...) it's whatever the caller passes when they call f. In (for [shape coll] ...) it's each element as the loop walks through. The binding form sits there, set up once, ready to catch every value that comes through.

The colon is the directive marker

Before diving into the patterns, one thing about destructuring syntax that bites everyone at least once: the colon prefix is what turns a keyword into an instruction the destructuring engine watches for. Without the colon, the same word is just a symbol — used as a regular map key like any other. With the colon, it's a directive — a special signal that says "this isn't data; this is a command to the destructuring engine."

The rule Inside a destructuring map, anything starting with : may be a directive (:keys, :strs, :syms, :or, :as). Without the colon, it's a plain symbol — and the destructuring engine treats the whole entry as a regular "lookup this key, bind to this name" pair. {:keys [name age]} and {keys [name age]} look almost identical but mean completely different things.
With colon — directive interpreted
(let [{:keys [name role]
       :or   {role "guest"}}
      {:name "Ada"}]
  (str name " | " role))
;=> "Ada | guest"

; The destructuring engine sees :keys and :or
; as directives. It pulls :name and :role out
; of the map and applies the default for :role.
No colon — treated as a map key
(let [{keys [name role]
       :or  {role "guest"}}
      {:name "Ada"}]
  (str name " | " role))
;=> ERROR or surprising nil

; The engine reads {keys [name role]} as
; "look up the key [name role] (a vector!)
;  and bind the result to local 'keys'."
; name and role are never bound at all.

A useful mental rephrasing of the same idea:

How the destructuring engine reads each entry:
    { :keys  [...] }   →  "directive: pull these keyword keys, bind same-named locals"
    { :strs  [...] }   →  "directive: pull these string keys, bind same-named locals"
    { :syms  [...] }   →  "directive: pull these symbol keys, bind same-named locals"
    { :or    {...} }   →  "directive: defaults for missing keys"
    { :as    name  }   →  "directive: also bind the whole value to this name"

    { keys   [...] }   →  "look up the map key [...] and bind result to keys"
    { name   :n    }   →  "look up :n in the map and bind result to name"

The five colon-prefixed names above are the entire destructuring vocabulary. Everything else in a destructuring map is interpreted as ordinary "bind this name to the value at this key" — that's how the workhorse pattern {name :name} works.

The sneaky failure mode The trap isn't usually a missing colon — that errors loudly. The trap is the opposite: a map whose keys aren't the type the directive assumes. {:keys [name]} looks for keyword keys; if the actual map has string or symbol keys, the lookup quietly returns nil and the local binds to nothing. No error — just unexpected nils. If a destructured local is mysteriously nil, check whether the map's keys are actually keywords (use :strs for strings, :syms for symbols).

Binding forms appear in many constructs — and the same destructuring rules apply in all of them. Set up the shape once; learn the patterns once.

Soft contract — missing pieces become nil, not errors The shape describes what works cleanly, not what's enforced. If the actual value doesn't match the shape — too few elements, missing keys — Clojure does its best and fills the unmatched names with nil. (let [[a b c] [10 20]] [a b c]) yields [10 20 nil]. No exception. This is sometimes a feature (graceful handling of optional fields) and sometimes a footgun (silent nils propagating where you expected a value). It's a Clojure-ism worth knowing.

Sequential destructuring — with […]

A vector on the left-hand side pulls apart a sequential collection (vector, list, seq, string) by position. It mirrors Python's tuple unpacking but with more features.

Python — tuple unpacking
point = (3, 4)
x, y = point                # 3, 4

# Nested
((a, b), c) = ((1, 2), 3)   # 1, 2, 3

# Rest with star
first, *rest = [1, 2, 3, 4] # 1, [2,3,4]

# Ignore with _
_, y = point                # only y

# No "keep the original" built in
Clojure — sequential destructuring
(let [[x y] [3 4]]          ; 3, 4
  ...)

; Nested
(let [[[a b] c] [[1 2] 3]]  ; 1, 2, 3
  ...)

; Rest with &
(let [[first & rest] [1 2 3 4]]  ; 1, (2 3 4)
  ...)

; Keep the whole thing with :as
(let [[x y :as pt] [3 4]]   ; x=3 y=4 pt=[3 4]
  ...)
; All four patterns together:
(let [[first second & rest :as all] [10 20 30 40 50]]
  {:first first          ; => 10
   :second second        ; => 20
   :rest rest            ; => (30 40 50)
   :all all})            ; => [10 20 30 40 50]

; Works on any seqable — strings, lists, vectors, lazy seqs
(let [[a b c] "xyz"]
  (str a b c))           ; => "xyz"   — characters destructured

; Ignore a position with _
(let [[_ second] [10 20]]
  second)                ; => 20

; Partial shapes — describe only what you want; the rest is dropped
(let [[first] [10 20 30 40]]
  first)                 ; => 10   — extras are simply ignored

; Shape mismatch — unmatched names are nil, no error thrown
(let [[a b c d] [10 20]]
  [a b c d])             ; => [10 20 nil nil]   (soft contract)

Associative destructuring — with {…}

A map on the left-hand side pulls values out of a map by key. This has no direct Python equivalent — Python requires either explicit assignment or dict unpacking with ** which is different in purpose. Clojure's map destructuring is extraordinarily common in real code.

; Full form — name → key
(let [{name :name, age :age} {:name "Ada" :age 36}]
  (str name " is " age))          ; => "Ada is 36"

; :keys shortcut — when local name = keyword name
(let [{:keys [name age]} {:name "Ada" :age 36}]
  (str name " is " age))          ; => "Ada is 36"

; :strs for string keys, :syms for symbol keys
(let [{:strs [name age]} {"name" "Ada" "age" 36}]   ; string keys
  ...)

; Defaults with :or
(let [{:keys [name role] :or {role :guest}}
      {:name "Ada"}]
  (str name " / " role))          ; => "Ada / :guest"

; :as to keep the original map
(let [{:keys [name] :as person} {:name "Ada" :age 36}]
  [name person])                  ; => ["Ada" {:name "Ada" :age 36}]

; Nested — maps inside maps
(let [{{:keys [city]} :address} {:address {:city "Oslo"}}]
  city)                           ; => "Oslo"

Nested destructuring — patterns inside patterns

When data is nested — a map inside a map, a vector inside a vector, a map inside a vector — the destructuring shape can mirror the data's structure exactly. The rule is one principle, applied recursively:

The one rule for nesting Anywhere you'd write a name on the left of a binding, you can write another destructuring shape instead. Sequential shapes nest. Associative shapes nest. They can mix. The pattern just keeps mirroring the data's shape, however deep it goes.

The infamous-looking case is nested map destructuring, because of the double curly brace. Let's pull it apart carefully — once you see it, the magic disappears.

; The data:
{:address {:city "D.C."}}    ; outer map → :address → inner map → :city → "D.C."

; Goal: get "D.C." in one destructuring step.

; Without nesting (two separate lets):
(let [{address :address} {:address {:city "D.C."}}]
  (let [{:keys [city]} address]
    city))                        ; => "D.C."

; With nesting (collapsed into one):
(let [{{:keys [city]} :address} {:address {:city "D.C."}}]
  city)                            ; => "D.C."

; The two are EXACTLY equivalent. Nesting just skips
; the intermediate name.

Reading the double-curly form

In a destructuring map, the layout is the opposite of a regular map: the binding form is on the left, the key to look up is on the right. When the binding form on the left is itself a shape, you get the double curly.

Anatomy of {{:keys [city]} :address}:
           { {:keys [city]}        :address }
             └──────┬──────┘        └────┬────┘
                    │                    │
              binding form          key to look up
              (what to do          (in the outer map)
               with what we find)

                                         │
                                         ▼
                          { :address {:city "D.C."} }     ← actual map
                                       └──────┬─────┘
                                              │
                                              ▼
                                     {:keys [city]}         ← applied to this
                                              │
                                              ▼
                                        city = "D.C."

The right-hand side :address says "look up this key." The left-hand side, instead of being a name like x, is a destructuring shape that gets applied to whatever was found.

Sequential nesting works the same way

; A vector with two pairs inside:
(let [[[a b] [c d]] [[1 2] [3 4]]]
  [a b c d])                      ; => [1 2 3 4]

; A vector containing a map:
(let [[{:keys [name]} score] [{:name "Ada"} 95]]
  (str name ": " score))          ; => "Ada: 95"

; A map containing a vector:
(let [{[x y] :point} {:point [10 20]}]
  [x y])                          ; => [10 20]

; Mix and match to any depth:
(let [{:keys [users]
       :as   data}
      {:users [{:name "Ada" :role :admin}
               {:name "Alan" :role :user}]}]
  [(count users) (-> users first :name)])
;=> [2 "Ada"]

A useful tactic when nesting confuses you

If a deeply nested destructuring is hard to follow, mentally (or actually) uncollapse it into multiple let bindings. They're equivalent, and the uncollapsed form names every intermediate value:

; Confusing nested form:
(let [{[{:keys [name]}] :users} data]
  name)

; Uncollapsed — every step has a name:
(let [users (:users data)             ; pull the :users vector
      first-user (first users)        ; grab the first one
      name (:name first-user)]        ; pull :name from it
  name)

; Both produce the same result. Use the uncollapsed form
; when reading; collapse when writing once it's clear.
When to nest, when to flatten Nested destructuring is most readable at two levels deep. Beyond that, prefer naming intermediate values with separate let bindings or threading (some->, get-in) — readability falls off a cliff once a destructuring pattern crosses three levels. The point of destructuring is clarity at the binding site; if it's no longer clear, it's no longer earning its place.

Destructuring in function parameters

The same patterns work directly in function parameter lists — no separate let needed. This is one of the most idiomatic ways Clojure code stays compact.

Python — manual unpacking in body
def distance(p1, p2):
    x1, y1 = p1
    x2, y2 = p2
    return ((x2-x1)**2 + (y2-y1)**2) ** 0.5

def greet(user):

@ deref — read the value out of a reference

A natural question on first encounter: why is this needed at all? If counter holds a value, why not just write counter and get the value? The answer reveals something fundamental about how Clojure models state. Worth pausing on, because once it clicks, atoms (and their cousins) stop feeling arbitrary.

The core idea — container vs contents An atom is not a value; it's a box that holds a value. The box has a stable identity that lasts forever. The value inside it can change over time. counter refers to the box; @counter says "open the box right now and give me whatever's inside." The two are deliberately kept separate.

This separation is core to Clojure's philosophy. Values are immutable — 0 is 0 forever, you can't "change" it into 1. But programs need state. Clojure's answer: don't model state by mutating values; model state by having a stable identity point at different immutable values over time. The atom is that identity. @ is how you ask it for its current state.

Container vs contents over time:
     counter       (the atom — a stable identity, lives forever)
        │
        │  at time t1:   ──►   0       ← @counter returns 0
        │
        │  (swap! counter inc)
        │
        │  at time t2:   ──►   1       ← @counter returns 1
        │
        │  (swap! counter inc)
        │
        │  at time t3:   ──►   2       ← @counter returns 2

     The atom never "becomes" a different atom.
     The values it points at are different immutable facts.
     @counter is a snapshot read at one moment.

Three reasons the unwrap is explicit

If counter auto-unwrapped to its current value, you'd lose three things — and each matters:

1. Passing the atom around

Functions that need to update state need the atom itself, not a snapshot. (swap! a inc) can't work if a auto-dereferenced — it'd see only the value, with no way to update what the atom points at.

2. Snapshot timing

Each @ is a point-in-time read. In multi-threaded code, the value can change between reads. Making each read explicit lets you see exactly when state is observed — vital for reasoning about concurrency.

3. Identity is preserved

Two atoms with the same value are still different atoms. Updating one doesn't update the other. The atom's identity is its own thing — meaningful and stable. Auto-unwrapping would erase that distinction.

The mental shortcut

counter asks "what container is this?"  @counter asks "what's in the container right now?" The same character that decorates Python functions is, here, the question mark you're putting to a reference type.

This same @ works on every reference type Clojure offers — atom, ref, agent, future, promise, delay. They all share the "stable identity holding a current value" pattern. Knowing the principle once carries you through all of them.

Rich Hickey's triad — identity, state, value This is the conceptual frame from Simple Made Easy: value = an immutable fact (0, {:a 1}); identity = a stable name for "the same thing across time" (the atom); state = the value an identity holds at a given moment (the result of @atom). OO languages fuse all three into "the object." Clojure pulls them apart on purpose, and @ is the syntax that says "I want to ask this identity what its state is right now."

In code

; @ extracts the current value from a reference type
(def counter (atom 0))
@counter          ; => 0              — same as (deref counter)
(swap! counter inc)
@counter          ; => 1

; @ also works on futures, promises, delays, agents
@(future (+ 1 2)) ; => 3              — blocks until ready

; The atom itself vs. its current value:
counter           ; => the atom itself (e.g. clojure.lang.Atom)
@counter          ; => 1              — what it holds right now

; Identity is preserved across time and copies:
(def x (atom 0))
(def y x)         ; y is the SAME atom as x (not a copy)
(swap! x inc)
@y                ; => 1              — y sees the change because y IS x
Python comparison — the @ specifically Python uses @ for decorators — a syntactic feature for functions. In Clojure, @ is deref — it reads the current value out of a reference type (atom, ref, future). Same character, completely different meaning. If you see @x in Clojure, think "get the value of the reference x right now".

Clojure code without threading can look like deeply-nested parentheses that you read inside-out. Threading macros — the arrow macros — rewrite nested calls into a top-to-bottom pipeline. They are all macros, not special forms, and they are among the most-used syntax in real code. You have probably already seen -> and ->> in Episode 04.

-> and ->> — the two you'll use every day

Nested — reads inside-out
; Without threading — hard to read:
(reduce +
  (map #(* % %)
    (filter even?
      (range 10))))
;=> 120

; With ->> — reads top-down:
Threaded — reads top-down
(->> (range 10)        ; the seed
     (filter even?)    ; then filter
     (map #(* % %))    ; then square
     (reduce +))       ; then sum
;=> 120

; Mechanically: take (range 10) and STUFF IT
; into the LAST position of each form below.
; ── -> threads into FIRST position ─────────────────────────────
; Most map/object APIs take the thing as the first arg:
(-> {:name "Ada"}
    (assoc :age 36)         ; (assoc _ :age 36)
    (update :age inc)       ; (update _ :age inc)
    (dissoc :name))         ; (dissoc _ :name)
;=> {:age 37}

; ── ->> threads into LAST position ─────────────────────────────
; Most collection APIs take the coll as the last arg:
(->> [1 2 3 4 5]
     (filter odd?)          ; (filter odd? _)
     (map #(* % 10))        ; (map #(* % 10) _)
     (reduce +))            ; (reduce + _)
;=> 90

; Rule of thumb:
;   ->  for maps and single values      (data flows through 1st arg)
;   ->> for collections and pipelines   (data flows through last arg)

as-> — when position varies

Some pipelines need the threaded value in different positions at different steps. as-> lets you name the "current" value and put it wherever you want.

; as-> names the current value, you decide where it goes
(as-> [1 2 3 4] $
  (map inc $)          ; $ in last position — (map inc _)
  (conj $ 99)          ; $ in first position — (conj _ 99)
  (reverse $))         ; $ in last — (reverse _)
;=> (99 5 4 3 2)

some-> and some->> — short-circuit on nil

; Nested lookup that might fail:
(-> {:address {:city "Oslo"}}
    :address :city :length)         ; :length of "Oslo" ... works
; But:
(-> {:no-address true}
    :address :city :length)         ; NullPointerException!

; some-> stops as soon as nil appears:
(some-> {:no-address true}
        :address :city :length)     ; => nil   — safe

(some-> {:address {:city "Oslo"}}
        :address :city count)       ; => 4

cond-> and cond->> — conditional steps

; Apply some steps only if a condition holds.
; Each pair: (predicate form) — form is threaded if predicate is truthy.

(defn build-query [{:keys [name active? limit]}]
  (cond-> {}                        ; start with empty map
    name       (assoc :name name)   ; only if name given
    active?    (assoc :active true) ; only if active?
    limit      (assoc :limit limit))) ; only if limit given

(build-query {:name "Ada"})                  ; => {:name "Ada"}
(build-query {:name "Ada" :active? true})    ; => {:name "Ada" :active true}
(build-query {:name "Ada" :limit 10})        ; => {:name "Ada" :limit 10}

; Replaces a lot of:   (if x (assoc m :k v) m)   chains

doto — for mutable Java objects

; doto calls multiple methods ON the same object, returning the object.
; It's for Java interop where you need to configure a mutable thing:

(doto (java.util.HashMap.)
  (.put "a" 1)
  (.put "b" 2)
  (.put "c" 3))
;=> {"a" 1, "b" 2, "c" 3}

; Expansion:
; (let [obj (java.util.HashMap.)]
;   (.put obj "a" 1)
;   (.put obj "b" 2)
;   (.put obj "c" 3)
;   obj)
Why threading is a separate design choice In most languages, chaining happens through method syntax — list.filter(p).map(f).reduce(g). That only works when everything is a method on the object. Clojure functions are free-floating, so it invents syntax (threading macros) to give you the same readability without coupling operations to types. (->> coll (filter p) (map f) (reduce g)) does the same thing, but it works on any collection, string, or seqable.

(if (> x 0) "positive" "not positive")

; No "else" needed; returns nil if no else given:
(if (> x 0) "positive")    ; if x=0, returns nil

; ── when ─── one branch with side effects ─────────────────────
; Use when the "else" is nil and body has multiple expressions:
(when (> x 0)
  (println "positive!")    ; multiple side effects
  (println "confirmed")
  x)                        ; returns x

; Equivalent to:  (if (> x 0) (do ... ... x) nil)

; ── cond ─── multi-branch — like Python elif ─────────────────
(defn grade [score]
  (cond
    (>= score 90) "A"
    (>= score 80) "B"
    (>= score 70) "C"
    (>= score 60) "D"
    :else         "F"))        ; :else is just a truthy keyword

; ── case ─── O(1) dispatch on constants ──────────────────────
; Like Python match but for literals only — no guards:
(defn day-type [day]
  (case day
    (:sat :sun)              "weekend"        ; list groups keys
    (:mon :tue :wed :thu :fri) "weekday"
    "unknown"))                                ; default

; case uses JVM tableswitch — fast, but keys MUST be compile-time
; constants (keywords, strings, numbers). Will NOT work with
; variables or expressions as keys.
Choosing between cond and case Use case when matching against fixed values (keywords, numbers, strings) — it compiles to a fast jump table. Use cond when branches depend on computed predicates or ranges. case requires literal keys; cond takes arbitrary expressions.

if-let, when-let, if-some, when-some

These are two-in-one forms: they bind a name and branch on the bound value in a single step. They are idiomatic when you want to both test-for-presence and use the value.

; ── Without if-let ─── verbose ──────────────────────────────
(let [user (find-user id)]
  (if user
    (greet user)
    (println "not found")))

; ── With if-let ─── combined ────────────────────────────────
(if-let [user (find-user id)]
  (greet user)                         ; user is bound here
  (println "not found"))

; ── when-let ─── no else branch ─────────────────────────────
(when-let [user (find-user id)]
  (greet user)
  (update-last-seen user))

; ── if-let vs if-some ── ─────────────────────────────────────
; if-let tests truthiness — false and nil are both "missing"
(if-let [x false] "yes" "no")     ; => "no"    (false is falsy)
(if-let [x nil]   "yes" "no")     ; => "no"

; if-some tests (not= nil) — only nil is "missing"
(if-some [x false] "yes" "no")    ; => "yes"   (false is a value!)
(if-some [x nil]   "yes" "no")    ; => "no"

; Use if-some when false is a meaningful value in your domain
; (e.g., a boolean flag that could legitimately be false)
Truthiness in Clojure — important Only false and nil are falsy. Everything else is truthy — including 0, empty strings, empty collections, and empty maps. This is different from Python where 0, "", [], and {} are all falsy. If you want Python-style "empty is falsy", use (seq coll) which returns nil for empty and the seq otherwise.

Metadata — the ^ symbol
Attaching information to a form without affecting its value

The caret (^) is one of the stranger things in Clojure — it looks like an operator but behaves like an annotation. It attaches a metadata map to the next form at read time. Metadata is separate from the value: two values that are = can have different metadata, and vice versa. Metadata never participates in equality, hashing, or printing (except in pr with *print-meta* on).

The three uses you will actually see (1) Type hints — ^String tells the compiler "this is a String" to avoid reflection. (2) Boolean flags on vars — ^:private, ^:dynamic, ^:const — set a flag on the next thing. (3) Arbitrary metadata — ^{:doc "..." :author "Ada"} — any map you want. Everything else is variations on these.

The three shorthand forms

; ── ^Symbol ─── shorthand for {:tag Symbol} ───────────────
; Used for type hints to avoid Java reflection:
(defn upper ^String [^String s]       ; both params and returns
  (.toUpperCase s))
; Expands to: {:tag String}

; ── ^:keyword ─── shorthand for {:keyword true} ────────────
(def ^:private helper 42)              ; {:private true}
(def ^:dynamic *config* {})             ; {:dynamic true}
(def ^:const PI 3.14159)                ; {:const true}

; ── ^{...} ─── full metadata map ──────────────────────────
(def ^{:doc "The answer" :added "1.0"} answer 42)

; All three can be combined by stacking:
(def ^:private ^String ^{:doc "greeting"} hello "hi")
; final meta: {:private true :tag String :doc "greeting"}

Reading metadata back

; The (meta x) function reads the metadata map
(def ^:private ^{:doc "counts things"} counter (atom 0))

(meta #'counter)
;=> {:private true
;    :doc "counts things"
;    :line 1
;    :column 1
;    :file "..."
;    :name counter
;    :ns #object[...]}

; Vars get line/column/file metadata automatically from the reader.

; Metadata on data structures:
(def v ^:immutable [1 2 3])
(meta v)                      ; => {:immutable true}
v                             ; => [1 2 3]   — metadata invisible
(= v [1 2 3])                 ; => true      — doesn't affect equality

Why type hints exist

Clojure is dynamically typed. When you call (.toUpperCase s), the compiler doesn't know at compile time what class s is, so it uses Java reflection at runtime — slow. A type hint tells the compiler "trust me, this is a String" so it can emit a direct method call instead. You will see type hints in performance-sensitive code and when the compiler warns you with *warn-on-reflection* on.

; Without hint — reflection warning (if enabled):
(defn slow [s]
  (.toUpperCase s))
; Reflection warning, call to toUpperCase can't be resolved.

; With hint — direct method call, much faster:
(defn fast [^String s]
  (.toUpperCase s))

; Enable warnings with:
(set! *warn-on-reflection* true)
Python comparison Python's closest analogue is type annotations: def f(s: str) -> str:. But Python annotations are checked by external tools (mypy), not by the runtime. Clojure type hints go to the compiler, which uses them to generate faster bytecode. Different purpose: Python hints are for correctness; Clojure hints are for performance (and the occasional ^:private for visibility).

The Quote Family — ' ` ~ ~@
Suppressing, templating, and punching holes in evaluation

The quote family is the single hardest thing about Lisp syntax for newcomers — but once it clicks, it's the gateway to understanding macros. Four characters, four related jobs, all about controlling when a form is evaluated.

' (quote) — "don't evaluate this"

; Without quote — Clojure evaluates (+ 1 2) and returns 3:
(+ 1 2)                 ; => 3

; With quote — Clojure returns the LIST, unevaluated:
'(+ 1 2)                ; => (+ 1 2)
(quote (+ 1 2))         ; => (+ 1 2)  — same, longhand

; Quote on a symbol — return the symbol, don't look it up:
'foo                    ; => foo   (the symbol)
foo                     ; ERROR — Unable to resolve symbol

; Quote is useful for:
;   (1) data that LOOKS like code but is just data
'(apple banana cherry)
;   (2) passing symbols to functions that take them
(resolve 'clojure.core/map)

` (syntax-quote) — quote with namespace resolution

Syntax-quote (the backtick) is like quote, with two extra features that make it indispensable for macros: it namespace-qualifies symbols, and it allows holes (~ and ~@) that evaluate inside the quoted form.

; ── Difference 1: symbols get namespace-qualified ─────────
'inc                    ; => inc                 — bare symbol
`inc                    ; => clojure.core/inc    — fully qualified!

; Why this matters for macros:
; When a macro generates code that uses inc, syntax-quote
; ensures the inc refers to clojure.core/inc even if the
; user's namespace has its own inc. This prevents name capture.

; ── Difference 2: ~ (unquote) evaluates in place ───────────
(let [x 10]
  '(+ 1 x))             ; => (+ 1 x)       — literal x
(let [x 10]
  `(+ 1 ~x))            ; => (+ 1 10)      — x was unquoted

; ── ~@ (unquote-splicing) evaluates AND splices ────────────
(let [xs [1 2 3]]
  `(+ ~xs))             ; => (+ [1 2 3])   — as one value
(let [xs [1 2 3]]
  `(+ ~@xs))            ; => (+ 1 2 3)     — spliced out!

Putting it together — a simple macro

; A macro that generates an if-not from an unless call:

(defmacro unless [condition & body]
  `(if (not ~condition)         ; ` freeze the template
     (do ~@body)                ; ~@ splice the body forms
     nil))

; Usage:
(unless (zero? 5)
  (println "not zero")
  42)

; Macroexpansion:
(macroexpand-1 '(unless (zero? 5) (println "not zero") 42))
;=> (if (clojure.core/not (zero? 5))
;     (do (println "not zero") 42)
;     nil)

; Notice:
;   - `  freezes the template structure
;   - ~condition  punches a hole for the condition
;   - ~@body      splices all body forms into the do
;   - not becomes clojure.core/not (namespace-resolved)
The mental model Think of ` as "template this, but allow holes". Think of ~ as "evaluate this one thing". Think of ~@ as "evaluate this collection and splice its contents". You won't write macros every day, but you will read them — every Clojure library uses them — and the quote family is the syntax you'll see.
Don't confuse ' with ` Plain quote ' is for data ('(1 2 3)). Syntax-quote ` is for macro templates. They look almost identical in some fonts — if code is inside a macro definition, it's likely `.

  (.put "b" 2)
  (.put "c" 3))
;=> {"a" 1, "b" 2, "c" 3}

; Reads as: "do TO this object: call these methods"
Why the leading dot is a signal Any time you see a symbol starting with ., you are looking at Java interop. Any time you see a symbol ending with ., you are looking at a constructor. Any time you see Class/something, you are looking at a static access. These three visual cues tell you "this code is talking to Java" without reading it in detail.
Python comparison Python has one dot — obj.attr and Class(...) and Class.method all look similar. Clojure has to distinguish instance methods, static methods, fields, and constructors because the language is different from the host (JVM), so syntax had to be invented to express Java concepts that Clojure itself doesn't have natively.

Regex
The #"…" literal, the four matching verbs, and the Java pattern flavor

Regular expressions in Clojure are a thin layer over Java. The Clojure-specific surface is small: one reader literal (#"…") and a handful of functions in clojure.core and clojure.string. The pattern language itself — \d, [abc], (?:...), lookahead, backreferences, and so on — comes straight from java.util.regex.Pattern. This section covers the Clojure-specific parts in full and lists the most common metacharacters as a courtesy reference.

The #"…" literal — pattern as a first-class value

A regex in Clojure is written with the #"…" reader literal. The reader compiles the pattern to a java.util.regex.Pattern object at read time — once, when the file is loaded — not every time the form is evaluated. The pattern is then a regular value you can pass around, store in vars, and use with the matching functions.

; The literal compiles ONCE, at read time:
#"\d+"            ; => #"\d+"   — a Pattern object
(class #"\d+")    ; => java.util.regex.Pattern

; You can store it like any other value:
(def email-rx #"[\w.+-]+@[\w-]+\.[\w.-]+")

; And build one programmatically with re-pattern:
(re-pattern "\\d+")   ; => #"\d+"   — note: string needs \\
The big win — no double-escaping Inside #"…", backslashes go straight to the regex engine. #"\s+" matches whitespace. Compare to a string-built pattern where you'd need "\\s+" (escape the backslash for the string parser, so the regex engine sees \s+). This is the same benefit Python's r"\s+" raw strings provide — but in Clojure it's the default for regex literals, no special prefix needed.

The four matching verbs

Where Python folds matching into one module with several method names (re.match, re.search, re.findall, re.finditer), Clojure exposes four free-standing functions. Each does one job. Knowing which to reach for is half of regex in Clojure.

FunctionReturnsUse when
re-findFirst match (string or vector)"Is there a match anywhere?" — most common
re-matchesMatch if pattern matches the WHOLE string, else nilValidating an entire string against a pattern
re-seqLazy seq of all matchesPulling out every match in a string
re-patternCompiles a string into a PatternWhen the pattern is built dynamically
; ── re-find ── first match anywhere in the string ───────────
(re-find #"\d+" "abc 42 def 99")    ; => "42"
(re-find #"xyz" "abc 42 def 99")    ; => nil   — no match

; ── re-matches ── whole string must match ──────────────────
(re-matches #"\d+"     "12345")     ; => "12345"   — match
(re-matches #"\d+"     "12abc")     ; => nil       — full match required
(re-matches #"\d+\w+"  "12abc")     ; => "12abc"   — now whole string matches

; ── re-seq ── lazy seq of all matches ──────────────────────
(re-seq #"\d+" "abc 42 def 99")     ; => ("42" "99")
(re-seq #"\w+" "hello world there") ; => ("hello" "world" "there")

; ── re-pattern ── build a pattern from a string ────────────
(def n 3)
(re-pattern (str "\\d{" n "}"))      ; => #"\d{3}"
(re-find (re-pattern (str "\\d{" n "}")) "abc 1234")  ; => "123"

The "what gets returned" gotcha

This trips up almost everyone the first time. The shape of what re-find and re-matches return depends on whether your pattern has capturing groups:

The rule Pattern with no capture groups → returns a string (or nil). Pattern with capture groups → returns a vector: whole-match first, then each group. 
; No groups — returns a string:
(re-find #"\d+" "age 42")              ; => "42"

; With one group — returns a vector [whole-match, group1]:
(re-find #"age (\d+)" "age 42")        ; => ["age 42" "42"]

; Multiple groups — vector with all of them:
(re-find #"(\w+)@(\w+)" "ada@example") ; => ["ada@example" "ada" "example"]

; re-matches follows the same rule:
(re-matches #"(\d+)-(\d+)" "10-20")    ; => ["10-20" "10" "20"]

; Destructure to grab the parts cleanly:
(let [[_ user host] (re-find #"(\w+)@(\w+)" "ada@example")]
  {:user user :host host})              ; => {:user "ada" :host "example"}

; re-seq when groups are present yields a seq of vectors:
(re-seq #"(\w+)=(\d+)" "x=1 y=2 z=3")
;=> (["x=1" "x" "1"] ["y=2" "y" "2"] ["z=3" "z" "3"])

Replacing and splitting — clojure.string

Replacement and splitting live in clojure.string, not clojure.core. Both functions accept a regex anywhere they accept a string, so you don't need a separate re-sub like Python's re.sub.

(require '[clojure.string :as str])

; ── str/replace — string OR regex OR character ─────────────
(str/replace "hello world" "world" "Clojure")   ; string match
;=> "hello Clojure"

(str/replace "abc 42 def 99" #"\d+" "N")        ; regex match
;=> "abc N def N"

; Replacement can be a function — receives each match:
(str/replace "abc 42 def 99" #"\d+"
             #(str "[" % "]"))
;=> "abc [42] def [99]"

; With groups, the function receives the whole vector:
(str/replace "ada@example.com" #"(\w+)@(\w+)"
             (fn [[_ user host]] (str host "/" user)))
;=> "example/ada.com"

; Backreferences with $1, $2 (string replacement form):
(str/replace "ada example" #"(\w+) (\w+)" "$2 $1")
;=> "example ada"

; ── str/split — split a string by a regex ─────────────────
(str/split "a,b,,c" #",")           ; => ["a" "b" "" "c"]
(str/split "  one   two  three"
           #"\s+")                   ; => ["" "one" "two" "three"]
(str/split "1-2-3-4-5" #"-" 3)       ; => ["1" "2" "3-4-5"]   ; limit

Named groups — the Java syntax

; Named groups use Java's syntax: (?<name>...)
; The match result is still a vector — names don't appear there.
; To access by name, use re-matcher + .group:

(let [m (re-matcher #"(?<user>\w+)@(?<host>[\w.]+)" "ada@example.com")]
  (when (.find m)
    {:user (.group m "user")
     :host (.group m "host")}))
;=> {:user "ada" :host "example.com"}

; In practice, destructuring the vector is far more common in Clojure code
; than named groups — they show up mostly in code shared with Java tooling.

Special-Position Keywords
Why :as, :keys, :or, :when, :else seem to "do something"

A keyword like :as looks identical to any other keyword you'd use as a map key. There's no syntactic distinction. Yet in certain places — inside a destructuring map, inside a for loop, inside an ns form — these keywords behave like structural markers that the surrounding macro recognises and acts on. Understanding why is simple once you see it.

The mental model These keywords have no special meaning by themselves — :as, :keys, :or, :when, :else are all just plain keywords. (keyword? :as) is true. You can put any of them in a map as a key. What makes them feel special is that specific macros and forms scan their arguments looking for these particular keywords in particular positions, and treat them as instructions when found. The "specialness" lives in the consumer (the macro), not in the keyword itself.

A useful way to think about it: these keywords are like named flags or section dividers that the macro reads as syntax. Outside the contexts where the macro looks for them, they're just data. This is also why you can introduce your own keywords with similar conventions — there's nothing privileged about :as; library macros routinely invent their own positional keywords.

Group 1 — Destructuring keywords

These appear inside binding forms (§4) and are recognised by let, fn, defn, loop, for, and any macro built on them.

KeywordWhereEffect
:asInside [...] or {...} destructureBind a name to the whole original value
:keysInside a map destructurePull keyword keys into same-named locals
:strsInside a map destructureSame, for string keys
:symsInside a map destructureSame, for symbol keys
:orInside a map destructureProvide defaults for missing keys
:ns/keysInside a map destructure (1.9+)Pull namespaced keys; e.g. :user/keys matches :user/name
; All five common destructuring keywords in one binding form:
(let [{:keys [name age]      ; pull :name and :age
       :strs [email]         ; pull "email" (string key)
       :or   {age 0}          ; default for missing :age
       :as   person}          ; capture the whole map too
      {:name "Ada" "email" "a@x.com"}]
  [name age email person])
;=> ["Ada" 0 "a@x.com" {:name "Ada" "email" "a@x.com"}]

; Namespaced keys variant — Clojure 1.9+:
(let [{:user/keys [name id]} {:user/name "Ada" :user/id 42}]
  [name id])
;=> ["Ada" 42]

Group 2 — for and doseq modifiers

Inside list comprehensions, three keywords act as in-line directives that modify the iteration. They appear between binding pairs and are recognised positionally by the for and doseq macros.

KeywordEffectRoughly equivalent Python
:let [bindings]Introduce locals for the rest of the iterationAssignment inside a comprehension's for clause
:when testSkip iterations where test is falsy (filter)if filter in a comprehension
:while testStop the inner loop the moment test is falsyNo direct equivalent — early break
; All three modifiers, working together:
(for [x (range 10)
      :let  [sq (* x x)]              ; bind sq for each x
      :when (even? x)                  ; only even x
      :while (< sq 50)]                ; stop when sq >= 50
  [x sq])
;=> ([0 0] [2 4] [4 16] [6 36])

; :when filters but keeps going; :while halts:
(for [x (range 10) :when (odd? x)]   x)  ;=> (1 3 5 7 9)
(for [x (range 10) :while (< x 5)]   x)  ;=> (0 1 2 3 4)
(for [x (range 10) :when  (< x 5)]   x)  ;=> (0 1 2 3 4)  ; same here
(for [x [1 3 5 4 7 9] :while (odd? x)] x);=> (1 3 5)       ; stops at 4
Python comparison Python comprehensions allow filters: [x for x in range(10) if x % 2 == 0]. That's roughly Clojure's :when. Python has no inline :let (you'd use a generator expression or a separate variable) and no :while for early termination from inside a comprehension (you'd break in a regular loop instead). Clojure's for packs all three into one form.

Group 3 — cond, case, condp sentinels

In cond and friends, two keywords appear as conventions, not as macro-recognised syntax. They work because they happen to be truthy — but they're written this way to make the "default" case visually obvious.

KeywordWhereWhy it works
:elseLast branch of condJust a truthy keyword — cond doesn't care which truthy value, but :else reads as "default branch"
:defaultDefault branch of condpSame idea — convention, not syntax
; :else is just a truthy keyword — cond evaluates branches top-to-bottom
; and takes the first whose test is truthy. The keyword :else is truthy, so
; it always matches if reached. ANY truthy value would work — :default,
; true, "anything" — but :else is the convention.

(defn grade [score]
  (cond
    (>= score 90) "A"
    (>= score 80) "B"
    (>= score 70) "C"
    :else         "F"))      ; convention; could also be (true "F")

; condp uses :default as the keyword for the default branch:
(condp = 5
  1  "one"
  5  "five"
  :default "other")          ;=> "five"
Don't try this with case case is different — its default is just a single trailing expression with no key. Writing :else as a key in case would make :else an actual key to match, not a default. Use the trailing-expression form instead: (case x 1 "one" 2 "two" "default").

The ns form is built almost entirely out of keyword markers. Each keyword introduces a different kind of declaration. These are recognised by the ns macro itself, and the same keywords appear in require libspecs.

Group 5 — Reader conditional platform tags

Inside #?(...) and #?@(...) reader conditionals (§5), keywords act as platform selectors. Each branch is keyed by which Clojure dialect is reading the file. Only the matching branch is read; the others vanish before the compiler ever sees them.

KeywordPlatform

  #?(:clj  (java.util.Date.)
     :cljs (js/Date.)

; Splicing variant: contents are spliced into the surrounding form:
(defn parse-int [s]
  #?@(:clj  [(Integer/parseInt s)]
      :cljs [(js/parseInt s 10)]))

The big-picture takeaway

Why this design works Clojure's macros routinely extend syntax by inventing new positional keywords. clojure.spec uses :req, :opt, :req-un. Test libraries use :fixtures, :setup. Web frameworks use :get, :post, :middleware. The pattern is the same: keywords are cheap (no syntax to learn — they're just data), self-documenting (you read :as and you know what it does), and any macro can recognise them. So when you see a new library, expect to encounter its own positional keywords. The mental tool is identical: the keyword is data; the macro decides what it means in this position.
Quick lookup — "I see this keyword, where does it apply?" :as & :keys & :strs & :syms & :or → destructuring (§4). :let & :when & :while → for/doseq modifiers. :else → cond default convention. :default → condp default; also reader conditional fallback. :require & :import & :refer & :as-alias & :exclude → namespace declarations. :clj & :cljs & :cljr & :bb → reader conditional platforms. :private & :dynamic & :const & :tag & :doc → compiler-recognised metadata (§8).
