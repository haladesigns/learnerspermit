OpenSpace

A self-evolving skill engine from HKUDS that treats skills as living entities — not static files. Every task execution feeds back into the skill library: broken skills auto-repair, successful patterns get captured, and improvements propagate to any connected agent via the cloud community at open-space.cloud.

GDPVal Benchmark Results (50 professional tasks, 6 industries)

Fewer tokens on real-world tasks vs baseline (cold→warm rerun)
4.2×
Higher GDPVal income vs ClawWork baseline (same LLM backbone)
70%
Quality score vs 40% baseline (220 tasks, 44 occupations)
60+
Skills self-generated from 6 seeds in the showcase dashboard demo
Cold start = no prior skills (reasons from scratch). Warm start = evolved skills reused. The gap between them is the value. GDPVal is HKUDS's own economic benchmark — read the arXiv paper before treating dollar figures as ground truth.
The Three Skill Evolution Modes
🔧
FIX
Skill is broken. OpenSpace auto-repairs it — detects the failure pattern and patches the SKILL.md. Agent cannot proceed until this resolves.
🌿
DERIVED
Creates an enhanced or specialized version of an existing skill. New skill directory, co-exists with the parent — lineage tracked with full diffs.
📸
CAPTURED
Extracts a new reusable pattern from a successful task execution. Turns one-off solutions into permanent skills the agent can reuse.
All three modes trigger automatically inside execute_task — no manual invocation needed. Evolution runs every N calls (default: 5).
The 4 MCP Tools Exposed to Your Agent
ToolWhat it doesNotes
execute_task Multi-step grounding agent loop — runs the task and triggers skill evolution automatically Main entry point; FIX / DERIVED / CAPTURED all fire here
search_skills Local + cloud skill search — finds the best matching skill before executing Falls back to local-only if no API key configured
fix_skill Explicitly repair a broken SKILL.md Usually triggered automatically, but can be called manually
upload_skill Push an evolved skill to the cloud community Requires OPENSPACE_API_KEY

The 4-Command Workflow
1

3
Refine — iterative gap analysis, non-destructive 
# Broad: agent finds all capability gaps
/cli-anything:refine ./gimp
# Focused: target a specific area

Each refine run is incremental — it adds new commands, tests, and docs without breaking what already works. Run as many times as needed.
4

# Unit tests (no backend needed)

# E2E tests (real software must be installed)
CLI_ANYTHING_FORCE_INSTALLED=1 python3 -m pytest cli_anything/<software>/tests/ -v -s
⚠ E2E tests require the actual backend software installed. The CLI is a wrapper TO the software — not a replacement. Hard dependency, not optional.

Claude Peers

A local MCP server that lets all your Claude Code sessions find each other and message each other ad-hoc — in real time, push-based, fully localhost. No cloud, no pre-planning. One clone to install.

Architecture — How It Works
         ┌───────────────────────────┐
         │       broker daemon        │
         │   localhost:7899 + SQLite  │   ← auto-launches on first session
         └──────┬───────────────┬────┘
                │               │
         MCP server A      MCP server B    (stdio — one per session)
            (stdio)           (stdio)
                │               │
           Claude A         Claude B
Polling: Each MCP server polls the broker every 1 second for new messages
Push delivery: Inbound messages are pushed into the session via claude/channel protocol — lands in context window without a user prompt
Self-healing: Broker cleans up dead peers automatically via PID checks. Everything is localhost-only.
Install & Run
1
Clone + install (requires Bun) 
git clone https://github.com/louislva/claude-peers-mcp.git ~/claude-peers-mcp
cd ~/claude-peers-mcp
bun install
2
Register as a user-scoped MCP (available in every session) 
claude mcp add --scope user --transport stdio claude-peers \
  -- bun ~/claude-peers-mcp/server.ts
3
Launch with channel push enabled 
claude --dangerously-load-development-channels server:claude-peers
💡 Add an alias: alias claudepeers='claude --dangerously-load-development-channels server:claude-peers' ⚠ The broker daemon starts automatically on first launch — no separate step needed.
4
Open a second terminal with the same command — they find each other
Ask either session: "list all peers" — it shows every running instance with working directory, git repo, and a summary of what it's doing.
What Each Session Can Do
list_peers
Discover all running instances — shows working directory, git root, and auto-generated summary of what each is working on
send_message
Send a message to any peer by ID — arrives in their context window instantly via channel push, no user prompt needed
set_summary
Manually set what this session is working on — visible to all other peers via list_peers. Auto-set if OPENAI_API_KEY is present
auto-summary (optional)
Set OPENAI_API_KEY → each instance auto-generates context from git branch, working dir, recent files via gpt-4o-mini (fractions of a cent per session)
CLI Debug Tools
bun cli.ts status       # broker status + all registered peers
bun cli.ts peers        # list peers with summaries
bun cli.ts send <id> <msg>  # inject a message into a Claude session
bun cli.ts kill-broker  # stop the broker daemon
DB path: ~/.claude-peers.db — override with CLAUDE_PEERS_DB env var. Port default 7899, override with CLAUDE_PEERS_PORT.
Bottom-Up vs Top-Down — The Key Philosophy Difference
✓ Claude Peers (bottom-up)
✅Sessions started independently discover each other on their own
✅Flat hierarchy — all agents are peers, no God-mode controller
✅Collaboration is ad-hoc — one instance notices overlapping work and reaches out spontaneously
✅No pre-planning required — works across projects you started for different reasons
✗ Built-in Claude Code Teams (top-down)
❌Requires pre-planning — you decide upfront who talks to whom
❌Central orchestrator model — one controller, subordinate agents
❌Collaboration is structured — must be configured before the sessions start
❌Sessions started independently remain blind to each other
The 3-Role Harness (Anthropic blog pattern)
🗺️
PLANNER
Breaks down the problem, coordinates the other sessions
⚙️
EXECUTOR
Actually writes and runs the code
🔍
EVALUATOR
Grades the output — separate session avoids self-hype bias
Claude Code is a poor evaluator of its own work. The 3-role harness separates creator from critic. With Claude Peers, the evaluator session's feedback arrives in the executor's context window as a push message — no copy-pasting, no manual bridging.
✓ Strong Use Cases
✅Frontend + backend in separate sessions — frontend asks backend what the JSON type looks like, answer lands instantly
✅Frontend design needing iterative subjective evaluation
✅Game development — complex, multi-system, needs external grading
✅Any project spanning multiple repos or workspaces simultaneously
✅Automated review pipelines where you're running 5+ sessions
✗ Overkill For
❌Single-session work that fits in one context window
❌Simple scripts with no cross-session coordination needed
❌Tasks without an evaluation or review loop
❌No-Bun environments (workarounds exist but add complexity)
⚠ Gotcha — Requires Bun

The broker uses bun:sqlite and Bun.serve() — not portable to Node.js without a rewrite. Enterprise / restricted environments can containerize the broker via Podman/Docker as a workaround, but it adds setup friction. Plain npm/Node.js is not natively supported.

💡 Key Insight

The push-based interruption is the differentiator. Instead of polling or copy-pasting, a message from Peer A physically appears in Peer B's context window without a user prompt — a digital shoulder tap that transforms AI from a passive tool into an active collaborator.

Google Workspace CLI gws

One CLI for all of Google Workspace — Drive, Gmail, Calendar, Sheets, Docs, Chat, Admin, and every Workspace API. Written in Rust, distributed via npm. 40+ agent skills included. Built by Google Developer Relations — not an officially supported Google product. ~22.9k stars.

The Architecture — Dynamic Command Surface
gws doesn't ship a static list of commands. It reads Google's Discovery Service at runtime and builds its entire command surface dynamically. When Google Workspace adds a new API endpoint, gws picks it up automatically — no waiting for a package update. Every response is structured JSON. 
# Command pattern: gws <service> <resource> [sub-resource] <method> [flags]
gws drive files list --params '{"pageSize": 5}'
gws gmail users messages get --params '{"userId": "me", "id": "MSG_ID"}'
gws calendar events list --params '{"calendarId": "primary"}'
gws sheets spreadsheets values get \
  --params '{"spreadsheetId": "ID", "range": "Sheet1!A1:D10"}'

# Every command has --help, --dry-run (preview), and --json output
gws drive --help
gws gmail users messages list --dry-run  # preview without executing
Install & Authenticate
1
Install 
npm install -g @googleworkspace/cli   # bundles pre-built binary, no Rust needed
# Or download binary from GitHub Releases page
# Or via Nix: nix run github:googleworkspace/cli
2
Set up a Google Cloud project + OAuth credentials 
gws auth setup   # interactive — walks you through GCP project + OAuth client creation
⚠ You need a Google Cloud project with OAuth credentials (Desktop app type). Each Google API (Gmail, Drive, etc.) must be enabled separately in your GCP project — one-time per API, but expect to click through the Cloud Console several times. ⚠ Windows: gws auth setup is broken on Windows due to Rust PATH resolution. Use env vars and manual OAuth flow instead.
3
Log in (OAuth browser flow) 
gws auth login   # opens browser, stores encrypted refresh token in OS keyring
# Subsequent runs are silent — no browser window
# For CI/CD: gws auth export --unmasked credentials.json
# Then: GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE=/path/to/credentials.json
💡 Credentials are encrypted at rest using AES-256-GCM and stored in your OS keyring.
Three Ways to Use gws with AI Agents
MODE A — MCP SERVER (recommended for Claude Code) 
gws mcp -s drive                  # expose Drive tools only
gws mcp -s drive,gmail,calendar   # multiple services
gws mcp -s all                    # ⚠ exposes hundreds of tools — stay under client limit
Starts an MCP server over stdio. Works with Claude Code, Claude Desktop, VS Code, Cursor — any MCP-compatible client. Each service adds roughly 10–80 tools. Most clients cap at 50–100 tools total, so scope carefully.
MODE B — AGENT SKILLS (Claude Code native) 
# Install all skills at once
npx skills add https://github.com/googleworkspace/cli

# Or install only what you need
npx skills add https://github.com/googleworkspace/cli/tree/main/skills/gws-drive
npx skills add https://github.com/googleworkspace/cli/tree/main/skills/gws-gmail
40+ pre-built skills tell Claude Code how to use gws effectively — which params to pass, how to handle pagination, common error patterns. Makes it agent-ready, not just agent-compatible. The gws-shared skill auto-installs the CLI via npm if gws isn't on PATH.
MODE C — GEMINI CLI EXTENSION 
gemini extensions install https://github.com/googleworkspace/cli
Gives Gemini CLI direct access to all gws commands and skills. Auth is inherited from gws auth login — no separate configuration needed.

Security — Model Armor + Prompt Injection
🛡️
The --sanitize flag pipes API responses through Google Cloud Model Armor before they reach your agent
📨
Defends against indirect prompt injection — e.g. a malicious email body instructing the agent to forward your messages
🔒
Additional hardening: resource-name validation, path-traversal protection, path-parameter encoding
⚠️
Model Armor only covers gws tools. Claude Code's other tools (web requests, file writes, code execution, other MCPs) are not protected by it
# Scan a Gmail message response before it reaches your agent
gws gmail users messages get --params '...' \
  --sanitize "projects/P/locations/L/templates/T"
Scoping Strategy — Don't Go Full Access Day One
1
Start scoped — pick services based on actual need 
gws mcp -s drive              # drive only to start
gws mcp -s drive,calendar     # expand incrementally
💡 Clone the repo and ask Claude: "Walk me through which skills make sense for my use case" before installing everything.
2
Use shared-folder sandboxing for Drive
Give access to a specific shared folder rather than your root Drive — limits blast radius if something goes wrong.
3
For Gmail — use a dedicated address and enable --sanitize
Email carries the highest prompt injection risk. A dedicated Gmail account + Model Armor before going near your primary inbox. ⚠ Even with --sanitize, Claude Code's other tools remain potential exfiltration vectors. Review SECURITY.md.
4
Stay under your MCP client's tool limit
Running gws mcp -s all exposes hundreds of tools. Most clients cap at 50–100. Pick only what you need.
Gotchas
⚠️
Pre-1.0 / breaking changes: README explicitly warns about this. Don't pin in production without expecting config updates.
⚠️
Windows auth setup broken: gws auth setup fails on Windows. Use environment variables and manual OAuth flow instead.
⚠️
Per-API enablement: Each Google API must be enabled in your GCP project individually. Expect several Cloud Console clicks on first setup.
⚠️
zsh Sheets ranges: Sheet1!A1 contains ! which zsh interprets as history expansion. Always use double quotes: "Sheet1!A1:D10"
⚠️
Tool count explosion: -s all exposes hundreds of tools — far exceeds typical MCP client limits. Scope to specific services.
⚠️
OAuth app in testing mode: Limited to ~25 scopes. Unverified app warning is expected — click "Advanced → Go to app" for personal use.
Best Use Case

Claude Code as a personal assistant operating system. Email triage, Drive organization, calendar management, Sheets automation — all from the terminal without touching a browser. The more of your daily work lives in Google, the higher the ROI. For enterprise: evaluate in a sandboxed Workspace environment first, identify 2–3 high-friction use cases, then expand.
