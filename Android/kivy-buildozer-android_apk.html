Kivy + Buildozer WSL2 + Android

Python → Android APK

A real-world walkthrough of building a Kivy Android app on Windows using WSL2 and Buildozer — including every error hit along the way and how to fix it. Built around the PhotoRename app: take a photo, get prompted for a name, save it.

The Full Pipeline
WRITEmain.py in Python
CONFIGUREbuildozer.spec
BUILDWSL2 + Buildozer
PACKAGEAPK compiled
DEPLOYsideload to phone

The first build downloads ~2GB of Android SDK/NDK tooling and takes 15–20 minutes. Every subsequent build is much faster.

What Each Tool Does
ToolWhat It IsWhy You Need It
KivyPython UI frameworkRenders your app UI on Android — buttons, popups, images
plyerPlatform bridge libraryLets Python talk to Android APIs — camera, GPS, notifications
pyjniusJava/Python bridgeCalls Android Java APIs directly from Python (media scan, etc.)
BuildozerAPK packaging toolCompiles Python + Kivy + dependencies into an installable .apk
python-for-androidCross-compiler (called by Buildozer)Converts Python code to ARM architecture that Android can run
WSL2Linux on WindowsBuildozer requires Linux — WSL2 provides it without a VM
⚡ Key Mental Model

Your phone doesn't know what main.py is. Buildozer's entire job is to bundle Python, the interpreter, all dependencies, and Android glue code into a single APK your phone can understand. Think of it as a compiler that speaks Python in and APK out.

Environment Setup

Getting WSL2 + Ubuntu installed and Buildozer ready. This is the foundation everything else sits on.

Step 1 — Install WSL2 + Ubuntu
1
Check if WSL has a distro installed
PowerShell (as Admin)
wsl --list --verbose
⚠ If output says "no installed distributions" — WSL is enabled but empty. That's fine, install one now.
2
Install Ubuntu 22.04 
wsl --install -d Ubuntu-22.04
💡 This downloads Ubuntu (~500MB). When prompted, set a Linux username and password — remember them.
3
Reboot, then verify 
wsl --list --verbose
# Should show: Ubuntu-22.04   Running   2
ℹ VERSION must be 2. If it shows 1, run: wsl --set-version Ubuntu-22.04 2
Step 2 — Install Build Dependencies
1
Update Ubuntu and install all build dependencies 
sudo apt update && sudo apt upgrade -y
sudo apt install -y \
  python3-pip zip unzip \
  openjdk-17-jdk cython3 \
  autoconf automake libtool \
  pkg-config libssl-dev \
  libffi-dev build-essential \
  git
💡 Install all of these upfront. Buildozer will hit missing tools one by one mid-build if you don't — autoconf, automake, and libtool are needed by libffi compilation inside python-for-android.
2
Install Buildozer 
pip3 install buildozer
⚠ If you get "normal site-packages is not writeable" — that's fine, it installs to ~/.local instead. See Gotchas tab.
3
Add ~/.local/bin to PATH 
# Fix for current session
export PATH=$PATH:~/.local/bin

# Make permanent — survives WSL restarts
echo 'export PATH=$PATH:~/.local/bin' >> ~/.bashrc
source ~/.bashrc
ℹ ~/.bashrc runs every time you open WSL. Anything in it is permanent. PATH tells Linux where to look for commands.
4
Verify Buildozer is found 
which buildozer
buildozer --version
💡 If both return output, you're ready. If not, check PATH and re-run step 3.
ℹ WSL ↔ Windows Filesystem

WSL and Windows share the same filesystem automatically. Your Windows D:\ drive is visible in WSL as /mnt/d/. Your WSL home folder is accessible from Windows Explorer at \\wsl$\Ubuntu-22.04\home\username\. No copying or syncing needed — edit in VS Code on Windows, build in WSL, they see the same files.

Project Setup

Creating the project folder, the app code, and the Buildozer spec. The spec is the config file that tells Buildozer everything about your app.

Create the Project — In WSL Linux Filesystem
!
Critical: Build inside the Linux filesystem, not /mnt/c or /mnt/d ⚠ Building on a Windows-mounted path (/mnt/c/, /mnt/d/) causes chmod failures. Buildozer needs Linux file permissions. Always build from ~/projects/ or similar. 
# Create project in Linux home — NOT on /mnt/
mkdir -p ~/projects/photorename
cd ~/projects/photorename
1
Copy your files from Windows into Linux 
# If your files are on D:\projects\photorename\
cp /mnt/d/projects/photorename/main.py .
cp /mnt/d/projects/photorename/buildozer.spec .  # if already created
💡 Or just create main.py directly here and edit it in VS Code — VS Code can open WSL paths via the Remote WSL extension.
2
Generate buildozer.spec 
cd ~/projects/photorename
buildozer init
This creates buildozer.spec in your project folder with all defaults. You edit the key lines next.
Configure buildozer.spec — Key Lines to Change

Open the generated spec in VS Code and find+replace these values. Everything else can stay as default.

# APP IDENTITY
title = PhotoRename
package.name = photorename
package.domain = org.hala

# SOURCE
source.dir = .
source.include_exts = py
version = 1.0

# DEPENDENCIES — must include all libraries your code imports
requirements = python3,kivy,plyer,pyjnius

# ANDROID PERMISSIONS
android.permissions = CAMERA,READ_EXTERNAL_STORAGE,WRITE_EXTERNAL_STORAGE,READ_MEDIA_IMAGES

# ANDROID API TARGETS
android.api = 33
android.minapi = 21
android.ndk = 25b
⚠ Requirements

Every Python library you import must be listed in requirements. Missing one = build succeeds but app crashes at runtime. pyjnius is needed for the media scan broadcast that makes renamed photos appear in your gallery.

✓ Iterating After First Build

Edit main.py on Windows, re-run buildozer android debug in WSL. The SDK/NDK are already downloaded. Only your changed Python code gets recompiled — much faster than the first run.

Gotchas — Errors We Actually Hit

Every error encountered during this build, what caused it, and the exact fix. In order of appearance.

🔴 Gotcha 1 — WSL has no installed distributions

Error: Windows Subsystem for Linux has no installed distributions

WSL the feature was enabled on Windows, but no Linux distro was ever installed inside it. The two are separate — enabling WSL doesn't give you Linux automatically.

wsl --install -d Ubuntu-22.04
🔴 Gotcha 2 — buildozer: command not found

Error: buildozer: command not found even after installing it.

pip installed Buildozer to ~/.local/bin/ because the system site-packages folder wasn't writeable. That folder isn't in PATH by default, so the shell can't find the command.

export PATH=$PATH:~/.local/bin && echo 'export PATH=$PATH:~/.local/bin' >> ~/.bashrc
🔴 Gotcha 3 — pip3: no such option: --break-system-packages

Error: no such option: --break-system-packages

This flag was introduced in pip 23.x for newer Ubuntu versions. Older Ubuntu (20.04 or below) doesn't recognize it. The flag isn't needed on older systems anyway.

pip3 install buildozer   # just drop the flag
🔴 Gotcha 4 — Cython not found

Error: Cython (cython) not found, please install it.

python-for-android (called by Buildozer) requires Cython to compile Python extensions for ARM. It's not installed by default.

sudo apt install -y cython3   # or: pip3 install cython
🔴 Gotcha 5 — chmod failed: Operation not permitted (THE BIG ONE)

Error:

error: chmod on /mnt/c/Users/.../photorename/.buildozer/... failed: Operation not permitted
fatal: could not set 'core.filemode' to 'false'

The project folder was on the Windows filesystem (/mnt/c/ or /mnt/d/). Linux cannot set file permissions (chmod) on NTFS-formatted Windows drives. Buildozer needs full Linux file permissions to work.

This one wastes the most time because the build starts fine, downloads things, then fails mid-way.

mkdir -p ~/projects/photorename && cp /mnt/d/.../main.py ~/projects/photorename/ — build from Linux home dir only
🟡 Gotcha 6 — Silent rename failure on Android 10+

Symptom: App builds and runs but renamed photos don't appear in gallery, or the rename fails silently.

Android 10 (API 29) introduced scoped storage restrictions. Direct writes to DCIM may be silently blocked. Two fixes: use MediaStore API via pyjnius for the rename, and always trigger a media scan broadcast after any file operation.

Wrap shutil.move in try/except and fall back to MediaStore. Always fire ACTION_MEDIA_SCANNER_SCAN_FILE after rename.
🔴 Gotcha 7 — autoreconf: not found (libffi build failure)

Error:

autogen.sh: 2: exec: autoreconf: not found
Command failed: pythonforandroid.toolchain create ...

python-for-android compiles libffi from source as part of the Android build. autoreconf is part of GNU autotools — required for that compilation — and isn't installed on a fresh Ubuntu. The build gets surprisingly far before hitting this.

Rather than fixing missing tools one by one, install the full set upfront:

sudo apt install -y \
  autoconf automake libtool \
  pkg-config libssl-dev \
  libffi-dev build-essential \
  git zip unzip
sudo apt install -y autoconf automake libtool pkg-config libssl-dev libffi-dev build-essential git zip unzip
🟡 Gotcha 8 — DESKTOP_TEST flag not working / NameError

Error: NameError: name 'DESKTOP_TEST' is not defined — or button click does nothing.

Two variations of this problem:

Variation A — DESKTOP_TEST defined inside the class but referenced without self. in take_photo. Class attributes need self.DESKTOP_TEST.

Variation B — DESKTOP_TEST = True added but the on_photo_taken guard silently exits because os.path.exists(TEST_IMAGE) returns False. The path is wrong or the file doesn't exist at that exact location.

# Diagnostic — add to on_photo_taken temporarily
def on_photo_taken(self, filename):
    print(f"on_photo_taken called with: {filename}")
    print(f"File exists: {os.path.exists(filename)}")
Define DESKTOP_TEST and TEST_IMAGE at the top of the file as globals (before the class). Use r"..." for Windows paths to avoid backslash escape issues.
⚡ Pattern

Most errors in this stack fall into three categories: missing Linux tools (Cython, autotools, PATH issues), filesystem permission conflicts (building on /mnt/ instead of ~/), and desktop testing scope issues (DESKTOP_TEST flag placement, wrong paths). Fix those three classes and 95% of problems disappear.

Desktop Testing Before You Build

Kivy runs on Windows and Linux desktops too. You can test your entire app — UI, popup, rename logic, thumbnail — without touching Buildozer or your phone. Do this first. Every bug you catch here saves a 5-minute rebuild cycle.

Install Kivy on Windows for Desktop Testing
1
Install Kivy + plyer on Windows
Windows PowerShell or CMD
pip install kivy plyer
💡 You don't need pyjnius on Windows — it's Android-only and already guarded by try/except in the code.
2
Run the app directly 
python main.py
A desktop window opens with your app running. Click buttons, test the popup, test the rename flow — all live.
What Works vs What Doesn't on Desktop
FeatureDesktopNotes
UI layout + popup✓ WorksButtons, TextInput, Image all render normally
Thumbnail preview✓ WorksLoads any local image file
Rename logic✓ Worksshutil.move works on desktop filesystem
plyer camera✗ FailsNo Android camera on desktop — mock it (see below)
pyjnius media scan✗ FailsAndroid-only — already wrapped in try/except, silently skipped
Mocking the Camera for Full Flow Testing

Add a DESKTOP_TEST flag at the top of main.py. This bypasses the camera intent and feeds a local image directly into the popup — giving you the full rename + thumbnail flow on desktop.

main.py — add near the top
# DESKTOP TESTING — set to False before buildozer android debug
DESKTOP_TEST = True
TEST_IMAGE = "C:/Users/halaf/Pictures/test.jpg"  # any image on your machine
main.py — update take_photo method
def take_photo(self, *args):
    if DESKTOP_TEST:
        self.on_photo_taken(TEST_IMAGE)
        return
    camera.take_picture(filename=TMP, on_complete=self.on_photo_taken)
⚠ Before Running Buildozer

Set DESKTOP_TEST = False before running buildozer android debug. If you forget, the APK will load a hardcoded Windows path on your Android phone — which obviously doesn't exist.

Recommended Testing Workflow
1
Set DESKTOP_TEST = True, point TEST_IMAGE at any local photo 💡 Use a real photo — it tests the thumbnail scaling and aspect ratio too.
2
Run python main.py — iterate freely Edit code → re-run → instant feedback. No WSL, no build time, no phone needed.
3
When logic is solid, set DESKTOP_TEST = False Then run buildozer android debug in WSL for the Android build. ℹ Only build for Android when you're confident the logic works. Each build takes 2–5 minutes minimum.
4
Android-only bugs (camera, media scan) — test on device These can't be tested on desktop. Install the APK and test on your actual phone for the final validation.
