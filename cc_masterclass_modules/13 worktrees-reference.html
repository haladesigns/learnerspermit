Parallel Sessions Without Conflicts

Git worktrees give each Claude Code agent its own isolated copy of your project — same history, separate files, separate branches. Run a hotfix and a feature build simultaneously without either session stepping on the other.

The Core Idea
Without Worktrees
Agent A
Agent B
Agent C
↓ ↓ ↓
⚡ /my-project — CONFLICT
Agents overwrite each other's work
VS
With Worktrees
Agent A
Agent B
Agent C
↓     ↓     ↓
/my-project
(main)
/proj-frontend
(frontend)
/proj-tests
(tests)
↘ ↓ ↙
Shared .git History — isolated work, clean merge
When to Use Worktrees
⚡
Parallel sessions — feature + hotfix running simultaneously
🔍
PR review while continuing development in another window
⏳
Long-running tests without blocking your main work
🗂️
Large-scale changes across many files
🤖
Subagent isolation — each agent gets its own tree
When NOT to Use Worktrees
🚫
Single task — one Claude session is all you need
🚫
Read-only inspection — use git show or git diff instead
🚫
Separate projects — parallel tasks in completely different repos
How a Worktree Relates to Your Repo
📁
Its own files — a full copy of the project at a specific branch
🌿
Its own branch — the same branch cannot be open in two worktrees simultaneously
📦
Its own staging area — git add is independent per worktree
🔗
Shared history — git fetch in any worktree updates remote info for all of them
💡 Mental Model

Think of worktrees as separate desks for each agent. Without worktrees, three agents share one desk and collide on the same files. With worktrees, each agent has their own desk with their own copy — they merge everything together cleanly at the end.

Feature + Hotfix in Parallel

You are deep into a dashboard feature when a critical bug lands. Here is the full step-by-step using worktrees so neither task blocks the other.

Step-by-Step Walkthrough
1
One-time setup — add worktrees to .gitignore
Prevents worktree contents from appearing as untracked files. Do this once per project and commit it. 
echo ".claude/worktrees/" >> .gitignore
git add .gitignore
git commit -m "Ignore Claude Code worktree directories"
⚠ If you skip this, every worktree's files show as untracked. Always do this first.
2
Start your feature in a worktree
Claude creates .claude/worktrees/dashboard-feature/ with a fresh branch and starts a session inside it. 
claude --worktree dashboard-feature
💡 Then prompt Claude: "Build a new analytics dashboard page at /dashboard with charts for monthly revenue, active users, and conversion rate."
3
Open a second terminal for the hotfix
Without closing the dashboard session, open a new terminal. Claude creates a second isolated worktree. 
claude --worktree hotfix-auth
💡 Then prompt: "There's a bug where expired JWT tokens aren't being rejected. Find the token validation logic and fix it. Add a test case for expired tokens."
4
Merge the hotfix
Once the fix is tested, push and PR from within that session. When you exit, Claude will ask: keep or remove? 
# Ask Claude inside the session:
"Commit these changes, push the branch, and create a PR for the auth fix."
⚠ If the PR is already merged, choose remove when prompted. If you may return, choose keep.
5
Resume feature work
Switch back to Terminal 1. Your dashboard session is right where you left it. Continue normally.
6
Push the feature branch
When ready, push from inside the worktree folder — or ask Claude to do it. 
cd .claude/worktrees/dashboard-feature
git push -u origin worktree-dashboard-feature
gh pr create --title "Add analytics dashboard" --body "New dashboard with charts"
💡 Or just ask Claude inside the session: "Push this branch and open a PR."
ℹ Mid-Session Worktree Creation

You can also ask Claude to switch into a worktree mid-session without starting with --worktree. Just say: "work in a worktree" or "start a worktree called refactor-api".

Starting Worktree Sessions
Command What It Does Notes
claude --worktree <name> Named worktree — creates branch worktree-<name> under .claude/worktrees/<name>/ Recommended. Easier to find later.
claude -w <name> Short flag, same as above Convenience alias
claude --worktree Auto-generated name — Claude picks a random name Less discoverable; use named form
claude --worktree <name> --tmux Runs session in background, survives terminal close Great for long-running tasks
claude --worktree <name> -p "..." Headless mode — runs one task non-interactively then exits Use for scripted / CI-style jobs
Manual Git Worktree Commands
Command What It Does Notes
git worktree add ../my-feature -b feature-branch Create a worktree at a custom path with a new branch Full control over location
git worktree list Show all active worktrees and their branches Use to audit what's open
git worktree remove ../my-feature Delete a worktree after merging or discarding Run after PRs are merged
git worktree prune Clean up stale/broken worktree references Use if Claude crashed mid-session
Subagent Worktree Isolation

Add isolation: worktree to a custom subagent definition to give each spawned agent its own isolated tree automatically.

--- (.claude/agents/my-agent.md)
name: code-migrator
description: Migrates deprecated API calls to the new SDK
isolation: worktree
tools:
- Read
- Edit
- Bash
---

You are a code migration specialist...
Prompt Alternative

Or just ask Claude mid-session: "Use worktrees for your agents. Split the codebase migration into three parallel tasks: update API calls, update data models, update tests."

Pattern 1 — Feature + Hotfix
Most common use case
# Terminal 1 — ongoing feature work
claude --worktree feature-payments

# Terminal 2 — urgent fix
claude --worktree hotfix-login-crash
Fix crash, commit, PR, merge. Exit and choose remove to clean up the hotfix worktree. Feature worktree keeps running untouched.
Pattern 2 — PR Review While Developing
Review without interrupting your work
# Terminal 1 — keep your feature going
claude  # (normal session)

# Terminal 2 — check out the PR branch
git worktree add ../review-pr-42 origin/pr-42-refactor-db
cd ../review-pr-42
claude
Prompt Claude: "Review the changes in this branch. Look for potential issues with database safety, missing indexes, and error handling."
Pattern 3 — Background Tasks with tmux
Long refactor? Don't block your terminal
# Kick off background task
claude --worktree refactor-api --tmux

# Keep working in main project
claude
The --tmux flag keeps the background session alive even if you close the terminal window. Attach to it later with tmux commands.
Pattern 4 — Headless Parallel Tasks
Fire-and-forget non-interactive jobs
# Each one-liner runs independently
claude -w update-api -p "Update all API calls to v2 SDK"
claude -w update-models -p "Update data models for new schema"
claude -w update-tests -p "Update tests for v2 changes"
Each agent works in isolation, then you merge the three branches. Ideal for large-scale mechanical refactors.
🤖 Ask Claude

Not sure if worktrees make sense? Just ask Claude: "Do worktrees make sense for this task?" — Claude can decide, set them up, and manage cleanup automatically.

✓ Do This
✅Add .claude/worktrees/ to .gitignore before first use
✅Run /init in each new worktree session so Claude loads CLAUDE.md and project context
✅Install dependencies (npm install etc.) separately in each worktree
✅Choose keep on exit if you might return; choose remove if the PR is merged
✅Use git worktree prune to clean up stale references after a crash
✗ Avoid This
❌Skipping the .gitignore step — worktree contents flood your git status
❌Trying to open the same branch in two worktrees — Git will reject it
❌Exiting and removing a worktree if you haven't committed or pushed — the files are gone permanently
❌Assuming .env files are shared — they are not; copy or symlink them manually
❌Leaving many old worktrees around — disk space adds up since each is a full file copy
Critical Gotchas
⚠️
Cannot resume after removal. If you exit and choose remove, the worktree files are deleted. There is no undo. When in doubt, choose keep.
⚠️
Dependencies are not shared. A new worktree is a fresh file copy — node_modules/ and similar installed packages are not present. Run your install command or ask Claude to set up the dev environment.
⚠️
.env files are not shared. Worktrees do not inherit your root .env. Either copy it or create a symlink: 
ln -s $(pwd)/.env .claude/worktrees/my-feature/.env
⚠️
Unique branch per worktree. Git enforces that no branch can be checked out in two worktrees at once. Claude's --worktree flag avoids this by auto-creating a fresh branch named worktree-<name>.
⚠️
Manual cleanup if Claude crashes. Use these commands to recover: 
git worktree list
git worktree remove .claude/worktrees/stale-feature
git worktree prune
ℹ️
git fetch is global. Running git fetch inside any worktree updates remote info for all worktrees — they share the same .git history.
💡 Run /init in Every New Worktree

Each worktree starts as a fresh session context. Run /init so Claude loads your CLAUDE.md, project settings, and understands the codebase — otherwise it starts blind.
