/dream — Memory Consolidation

Dream is Anthropic's (currently limited-access) reflective pass over Claude Code's auto-memory files. It merges duplicates, resolves contradictions, converts relative dates to absolute, and keeps the memory index under 200 lines — so Claude gets smarter the longer you use it, not bloated.

Problems Dream Solves
🪞
Duplicates — multiple files saying the same thing pile up over time.
⚡
Contradictions — "always use React" vs "never use React" both live in memory.
🗓️
Relative dates — "next Friday" becomes meaningless as time passes; Dream rewrites to absolute dates.
📦
Stale info — outdated facts that no longer reflect real usage get pruned.
📄
Bloated index — MEMORY.md is loaded every session; Dream keeps it tight (≤200 lines).
Why This Matters
200
Max lines in MEMORY.md index
∞
Sessions accumulating without Dream
The master memory file is loaded at the start of every Claude Code session. Context bloat from an uncleaned memory has the same compounding cost as a bloated CLAUDE.md — less room for actual work.
💡 Key Takeaway

Dream is available as a public skill even if Anthropic hasn't given you native access. Copy the prompt into a skill called dream and invoke it manually whenever memory feels noisy. Less is more — a tight index beats a comprehensive one.

Auto-Memory System
🧠
Auto-memory lets Claude Code accumulate knowledge across sessions without any manual effort — you just talk, and Claude flags what's worth remembering.
📁
Memory files live in ~/.claude/projects/<project-slug>/memory/ — not inside your actual project directory.
📋
MEMORY.md is the master index — it references all individual topic files. Loaded at the start of every session.
📂
Individual topic files (e.g. testing-approach.md, user-preferences.md) hold the actual detail. The index links to them.
🎯
Session transcripts (JSONL files) live in the parent of the memory folder — Dream reads these to see how memory maps to real usage.
Memory Scope: Project
Each project has its own memory folder. Claude knows which project you're in and matches it to the correct slug. 
~/.claude/projects/
C--Users-Chase-the-vault/
  memory/
    MEMORY.md           # master index
    testing-approach.md
    user-preferences.md
    *.jsonl             # session transcripts
Memory Scope: User
A user-level memory folder applies across every project — good for persistent personal preferences. 
~/.claude/projects/
C--Users-Chase/
  memory/
    MEMORY.md           # applies globally
    coding-style.md
    ...
🔗 Relationship to CLAUDE.md

Auto-memory and CLAUDE.md are complementary but separate. CLAUDE.md is hand-authored and lives in your repo. Auto-memory is Claude-authored and lives in ~/.claude. Both are loaded each session — both can bloat. Dream only touches auto-memory.

Dream Runs in 4 Phases
1
Orient — read what exists
ls the memory directory, read MEMORY.md, skim all existing topic files. If logs/ or sessions/ subdirectories exist, review recent entries there too. 💡 Goal: understand what's already there before creating duplicates.
2
Gather Recent Signal — sources in priority order
💡 Priority 1: Daily logs (logs/YYYY/MM/YYYY-MM-DD.md) if present — these are the append-only stream of ground truth. 💡 Priority 2: Existing memories that have drifted — facts that contradict what you now see in the codebase. 💡 Priority 3: Transcript search — grep JSONL only for specific context you already suspect matters. 
grep -rn "<narrow term>" ~/.claude/projects/<project>/ --include="*.jsonl" | tail -50
⚠ Don't exhaustively read all transcripts — narrow grep only.
3
Consolidate — write / update memory files
For each thing worth remembering, write or update a topic file. Three rules from the skill: 💡 Merge into existing files first — don't create near-duplicate files. 💡 Convert all relative dates to absolute — "last week" → "2026-03-24". 💡 Delete contradicted facts at the source — if new info disproves an old memory, fix it there, don't add a new file. Follow the memory file format (see Memory Format tab).
4
Prune & Index — keep MEMORY.md under 200 lines
Remove stale pointers. Demote verbose entries — keep the gist in the index, move the detail into the topic file. Add pointers to new memories. Resolve any remaining contradictions between files. ⚠ MEMORY.md is an index, not a dump. Never write memory content directly into it.
✅ Expected Output Format

Dream always returns a structured summary:

## Dream Complete — [scope: project / user / all]

### Consolidated
- what was merged/created

### Updated
- what was modified

### Pruned
- what was removed or cleaned up

### No Change
- memories already tight (if nothing needed fixing)
Memory File Format
Every individual memory file follows this frontmatter + body structure:
---
name: testing-approach
description: Chase's preferences for test strategy — use real integrations, no mocks
type: feedback   # user | feedback | project | reference
---

# Memory content goes here
Rule: never use mock tests; always use real integrations.
Why: mocks hide real failure modes and slow down debugging.
How to apply: when writing tests, connect to actual APIs/DBs in the test environment.
Memory Types
user
Role, goals, preferences, domain knowledge. Who this person is and how they work.
feedback
Approach guidance — what to avoid, what to keep doing. Structure: rule → Why: → How to apply:
project
Ongoing work, goals, initiatives, known bugs. Structure: fact/decision → Why: → How to apply:
reference
Pointers to external resources, docs, or links that Claude should know exist.
✓ Save to Memory
✅User preferences and working style
✅Persistent project goals and decisions
✅Approach rules with clear "why" and "how"
✅Absolute dates for any time-bound facts
✅Known bugs or ongoing initiatives
✗ Do NOT Save to Memory
❌Code patterns / architecture (derivable from code)
❌File paths (grep the repo)
❌Git history (use git log)
❌Debugging solutions (fix is in the code)
❌Anything already in CLAUDE.md
❌Ephemeral task details
⚠ Description Field Is Critical

The description field in each memory file is used by Claude to decide whether that file is relevant to the current conversation. Make it specific and actionable — vague descriptions mean the memory gets loaded unnecessarily or missed entirely.

Installing the Skill
1
Get the Dream prompt
The prompt is public — courtesy of Pyball-AI. Copy the full skill markdown (see the Dream-Skill.md file).
2
Create the skill in Claude Code
Tell Claude Code: "I want to create a new skill called dream. Use this prompt to build it." Paste the skill content. Claude Code creates the skill file automatically. 💡 Claude Code will generate the skill in your .claude/skills/ directory.
3
Invoke whenever memory feels noisy
Run /dream, /dream user, or /dream all from within Claude Code. Dream will ask permission before editing its own .claude folder — even with bypass-permissions enabled. ⚠ Always approve the permission prompt — Dream needs to write to the memory directory.
Gotchas
⚠️
Permission prompt always appears: Dream edits files in ~/.claude. Claude Code will ask for permission to edit its own settings even if bypass-permissions is on. You must approve.
⚠️
Don't over-run Dream: It's a maintenance pass, not a real-time tool. Run it when memory feels large or contradictory — not every session.
⚠️
MEMORY.md hard limit is 200 lines — but less is always better. Dream targets the tightest possible index, not the longest allowed one.
⚠️
Dream does NOT touch CLAUDE.md — only auto-memory files. If you have redundancy between CLAUDE.md and memory, resolve that manually.

Title & Purpose
# Dream: Memory Consolidation

You are performing a dream — a reflective pass over your memory files.
Synthesize what you've learned recently into durable, well-organized
memories so that future sessions can orient quickly.
## Targeting
The user can specify a scope. Parse the argument after `/dream`:
Command Target
/dream (no args) Current project memory only
/dream user User-level memory only (~/.claude/projects/C--Users-Chase/memory/)
/dream all Both user-level AND current project memory (run phases on each separately, report combined results)
## Locating the Memory Directory
User-level memory is always at:
~/.claude/projects/C--Users-Chase/memory/

Project-level memory — find it by running:
ls ~/.claude/projects/

Match the current working directory to the project folder name
(dashes replace path separators, e.g.
`C:\Users\Chase\the vault` → `C--Users-Chase-The-Vault`).
The memory directory is at:
~/.claude/projects/<matched-project>/memory/

Session transcripts (JSONL files) are in the parent directory of the memory folder.
If the user specifies a different project or path, use that instead.

Index max lines: 200
## Phase 1 — Orient
·
ls the memory directory to see what already exists
·
Read MEMORY.md to understand the current index
·
Skim existing topic files so you improve them rather than creating duplicates
·
If logs/ or sessions/ subdirectories exist, review recent entries there
## Phase 2 — Gather Recent Signal
Look for new information worth persisting. Sources in rough priority order:

1. Daily logs (`logs/YYYY/MM/YYYY-MM-DD.md`) if present — these are the append-only stream
2. Existing memories that drifted — facts that contradict something you see in the codebase now
3. Transcript search — grep the JSONL transcripts for narrow terms:

grep -rn "<narrow term>" ~/.claude/projects/<project>/ --include="*.jsonl" | tail -50

Don't exhaustively read transcripts. Look only for things you already suspect matter.
## Phase 3 — Consolidate
For each thing worth remembering, write or update a memory file at the
top level of the memory directory. Follow the memory file format:

---
name: {{memory name}}
description: {{one-line description — used to decide relevance in future
             conversations, so be specific}}
type: {{user, feedback, project, reference}}
---

{{memory content}}

user — role, goals, preferences, knowledge
·
feedback — guidance on approach (what to avoid, what to keep doing). Structure: rule, then Why: and How to apply:
·
project — ongoing work, goals, initiatives, bugs. Structure: fact/decision, then Why: and How to apply:

### What NOT to save

·
Git history (use git log)
·
Debugging solutions (fix is in the code)

Ephemeral task details

·
Converting relative dates ("yesterday", "last week") to absolute dates so they remain interpretable after time passes
·
Deleting contradicted facts — if today's investigation disproves an old memory, fix it at the source
## Phase 4 — Prune and Index
Update `MEMORY.md` so it stays under 200 lines. It's an index, not a dump —
link to memory files with one-line descriptions. Never write memory
content directly into it.
·
Remove pointers to memories that are now stale, wrong, or superseded
·
Demote verbose entries: keep the gist in the index, move the detail into the topic file
·
Add pointers to newly important memories
·
Resolve contradictions — if two files disagree, fix the wrong one
