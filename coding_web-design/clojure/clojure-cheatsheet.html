clojure reader · types · collections — Rich Hickey reference

The Reader, Types & Evaluation

What happens before your code runs. The reader turns text into data structures; the evaluator acts on them. These are separate, independent stages — and that separation is what makes Clojure macros possible.

The read → eval pipeline
read vs eval reader Two separate stages — text → data, then data → value ▶

The reader parses text into data structures. The evaluator then acts on those data structures. Most of the time they feel identical — but they're independent stages you can use separately.

; Atoms: read and eval give the same result 42 ; read → Long eval → Long 42 :name ; read → Keyword eval → Keyword (itself) "hello" ; read → String eval → String ; Symbols & lists: eval does extra work a ; read → Symbol 'a eval → looks up its value (1 2 3) ; read → List eval → tries to CALL 1 (error!)
The language is defined in terms of data structures, not grammar. A Clojure program is data. This is what makes macros possible — macros transform data before eval sees it.
quote / ' special form Suppress evaluation — return the form itself ▶

Tells the evaluator: return this thing as-is. The single quote ' is shorthand for (quote ...). Suppresses evaluation of the entire form — not just the outer wrapper.

'w ; => symbol w (not its value) (quote w) ; same thing '(1 2 3) ; => list (1 2 3) — not a function call '(a b c) ; => list of symbols — nothing evaluated
Returns in W6D3 — macros. A quoted list is just a list; you can manipulate it as data, then hand it to eval. That's the macro system.
Numbers
integer / BigInt auto-promoting No silent overflow — grows to BigInt automatically ▶
42 ; Long (small int) 42N ; BigInteger (explicit) ; Integers auto-promote — no overflow, ever (* 1000000 1000000 1000000) ; => 1000000000000000000 (fine!)
Unlike Java/C, integer math never silently wraps. If the result can't fit in 64 bits, Clojure transparently promotes to BigInt and keeps going.
Python int: also arbitrary precision. Java/C int: wraps at 2³¹. Clojure: starts as Long, auto-promotes.
Ratio no Python equivalent Exact fractions — no floating-point loss ▶

When you divide two integers that don't divide evenly, Clojure stores the result as a Ratio — two integers held together. No precision loss until you mix in a Double.

22/7 ; => 22/7 (Ratio — exact) (* 22/7 7) ; => 22 (reduces to integer!) 5/20 ; => 1/4 (auto-reduces) (+ 1/3 1/3) ; => 2/3 (exact) ; Mix with a Double → everything becomes Double (+ 1/3 0.1) ; => 0.4333... (precision lost) (* 22/7 1.0) ; => 3.1428... (Double territory)
As soon as a Ratio touches a Double, everything converts to Double. Keep computations in Ratio-land for exact arithmetic; only convert at the output boundary.
variadic math n-arity +, -, *, / take any number of arguments ▶
(+) ; => 0 (identity for addition) (*) ; => 1 (identity for multiplication) (+ 1 2 3 4) ; => 10 (< 4 5 6) ; => true (chained comparison!) (< 4 5 3) ; => false (min 3 1 4 1 5 9) ; => 1 (inc 7) ; faster + clearer than (+ 7 1) (zero? 0) ; faster + clearer than (= 0 n)
Chained comparisons like (< 4 5 6) check the relationship across the whole set — there's no Python equivalent without writing 4 < 5 < 6. Prefer inc/dec/zero? — they're faster and more descriptive.
Keywords & Symbols
keyword no Python equivalent Symbolic identifiers that always evaluate to themselves ▶

Keywords evaluate to themselves — no lookup, no error. Any two keywords with the same name are the exact same object (interned). This makes them extremely fast as map keys.

:name ; a keyword — means itself :status/active ; namespaced keyword ; Interning — same name = same object (= :name :name) ; => true (identical? :name :name) ; => true (same object!) ; Strings: equal in value, NOT same object (identical? "foo" "foo") ; => false ; Keywords are also functions — look themselves up in a map (def person {:name "Ada" :age 36}) (:name person) ; => "Ada" (name :name) ; => "name" (string, no colon)
Because any two :name keywords are the same object, equality is an identity (pointer) check — the fastest possible comparison. This is why keywords are the idiomatic choice for map keys.
Closest Python analogy: Enum members — named constants that mean themselves. But keywords need no class, no boilerplate, and are first-class function values.
symbol no Python equivalent Names of things — eval looks up their value ▶

Symbols are objects with names. The evaluator treats them as names of things and tries to resolve what they refer to. Quote suppresses that resolution.

a ; symbol → eval looks up its value 'a ; quoted symbol → suppresses eval 'foo/bar ; namespaced: ns=foo, name=bar (name 'foo/bar) ; => "bar" (namespace 'foo/bar) ; => "foo" (gensym "id") ; => id__123 (unique symbol — for macros) (gensym) ; => G__456 (no prefix)
gensym used in Phase 3 macros — when writing code that writes code, you need guaranteed-unique symbol names to avoid name collisions (macro hygiene).
No Python equivalent. A symbol is a variable name treated as a value — a concept Python doesn't expose.
nil & truthiness
nil falsy Nothing. Not an empty collection — nothing. ▶

nil means the absence of anything. Same as Java null. Sequence functions return nil when there's nothing left to iterate through.

; Only nil and false are logically false (if nil "yes" "no") ; => "no" (if false "yes" "no") ; => "no" ; Everything else is truthy — including 0 and "" (if 0 "yes" "no") ; => "yes" (0 is TRUTHY!) (if "" "yes" "no") ; => "yes" ("" is TRUTHY!) (if [] "yes" "no") ; => "yes" (empty vec is TRUTHY!)
0 and "" are truthy in Clojure. Python developers expect 0, "", [], {} to be falsy — they are not. Only nil and false are falsy. This will catch you.
Python: 0, "", [], {}, None are all falsy. Clojure: only nil and false. Everything else — including empty string, 0, empty vector — is truthy.
Immutability & equality
value equality immutable All data is immutable → = is always safe and meaningful ▶

Because Clojure data structures never change, comparing them by value is always safe — the result of a comparison can never be invalidated by a later mutation.

; Strings: equal in value, but different objects (= "foo" "foo") ; => true (value equality) (identical? "foo" "foo") ; => false (diff objects) ; Keywords: equal AND same object (interned) (= :foo :foo) ; => true (identical? :foo :foo) ; => true ; Even large maps compared by value — hashes cached (= {:a 1 :b 2} {:a 1 :b 2}) ; => true
Mutable objects should never be compared by value — the result could differ between two calls. Clojure's immutability makes = universally meaningful. Hash values are cached internally — comparing large structures is fast.

Collections — Immutable & Persistent

All Clojure collections are immutable and persistent. "Persistent" means creating a modified version shares structure with the original — no full copy. Both copies can share safely because neither can ever change.

The big idea
persistence structural sharing Modified version shares structure with original — no full copy ▶

When you "modify" a collection you get a new value that shares most of its internal structure with the old one. Safe because neither can ever change.

; Original is NEVER changed (def v [1 2 3]) (conj v 4) ; => [1 2 3 4] (new vector) v ; => [1 2 3] (untouched) ; List: new node points to existing list — zero copy (def lst '(a b c)) (def lst2 (conj lst :x)) (identical? (rest lst2) lst) ; => true! (shared tail)
Thread safety is a free side-effect. Any number of threads can read the same data structure simultaneously — nothing races because nothing changes. No locks needed.
Performance: most operations are O(1) or near-O(1) for hash maps/vectors. Sorted maps/trees are O(log n). It's not a full copy — internally it's a 32-way branching tree (hash array mapped trie).
conj all collections Universal add — where it goes depends on the collection type ▶

Short for "conjoin". Returns a new collection with the item added. The collection comes first. Where it lands is intentional design, not inconsistency.

(conj '(a b c) :x) ; => (:x a b c) — FRONT (list) (conj [1 2 3] 4) ; => [1 2 3 4] — BACK (vector) (conj {:a 1} {:b 2}) ; => {:a 1 :b 2} — merge (map) (conj #{1 2} 3) ; => #{1 2 3} — add (set) ; Collections are heterogeneous — mix types freely (conj [1 "two"] :three) ; => [1 "two" :three]
Use assoc when you care about keys/values in maps. Use conj when you're writing code that works generically across collection types.
seq abstraction bridge Turn any collection into a sequence of items ▶

A sequence is the simplest interface: first and rest. That's it. When nothing remains, rest returns nil.

(seq '(a b c)) ; => (a b c) — list IS its own seq (seq [1 2 3]) ; => (1 2 3) — prints like list (it's a seq) (seq {:a 1 :b 2}) ; => ([:a 1] [:b 2]) — key-value pairs (seq #{1 2 3}) ; => (1 2 3) — unordered (first [1 2 3]) ; => 1 (rest [1 2 3]) ; => (2 3) — a seq, not a vector (first nil) ; => nil (no exception) (rest nil) ; => () (empty seq)
When sequences print with parentheses it doesn't mean they're lists. Parentheses = "this is a sequence". The underlying type could be anything.
map, filter, reduce, take, drop — all work on any sequence. Write to the seq abstraction and your code works on lists, vectors, maps, sets, and anything future.
List
list singly linked Linked list — fast at the front, slow random access ▶
count O(1)
conj O(1) front
nth O(n)
peek/pop O(1) front
'(1 2 3) ; literal (must quote — else it's a call) (list 1 2 3) ; constructor (list* 1 2 [3 4]) ; => (1 2 3 4) — prepend + flatten seq tail (peek '(1 2 3)) ; => 1 (first — stack front) (pop '(1 2 3)) ; => (2 3) (rest — stack front) ; Structural sharing — zero copy on conj (def a '(b c)) (def a2 (conj a :x)) (identical? (rest a2) a) ; => true (shared tail)
Use peek/pop instead of first/rest when thinking of a list as a stack — same result, clearer intent. Lets you swap list ↔ vector without changing algorithm logic.
Vector
vector 32-way hash trie Indexed collection — near-O(1) everywhere, including random access ▶
count O(1)
nth ≈O(1)
conj back ≈O(1)
peek/pop O(1) back
rseq O(1)
subvec O(1)
[1 2 3] ; literal (vector 1 2 3) ; constructor ; Index access (nth [1 2 3] 1) ; => 2 (throws if out of range) (get [1 2 3] 9) ; => nil (safe, no throw) (get [1 2 3] 1) ; => 2 ; Vector is associative — indices are keys (assoc [1 2 3] 1 99) ; => [1 99 3] (replace at index) (assoc [1 2 3] 3 4) ; => [1 2 3 4] (extend by exactly 1) ; Stack operations — back end (peek [1 2 3]) ; => 3 (last) (pop [1 2 3]) ; => [1 2] (all-but-last) ; O(1) operations — no copying (rseq [1 2 3]) ; => (3 2 1) reverse, O(1)! (subvec [1 2 3 4 5] 1 4) ; => [2 3 4] O(1) window, no copy
Internally a 32-way branching trie. log₃₂(4 billion) = 6. That branching factor is so small operations are effectively constant time — hence "near-O(1)".
nth vs get: use nth when out-of-range means something went wrong (want an exception). Use get when nil is a reasonable "not found" signal.
subvec shares structure with the original — a 1000-item window over 10M elements is instantaneous. Use this freely for sliding windows, chunking, etc.
Python list: mutable, O(1) append, O(n) insert. Clojure vector: immutable, O(1) add-to-end, O(1) random access — but every "change" returns a new vector.
Map
hash-map log₃₂ n ≈ O(1) Default map type — fast, unordered ▶
{:a 1 :b 2} ; literal → HashMap {:a 1, :b 2} ; same — commas are whitespace ; assoc — add/replace, takes multiple k/v pairs (assoc {:a 1} :b 2 :c 3) ; => {:a 1 :b 2 :c 3} ; dissoc — remove keys (dissoc {:a 1 :b 2} :b) ; => {:a 1} (dissoc {:a 1 :b 2} :z) ; => {:a 1 :b 2} (no error) (dissoc {:a 1 :b 2} :a :b) ; => {} (multiple keys) ; get — safe lookup (get {:a 1} :a) ; => 1 (get {:a 1} :z) ; => nil (get {:a 1} :z :not-found) ; => :not-found (default value) ; contains? — tests key PRESENCE (not value) (contains? {:a 1 :b nil} :b) ; => true (key exists!) (contains? {:a 1 :b nil} :z) ; => false ; find — returns [k v] pair or nil (find {:a 1 :b nil} :b) ; => [:b nil] (key exists) (find {:a 1 :b nil} :z) ; => nil (key absent) ; keys / vals (keys {:a 1 :b 2}) ; => (:a :b) (vals {:a 1 :b 2}) ; => (1 2) ; merge — last value wins on conflicts (merge {:a 1 :b 2} {:b 9 :c 3}) ; => {:a 1 :b 9 :c 3} ; merge-with — resolve conflicts with a function (merge-with + {:a 1 :b 2} {:b 3 :a 4}) ; => {:a 5 :b 5} (merge-with list {:a 1} {:a 2}) ; => {:a (1 2)}
If you store nil as a value, (get m :k) returns nil whether the key exists or not. Use contains? to test key presence, find to get the [k v] pair, or get with a sentinel third arg.
Commas are whitespace. {:a 1, :b 2} and {:a 1 :b 2} are identical. Clojure prints them for readability; you don't have to write them.
sorted-map O(log n) Red-black tree — maintains key sort order ▶
; sorted-map — keys sorted by natural order (sorted-map :b 2 :a 1 :c 3) ; => {:a 1 :b 2 :c 3} (always sorted) ; sorted-map-by — custom comparator (sorted-map-by > 3 "c" 1 "a" 2 "b") ; => {3 "c" 2 "b" 1 "a"} (descending) ; rseq works on sorted maps too (O(1) reverse) (rseq (sorted-map :a 1 :b 2 :c 3)) ; => ([:c 3] [:b 2] [:a 1])
Use sorted-map when you need iteration in key order. For pure lookup performance, hash-map is faster (log₃₂ n vs log₂ n). Both implement the same map abstraction — code written for one works on the other.
Quick reference — all collections
collection hierarchy reference What every collection supports and what's type-specific ▶
countEvery collection. O(1). (count nil) = 0, no exception.
conjEvery collection. Front for lists, back for vectors, merge for maps/sets.
seqEvery collection → sequence. Maps yield [:k v] pairs.
first / restAny sequence. first nil = nil, rest nil = () — never throws.
peek / popLists (front) and vectors (back). Swap freely for stack algorithms.
assoc / dissocMaps and vectors (index = key). Core of map manipulation.
getMaps and vectors. nil on miss — safe by design.
nthVectors + seqs. Throws on out-of-range.
contains?Maps and sets — tests key/member presence. For vectors, tests index range.
findMaps only. Returns [k v] or nil — distinguishes nil values from missing keys.
keys / valsMaps only. Returns sequences of keys or values.
merge / merge-withMaps only. merge-with resolves conflicts with a function.
rseqVectors and sorted-maps only. O(1) reverse sequence.
subvecVectors only. O(1) windowing — shares structure, no copy.
Everything is defined in terms of abstractions, not concrete types. Code written against the map abstraction works on hash-map, sorted-map, and any future map type — including ones you write yourself.
