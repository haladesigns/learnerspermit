About the built-in REPL

Each card has a ▶ try it button that opens a live editor powered by Klipse — a self-hosted ClojureScript evaluator that runs entirely in your browser. No server involved.

The first REPL you open triggers a one-time load of ~1.5MB from Klipse's CDN. After that, all other REPLs open instantly. Requires an internet connection.

Code re-evaluates automatically after you stop typing for ~600ms. You can write anything — not just the pre-loaded example.

What works in the REPL

✓  All pure ClojureScript — def, defn, let, map, filter, reduce, threading macros

✓  All data structures — vectors, maps, sets, lists, keywords

✓  Higher-order functions, closures, anonymous functions #()

✓  Multi-line expressions and defn definitions

✓  println output (appears in result area)

✓  clojure.string functions with (require '[clojure.string :as str])

Known differences from real Clojure

✗  (type x) returns JS types, not JVM types — e.g. number not java.lang.Long

✗  Ratio type — (/ 22 7) gives 3.142... not 22/7. One of Clojure's nicest features, absent here.

✗  Java interop — (.method obj), (new ClassName), import — no JVM in the browser

✗  Infinite sequences without take — (range) will hang. Always wrap with (take n ...)

✗  doc — (doc map) won't work in the browser REPL, use this directory instead

Everything else you'll encounter in Phase 1 and Phase 2 of the curriculum works identically.

How the REPL was built — what's in your code

The embedded REPL uses Klipse (v7.12.0), an open-source JavaScript plugin by Yehonathan Sharvit. It evaluates ClojureScript in the browser using the self-hosted ClojureScript compiler — meaning the compiler itself is written in ClojureScript and runs in JavaScript, with no server involved.

What loads and when:

page load Only the Klipse CodeMirror CSS loads immediately — one lightweight stylesheet from storage.googleapis.com
first ▶ try it The main Klipse JS plugin (~1.5MB) is injected into the page dynamically — deferred until actually needed
subsequent REPLs No further network requests — Klipse reuses the already-loaded compiler state

External resources loaded (2 total):

CSS https://storage.googleapis.com/app.klipse.tech/css/codemirror.css
JS (lazy) https://storage.googleapis.com/app.klipse.tech/plugin/js/klipse_plugin.js

Google Fonts (JetBrains Mono + DM Sans) also load on page open — those are for typography only, unrelated to the REPL.

How evaluation works step by step:

1. You click ▶ try it — toggleRepl() fires, opens the panel, calls loadKlipse()

2. loadKlipse() injects a <script> tag pointing at the Klipse plugin JS. This is a lazy load — the script only hits the network once, on first use

3. Klipse scans the DOM for any element with class language-klipse and wraps each one in a CodeMirror editor

4. window.klipse_settings (set before the script loads) tells Klipse: which selector to use, idle delay before eval (600ms), and CodeMirror options

5. When you stop typing, Klipse passes your code to the self-hosted ClojureScript compiler (cljs.js/eval-str internally), which compiles it to JavaScript and evaluates it via eval()

6. The result is rendered into the output CodeMirror pane below your code

State and isolation:

All REPL sessions on the page share a single compiler state — so (def x 10) in one card's REPL is visible if you type x in another card's REPL. This mirrors how a real Clojure REPL session works. Refreshing the page resets everything.

No data leaves your browser. Evaluation is 100% client-side. Nothing is sent to any server. The only network calls are the two asset loads above.
