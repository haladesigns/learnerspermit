Claude Code Commands & Skills

Commands and Skills are the same thing — Markdown files that become /slash-commands. Write instructions once, invoke them forever. They're how you stop re-typing the same multi-step workflows and start building a reusable AI toolkit that compounds across every project.

⚠️ Terminology Update (2025)

Claude Code merged commands and skills into a single system. A file at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md both create /deploy and work identically. The old .claude/commands/ path still works — Skills are recommended because they support supporting files and live-reload.

What a Command/Skill Does
📄
A Markdown file with YAML frontmatter. The filename (or name field) becomes the /slash-command.
⚡
Claude follows the instructions inside when the command is invoked — by you with /name, or automatically by Claude when the description matches the task.
💰
Lazy loading: the skill body only enters context when it's actually used. Long reference material costs zero tokens until invoked.
🔁
Version-controllable, shareable, evolvable. Check them into your repo and they're available to everyone on the project.
Commands vs Skills — Mental Model
Commands — you invoke
⚡You type /commit, /prime, /deploy — you decide when to run them.
⚡Multi-step workflows: deployments, commits, test runs, code generation.
⚡Use disable-model-invocation: true to prevent Claude from auto-running them.
Skills — Claude decides
📖Claude reads the description and loads the skill automatically when it's relevant.
📖Reference content: conventions, patterns, style guides, API playbooks.
📖Omit disable-model-invocation (default false) to let Claude decide.
Both are the same file format. The distinction is in how you think about what triggers them.
Where to Put Them — Scope
ScopePathAvailable To
Project PROJECT .claude/skills/<name>/SKILL.md
or legacy: .claude/commands/<name>.md		 Anyone working in this repo
Personal PERSONAL ~/.claude/skills/<name>/SKILL.md

Enterprise ENTERPRISE Managed settings (admin configured) All users in the organization
When names conflict: Enterprise > Personal > Project. Plugin skills use a plugin-name:skill-name namespace and never conflict.
💡 When to Create a Command

When you keep pasting the same playbook into chat — or when a section of CLAUDE.md has grown into a procedure rather than a fact — extract it into a command. If you do something more than twice, it should be a command.

Anatomy of a Command File

Every command is a Markdown file. The YAML block at the top controls behavior. Everything below it is the prompt Claude follows. That's it.

Skills Directory Structure
.claude/ └── skills/ └── commit/ ← skill name = /commit command ├── SKILL.md ← required — instructions + frontmatter ├── template.md ← optional — template for Claude to fill in ├── examples/output.md ← optional — expected output format └── scripts/validate.sh ← optional — script Claude can execute Legacy (still works): .claude/commands/ └── commit.md ← same as above, no supporting files
Complete SKILL.md Anatomy
---                                    ← YAML frontmatter block start
name: commit                           ← becomes /commit (or use dirname)
description: Create a standardized git commit with conventional message format.
                                       ↑ Claude reads this to decide when to auto-load
argument-hint: [message-prefix]       ← shown in autocomplete hint
disable-model-invocation: true        ← prevent Claude from auto-running this
allowed-tools: Bash(git *) Read       ← tools pre-approved, no per-use prompt
---                                    ← frontmatter block end

# Commit Command

## Everything below this is the prompt Claude follows.

Stage all changes and create a commit following conventional commits format:
- Use feat:, fix:, chore:, refactor:, docs: prefixes
- Keep subject line under 72 characters
- If $ARGUMENTS is provided, use it as the scope: feat($ARGUMENTS): ...

Run:
1. git status — confirm what will be staged
2. git add -A
3. git commit -m "<generated message>"
4. Output the final commit hash and message
The body is plain Markdown. Numbered lists, headers, code blocks — all valid. Claude follows it as a step-by-step procedure.
Arguments — How to Pass and Use Them
📥
$ARGUMENTS — captures everything typed after the command name as a single string. /commit feat → $ARGUMENTS = "feat"
📥
$1, $2, $3 — positional arguments. /fix-issue 42 high → $1 = "42", $2 = "high"
📋
Use argument-hint in frontmatter to show a hint in autocomplete: argument-hint: [issue-number] [priority]
# Example: fix-issue.md
---
argument-hint: [issue-number] [priority]
---
Fix GitHub issue #$1 with priority $2.
1. Read the issue details
2. Implement the fix
3. Write a test that covers it
4. Run /commit fix: resolve issue #$1
Dynamic Context Injection — Live Bash Output in Prompts

Prefix a line with ! to run a bash command and embed its output directly into the skill prompt at invocation time. Requires allowed-tools: Bash in frontmatter.

---
allowed-tools: Bash(git diff:*) Read Grep Glob
---
## Changed Files
!`git diff --name-only HEAD~1`

## Detailed Diff
!`git diff HEAD~1`

## Review Checklist
Review the above changes for:
1. Code quality and readability
2. Security vulnerabilities
3. Performance implications
4. Test coverage
The ! lines execute at invocation — Claude sees the actual diff output, not the command string.

Frontmatter Reference

All fields are optional. Only description is strongly recommended — it's how Claude knows when to load the skill automatically.

All Frontmatter Fields
FieldType / DefaultWhat It Does
name string / dirname The /slash-command name. If omitted, uses the directory name. Lowercase, hyphens, numbers only. Max 64 chars.
description string / first paragraph Strongly recommended. Claude uses this to decide when to auto-load the skill. Front-load the key use case. Truncated at 250 chars in the skill listing.
argument-hint string / none Shown during autocomplete to hint what arguments are expected. Example: [filename] [format] or [issue-number]
disable-model-invocation bool / false Set true to prevent Claude from auto-loading this skill. Use for commands you always want to trigger manually (deployments, commits, etc.).
user-invocable bool / true Set false to hide from the / menu. Use for background knowledge skills that users shouldn't invoke directly.
allowed-tools string or list / none Tools Claude can use without asking permission when this skill runs. Space-separated string or YAML list. Example: Read Grep Bash(git *)
context string / inline Set fork to run the skill in a subagent with its own context window. Results are summarized back to the main conversation.
agent string / general-purpose Which subagent to use when context: fork. Options: Explore, Plan, general-purpose, or any custom agent name from .claude/agents/.
model string / default Override the model for this skill. Example: claude-opus-4-6 for complex reasoning tasks.
allowed-tools Syntax
🔧
Read Write Edit Grep Glob — core file tools, no restrictions
🔧
Bash(git *) — allow any git command. Bash(npm run:*) — allow npm scripts only. Wildcards scope bash access.
🔧
Skill(commit) — allow invoking a specific other skill. Skill(review-pr *) — prefix match.
🚫
Your permission settings still govern everything else. allowed-tools only pre-approves — it doesn't grant access beyond your baseline permissions.
# Space-separated string
allowed-tools: Read Grep Bash(git diff:*)

# YAML list
allowed-tools:
  - Read
  - Grep
  - Bash(git diff:*)
⚠️ Description Budget

Skill descriptions are loaded into context so Claude knows what's available. The budget scales at 2% of the context window (fallback: 16,000 chars). If you have many skills and hit the limit, run /context to see a warning about excluded skills. Set SLASH_COMMAND_TOOL_CHAR_BUDGET env var to override.

Patterns & Advanced Usage

Beyond simple single-file commands — subagent delegation, supporting files, permission control, and the three content archetypes.

Three Skill Content Archetypes
1
Reference content — knowledge Claude applies to your current work
Conventions, patterns, style guides, domain knowledge. Runs inline alongside conversation context. Let Claude auto-load (no disable-model-invocation). 
---
description: API design conventions for this codebase. Use when writing endpoints.
---
When writing API endpoints:
- Use RESTful naming conventions
- Return consistent error formats
- Include request validation
2
Task content — step-by-step instructions for a specific action
Deployments, commits, code generation, migrations. These should be manually invoked — use disable-model-invocation: true. 
---
disable-model-invocation: true
description: Deploy to production
---
1. Run the test suite
2. Build the application
3. Push to production
3
Research content — subagent-powered exploration
Use context: fork to run in an isolated subagent. The research agent can load 100k+ tokens; only its summary returns to your main context. 
---
context: fork
agent: Explore
allowed-tools: Read Grep Glob Bash(git log:*)
---
Research the codebase for patterns related to: $ARGUMENTS
Return a concise summary of findings only.
Supporting Files — Beyond the Single SKILL.md

Skills (not legacy commands) support a full directory with optional supporting files. Reference them from SKILL.md so Claude knows what they contain.

📋
template.md — a document template Claude fills in. Reference with: "Fill in the template at template.md with the following information..."
🖼️
examples/output.md — expected output format showing Claude exactly what the result should look like. Few-shot examples in file form.
⚙️
scripts/validate.sh — a script Claude can execute (requires Bash in allowed-tools). Powerful for validation steps that need real shell output.
📚
reference.md — detailed reference documentation too large for SKILL.md itself. Claude loads it only when executing this specific skill.
Controlling Who Invokes a Skill
SettingEffectUse Case
(default) Claude auto-loads AND you can invoke manually Reference content, context-aware patterns
disable-model-invocation: true Manual /invoke only — Claude never auto-loads Deployments, commits, dangerous operations
user-invocable: false Hidden from / menu — background knowledge only Internal context Claude reads but users don't trigger
context: fork Runs in subagent — isolated context window Heavy research, large codebase analysis
Restricting Skill Access via Permissions
# Allow only specific skills
Skill(commit)
Skill(review-pr *)     ← prefix match with any arguments

# Deny specific skills
Skill(deploy *)        ← block all deploy variants

# Deny the entire Skill tool (disable all skills)
# In /permissions: deny the Skill tool
Use Skill(name) for exact match, Skill(name *) for prefix match with any arguments. Built-in commands like /compact and /init are NOT available through the Skill tool.

Built-in Commands Reference

Claude Code ships with built-in commands (fixed logic) and bundled skills (prompt-based). Both invoked with /name. You can't override built-in commands — but you can complement them with custom skills of different names.

Built-in Commands
CommandWhat It Does
/helpLists all available commands including your custom ones
/clearClears conversation history — use between tasks to protect context
/compactSummarizes conversation to free up context window
/contextShows context window usage and any excluded skills
/initInitializes Claude Code in a new project, creates CLAUDE.md
/modelSwitches the active model (Opus, Sonnet, Haiku)
/permissionsOpens the permissions management interface
/hooksConfigure hooks through an interactive menu
/install-github-appSets up Claude to auto-review pull requests on GitHub
Bundled Skills (Prompt-Based, Shipped with Claude Code)
SkillWhat It Does
/debugStructured debugging playbook — reproduces, diagnoses, and fixes bugs systematically
/simplifyRefactors selected code for readability following clean code principles
/batchRuns the same operation across multiple files in parallel
/loopRepeatedly executes a task until a condition is met
/claude-apiGenerates Anthropic API integration code for the current codebase
Bundled skills are prompt-based — Claude orchestrates the work using its tools. They're invoked the same way as your custom skills.
📐 Namespace tip

Use namespacing for related commands: /project:deploy, /project:rollback, or /git:commit, /git:pr. Claude Code supports this natively via subdirectories in .claude/skills/.

Example Command Library

Production-ready examples covering the most common use cases. Each shows the complete SKILL.md — copy, adapt, and add to your .claude/skills/ directory.

📍 /prime — Session Boot Command

Run at the start of every new implementation conversation. Reads PRD + rules, explores structure, checks git log, outputs mental model.

---
name: prime
description: Boot command for a new coding session. Reads PRD, explores structure, checks git log, builds mental model.
disable-model-invocation: true
allowed-tools: Read Grep Glob Bash(git log:*) Bash(git diff:*)
---

## Prime — Session Initialization

1. Read PRD.md and CLAUDE.md (global rules + on-demand context pointers)
2. Explore the top-level directory structure
3. Read key entry points (main, index, app, etc.)
4. Run: !`git log --oneline -20`
5. Run: !`git diff HEAD~3 --stat`

Output a written mental model summary:
- Current state of the codebase
- What was done recently (from git log)
- What the PRD says to build next
- Any risks or blockers you've identified
📋 /create-prd — Generate Product Requirements Doc

Converts a brain-dump conversation into a structured PRD. Run after the Q&A session.

---
name: create-prd
description: Convert the current conversation into a structured PRD document.
disable-model-invocation: true
allowed-tools: Write Read
---

Using the full conversation as context, create PRD.md with these sections:

## Required Sections
1. **Overview** — what we're building and why
2. **MVP Scope** — features included in v1
3. **Out of Scope** — explicitly what we are NOT building
4. **Tech Stack** — chosen technologies and rationale
5. **Architecture** — high-level structure and key decisions
6. **Directory Structure** — planned file/folder layout
7. **Phases of Work** — ordered implementation phases, each as a PIV loop
8. **Open Questions** — anything still unresolved

Write to PRD.md. Be comprehensive — the conversation context is temporary, only the PRD survives.
🗺️ /plan-feature — Feature Planning Document

Converts a vibe-planning conversation into a structured execution plan for a single PIV loop.

---
name: plan-feature
argument-hint: [feature-name]
description: Create a structured feature plan document for the next PIV loop.
disable-model-invocation: true
allowed-tools: Write Read Grep
---

Feature: $ARGUMENTS (or infer from conversation if not provided)

Create plans/<feature-name>.md with:

1. **Problem Statement** — what we're solving
2. **Context & References** — relevant docs, files, or APIs to load
3. **Implementation Plan** — ordered task list, specific enough to include filenames
4. **Validation Strategy** — exact steps to verify success before execution completes
    - Type checking / linting
    - Unit tests
    - Integration tests
    - End-to-end user journeys (be specific: list each flow)
5. **Environment Variables** — any new env vars required (add to .env.example)
💾 /commit — Standardized Git Commit

Creates a conventional commit message. Consistency = better git log = better /prime context.

---
name: commit
argument-hint: [scope]
description: Stage all changes and create a conventional commit message.
disable-model-invocation: true
allowed-tools: Bash(git *)
---

1. Run git status to see what will be staged
2. Run git add -A
3. Analyze the diff and choose the right prefix:
   - feat:     new feature
   - fix:      bug fix
   - refactor: code change, no feature/fix
   - chore:    maintenance, deps, config
   - docs:     documentation only
4. If $ARGUMENTS provided, use as scope: feat($ARGUMENTS): ...
5. Subject line: under 72 chars, imperative mood
6. Run git commit -m "<message>"
7. Output the commit hash and message
🔍 /review — Code Review

Injects the actual diff at invocation time, then runs a structured review checklist.

---
name: review
description: Code review of current changes: quality, security, performance, tests.
disable-model-invocation: true
allowed-tools: Read Grep Glob Bash(git diff:*)
---

## Changed Files
!`git diff --name-only HEAD~1`

## Full Diff
!`git diff HEAD~1`

## Review Checklist
Provide specific, actionable feedback organized by priority:

1. **Code quality** — readability, naming, structure
2. **Security** — injection risks, exposed secrets, auth bypasses
3. **Performance** — unnecessary queries, missing indexes, N+1s
4. **Test coverage** — are the changed paths tested?
5. **Compliance** — does this follow CLAUDE.md conventions?

Format: group by severity (Critical / Warning / Suggestion)
