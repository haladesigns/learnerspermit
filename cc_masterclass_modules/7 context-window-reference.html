Conversation Memory Management

Your context window is a finite resource — the single most important thing to manage in Claude Code. Every file read, command output, and back-and-forth fills it up. The 99% solution: keep your eye on usage and /clear aggressively. Update: Opus 4.6 and Sonnet 4.6 on the 1M token window show only ~2% drop per 100K tokens — the old "cliff at 50%" no longer applies to these models. Clear early and often still wins, but you now have real wiggle room when you need it.

💡 The 50% Rule — Updated

When memory hits 50%, ask: "Do I need this context for the next task?" If no → /clear. If yes and you're on Opus/Sonnet 4.6 with 1M window → you can keep going with confidence. If you're on an older model → /compact with focus now. Clear early and often still wins — but it's no longer a cliff if you don't.

Status Line Setup — Live Context Bar
Add a persistent context meter to the Claude Code prompt bar showing dir | model | ctx: X% used with warning markers at 50%, 75%, and 90%. One-time setup — three steps.
1
Create the script at ~/.claude/statusline-command.sh 
cat > ~/.claude/statusline-command.sh << 'EOF'
#!/usr/bin/env bash
# Claude Code pipes a JSON blob to this script on stdin every message.
# We slurp it all into $input so we can query it multiple times without re-reading stdin.
input=$(cat)

# Extract a quoted string value by key name from the JSON.
# grep -o isolates the "key": "value" pair; sed strips the key and both quotes.
# head -1 ensures we only get the first match (no duplicates).
# Used for string fields like "current_dir" and "display_name".
get_val() {
  echo "$input" | grep -o "\"$1\"[[:space:]]*:[[:space:]]*\"[^\"]*\"" \
    | head -1 | sed 's/.*:[[:space:]]*"//;s/"$//'
}

# Extract an unquoted numeric value by key name.
# Identical approach but the regex matches bare digits/decimals (no surrounding quotes).
# Used for "used_percentage", which arrives as a float like 43.7.
get_num() {
  echo "$input" | grep -o "\"$1\"[[:space:]]*:[[:space:]]*[0-9.]*" \
    | head -1 | sed 's/.*:[[:space:]]*//'
}

# Try "current_dir" first; fall back to legacy key "cwd"; default to "unknown".
# Claude Code has used both key names across versions — this handles either.
cwd=$(get_val "current_dir")
[ -z "$cwd" ] && cwd=$(get_val "cwd")
[ -z "$cwd" ] && cwd="unknown"

# "display_name" is the human-readable model label, e.g. "claude-opus-4-6".
model=$(get_val "display_name")
[ -z "$model" ] && model="unknown"

# "used_percentage" is a float (e.g. 43.7) representing context window fill level.
used=$(get_num "used_percentage")

# basename strips the full path to just the last directory component.
# e.g. /Users/you/projects/myapp → myapp
dir=$(basename "$cwd")

if [ -n "$used" ]; then
  # Bash's [ -ge ] only handles integers, not floats.
  # ${used%.*} is parameter expansion that drops everything from the decimal point onward.
  # e.g. "43.7" → "43". Safe and dependency-free (no awk/bc needed).
  used_int=${used%.*}

  # Threshold ladder: highest first so the most severe match wins.
  # 2>/dev/null suppresses the arithmetic error if $used_int is somehow empty/non-numeric.
  # [!] = heads-up at 50% · [!!] = act now at 75% · [!!!] = critical at 90%
  if   [ "$used_int" -ge 90 ] 2>/dev/null; then ctx_display="ctx: ${used}% used [!!!]"
  elif [ "$used_int" -ge 75 ] 2>/dev/null; then ctx_display="ctx: ${used}% used [!!]"
  elif [ "$used_int" -ge 50 ] 2>/dev/null; then ctx_display="ctx: ${used}% used [!]"
  else                                           ctx_display="ctx: ${used}% used"
  fi

  # printf (not echo) produces no trailing newline — correct for a status line widget.
  # Output format: "myapp | claude-opus-4-6 | ctx: 43.7% used"
  printf "%s | %s | %s" "$dir" "$model" "$ctx_display"
else
  # used_percentage was absent from the JSON (e.g. during session startup).
  # Show "--" so the bar still renders rather than going blank.
  printf "%s | %s | ctx: --" "$dir" "$model"
fi
EOF
2
Make it executable 
chmod +x ~/.claude/statusline-command.sh
3
Register it with Claude Code

📋 9 Steps to Master Memory Management

Follow these in order. Steps 1-4 are the core habit loop. Steps 5-9 are the level-up moves.

Core Habit Loop
1
Build the habit of checking
Run /context regularly — before a sub-task, after heavy file exploration, and whenever Claude's responses feel "off." 💡 Think of it like checking your rearview mirror. Make it reflexive.
2
Learn the three memory commands
/compact to summarize and continue · /compact focus on X for guided summarization (power move) · /clear for a completely fresh start.
3
Compact at strategic moments
After completing a task · after heavy file exploration · every 30-45 min in long sessions · after a successful test run. At ~50% on older models, compact immediately. On Opus/Sonnet 4.6 with 1M window, 50% is ~500K tokens — you have real room but peak quality is still at lower usage. ⚠ Don't compact mid-implementation or when Claude is already confused — clear instead.
4
Use /clear decisively
If you've corrected Claude more than twice on the same issue, the memory is polluted. Don't try to fix it — clear and rewrite a better prompt. Also: always clear between unrelated tasks. 💡 A clean session with a precise prompt outperforms a long session with corrections every time.
Level-Up Moves
5
Structure sessions as focused units
One session per task. Kitchen-sink sessions load Claude with irrelevant memory from three different tasks and cause drift. Finish a task → /clear → new task.
6
Be deliberate about what Claude reads
Name exact files in your prompts. Use .claudeignore to block irrelevant directories (~25% savings). Specific prompts prevent "explore everything" spirals. ⚠ "Improve this codebase" triggers broad scanning. "Add input validation to src/auth/login.ts" does not.
7
Use CLAUDE.md for persistent instructions
Anything in your conversation gets summarized or lost during compaction. Anything in CLAUDE.md survives every compaction and every new session. Keep it under 150-200 lines — ruthlessly pruned.
8
Use subagents to protect your memory
Delegate exploration to subagents. They run in their own memory and return only a summary to your main context. Ideal for codebase research, code review, edge case checking, and understanding unfamiliar code. 💡 "Use a subagent to investigate how our auth system handles token refresh. Report back with a summary."
9
Use /rewind for partial compaction
Press Esc twice (or run /rewind) to open a checkpoint menu. Select a message, choose "Summarize from here." Compresses a detour while keeping earlier setup intact.
🔄 5 Reusable Patterns

These are the most common session architectures for memory-efficient Claude Code usage. Pick the one that matches your task type.

P1 The Task Sandwich
Bookend every task with memory hygiene. The simplest and most universal pattern.
1./clear — clean slate before starting
2.Give your specific, scoped task prompt
3.Claude works → task complete → tests pass
4./compact keep only summary of what changed or /clear if done
P2 Research → Implement Split
Separate exploration from execution. Implementation starts clean, not loaded with 50 files of research.
S1.Enter Plan Mode → explore → create PLAN.md
 /clear
S2."Implement the OAuth flow from PLAN.md." → write tests → run them
P3 Checkpoint Compact
Compact at achievement milestones. Each compaction is a "save point" that preserves exactly what the next phase needs.
1."Set up database schema" → tests pass
 /compact preserve: schema done, tables are X and Y
2."Build API endpoints" → tests pass
 /compact preserve: GET/POST/PUT/DELETE all passing
3."Build frontend components"
P4 The Subagent Shield
Delegate exploration to protect main memory. Subagent explores 30+ files; your main memory only grows by the summary size.
1."Use a subagent to: 1) map files in src/payments/, 2) identify external services, 3) list tests. Report back with a summary."
2."Based on that report, refactor src/payments/processor.ts to…"
P5 Multi-Session Pipeline — Large Features
Break large features into focused sessions. Each starts fresh, reads only what it needs.
S1.Planning — analyze requirements, write plan to docs/plan.md
S2.Backend — implement per plan, commit when tests pass
S3.Frontend — implement per plan, commit when tests pass
S4.Integration — run full tests, fix issues, create PR
Gotchas — Common Traps
⚠️
Auto-compact at worst time: Triggers at ~95% when you're mid-task. Claude chooses what to keep — you don't. Fix: compact manually before hitting 95%. On older models, 50-60% is the target. On Opus/Sonnet 4.6 with 1M window, you have more runway — but manual compaction at a good save point always beats forced compaction mid-task.
⚠️
MCP servers silently eat memory: Each server adds tool definitions to every message. 3-4 servers = 10-20K tokens gone before you type anything. Fix: run /mcp to check costs; disconnect unused servers.
⚠️
"Explore everything" trap: Asking Claude to "look through the codebase" without constraints reads dozens of files. Fix: be specific. Use subagents for broad exploration.
⚠️
Compacting doesn't fix confusion: If Claude is going in circles, /compact compresses the confusion but doesn't resolve it. Fix: use /clear and start fresh with a better prompt.
⚠️
Bloated CLAUDE.md backfires: Over 150-200 lines and Claude can't hold it all in active focus — important rules get lost. Fix: ruthlessly prune. Ask "would removing this cause mistakes?" If not, cut it.
⚠️
Resuming a bloated session: claude --continue brings back the full history, including bloat. Fix: run /compact immediately after resuming.
✓ Good Prompts (Specific)
✅"Fix the validation bug in src/payments/checkout.ts"
✅"Add input validation to the login function in src/auth/login.ts"
✅"The checkout flow fails for expired cards. Check src/payments/, especially token refresh."
✅"Read src/auth/session.ts and explain the token refresh flow."
✗ Bad Prompts (Vague)
❌"Improve this codebase" — triggers broad file scanning
❌"What's wrong with the app?" — reads everything
❌"Understand the auth system" — reads auth + everything it imports
❌"Explore the codebase" — worst possible trigger for memory overflow
🆕 4.6 Update — The Cliff Is Gone (For These Models)

MRCR v2 8-needle benchmark (long context retrieval): Opus 4.6 scores 91.9 at 256K tokens and 78.3 at 1M tokens — a gradual ~2% drop per 100K, not a cliff. Sonnet 4.6 holds similarly (90.6 → 65.1). Compare: GPT-5.4 drops from 79.3 to 36.6, Gemini 3.1 Pro from 59.1 to 25.9, Sonnet 4.5 from 10.8 (barely registers). The mantra stays the same — clear early, clear often — but on Opus/Sonnet 4.6 with the 1M window, not clearing isn't a disaster if you genuinely need the full context.

💡 Prompt Caching vs. Memory

Claude Code caches repeated content to reduce costs. But cached tokens still count against memory space. Don't confuse cost savings with memory savings — they are different things.

💡 Plan Mode Is Free Memory

Plan Mode (Shift+Tab twice) lets Claude read files and think without committing to implementation. This prevents wasting memory on failed approaches before you've agreed on a direction.
