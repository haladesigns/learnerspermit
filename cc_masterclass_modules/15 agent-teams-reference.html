Agent Teams

Sub-agents on steroids. Instead of isolated workers reporting back to one boss, agent teams give every member a shared task list and direct messaging — so teammates talk to each other, not just the lead. Real engineers in a room, not interns at the library.

DIFFICULTY: ADVANCED PREREQS: MODULES 1–10, 16 READ TIME: ~12 MIN

📋 SHARED TASK LIST + MAILBOX
The lead creates teammates, assigns work, and pulls results together. You only talk to the lead — no managing chaos. Teammates communicate with each other directly via the shared mailbox, not through the lead as a relay.
Four Components
🧠
Team Lead — Your main Claude Code session. Creates the team, assigns work, coordinates results. You talk only to this.
⚙
Teammates — Separate Claude Code instances. Each has its own conversation memory, works independently, messages other teammates directly.
📋
Task List — Shared work items. States: pending → in progress → completed. Dependencies auto-unblock when prerequisites finish.
📬
Mailbox — Direct messaging between any agents. No bottleneck through the lead.
When Agent Teams Shine
🔬
Research with debate — Multiple agents test competing theories and actively try to disprove each other. Finds root causes faster than one agent guessing.
⚡
Parallel work across layers — Frontend, backend, and tests each owned by a different teammate working simultaneously.
🔍
Multi-angle review — Security, performance, and test coverage reviewers all examining the same code at the same time.
🆕
New models or features — Research and recommendation across multiple competing approaches.
Sub-Agents vs. Agent Teams
Dimension Sub-Agents Agent Teams
Communication Report back to main agent only Message each other directly
Coordination Managed entirely by main agent Self-coordinate via shared task list
Best for Focused tasks where only the result matters Work requiring discussion + collaboration
Cost Cheaper (results summarized back) More expensive (each = full instance)
Nesting Cannot create sub-agents Cannot create sub-teams
Analogy Interns sent to the library Senior engineers in a room
💡 Rule of Thumb

Use sub-agents when you need quick, focused workers that report back. Use agent teams when teammates need to share findings, challenge each other, and coordinate on their own.

Enabling Agent Teams

Agent teams are disabled by default. One environment variable is all it takes. Recommend settings.json — it persists across sessions.

Option 1 — settings.json (Recommended)
// Add to your Claude Code settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
Persists across all sessions. Recommended approach.
Option 2 — Shell Environment
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
Session-only. Must re-export each terminal session.
💡 Quick Enable Trick

Copy the documentation page, paste it into Claude Code, and say: "Enable agent teams for me." Claude will update the settings automatically.

Step-by-Step: Your First Team
1
Enable the feature
Add CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 to your settings.json (see Setup tab).
2
Use explicit language
You must say "create an agent team" — Claude will not spawn a team on its own just because the task is large.
Try this: "I'm designing a CLI tool that helps developers track TODO comments. Create an agent team to explore this from different angles: one teammate on UX, one on technical design, one playing devil's advocate."
⚠ Saying "do multiple tasks" or "use sub-agents" will NOT create a team. You must explicitly say "agent team."
3
Watch the team form
The lead's terminal shows all teammates, their tasks, and status. Claude creates the team automatically — no manual management needed. 💡 The lead shows a breakdown of the team structure and scope as it creates each teammate.
4
Navigate between teammates
ActionShortcut
Cycle through teammates Shift+Down (wraps back to lead)
View teammate's full session Enter
Interrupt a teammate's turn Escape
Toggle the task list Ctrl+T
5
Message a specific teammate directly
Press Shift+Down to cycle to the teammate you want, then type your message.
Example: "Also check the session token rotation logic — I think there's a race condition when two tabs are open simultaneously."
6
Clean up when done
Always use the lead to clean up, not a teammate. Shut down all teammates first.

Task Sizing
✅
Just right — Self-contained units with clear deliverables. E.g., build one API endpoint, write tests for one module. Aim for 5–6 tasks per teammate.
⚠️
Too small — Coordination overhead exceeds benefit. E.g., renaming a variable.
⚠️
Too large — Teammates work too long without check-ins. E.g., "refactor the entire app."
Team Sizing
⭐
Start with 3–5 teammates for most workflows. Three focused teammates often outperform five scattered ones.
📈
Scale up only when the work genuinely benefits from more parallel sessions.
🚫
No nested teams — teammates cannot create their own teams or sub-teams.
Context and Permissions
🔐
Permissions — Teammates start with the lead's permission settings. You can change individual teammate modes after creating them. Pre-approve permissions before creating teammates to avoid friction.
📚
Context loading — Each teammate loads the same project context (CLAUDE.md files, MCP servers, Skills) as a regular session.
⚠️
No conversation inheritance — Teammates do NOT inherit the lead's conversation history. Include enough context in the task description for each teammate to do its job.
Good context example: "Create a security reviewer teammate with the prompt: 'Review the authentication module at src/auth/ for security vulnerabilities. Focus on token handling, session management, and input validation. The app uses JWT tokens stored in httpOnly cookies.'"
📖 Usage Note

These are the five core patterns from the official docs. Start with Pattern 1 or 2 if you're new to agent teams — review and investigation are safer than parallel code changes for beginners.

📝 Pattern 1 — Parallel Code Review
Split review criteria into independent areas so security, performance, and test coverage all get thorough attention simultaneously.
Try this prompt: "Create an agent team to review PR #142. Three reviewers: one focused on security implications, one checking performance impact, one validating test coverage. Have them each review and report findings."
🔬 Pattern 2 — Adversarial Debugging
When the root cause is unclear, multiple agents testing competing theories — and actively trying to disprove each other — find the answer faster than one agent guessing.
Try this prompt: "Users report the app exits after one message instead of staying connected. Create 5 agent teammates to investigate different hypotheses. Have them talk to each other to try to disprove each other's theories, like a scientific debate. Update the findings doc with whatever consensus emerges."
🏗 Pattern 3 — Cross-Layer Feature Build
When a feature spans frontend, backend, and tests, each layer can be owned by a different teammate. Use task dependencies to sequence work — backend schema first, then frontend and tests.
Try this prompt: "Build the user notification system. Create an agent team: 'backend' teammate builds the notification API endpoints and database schema, 'frontend' teammate builds the notification bell component and dropdown UI, 'tests' teammate writes integration tests for the notification flow. The backend teammate should finish the API schema first, then message the frontend and test teammates so they know what endpoints to hit."
⚠ Make sure each teammate works on different files to avoid conflicts — two teammates editing the same file leads to overwrites.
🔭 Pattern 4 — Research and Recommendation
Multiple teammates investigate different options simultaneously. The lead picks the best approach based on findings.
Try this prompt: "We need to add real-time updates to the dashboard. Create an agent team: one teammate researches WebSockets, one teammate researches Server-Sent Events, one teammate researches polling with smart intervals. Have each write a recommendation, then the lead picks the best approach based on our Next.js + Supabase stack."
✅ Pattern 5 — Plan Approval Workflow
For risky tasks, require teammates to get their plan approved before implementing. Useful when you need to validate approach before any files are touched.
Try this prompt: "Create a teammate to refactor the authentication module. Require plan approval before they make any changes."
With criteria: "Only approve plans that include test coverage. Reject any plan that modifies the database schema."
Gotchas
⚠️
No session resumption with teammates. After resuming a session, the lead may try to message teammates that no longer exist. Tell it to create new ones.
⚠️
Task status can lag. Teammates sometimes fail to mark tasks as completed. Check manually and update if needed.
⚠️
One team per session. You cannot have multiple active teams in one session. Clean up before starting a new one.
⚠️
File conflicts are real. Two teammates editing the same file leads to overwrites. Break work so each teammate owns different files.
⚠️
The lead sometimes starts doing work itself. If the lead starts implementing tasks instead of delegating, say: "Wait for your teammates to complete their tasks before proceeding."
⚠️
Lead may finish too early. It may decide work is done before all tasks complete. Tell it to keep going or explicitly wait for teammates.
⚠️
No nested teams. Teammates cannot create their own teams. One level only.
✓ Best Practices
✅Start with research/review tasks before trying parallel code changes.
✅Pre-approve permissions before creating teammates.
✅Monitor actively — check progress, redirect bad approaches, review findings as they arrive.
✅Include full context in each task description (teammates don't inherit lead's history).
✅Aim for 5–6 tasks per teammate, 3–5 teammates per team.
✅Assign each teammate distinct files to own.
✗ Things to Avoid
❌Letting a team run unattended too long.
❌Using for routine or small tasks — single session is more cost-effective.
❌Assigning overlapping file ownership to multiple teammates.
❌Expecting teammates to know the lead's conversation history.
❌Cleaning up from a teammate instead of the lead.
❌Forgetting to shut down all teammates before cleaning up.
Troubleshooting
Problem Fix
Teammates not appearing Press Shift+Down — they may be running but not visible. Or task may not be complex enough. For split panes, run: which tmux
Teammates stopping on errors Check their output (Shift+Down), give recovery instructions, or create a replacement teammate.
Lead finishes too early Tell it to keep going or explicitly say "Wait for all teammates to complete before proceeding."
Teammates messaging dead instances Occurs after session resume. Tell the lead to create new teammates.
