Karpathy's Obsidian "RAG"

A personal knowledge base using Obsidian + Claude Code — no vector database, no embeddings, no code. The LLM navigates a structured markdown wiki via hierarchical index files, answering questions with 3–4 targeted reads instead of semantic search.

Core Insight
🗂️
Hierarchy replaces vectors. An _master-index.md and per-topic _index.md files let the LLM navigate to any answer in 3–4 reads.
📥
raw/ is the inbox. Dump anything — articles, PDFs, tweets. Claude Code processes ("compiles") them into structured wiki articles.
📚
wiki/ is the LLM's domain. You rarely touch it directly. Claude writes and maintains all articles, links, and indexes.
🔁
Queries compound. Every answer can be saved back into the wiki — the knowledge base gets smarter over time.

How the LLM Navigates (No Vector DB)

When you ask a question, Claude Code does 3–4 targeted reads — not a full scan:

Read 1 _master-index.md "What topics exist?"
→
Read 2 topic/_index.md "What articles exist?"
→
Reads 3–4 specific articles 1–3 targeted docs
→
Output synthesized answer optionally saved to wiki
💡 Key Takeaway

Scalability comes from the index structure, not embeddings. Each read is cheap. This works reliably for hundreds or low thousands of documents.

7-Step Setup
1
Install Obsidian
Download from obsidian.md (free, Mac/Windows/Linux). Open → "Create new vault" → name it → pick a save location. 💡 A vault is just a folder of markdown files on your computer — nothing proprietary.
2
Create the folder structure via Claude Code
Open a terminal in your vault folder, run claude, and send this prompt: 
# Tell Claude Code to scaffold the vault
Create this folder structure for my knowledge base:

- raw/       — inbox where I'll dump source material
- wiki/      — your domain, you'll maintain everything here
  - Create wiki/_master-index.md with heading "Knowledge Base Index"
    and note "Topics will be listed here as they are created."
- output/    — query results and reports

Just create the folders and that one file. Nothing else yet.
3
Create CLAUDE.md in vault root — the most important step
This file is read every Claude Code session. It makes Claude your permanent librarian. See the CLAUDE.md tab for the full template. ⚠ Without CLAUDE.md, Claude won't know the conventions. Do this before compiling anything.
4
Install Obsidian Web Clipper (browser extension)
Extension store → search "Obsidian Web Clipper" → install → gear icon → General → add vault.
Then: Templates → edit Default template: 
Vault:         your-vault-name
Note location: raw
Note name:     {{date|date:"YYYY-MM-DD"}}-{{title|safe_name}}
✓ Any article on the web — one click — markdown file in raw/.
5
Install Local Images Plus (optional, recommended)
Obsidian: Settings → Community Plugins → Browse → search "Local Images Plus" → Install → Enable.
💡 When you open a clipped note, this plugin auto-downloads all images into your vault so nothing breaks when offline. ⚠ Community Plugins must be enabled first: Settings → Community plugins → toggle off "Restricted mode".
6
Your first compile
Clip 3–5 articles into raw/, then tell Claude Code: 
Compile everything in raw/ into the wiki. For each file:
1. Read it and identify the core topic
2. Create or find the right topic folder in wiki/
3. Write a wiki article with summary, key takeaways, and [[wiki links]]
4. Update the topic's _index.md
5. Update wiki/_master-index.md

Cross-link between topics wherever relevant.
✓ Open Obsidian — you'll see a structured wiki appear.
7
Audit / health check (run periodically)
Audit the wiki. Review every topic for:
- Inconsistent or contradictory information across articles
- Missing cross-links between related concepts
- Gaps in coverage — topics mentioned but no articles yet
- Suggest 3–5 new articles that would strengthen the knowledge base

Don't make changes yet — just give me the report.
💡 The audit prompt is the maintenance step that keeps quality compounding over time.
✓ Setup Complete When

You have raw/, wiki/ with _master-index.md, output/, a CLAUDE.md in vault root, and Web Clipper configured to drop into raw/.

CLAUDE.md

Place this file in your vault root. Claude Code reads it at the start of every session — it's how you set the rules once and have them followed forever. This is the most important file in the system.

Full Template (copy-paste ready)
# Knowledge Base — Vault Conventions

## Vault Structure
- /raw    — source material, clipped articles, research (input zone)
- /wiki   — LLM-compiled knowledge base (see Wiki System below)
- /output — query results and generated reports

## Wiki System
You are the librarian of the wiki/ folder. You write and maintain everything in it.

### Structure
- wiki/_master-index.md is the entry point — lists every topic with a one-line description.
- Each topic gets its own subfolder with its own _index.md listing all articles.

### Compiling
When I say "compile" or dump new material in raw/:
1. Read each raw file
2. Decide which topic it belongs to (or create a new topic folder)
3. Write a wiki article with key takeaways and [[wiki links]] to related concepts
4. Update that topic's _index.md
5. Update wiki/_master-index.md
6. If a raw file spans multiple topics, create articles in both and cross-link

### Querying
When answering questions against the knowledge base:
1. Read wiki/_master-index.md first to find the right topic
2. Read that topic's _index.md to find relevant articles
3. Read the specific articles
4. Synthesize the answer

### Auditing
When I say "audit" or "lint", review the wiki for:
- Inconsistent or contradictory information
- Missing cross-links between related concepts
- Gaps in coverage
- Suggest improvements, but don't make changes without confirmation

## Conventions
- Always use [[wiki links]] when referencing other notes
- File names: lowercase with hyphens (e.g., ai-agent-overview.md)
- Keep articles concise — bullet points over paragraphs
- Always include a ## Key Takeaways section in every wiki article
What Each Section Does
🗺️
Vault Structure — tells Claude which folder does what. Without this, it may write to the wrong place.
📚
Compiling section — the 6-step algorithm Claude follows every time you say "compile." Ensures consistent article format.
🔍
Querying section — enforces the 4-read navigation pattern so Claude doesn't try to read everything at once.
🔎
Auditing section — keeps Claude from making unsanctioned edits during an audit. Report-only mode.
📝
Conventions — ensures [[wiki links]] are used (enables Obsidian graph view), consistent filenames, and article structure.
Gotchas
⚠️
CLAUDE.md must be in vault root, not inside raw/ or wiki/. Claude Code looks for it at the project root.
⚠️
Don't skip the Querying section. Without it Claude may scan every file for each question, which is slow and expensive.
⚠️
Audit ≠ edit. The audit prompt says "don't make changes without confirmation" — include that line or Claude will start editing.
⚠️
_master-index.md freshness. If Claude skips updating it, later queries will miss new topics. Check it after compiles.
Query Patterns
Simple Lookup

Cross-Reference
How does [concept A] relate to [concept B] based on everything in the wiki?
Synthesis — saves back to wiki ⭐
Based on everything in the wiki, what are the main approaches to [topic] and how do they compare? Save your answer as a new wiki article and link it to the sources you referenced.

Obsidian RAG vs True RAG

Obsidian RAG is a personal knowledge base for solo operators. True RAG (vector DB + embeddings) is infrastructure for teams and production apps. Pick based on scale and use case.

✓ Obsidian RAG Wins
✅Zero infrastructure — markdown files in a folder. No vector DB, no embeddings, no hosting.
✅Setup in 15 minutes — you just did it.
✅Human-readable — you can browse and read your own wiki. A vector store is a black box.
✅Self-improving — audit step means quality gets better over time.
✅Outputs compound — every query adds to the knowledge base.
✅You get smarter — you actually read and learn from the wiki, not just get chunks back from a database.
✗ When It Breaks Down
❌5,000+ documents — cross-document discovery needs semantic search that indexes can't provide.
❌Bulk ingestion — compiling 500 new docs takes hours of LLM processing. True RAG chunks and embeds fast.
❌No semantic fallback — if the index missed it, your query misses it too.
❌Multi-user / production — this is a personal system. Teams and production apps need real infrastructure.
Comparison Table
Dimension Obsidian RAG True RAG (vector DB)
Setup time 15 minutes Hours to days
Infrastructure Markdown folder Vector DB + embedding pipeline + hosting
Doc scale sweet spot 0–2,000 docs Thousands to millions
Ingestion speed Slow (LLM per doc) Fast (chunk + embed)
Search type Hierarchical index nav Semantic similarity
Human-readable? Yes — it's markdown No — vector embeddings
Self-improving? Yes — audit + compile Requires re-indexing
Multi-user Personal only Designed for teams
Cost LLM tokens only Tokens + DB hosting + embeddings
Bottom Line

For solo operators, creators, researchers, and small teams — Obsidian RAG is the move. If you ever outgrow it (5,000+ docs, multi-user, production), that's a good problem to have. Start here.
