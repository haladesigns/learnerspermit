Concepts

What does it mean to be an IdP or SP?

The roles are defined by ownership — not by flow direction. One party owns the identity store. The other owns the protected resource. They never share secrets directly: trust is established out-of-band via metadata exchange, then verified cryptographically at runtime.

Identity Provider (IdP)
Okta — "I know who you are"
Owns the user directory. Provides authentication (password, MFA). Signs SAML assertions with its private key. Knows which Service Providers it is allowed to assert identity to.
Service Provider (SP)
F5 APM — "I have what you want"
Owns the protected resource (app, VPN, webtop). Has no user database. Outsources authentication entirely to the IdP. Accepts the IdP's signed assertion as proof of identity.
The passport analogy
🏛️
Okta
Passport office — issued your credential, signs it
🚪
F5 APM
Bouncer — doesn't know you, trusts the passport office
📄
SAML assertion
Your passport — signed proof of identity
👤
User
Just trying to get in — browser carries everything

The bouncer (F5) doesn't re-verify identity from scratch. He checks the passport is genuine (valid signature), checks it's not expired, and lets you in. The user's browser acts as the carrier — it never sees the raw assertion content in a meaningful way, it just forwards the signed blob from Okta to F5.

Flow comparison

SP-initiated vs IdP-initiated

The roles never swap. Okta is always the IdP, F5 is always the SP. What changes is which party fires first — and that has a concrete security implication on the F5 side.

SP-initiated (recommended)
User hits F5-protected URL with no session
F5 builds a signed AuthnRequest and 302-redirects to Okta
Okta authenticates user, builds assertion with InResponseTo referencing the AuthnRequest ID
F5 can correlate the assertion to a specific request — stronger security
F5 remembers the original URL and redirects back to it after auth
IdP-initiated (use with care)
User clicks app tile in Okta portal
Okta builds assertion unprompted — no AuthnRequest was made
Assertion has no InResponseTo attribute — F5 cannot correlate it to a prior request
F5 must have allow unsolicited responses enabled on the SP Service object
No original URL to return to — F5 redirects to a configured default landing URI
Security trade-off
IdP-initiated flows are weaker because F5 accepts an assertion it never asked for. An attacker who can replay a captured assertion has a wider window to exploit. If IdP-initiated is required, ensure assertion validity windows are tight (Okta default is 5 minutes) and that HTTPS is enforced end-to-end.
Stage 1

Certificates — three distinct roles

Three separate certificates are involved. They serve completely different purposes. Conflating them is one of the most common setup mistakes.

Certificate Where it lives on F5 Purpose Key pair?
SP signing cert + key
 System → Certificate Management → Traffic Cert Management APM signs outbound AuthnRequests. Public cert exported in SP metadata and given to Okta so Okta can verify the request came from this SP. Yes — cert + key
Okta IdP signing cert
 Imported via IdP metadata XML into External IdP Connector APM uses this to verify the cryptographic signature on every incoming SAML assertion. Proves the assertion genuinely came from Okta and wasn't tampered with. No — public cert only
VS SSL cert + key
 Traffic Cert Management → used in Client SSL Profile Terminates HTTPS for the client-facing virtual server. Completely separate from SAML signing. Must be CA-trusted. Yes — cert + key
Self-signed is fine for the SP cert
Okta does not validate the certificate authority of the SP signing cert — it only uses the public key to verify signatures. A self-signed cert is fully acceptable here. The VS SSL cert, however, must be CA-trusted because browsers validate it.
Stages 2 – 4

Federation objects — the trust chain

Three objects built in sequence. The order matters — the binding in Stage 4 references both objects from Stages 2 and 3, so both must exist first.

02 External IdP connector Access → Federation → SAML: SP → External IdP Connectors ▶

Tells APM who Okta is. Import Okta's metadata XML to populate all fields automatically — recommended over manual entry.

IdP entity ID
Populated from metadata XML
Okta's unique identifier, e.g. http://www.okta.com/<app-id>. Used to verify the Issuer field in incoming assertions.
SSO URL
Populated from metadata XML
The Okta endpoint APM redirects users to during SP-initiated flow, with the SAMLRequest query parameter attached.
SSO binding
Typically: HTTP Redirect
HTTP Redirect binding is standard for the AuthnRequest (GET). HTTP POST binding is used for the assertion response in the opposite direction.
IdP signing certificate
Imported from Okta metadata XML
Okta's public cert. APM uses this to verify the signature on every assertion. No private key on the F5 side.
03 Local SP service Access → Federation → SAML: SP → Local SP Services ▶

Defines F5 APM's identity as a service provider. After saving, export the SP metadata XML and upload it to Okta — this completes the out-of-band trust exchange.

SP entity ID
Must match Okta exactly — case sensitive
Typically the VS hostname, e.g. https://myapp.example.com. Trailing slashes matter. Character-for-character match required.
ACS URL
Must match Okta exactly
Where Okta POSTs the assertion. Format: https://myapp.example.com/saml/sp/profile/post/acs
SP cert + key
References cert from Stage 1
The keypair from Stage 1. Used to sign outbound AuthnRequests when "Sign AuthnRequest" is enabled.
Security settings
Signing + assertion requirements
Sign AuthnRequest: yes. Require signed assertions: yes. Allow unsolicited responses: only for IdP-initiated flows.
! After saving, export the SP metadata XML and upload to Okta before testing. Without this, Okta will reject the AuthnRequest as coming from an unknown SP.
04 SP ↔ IdP binding Local SP Services → [edit] → IdP Connectors tab ▶

The binding joins the two objects together and completes the trust pairing. Without it, both objects can be perfectly configured and SAML will still fail silently.

IdP connector reference
Local SP Service → IdP Connectors tab → Add
Select the External IdP Connector from Stage 2. This tells APM: "when a user needs to authenticate via this SP, redirect them to that IdP." The binding is directional and does not apply automatically in reverse.
Stage 5

Access policy — the VPE flow

The Visual Policy Editor (VPE) is where authentication logic lives. The SAML Auth agent is the engine; Variable Assign is the post-processing step that makes assertion attributes usable downstream.

— Access profile + VPE agents Access → Profiles/Policies → Access Profiles ▶

The policy flow executes on every new connection before a session exists. Place the SAML Auth agent first, then Variable Assign, then Allow/Deny endings.

SAML auth agent
+ Add Item → Authentication → SAML
References the Local SP Service from Stage 3. On no-session: triggers the AuthnRequest redirect to Okta. On assertion POST: resumes the policy with session.saml.last.* populated.
Variable assign agent
+ Add Item → General Purpose → Variable Assign
Maps assertion attributes to standard APM session variables used for group-based access control and downstream policy decisions.
Key session variables after SAML auth
session.saml.last.nameIDValue
→ username / subject identifier
session.saml.last.attr.email
→ email attribute (if sent by Okta)
session.saml.last.attr.groups
→ group membership for ACLs
session.saml.last.authinstant
→ timestamp of authentication
! Okta sends attribute names as full URI strings by default (e.g. http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress). Check the exact attribute name in the SAML tracer before referencing it in Variable Assign — the friendly Okta alias does not carry through to the assertion XML.
Stage 6

Virtual server — wiring it together

The virtual server is the entry point for all user traffic. Every configuration object built in Stages 1–5 ultimately connects here, directly or via the Access Profile attachment.

VS type + address
Local Traffic → Virtual Servers → General Properties
Standard type. IP:port matches the hostname in your ACS URL and SP Entity ID. Port 443 required for HTTPS.
HTTP profile
Local Traffic → Profiles → Services → HTTP
Required for APM to process HTTP-layer elements: session cookies, redirect responses, and the POST body containing the SAMLResponse.
SSL profile (client)
Local Traffic → Profiles → SSL → Client
Terminates HTTPS from the client. References the VS SSL cert+key from Stage 1. Mandatory — assertions must be encrypted in transit.
Access profile (attached)
VS → Access Policy section → Access Profile field
The attachment point for the Access Profile from Stage 5. This is what connects live traffic to the SAML policy. Without this, no authentication occurs.
Reference

Common failure points

The most frequent causes of SAML failures that work in config but break at runtime. Most produce unhelpful generic errors — knowing these in advance saves hours of tracing.

Entity ID / ACS URL mismatch
The SP Entity ID and ACS URL on F5 must be character-for-character identical to what is registered in Okta. Trailing slashes, HTTP vs HTTPS, subdomain casing — all count. Okta will reject the assertion or return a generic error if there is any discrepancy.
Fix: compare the SP metadata XML exported from F5 against the Okta app settings side-by-side.
Missing SP ↔ IdP binding
Both the External IdP Connector and Local SP Service can be perfectly configured, and SAML auth will still silently fail if the binding step (Stage 4) was skipped. APM has no IdP to redirect users to.
Fix: Local SP Services → edit → IdP Connectors tab → confirm the connector is listed.
Clock skew between F5 and Okta
SAML assertions include a NotBefore and NotOnOrAfter timestamp with a validity window (typically 5 minutes). If BIG-IP's system clock is more than a few minutes out of sync with Okta's, assertions will be rejected as expired or not-yet-valid.
Fix: NTP is not optional. Verify with date on the BIG-IP bash prompt and compare against an external time source.
Attribute name format mismatch
Okta sends attribute names in full URI format by default. Variable Assign agents in the VPE must reference the exact wire-format attribute name — the friendly Okta alias does not carry through to the assertion XML.
Fix: use the browser SAML tracer extension to inspect the raw assertion and copy the exact attribute name strings.
Unsolicited response not enabled for IdP-initiated
If IdP-initiated flow is required, the Local SP Service must have "Allow Unsolicited Responses" enabled. Without it, APM rejects assertions that arrive without a corresponding InResponseTo value — which is all IdP-initiated assertions.
Fix: Local SP Services → [edit] → Security Settings → Allow Unsolicited Responses: enabled.
SP metadata not uploaded to Okta
After creating the Local SP Service, the SP metadata XML must be exported and uploaded to the Okta app configuration. Okta uses it to validate the AuthnRequest signature and to know which ACS URL to POST the assertion to.
Fix: Local SP Services → [select] → Export Metadata → upload to Okta app → Sign On → SAML settings.
