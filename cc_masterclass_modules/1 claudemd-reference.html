CLAUDE.md

A markdown file that Claude Code reads automatically at session start. It's injected into every single prompt — so whatever's in it, Claude always has it. Think of it as a cheat sheet handed to a brilliant assistant on day one.

The Core Problem It Solves
🔄
Every Claude Code session starts with a blank slate — no memory of your stack, commands, or conventions.
📄
CLAUDE.md is auto-loaded before you type anything. It becomes Claude's background knowledge.
⚡
Appended to every prompt — even just typing "hi" sends the full file along with it.
The Research Warning
>20%
Inference cost increase when using agents.md
↓
Agent task completion rate reduced
Feb 2026 study: unnecessary context files made tasks harder, not easier. Like handing someone a policy doc to read before every single task — relevant or not.
🧪 The Litmus Test (for every line)

Ask yourself: "If I deleted this line, would Claude actually do something wrong?" If the answer is no — delete it.

The Honest Question to Ask
🤔
"Does this need to be in every prompt?" If you'd only need to say it occasionally, don't put it in CLAUDE.md — say it when you need it.
📏
Less is more. A lean 35-line file outperforms a bloated 300-line one. Context pollution degrades performance.

What Goes Wrong
📏
Hard limit: ~60 lines of content. Claude can follow ~100–150 instructions total; the system already uses some of that budget. Go over and Claude starts ignoring rules across the board — not just the new ones.
🔧
Don't use CLAUDE.md as a linter. If a tool can enforce a rule (formatting, import order, semicolons), use the tool. Put it in pre-commit hooks, not here.
🤖
Don't just accept /init output. Auto-generated files include obvious stuff Claude can infer from your code. Treat it as a rough draft — manually review every single line.
🚨
Use emphasis sparingly. Adding "IMPORTANT" or "YOU MUST" helps for critical rules. But if everything is marked IMPORTANT, nothing is.
🕰️
Stale files degrade performance. Review every few weeks. Delete stale instructions, consolidate duplicates, clarify anything confusing.
🧪 The Deletion Test

For every line: "If I deleted this, would Claude actually do something wrong?" If no → delete it. This is the single most effective way to keep the file lean.

What to Do Right
✓
Commit CLAUDE.md to git. It's a team resource. Everyone benefits from shared context.
✓
Use # shortcut during sessions to grow the file organically from real pain points.
✓
Use @imports to keep the root file short while giving Claude access to detailed docs.

Setup: Steps 1–7

The full sequence for creating your first CLAUDE.md from scratch — or validating an existing one.

Step-by-Step Walkthrough
1
Bootstrap with /init (recommended)
Open Claude Code in your project and run /init. Claude scans your codebase and generates a starter CLAUDE.md.
⚠ Treat it as a rough draft — auto-generated files include too much. Review and cut every line.
2
Or create the file manually
If starting from scratch: touch CLAUDE.md in your project root. Skip if you used /init.
3
Add your project identity
One short paragraph: what the project is, the tech stack, and any key architectural patterns Claude needs to know upfront. 
# Project
Next.js 14 e-commerce platform with TypeScript,
Tailwind CSS, and Supabase. Uses App Router with
server components by default.
4
Add essential commands
The commands Claude needs to build, test, lint, and type-check your project. 
# Commands
- npm run dev      — Start dev server on port 3000
- npm run build    — Production build
- npm test         — Run test suite
- npm run typecheck — TypeScript type checking
5
Add code style rules
Only include rules Claude wouldn't figure out on its own from reading your codebase. Skip anything a linter enforces automatically. 
# Code Style
- Use ES modules (import/export), never CommonJS
- Named exports only, not default exports
- Prefer interface over type for object shapes
6
Add workflow rules and gotchas
Non-obvious process rules and traps that would cause real mistakes if Claude didn't know them. 
# Workflow
- Always run typecheck after making code changes
- Never commit directly to main

# Gotchas
- NEXT_PUBLIC_ prefix required for client-side env vars
- Supabase RLS enabled — new tables need policies
  before any queries will work
7
Point to detailed docs + commit
Don't cram everything in. Reference docs for deeper detail, then commit so the whole team shares the same baseline. 
# Further Reading
IMPORTANT: Read relevant doc before starting work there.
- @docs/api-conventions.md   — API patterns, error handling
- @docs/database-schema.md  — Full schema + relationships
Then: git add CLAUDE.md → git commit -m "Add CLAUDE.md for project context"
✅ Done? Run the deletion test on every line

"If I deleted this line, would Claude actually do something wrong?" If no — cut it. A lean file that passes this test beats a thorough one that doesn't.
