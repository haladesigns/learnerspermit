01
Your Hardware
What the AGX Orin 64GB actually gives you

The AGX Orin 64GB is the high-end Jetson — the same silicon NVIDIA points to when they say "frontier-class edge AI." The defining feature isn't raw TOPS; it's the 64 GB of unified LPDDR5 shared between CPU and GPU. No PCIe transfers, no VRAM partition. Whatever fits in system memory can be operated on by the GPU at ~205 GB/s.

The bandwidth-bound truth LLM inference is memory-bandwidth-bound, not compute-bound. The AGX Orin's 205 GB/s is what really sets your token/sec ceiling — not the 275 TOPS marketing figure. For comparison: an RTX 4090 has ~1 TB/s, ~5× higher. That's why a 4090 generates more tokens/sec than an AGX Orin even though the Orin can hold larger models.
Why unified memory wins On a discrete GPU, model weights have to fit in VRAM (24 GB on a 4090). On the AGX Orin you have 64 GB to play with — you can comfortably load a quantized 31B dense model with room left over for KV cache, system processes, and a desktop environment. This is the AGX Orin's superpower.

Model Fit & Expected Speed
What runs, how fast, how much memory

Three numbers matter for any model: memory footprint (weights + KV cache), memory bandwidth needed per token (roughly = footprint), and active parameters (different from total for MoE). Divide bandwidth ÷ footprint and you get a rough tokens/sec ceiling.

ModelQuantVRAMtok/s (est)Verdict
Gemma 4 E2B Q4 ~3 GB 45–60 Snappy. Good for testing pipelines.
Gemma 4 E4B Q4 ~5 GB 25–35 Sweet spot for chat-quality + speed.
Gemma 3 12B Q4 ~8 GB 15–22 Solid quality, comfortable.
Gemma 4 26B-A4B (MoE) Q4 ~16 GB 15–25 Big-model quality at small-model speed (3.8B active).
Gemma 3 27B Q4 ~17 GB 7–12 Usable but slow. MoE is faster for similar quality.
Gemma 4 31B Dense Q4 ~20 GB 5–10 Highest quality. Patience required.
Gemma 4 31B Dense BF16 ~62 GB 2–4 Fits, but tight. Quantize.
Recommendation: start with E4B, end with 26B-A4B The Gemma 4 26B-A4B Mixture-of-Experts is the standout for the AGX Orin. It activates only 3.8B parameters per token, so it runs at small-model speed — but the routing across 26B total parameters gives it big-model knowledge. Better tokens/sec than the 27B dense and often better answers.
What "VRAM" really means here These figures are weights only. Real usage adds KV cache (grows with context length), the runtime, and system overhead. Rule of thumb: budget 30–40% extra for a 32K context window. For the 31B dense at long contexts, expect total memory pressure around 28–30 GB.

Three Paths to Inference
Pick one, not all

There are three reasonable ways to run an LLM on a Jetson, ordered by setup effort. They all produce tokens; they differ in optimization headroom and operational complexity.

easiest

Ollama

One-line install. Auto-detects CUDA. Pull a model, chat. Best for prototyping and "is this thing fast enough?" sanity checks. Performance is good but not optimal — abstracted GGUF runtime.

recommended

Jetson AI Lab containers

NVIDIA-curated Docker containers with vLLM or llama.cpp, pre-tuned for each Jetson module. Auto-generated launch commands. Best balance of performance and effort. Use this for Gemma 4.

advanced

vLLM / TensorRT-LLM direct

Manual install, manual tuning, manual everything. Highest throughput, full control over batching and serving. Worth it only when you're building a real service and need maximum tokens/sec/$.

Just curious? → Ollama → Hit a wall? → Jetson AI Lab → Production?

First-Boot Checklist
Things to do before pulling any models

Before you download a single byte of model weights, do these four things. They make every subsequent benchmark believable.

1. Confirm JetPack version & CUDA. Models and containers are built against specific JetPack versions; mismatches cause silent failures.

# Check JetPack version (look for L4T release)
sudo apt show nvidia-jetpack 2>/dev/null | grep Version

# Check CUDA toolkit
nvcc --version

# Check Tegra release
cat /etc/nv_tegra_release

2. Set max performance mode. By default the Jetson throttles. For benchmarks and demos, unlock everything.

# MAXN power profile (mode 0 on AGX Orin = unrestricted)
sudo nvpmodel -m 0

# Lock CPU/GPU/EMC clocks to max
sudo jetson_clocks

# Verify
sudo nvpmodel -q
sudo jetson_clocks --show | head -20

3. Install jtop. The only monitoring tool you'll actually use.

sudo apt install python3-pip -y
sudo pip3 install -U jetson-stats
sudo systemctl restart jtop.service

# Run it
sudo jtop

4. Free up disk space. 64 GB eMMC fills fast. Move model storage to NVMe (M.2 Key M slot) if you have one.

# Check free space
df -h /

# If you have an NVMe drive at /mnt/nvme, point Ollama there
echo 'OLLAMA_MODELS=/mnt/nvme/ollama' | sudo tee -a /etc/environment
The eMMC trap The 64 GB eMMC is shared with the OS. After JetPack install you have ~50 GB free. A single Gemma 4 31B model is ~20 GB. Plan for an NVMe SSD if you'll keep more than two large models around. The M.2 Key M slot supports NVMe directly.

Ollama Quickstart
~10 minutes to your first token

Ollama is the fastest path. It detects the Jetson's CUDA stack, downloads quantized GGUF weights, and serves them. The trade-off is some performance left on the table compared to optimized runtimes.

# Install (auto-detects ARM64 + CUDA)
curl -fsSL https://ollama.com/install.sh | sh

# Verify it's using the GPU
ollama --version
journalctl -u ollama --no-pager -n 20 | grep -i cuda

# Pull a small model first to validate the pipeline
ollama pull gemma3:4b
ollama run gemma3:4b --verbose
# > type a prompt, then exit with /bye

# Step up
ollama pull gemma3:12b
ollama run gemma3:12b --verbose

# Big model (will take a few minutes to download)
ollama pull gemma3:27b
ollama run gemma3:27b --verbose

The --verbose flag is the magic ingredient — it prints timing data after each response:

# What you'll see after each generation:
total duration:       4.821s
load duration:        38ms
prompt eval count:    24 token(s)
prompt eval rate:     412.50 tokens/s   # prompt processing
eval count:           187 token(s)
eval duration:        4.78s
eval rate:            39.12 tokens/s   # this is your real number
Useful Ollama env vars on Jetson Set these in /etc/systemd/system/ollama.service.d/override.conf (then sudo systemctl daemon-reload && sudo systemctl restart ollama):
OLLAMA_FLASH_ATTENTION=1 — faster attention, lower memory
OLLAMA_KV_CACHE_TYPE=q8_0 — quantize KV cache (saves ~50% cache memory at minor quality cost)
OLLAMA_NUM_PARALLEL=1 — single-stream is faster on Jetson than parallel

Jetson AI Lab Containers
NVIDIA-tuned Docker images for Gemma 4

The Jetson AI Lab is NVIDIA's official catalog of pre-built containers for edge LLM inference. It auto-generates the exact docker run command for your specific Jetson module. This is the recommended path for Gemma 4 specifically, since Ollama support lags behind day-one model releases.

jetson-ai-lab.com/models — pick AGX Orin 64GB, pick the model, pick vLLM or llama.cpp container, copy the command.

# Typical pattern for Gemma 4 26B-A4B via vLLM container:
docker run --runtime nvidia -it --rm \
  --network=host \
  --volume /mnt/nvme/cache:/root/.cache \
  --env HF_TOKEN=$HF_TOKEN \
  dustynv/vllm:0.8-r36.4.0 \
    vllm serve google/gemma-4-26b-a4b \
    --max-model-len 32768 \
    --gpu-memory-utilization 0.85

Once running, vLLM exposes an OpenAI-compatible HTTP API on port 8000. From any other terminal:

curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "google/gemma-4-26b-a4b",
    "messages": [{"role": "user", "content": "Explain unified memory in one sentence."}]
  }'
Why containers, not bare-metal The CUDA / cuDNN / TensorRT / PyTorch / vLLM dependency graph on Jetson is delicate. Containers freeze a known-good combination. Bare-metal installs work, but every JetPack upgrade can break them. Containers survive upgrades.

Monitoring While Running
Confirm the GPU is actually doing the work

The number one Jetson failure mode is "it's running, but on the CPU." Tokens still come out, just 5–10× slower than they should. jtop tells you instantly.

sudo jtop
What to checkHealthySuspicious
GPU %80–99% during gen< 20% → CPU fallback
RAMStable around model sizeClimbing → memory leak / swap
SWAP0 MB> 0 → kill, you're thrashing
Temp< 80 °C> 90 °C → throttling soon
Power40–60 W under load< 25 W → throttled or idle
EMC freq3199 MHz (locked)Lower → run jetson_clocks
Don't trust nvidia-smi nvidia-smi doesn't work on Jetson — Tegra GPUs aren't supported by the standard driver query path. Use jtop or tegrastats:
sudo tegrastats --interval 1000 — text-only one-line-per-second stats.

The 80/20 Run sudo nvpmodel -m 0 && sudo jetson_clocks on every boot, use Q4 quantization, and prefer MoE models. That gets you ~80% of the achievable throughput. Everything else is incremental.

Known Gotchas
Things that will bite you, in order of likelihood
1. llama.cpp + Gemma 3n full GPU offload broken on AGX Orin (CUDA 8.7) Open issue: setting n_gpu_layers at or above the layer count on Gemma 3n produces empty/garbage output. Partial offload also corrupts. Workaround: CPU-only, or use vLLM instead. Check before committing to llama.cpp for Gemma 3n. github.com/ggml-org/llama.cpp/issues/16370
2. Silent CPU fallback If the CUDA allocator fails (often on first load of a borderline-sized model), some runtimes silently fall back to CPU inference instead of erroring. Symptom: tokens/sec is half what you expected. Always verify GPU% in jtop.
3. Swap will destroy you The eMMC is slow. If a model spills to swap, generation drops from 20 tok/s to 0.5 tok/s. Either fits in RAM or it doesn't — don't run anything if jtop shows swap usage.
4. Day-one model availability Ollama's registry typically lags 1–4 weeks behind brand-new models like Gemma 4. For day-one, use llama.cpp or vLLM via the Jetson AI Lab containers. Check ollama.com/library for current availability.
5. Container runtime mismatch Containers built for JetPack 5 won't run on JetPack 6 and vice versa. Always pull the tag matching your L4T version (r36.x = JetPack 6, r35.x = JetPack 5).
6. Power supply matters The dev kit ships with a 90 W adapter. It's enough — barely. If you see unexplained throttling under sustained load, check that you're using the supplied adapter and a good USB-C cable, not a phone charger.
