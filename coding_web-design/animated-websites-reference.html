4

Supply start frame, end frame, and the Claude-written prompt. Download the output video. ⚠ Results may need iteration — imperfect is fine for demos.
Phase 3 — Build the Website (Claude Code)
5

Create a new folder (e.g. animated-websites/). Place skills inside .claude/skills/. Drag your video file into the project.
6
Switch to Plan Mode
Prevents Claude from doing anything yet — it reads and plans only. Results in much better first-try output. 💡 Claude will ask clarifying questions in plan mode. Answer them.
7
Send the initial prompt
"I dropped in a video called [filename]. Help me create a one-page product landing page. Modern, professional, smooth animations. Background completely black, dark mode, blends with the product video." 💡 More product detail = better copy. But Claude invents fictional branding if you skip it.
8
Answer plan-mode questions + refine
Product name/brand → "create fictional branding." Content sections → "full premium." Add any layout tweaks (e.g. "product video right-aligned, 2/3 of page, text left").

⚠️ Context Management

At ~50% context used, clear the conversation and start a new plan-mode session for iterations. Letting context rot causes Claude to lose track of what it already built.

The Discovery Interview (6 Rounds)
R1
Goal & Name
What does the skill do? What problem does it solve? What's the /slash-command name? (lowercase, hyphens, max 64 chars)
R2
Trigger
What would someone say to trigger it? (2–3 natural phrases.) User-only, Claude-auto, or both? Does it take arguments? 💡 The description field is how Claude decides whether to load the skill — bad keywords = skill never fires.
R3
Step-by-Step Process
Walk through every step from trigger to output. Does Claude do it directly or delegate to a subagent? Conversational or fire-and-forget? ⚠ Vague steps = vague results. Specific steps = consistent output.
R4
Inputs, Outputs & Dependencies
What files/APIs/data does it need? Where does output go? Any external scripts, APIs, or reference files required?
R5
Guardrails & Edge Cases
What can go wrong? What should this skill NOT do? Any cost concerns (API calls, image generation)? Ordering constraints?
R6
Confirmation
Skill Builder summarizes everything back to you. Confirm or correct before building starts. 💡 Rounds are skipped if you provide enough context upfront.

🔑

🤖

1. Natural language — say something matching the description. 2. Direct invoke — run /skill-name with test args. 3. Edge cases — missing args, unusual input. 4. Character budget — run /context to confirm skill descriptions are loaded (max 2% of context window / 16,000 chars across ALL skills).

GitHub + Vercel Deploy Pipeline

Local → GitHub → Vercel. Every code change is tested locally first, then pushed to auto-deploy. This gives you a safe testing environment separate from live production.

Deploy Steps
1
Create accounts
GitHub (free, industry standard) + Vercel (free). Sign into Vercel with your GitHub account.
2
Connect Claude Code to GitHub
In a fresh conversation: "Help me connect to GitHub so I can push this codebase to my GitHub repository." Claude uses the CLI to authenticate — just sign in at the popup.
3
Create repo + push
"Create a new GitHub repository called [project-name] and push the codebase into it." ⚠ Check for API keys before pushing — remove them first if present.
4
Import to Vercel
Vercel dashboard → Projects → Add New → Import GitHub Repository → select your repo → Deploy. 💡 Repo can be set to private in GitHub settings — Vercel still auto-deploys.
5
Verify frames are included
Check that the frames folder is NOT grayed out in VS Code (grayed = excluded from push). If frames are missing from the repo, the animation won't render on Vercel. ⚠ The #1 deploy gotcha: animation frames excluded from the GitHub push. Tell Claude to fix and re-push.
6
Roll back or redeploy if needed
Vercel → Deployments → three dots menu on any deployment → Redeploy. You can roll back to any previous commit.

Quick Start — What Is a Skill?

A skill is a reusable set of instructions that tells Claude Code how to handle a specific task. Skills live in .claude/skills/[skill-name]/SKILL.md inside your project. When you type /skill-name or describe what you need in natural language, Claude loads the skill's instructions and follows them.

Think of skills as SOPs for Claude. Instead of re-explaining a workflow every conversation, you write it once and invoke it forever.

📋
Your project's CLAUDE.md instructions are always loaded, every conversation.
🏷️
Skill descriptions (from frontmatter) are always loaded so Claude knows what's available.
⚡
The full skill content only loads when the skill is actually invoked.
🔗
Once loaded, Claude follows the skill's instructions while still respecting your CLAUDE.md rules.
Mode 1 — Build a New Skill
Discovery Interview

Ask questions one round at a time. Move to the next round only after the user answers. Keep going until 95% confident you understand the skill well enough to build it. Do NOT start writing files until discovery is complete.

R1
Goal & Name Why: A clear goal prevents scope creep. The name becomes the /slash-command.
• What does this skill do? What problem does it solve?
• What should we call it? (lowercase, hyphens, max 64 chars)
R2
Trigger Why: The description field is how Claude decides whether to load your skill. Bad trigger words = skill never fires. Too broad = unwanted fires.
• 2–3 natural language phrases that would trigger it
• User-only (/slash-command), Claude-auto-invocable, or both?
• Does it accept arguments? (topic, URL, file path?)
R3
Step-by-Step Process Why: Claude follows instructions literally. Vague steps → vague results. Specific steps → consistent output every time.
• Walk through exactly what happens from trigger to output
• For each step: Claude directly, or delegate to a subagent/script?
• Conversational (back-and-forth) or fire-and-forget?
R4
Inputs, Outputs & Dependencies Why: Skills that don't specify where to find inputs or put outputs produce inconsistent results.
• What inputs does it need? (Files, API responses, user args, live data)
• What does it produce? Where do outputs go?
• External APIs, scripts, tools? Reference files, templates?
R5
Guardrails & Edge Cases Why: Skills without guardrails can produce wrong outputs, unnecessary API costs, or unintended actions.
• What could go wrong? Common failure modes?
• What should this skill NOT do? Hard boundaries?
• Cost concerns? Ordering / dependency constraints?
R6
Confirmation — Skill Summary Why: Misunderstandings caught here save you from rebuilding the skill later. 
## Skill Summary: [name]
**Goal:**       [one sentence]
**Trigger:**    `/name` + [natural language phrases]
**Arguments:** [what it accepts, or "none"]

**Process:**
1. [step]
2. [step]

**Inputs:**       [what it reads/needs]
**Outputs:**      [what it produces + where]

Ask: "Does this capture it? Anything to add or change?" — Only build once the user confirms. 💡 Skip rounds if the user provides enough context upfront. Don't re-ask what you already know.

Complete Example — meeting-notes

A minimal but complete skill you can use as a starting template. File: .claude/skills/meeting-notes/SKILL.md

---
name: meeting-notes
description: Use when someone asks to summarize meeting notes,
             recap a meeting, or format meeting minutes.
argument-hint: [topic or date]
---

## What This Skill Does

Takes raw meeting notes and produces a structured summary with action items.

## Steps

1. Ask the user to paste their raw meeting notes (or provide a file path).
2. Extract the following from the notes:
   - Attendees — who was in the meeting
   - Key decisions — what was decided
   - Action items — who owes what, with deadlines if mentioned
   - Open questions — anything unresolved
3. Format the output using the template below.
4. If $ARGUMENTS is provided, use it as the meeting title.
   Otherwise, infer a title from the content.

## Output Template

# Meeting: [title]
**Date:** [date if mentioned, otherwise "Not specified"]
**Attendees:** [comma-separated list]

## Key Decisions
- [decision]

## Action Items
- [ ] [person]: [task] (due: [date or "TBD"])

## Open Questions
- [question]

## Notes
- Keep summaries concise. Don't add commentary or embellish.
- If notes are too vague to extract action items, flag that
  to the user instead of making them up.
Mode 2 — Audit an Existing Skill
Audit Instructions

Read the skill file first before running through the checklist. Fix issues before marking the audit complete.

Frontmatter Audit
☐name matches the directory name
☐description uses natural keywords someone would actually say
☐description specific enough to avoid false triggers, broad enough to catch real requests
☐disable-model-invocation: true set if skill has side effects (generates files, calls APIs, costs money)
☐argument-hint set if skill accepts arguments via /name
☐allowed-tools set if skill should NOT have access to all tools
☐context: fork used if skill is self-contained and produces verbose output
☐model set only if a specific model capability is needed
☐No unnecessary fields set — don't add frontmatter just because you can
Content Audit
☐SKILL.md is under 500 lines (detailed reference moved to supporting files)
☐Clear step-by-step workflow with numbered steps (for task skills)
☐Output format specified with templates or examples
☐All file paths and locations documented
☐Agent delegation includes the actual prompt text to send
☐Notes section covers edge cases, constraints, and what NOT to do
☐No vague instructions — every step tells Claude exactly what to do
☐String substitutions ($ARGUMENTS, $N) used where skill takes input
Integration Audit
☐Skill documented in CLAUDE.md (recommended, not required)
☐Supporting files referenced from SKILL.md, not orphaned
☐Scripts have correct file paths and are executable
☐API keys stored in environment variables, never hardcoded
Quality Audit
☐A beginner could follow instructions without prior context
☐Instructions are actionable, not abstract
☐Delegates to subagents when appropriate to keep main context clean
☐Doesn't duplicate info that lives elsewhere (CLAUDE.md, other skills)
☐Output paths follow a predictable convention
Recommended Conventions
📁
Skills live in .claude/skills/[skill-name]/SKILL.md
📤
Output files go in a predictable location, e.g. output/[skill-name]/
🔐
API keys go in environment variables, never hardcoded in skill files
📝
Document all active skills in your project's CLAUDE.md
💬
description written as: "Use when someone asks to [action], [action], or [action]."
