The Agentic Engineering Loop

Replacing pure vibe coding. You run the loop — the AI serves it.

1
Define intent clearly before prompting
Write down what you want to build, why, and what "done" looks like. Break work into discrete tasks a junior dev could understand. Vague prompts → verbose over-engineered output. 💡 If you can't explain the task in 2 sentences, you don't understand it well enough to delegate it.
2
You own the architecture — then delegate implementation
Design the structure, data model, and core decisions yourself (with AI as thinking partner). Then hand off implementation of well-defined chunks. Never ask the AI to "design a scalable architecture" and follow it blindly. ⚠ High-level design is not a prompt job. It requires your judgment.
3
Generate → Review → Understand → Commit
Every line that ships must be understood by you. If another developer could tell which lines came from an LLM, it's not ready. Internalize the code as if you wrote it. ⚠ Reviewers shouldn't be working harder than authors. If you didn't understand what you merged, you didn't review it.
4
Force the AI to review its own work (Rule of Five)
Before accepting any non-trivial output, prompt the model to review its own plan or code 5 times. AI performance converges on quality through recursive review, not a single perfect prompt. 💡 Ask: "Review this for security issues." "Review this for maintainability." "Is there a simpler approach?" — separately.
5
Write tests as survival mechanism, not afterthought
You can't manually review 12,000 AI-generated lines. Tests are your tripwires. Automated tests catch what confident hallucinations miss. Check behavior and side effects, not just whether it runs. ⚠ An agent that checks off "tests written" without actually writing tests is not a safety net — it's a liability.
6
Actively refactor, don't let patches accumulate
The LLM doesn't care how messy the code is. It will apply patches on patches indefinitely. Refactoring must be a deliberate human choice. Code smells don't clean themselves. 💡 If a PR is net-negative on lines (removing more than adding), that's often a sign of real health.
7
Document intent, not just implementation
AI-generated code needs more documentation, not less. Future maintainers (including you) need to know why a solution was chosen, not just what it does. Document the reasoning. ✅ Treat AI-generated code as you would a new hire's PR: understand it, document it, own it.
8
Know when to take back the wheel
If the AI keeps producing convoluted solutions to a simple problem — stop and write it yourself. Sometimes manual is faster, cleaner, and better understood. Critical security modules, custom optimizations, and nuanced algorithms may need human-only authorship. 💡 "AI did it" is not an explanation. It's an admission of abdication.
🔁 The Feedback Loop That Works

Prompt clearly → Generate → Review for behavior/side effects → Interrogate AI reasoning → Refine or rewrite → Document why → Commit with understanding. Speed lives in the generate step. Quality lives in every other step.

The Hidden Traps (Details)
⚠️
The Productivity Tax trap: 66% of developers report AI tools generate code that is "almost but not quite right," requiring cleanup. The time saved generating is often spent cleaning. Budget for this explicitly.
⚠️
The over-engineering trap: LLMs are trained on verbose codebases and tend to produce defensive, over-commented, helper-function-heavy output. Where an expert uses 10 lines, the model may use 100. Push back on verbosity.
⚠️
The "ambient competence" trap: Feeling capable of a task because the AI can do it is not the same as being capable. When the AI fails, your actual skill level is what catches it. Build real skills, not just prompting confidence.
⚠️
The PR asymmetry trap: It now takes less effort to produce a PR than to review one. If authors aren't internalizing the code, reviewers are doing more work than authors — and will start missing things. Authors must understand what they ship.
⚠️
The "nuke from orbit" trap: AI directed to "fix the setup" may choose a factory reset. Agents optimize for task completion — not physical/business consequences. Always review drastic actions. Never delegate irreversible operations blindly.
⚠️
The security invisibility trap: Security quality in LLM-generated code has not improved with model size. Larger models are not safer. Security-critical code must be reviewed by someone who understands security — not assumed safe because the model is smart.
🚨 The Deletion Test

Before shipping, ask: if this code disappeared tomorrow, would I be able to recreate it? If the answer is no — you don't own it. You're just hosting it. Own your code.

Who Owns What — The Responsibility Split

In the post-vibe world, every part of the stack has a clear owner. The split isn't about distrust of AI — it's about knowing who's accountable for what.

Area Owner Why
Product vision & intent HUMAN AI can't know what your users actually need or what makes you different
System architecture HUMAN Structural decisions compound; agents only see the current task
Security-critical code HUMAN LLM security quality flat across model sizes. Don't trust it here.
Code review & merge decision HUMAN Ownership transfers at merge. You signed it.
Refactoring decisions HUMAN The model won't clean up smells it created. Only you will.
Test strategy & edge cases HUMAN AI writes tests that pass, not tests that catch real failure modes
Boilerplate & repetitive code AI High-volume, low-judgment work — exactly where models shine
Known pattern implementation AI CRUD, standard API shapes, well-defined schemas
Documentation drafts AI Generate then verify; human edits for accuracy and intent
Bug hypothesis generation BOTH AI surfaces candidates; human validates against system knowledge
Code explanation & learning BOTH AI explains; human confirms understanding by paraphrasing back
Research & exploration BOTH Effective when human steers exploration with domain knowledge
Build the Foundation

Learn systems thinking — how parts interact, not just how to write syntax. This is what makes AI output legible.
🔍
Debug manually sometimes. The discipline of finding bugs without AI builds the pattern-recognition that catches AI mistakes.
🎨
Develop taste through exposure. Study what works and why. Foundational taste is the edge that can't be prompted into existence.
✍️
Write some code by hand. Students who don't struggle through concepts don't build the heuristics needed to evaluate AI output.
