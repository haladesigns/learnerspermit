⚗ ALPHA API 36 AppFunctions — Ground-Up Reference
alpha08 · Android 16+

AppFunctions is on-device MCP. It lets any Android app expose named, typed, self-describing functions that Gemini — or any agent holding the right permission — can discover and invoke at runtime. No deep-link hacks. No Accessibility Service abuse. Just structured tool-calling, locally.

🧠
The one-sentence model

AppFunctions : Android apps :: MCP tools : LLM servers — same idea, on-device, with stricter IPC security.

⚠️
Right now Gemini is the only caller in the wild

The EXECUTE_APP_FUNCTIONS permission is privileged on most Android 16 devices — meaning only system-signed apps can hold it. Only Gemini (+ Samsung S26 / Pixel 10 ships) currently qualifies. Writing your own agent caller requires root-level install for now.

WHYWhy not just use deep links or Intents?
INTENT
Deep links / Intents

Launch an Activity. Agent can't get structured data back. User must navigate UI. No schema for LLM to reason about parameters.

A11Y
Accessibility Service automation

Brittle to UI changes. Requires invasive permissions. Can't return structured data. Banned by Google Play for most use-cases.

APPFN
AppFunctions

Typed parameters + typed return. Schema readable by LLM. On-device IPC — no network. Secure: only system-privileged callers. Bidirectional: agent gets structured response back.

The Tool App (Provider) is the app that exposes functionality. You write a class with annotated functions, declare a service in the manifest, and the KSP compiler generates everything the OS needs to index and route calls.

STEP 1Annotate your functions
NoteAppFunctions.kt
import androidx.appfunctions.AppFunction
import androidx.appfunctions.AppFunctionContext
import androidx.appfunctions.AppFunctionSerializable

/**
 * Exposed AppFunctions for the Notes app.
 * KDoc here doubles as LLM-readable descriptions — be precise.
 */
class NoteAppFunctions(private val repo: NoteRepository) {

    /**
     * Creates a new note with the given title and content.
     *
     * @param appFunctionContext System-provided execution context.
     * @param title The title for the new note.
     * @param content The body text of the note.
     * @return The created [Note] with its assigned ID.
     */
    @AppFunction(isDescribedByKDoc = true)
    suspend fun createNote(
        appFunctionContext: AppFunctionContext,
        title: String,
        content: String,
    ): Note {
        return repo.createNote(title, content)
    }

    /**
     * Edits an existing note. Only supplied fields are updated.
     *
     * @param appFunctionContext System-provided execution context.
     * @param noteId ID of the note to update.
     * @param title New title, or null to leave unchanged.
     * @param content New body text, or null to leave unchanged.
     * @return The updated [Note], or null if not found.
     */
    @AppFunction(isDescribedByKDoc = true)
    suspend fun editNote(
        appFunctionContext: AppFunctionContext,
        noteId: Int,
        title: String?,   // nullable = optional param
        content: String?,
    ): Note? {
        return repo.updateNote(noteId, title, content)
    }
}
💡
isDescribedByKDoc = true is your LLM interface

The KSP compiler extracts your KDoc comments and bakes them into the XML schema. What you write in @param and the class summary is literally what Gemini reads when deciding whether to call this function. Write it like a tool description, not a code comment.

STEP 2Define your return type with @AppFunctionSerializable
Note.kt
import androidx.appfunctions.AppFunctionSerializable

/**
 * A note in the app.
 */
@AppFunctionSerializable(isDescribedByKDoc = true)
data class Note(
    /** The note's unique identifier. */
    val id: Int,
    /** The note's title. */
    val title: String,
    /** The note's body text. */
    val content: String
)
⚠️
Only @AppFunctionSerializable types can cross the IPC boundary

Custom return types must be annotated. Primitives (String, Int, Boolean, List<primitive>) work out of the box. Arbitrary data classes do not — you'll get a compile-time error from the KSP processor.

STEP 3Implement AppFunctionService
MyAppFunctionService.kt
import androidx.appfunctions.AppFunctionService

// The Jetpack compat base class (works on API 36+ and backport)
class MyAppFunctionService : AppFunctionService() {

    // Inject or construct your functions class here
    private val noteFunctions = NoteAppFunctions(NoteRepository())

    // Wire functions up — the KSP compiler generates a router for you
    override val appFunctionConfiguration = AppFunctionConfiguration
        .Builder()
        .addAppFunctions(noteFunctions)
        .build()
}
🔧
Using Hilt / DI?

If your functions class has injected dependencies, implement AppFunctionConfiguration.Provider in your Application class and provide the instance there. See the Annotations tab for the Hilt pattern.

🔒
The android:permission attribute is mandatory

Without BIND_APP_FUNCTION_SERVICE on the <service> tag, any app could bind to your service. The permission restricts binding to the system process only. Do not omit this.

DISCOVERQuery available functions
AgentActivity.kt (caller)
import androidx.appfunctions.AppFunctionManager

val manager = getSystemService(AppFunctionManager::class.java)

// Observe all AppFunctions from all installed apps
lifecycleScope.launch {
    manager.observeAppFunctions(packageName = null)
        .collect { appPackageMetadata ->
            appPackageMetadata.forEach { pkg ->
                pkg.appFunctions.forEach { fn ->
                    // fn.functionId  — unique ID to invoke later
                    // fn.description — LLM-readable KDoc summary
                    // fn.parameters  — typed param metadata
                    // fn.returnType  — serializable return type info
                    Log.d("AF", "Found: ${fn.functionId} in ${pkg.packageName}")
                }
            }
        }
}
INVOKEExecute a function
AgentActivity.kt — execution
import androidx.appfunctions.ExecuteAppFunctionRequest
import androidx.appfunctions.AppFunctionData

// Build the request — function ID comes from metadata query above
val request = ExecuteAppFunctionRequest.Builder(
    functionId = "com.example.notes/createNote",
    targetPackageName = "com.example.notes",
).setParameters(
    AppFunctionData.Builder()
        .put("title", "Meeting notes")
        .put("content", "Discuss Q2 roadmap")
        .build()
).build()

// Execute — always async, callback on main thread
manager.executeAppFunction(
    request = request,
    executor = Executors.newSingleThreadExecutor(),
    cancellationSignal = CancellationSignal(),
    callback = object : OutcomeReceiver<ExecuteAppFunctionResponse, AppFunctionException> {
        override fun onResult(result: ExecuteAppFunctionResponse) {
            // result.getReturnValue() → AppFunctionData with your Note fields
            val noteId = result.returnValue?.getLong("id")
            Log.d("AF", "Created note id=$noteId")
        }
        override fun onError(error: AppFunctionException) {
            Log.e("AF", "Error: ${error.errorCode}")
        }
    }
)
LLMTypical agent loop (pseudo)
Query AppFunctionManager for all installed functions

Build a list of { functionId, description, parameters } objects.

Pass function list as tools to your LLM

Same pattern as MCP tool definitions — name, description, JSON schema of params.

LLM selects a function and outputs params

Model outputs { tool: "createNote", args: { title: "...", content: "..." } }

Build ExecuteAppFunctionRequest and call manager

Map LLM output args into AppFunctionData key-value pairs.

Return structured result to LLM for next step

The response AppFunctionData maps back to JSON for the next turn.

COREThe annotation toolkit
@AppFunction
Marks a suspend fun as callable by agents

Required on every function you want to expose. First param must always be AppFunctionContext. Set isDescribedByKDoc = true to pull descriptions from KDoc comments.

@AppFunctionSerializable
Marks a data class as IPC-safe

Applied to any custom return type or parameter type. Only fields with primitive, String, or other @AppFunctionSerializable types allowed. Set isDescribedByKDoc = true for field descriptions.

@AppFunctionIntValueConstraint
Constrains Int parameter range

@AppFunctionIntValueConstraint(minValue = 1, maxValue = 100) — surfaces to the LLM as valid range hint.

@AppFunctionStringValueConstraint
Restricts String to an allowlist

@AppFunctionStringValueConstraint(allowedValues = ["low", "medium", "high"]) — models will respect this as an enum hint.

AppFunctionUriGrant
Auto-grants URI permission to caller

Wrap return URIs in AppFunctionUriGrant to give the calling app read access to content URIs without manual permission grants.

AppFunctionSchemaDefinition
Agent-side: pre-defines a known function schema

Used by caller apps to declare what functions they expect to find. Enables validation and IDE tooling. Available since alpha02.

KDOCWriting LLM-effective KDoc
✓ DO

First line = one-sentence action summary: "Creates a new note and returns it."

Each @param explains semantics, not types: "@param title User-visible heading — keep under 100 chars."

Describe nullable params as optional with consequences: "@param content Body text; null creates an empty note."

✗ DON'T

Don't repeat type info Kotlin already expresses: "@param title String the title"

Don't skip KDoc on @AppFunctionSerializable fields — LLMs need field semantics to fill them correctly.

Don't leave the class-level KDoc blank — it becomes the tool description the model sees first.

DIHilt pattern for injected dependencies
MyApplication.kt (Hilt)
@HiltAndroidApp
class MyApp : Application(), AppFunctionConfiguration.Provider {

    @Inject
    lateinit var noteAppFunctions: Provider<NoteAppFunctions>

    override val appFunctionConfiguration: AppFunctionConfiguration
        get() = AppFunctionConfiguration.Builder()
            .addAppFunctions(noteAppFunctions.get())
            .build()
}
ℹ️
Why Provider<T> and not direct injection?

The AppFunctionConfiguration is accessed before Hilt finishes its component setup in some cases. Using Provider<T> (Dagger lazy injection) defers construction until .get() is called, avoiding a crash on early access.

OK
Primitives

String, Int, Long, Float, Double, Boolean, ByteArray

OK

List<String>, List<Int>, etc. List<ByteArray> has a known bug — avoid until fixed.

OK
@AppFunctionSerializable data classes

Nested serializable types work. Keep hierarchies shallow.

OK

Nullable = optional from the caller's perspective. Always document the null case in KDoc.

OK

Can be returned to let the caller launch a UI if needed. List<PendingIntent> had a compile bug — check your alpha version.

NO
Arbitrary Android types

Uri, Bitmap, custom non-serializable classes — use AppFunctionUriGrant wrapper for URIs.

SECThe two critical permissions
BIND
android.permission.BIND_APP_FUNCTION_SERVICE

Who declares it: Tool App, on the <service> tag.
Effect: Only the system process can bind to your service. Any other app attempting to bind gets a SecurityException. If you omit this, any app can call your functions.

EXEC
android.permission.EXECUTE_APP_FUNCTIONS

Who holds it: Agent/Caller app (Gemini, system agent).
Level: Privileged on Android 16 (most builds). Normal level only on select devices with flag enabled.
Effect: Without this the caller cannot query or invoke any AppFunction.

TRUSTThe trust model
1
Only system can bind your service

BIND_APP_FUNCTION_SERVICE is held by the system — not by Gemini directly. The system acts as a trusted broker between caller and provider.

2
Caller identity is passed to your service

onExecuteFunction receives a SigningInfo parameter identifying which app triggered the call. You can check this if you want to restrict which agents can call specific functions.

3
User consent is system-managed

Sensitive actions require a user confirmation prompt surfaced by the system — not something your app implements. Gemini shows "live view" notifications for active function calls.

4
AppFunctions can be disabled per-app

Users can disable AppFunctions for a specific app in system settings. Your service receives a disabled state — handle it gracefully, don't crash.

ONDEVICEPrivacy model
🛡️
Everything runs on-device

Unlike cloud MCP, AppFunctions never leave the device. The execution path is: Gemini on-device model → AIDL IPC → your AppFunctionService → AIDL IPC back. No network call, no cloud relay. This is a key differentiator Google emphasizes for privacy-sensitive use cases (gallery search, medical notes, etc.).

TRAPSKnown gotchas
TRAP
No XML schema generated = silent failure

If KSP is not wired correctly, the compiler artifact produces nothing. The app installs, the service starts, but the OS has nothing to index — Gemini will never see your functions. Always verify the generated XML after build (see Setup tab).

TRAP
USE_CONTENT_PROVIDER=true deprecated in alpha08

Earlier workaround for devices without the schema-parser flag. Starting alpha08 this no longer supports function calling — only metadata retrieval returns incomplete data. Don't rely on it for execution.

TRAP
List<ByteArray> generates wrong metadata

Known bug — metadata is generated as List<List<ByteArray>> incorrectly. Avoid this type until fixed. Use a serializable wrapper class instead.

TRAP
Nullable required field crash

Fixed in alpha06. If you're on earlier alpha, ignoring a nullable required field in AppFunctionData construction will throw at runtime, not compile time.

TRAP
onExecuteFunction is on the main thread

The system invokes this on the main thread. Any blocking or IO work must be dispatched to a coroutine or executor immediately. The Jetpack AppFunctionService handles this for suspend fun automatically — but if you extend the platform class directly, you must handle threading yourself.

TRAP
R8/ProGuard strips generated code

Fixed in alpha04 with proper R8 rules. If you're on alpha03 or earlier and using release builds, your schema and service wiring will be stripped. Always test in release mode.

WARN
Empty XML file still required in manifest

Even if your app has no AppFunctions, AppSearch expects the schema XML to exist. The KSP compiler generates an empty one — but if you manually exclude generated resources you'll get a crash at index time.

WARN
Interface overrides don't inherit optional state

Fixed in alpha05. If your AppFunctions class implements an interface and overrides a function, the optional/required state of params was not respected from the interface definition.

TESTTesting with Robolectric
NoteAppFunctionsTest.kt
import androidx.appfunctions.testing.AppFunctionTestRule

@RunWith(RobolectricTestRunner::class)
class NoteAppFunctionsTest {

    @get:Rule
    val appFunctionRule = AppFunctionTestRule(NoteAppFunctions(FakeNoteRepository()))

    @Test
    fun createNote_returnsNoteWithId() = runTest {
        val result = appFunctionRule.executeFunction(
            functionId = "createNote",
            params = mapOf("title" to "Test", "content" to "Body")
        )
        assertThat(result.getLong("id")).isGreaterThan(0L)
    }
}
🧪
AppFunctionTestRule (since alpha03)

Lets you unit-test your AppFunction implementations with Robolectric without needing a device or system privilege. It bypasses the IPC layer and calls your function directly through the Jetpack service abstraction.
