Permission System

Every time Claude Code writes a file, runs a command, or makes a network request, a permission system decides whether to proceed, block, or wait for your approval. Getting this right is the difference between slow frustration, risky automation, and smart defaults that match how you work.

Three Tool Levels
Tool Type
Examples
Behavior
Read-only
Reading files, searching code
ALWAYS OK

Writing or changing files

Shell cmds
Running terminal commands

Approval Persistence
📁
File edit approvals last until you close the session.
💾
Shell command approvals ("Yes, don't ask again") stick permanently for that project.
🔍
Read-only tools never ask — always allowed.
💡 Key Principle

Use Normal mode while learning — every prompt Claude Code asks is a chance to understand what it's doing. Switch to Auto-Accept Edits or Bypass once you're comfortable and moving fast.

The Five Permission Modes
⬤ Normal (Default)
DEFAULT
Asks permission for most actions. Nothing runs automatically except reads.
Best for: Learning, sensitive projects, or when you want to understand every action Claude takes.
⬤ Auto-Accept Edits
ACCEPTEDITS
File changes happen automatically. Shell commands still ask for permission.
Best for: Everyday coding when you trust Claude's edits but still want control over what runs in your terminal.
⬤ Plan Mode
READ-ONLY
Claude can look at code and answer questions. Cannot change anything.
Best for: Code review, exploring a new project, security audits, or planning before a big change. Use this before attacking complicated tasks.
⬤ Don't Ask
AUTOMATED
Automatically denies everything unless you've pre-approved it in settings.
Best for: Automated pipelines where you want to enforce a strict allowlist.
Bypass Mode — Handle With Care
⚡
Skips all permission checks. Everything is auto-approved — file writes, shell commands, downloads, deletions.
🔒
Intentionally excluded from Shift+Tab cycle. Must be enabled with the --dangerously-skip-permissions flag.
🚀
Significantly speeds up build processes. Most experienced users work this way for the speed gains.
⚠️
Only use in isolated environments — containers or VMs. Never on your actual machine for untrusted prompts.
claude --dangerously-skip-permissions -p "Run the full test suite and fix any failures"
Switching Modes Mid-Session
⌨
Shift+Tab cycles through: Normal → Auto-Accept Edits → Plan Mode → back to Normal 💡 The current mode shows at the bottom of your terminal.
📍
Practical workflow: Drop into Plan Mode to explore code, then Shift+Tab to Auto-Accept to start building.
🚫
Bypass is not in the cycle. You can only enable it via --dangerously-skip-permissions at startup. ⚠ Bypass overrides all other permission settings entirely.
Setting a Default Mode
1
Global default — applies to all your projects
Edit ~/.claude/settings.json: 
{
  "permissions": {
    "defaultMode": "acceptEdits"
  }
}
2
Per-project default — overrides your global default
Edit .claude/settings.json inside the project: 
{
  "permissions": {
    "defaultMode": "plan"
  }
}
💡 Valid values: "default", "acceptEdits", "plan"
Creating Allow, Ask, and Deny Rules
1
Rule evaluation order: deny → ask → allow. Deny always wins. 💡 Create broad allow rules, then carve out exceptions with deny rules.
2
Rule syntax — tool name + optional pattern: 
Bash                   -- matches ALL shell commands
Bash(npm run build)    -- matches this exact command
Bash(npm run *)        -- matches anything starting with "npm run "
Bash(git * main)       -- matches "git checkout main", "git merge main", etc.
Read(./.env)           -- blocks reading this specific file
Read(./secrets/**)    -- blocks reading anything in secrets/
⚠ The space before * matters. Bash(ls *) matches "ls -la" but NOT "lsof".
3
Full example rules config: 
{
  "permissions": {
    "allow": [
      "Bash(npm run build)",
      "Bash(npm run test *)",
      "Bash(git commit *)",
      "Bash(npx prettier *)"
    ],
    "ask": [
      "Bash(git push *)"
    ],
    "deny": [
      "Bash(git push * --force)",
      "Bash(rm -rf *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
🔒 Security Note

Claude Code is smart about injection: a rule like Bash(safe-cmd *) will NOT match safe-cmd && malicious-cmd. The permission system protects against chained commands.

Pattern 1: The Cautious Approach
🎯
Start in Normal mode. Review everything. Allow specific commands over time as you trust them.
👤
Best for: Beginners, sensitive codebases, or anything where understanding every action matters.
{
  "permissions": {
    "defaultMode": "default",
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test)"
    ],
    "deny": [
      "Bash(rm *)",
      "Read(./.env)",
      "Read(./secrets/*)"
    ]
  }
}
Pattern 2: The Speed Runner
🚀
Auto-accept file edits. Pre-approve common commands. Block dangerous ones.
👤
Best for: Experienced users who want maximum speed while keeping guardrails on destructive actions.
{
  "permissions": {
    "defaultMode": "acceptEdits",
    "allow": [
      "Bash(npm *)", "Bash(npx *)",
      "Bash(git add *)", "Bash(git commit *)",
      "Bash(git checkout *)", "Bash(git branch *)",
      "Bash(* --version)", "Bash(* --help)"
    ],
    "ask": [
      "Bash(git push *)",
      "Bash(git merge *)"
    ],
    "deny": [
      "Bash(git push * --force)", "Bash(rm -rf *)",
      "Read(./.env)", "Read(./.env.*)",
      "Read(./secrets/**)",
      "Bash(curl *)", "Bash(wget *)"
    ]
  }
}
Pattern 3: The Code Reviewer
🔍
Lock Claude into read-only Plan mode. Analysis and review with zero risk of changes.
👤
Best for: Security audits, onboarding to a new codebase, PR reviews, exploring before a big refactor.
# settings.json
{ "permissions": { "defaultMode": "plan" } }

# Then run:
claude -p "Review the auth flow in src/auth/ and identify security issues"
✓ Do This
✅Use /permissions to debug why Claude keeps asking when it shouldn't.
✅Create broad allow rules and carve out exceptions with deny rules.
✅Only use Bypass mode in isolated containers or VMs.
✅Start in Normal mode while learning — permission prompts are teaching moments.
✅Always include a space before * when targeting subcommands: Bash(ls *).
✗ Avoid This
❌Running Bypass mode on your actual machine with unfamiliar prompts.
❌Forgetting that Bash(ls*) (no space) matches "lsof" — use Bash(ls *) to be safe.
❌Assuming project settings and personal settings conflict — they stack up and add together.
❌Expecting a deny+allow conflict to resolve in allow's favor — deny always wins.
❌Putting sensitive files anywhere except under explicit deny rules.
Critical Gotchas
⚠️
Deny always wins. A command matching both a deny and an allow rule gets denied. No exceptions.
⚠️
The space before * matters. Bash(ls *) matches "ls -la" but NOT "lsof". Use Bash(ls*) (no space) if you want both.
⚠️
Project settings beat personal settings. If your global settings allow something but the project settings deny it, the project wins. Teams can enforce safety rules this way.
⚠️
Bypass overrides everything. --dangerously-skip-permissions ignores all other permission settings, including deny rules.
⚠️
Shell approval persistence. "Yes, don't ask again" for shell commands sticks permanently for that project — across sessions.

CLI Flags Reference
Flag Effect
--dangerously-skip-permissions Enable bypass mode for this session
--permission-mode <mode> Set mode (default, plan, acceptEdits)
--allowedTools <tool> Add a tool to the allow list
--disallowedTools <tool> Add a tool to the deny list
🏗️ Team Setup Tip

Commit .claude/settings.json to enforce safety rules across your team. Each developer can still have personal overrides in .claude/settings.local.json (gitignored) without breaking team rules — but project deny rules will always win over personal allow rules.
