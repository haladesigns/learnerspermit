Building Complex Codebases with AI

A dead-simple, greenfield-focused framework for AI coding assistants. The goal is something simple, yours, and evolvable — not an overengineered multi-agent system. You can adapt this starter kit faster than you could learn BMAD or the GitHub spec kit, and it'll compound across every project you build.

Primary Mental Model

Each feature/task runs its own PIV loop. The loop is not optional — skipping validation is how AI-assisted projects accumulate hidden debt.
Three Pillars
📄
PRD-First: Document before you code. The PRD is source of truth for every AI conversation and each granular feature.
📐
Modular Rules: Split rules by domain — agents, references, components. Load only what's relevant per conversation.
🔁
PIV Loop: Plan → Implement → Validate per feature. Context resets between planning and execution conversations.
System Evolution Mindset
🐛
Every bug is an opportunity to evolve your system rules — not just fix the immediate issue.
📝
When AI makes a mistake, ask: "How do I prevent this class of mistake forever?" Then encode the answer into rules.
📈
Your SYSTEM improves over time. Each project leaves you with better rules, commands, and context that compound on the next.
💡 Core Principle

AI doesn't replace engineering judgment — it amplifies it. The framework exists to give AI enough structured context that it can act reliably, while keeping humans in control of architecture, goals, and validation.

Greenfield Project Planning

Before writing any code, build the AI layer for your codebase. This means creating the PRD, rules architecture, and prime command that will guide every subsequent conversation.

PRD-First Development — The Creation Flow
1
Brain dump — start completely unstructured
Tell the agent your idea, rough tech stack ideas, anything you have in mind. Use speech-to-text for speed. Don't worry about structure yet — the goal is to get context into the conversation. 💡 Use speech-to-text (Whisper, AquaVoice, etc.) to dump ideas fast. Raw is fine here.
2
Spin up subagents for research
Ask the agent to spin off subagents to research best practices, relevant libraries, and architecture patterns. You only get back the distilled summary — not the raw exploration.
3
Ask the agent to ask YOU questions
Explicitly prompt: "Come back to me with a bunch of questions so we can get on the same page on the details." This is the single most powerful assumption-reduction technique. With Claude Code's multi-choice question tool, you can blitz through 20–25 questions quickly. ⚠️ One line of a bad PRD could be 1,000 lines of bad code. Every question answered removes one assumption.
4
Run /create-prd to produce the structured document
The entire conversation becomes context for the PRD. The command has a fixed template with sections: MVP scope, out-of-scope, directory structure, tech stack, and phases of work. 💡 The conversation context is temporary. Only what's captured in the PRD survives to future sessions.
5
Read through and validate alignment
Don't skip this. Every misalignment here multiplies into bad code later. Confirm MVP scope, out-of-scope items, phases, and directory structure all match your intent. ⚠️ For existing codebases: research/document the existing code first before generating the PRD. Brownfield is a different flow.
⚠️ The Assumption Cascade

One line of bad code = one line of bad code. One line of a bad plan = ~100 lines of bad code. One bad line in a PRD = ~1,000 lines of bad code — because misalignment in the spec propagates through the entire implementation.

Modular Rules Architecture — Progressive Disclosure

Stop dumping everything into one massive rules file. Use progressive disclosure: CLAUDE.md auto-loads and stays concise (~233 lines), then points to deeper files the agent loads only when it needs them.

.claude/ ├── agents/ ← agent-specific rules (e.g. AGENTS.md) ├── references/ ← on-demand context (loaded when relevant) │ ├── components.md ← UI component layout conventions │ ├── api.md ← API endpoint rules and gotchas │ └── styles.md ← Tailwind/CSS style guide (added via evolution) └── commands/ ← /prime, /create-prd, /plan-feature, /commit, etc. CLAUDE.md (or AGENTS.md) ← auto-loaded; pointers only; keep <250 lines
What CLAUDE.md should contain
⚙️
Tech stack — frameworks, DB, ORM, auth library
▶️
Commands to run/test the app — dev server, migrations, test runner
🗂️
Project structure index — where things live in the codebase
📐
Code patterns — naming conventions, architecture decisions, never-do-X rules
🔗
On-demand context pointers — "If working on front end, read references/components.md. If building API routes, read references/api.md."
🔧 /create-rules command

Run /create-rules after your PRD is done. It reads the PRD for tech stack context, does web research on testing/logging best practices for that stack, and generates the initial CLAUDE.md + reference files using your template structure.

Subagents for Context Isolation

Delegate research to subagents. They explore everything (files, web, git) and load tens or hundreds of thousands of tokens — but only their summary returns to your main context.

✅
Use subagents for: PRD research, architecture exploration, documentation lookup, codebase analysis before planning.
❌
Don't use subagents for implementation. Implementation needs the full file-editing context in the main agent. Subagents for implementation leads to hallucinations — this is why Claude Code's built-in subagents are research-only.
🔒
Context isolation is the point. Research loads massive token counts — if that all hits your main context, it drowns your actual task with noise.

The PIV Loop — Core Feature Development

Every feature runs through Plan → Implement → Validate. Planning and execution are separate conversations (see Golden Rules). The goal of planning is to front-load context so implementation is execution, not exploration.

PLANstructured context
IMPLEMENTexecute task by task
VALIDATEtrust but verify
ITERATEnext task / bugfix
Steps to Planning (per PIV Loop)
1
Run /prime to start the session
Always begin a new conversation with /prime. It reads the PRD, explores the codebase structure, checks the git log, and outputs a mental model summary. Validates the agent knows where things stand before it touches anything.
2
"Vibe planning" — unstructured exploration for this feature
Explore the specific feature: architecture, edge cases, sub-tasks. Spin off subagents for codebase analysis and relevant documentation. Keep it conversational — not structured yet. 💡 For phase 1 of a new project this is very lightweight. Gets more detailed as the codebase grows.
3
Run /plan-feature to create a structured plan document
Converts the conversation into a structured plan with: problem statement, context/references, ordered task list, and — most importantly — a detailed validation strategy defined before any code is written. 💡 Validation strategy = test-driven development thinking. Define success criteria before execution. Forces both you and the agent to be specific about what "done" means.
4
Set up environment variables in parallel
Create a .env.example and set your real .env values before handing off to execution. ⚠️ Critical gotcha: if env vars aren't set, the agent will mock-test and report everything as passing when it isn't. It looks validated but it's not. Set them up during planning, not after.
5
Validate the plan before executing
Read through the structured plan. Confirm task list matches PRD phase scope. Verify the validation strategy is specific enough — if the success criteria are vague, push back before running /execute.
Context Engineering — What Goes in the Plan

Context Engineering = RAG + Memory + Task Management + Prompt Engineering. A complete plan covers all four zones.

RAG

External knowledge pulled in: docs, API specs, existing codebase analysis, architecture references.

Memory

What the AI should know about prior decisions, project history, constraints already established.

Task Management

Ordered task list, dependencies, what "done" looks like, validation criteria per task.

Prompt Engineering

How you frame the problem, persona, output format, constraints on behavior during execution.

Plan Components (Can Include)
🎯
Goal(s) — what success looks like for this feature
✅
Success criteria — measurable, testable outcomes
📚
Documentation to reference — which files, specs, or APIs are in scope
📋
Task list — ordered steps the AI will execute
🔬
Validation strategy — how each task gets verified
🏗️
Desired codebase structure — file layout, naming conventions, patterns
Two Layers of Planning
Layer 1 — Project Planning
Done once, updated rarely
Tech stack decisions
Architecture patterns
Constraints & conventions
Creating stable resources (PRD, global rules)
Layer 2 — Task Planning
Done per feature/task
Codebase analysis for this feature
Documentation analysis
Integration points
Task-specific rules
Steps to Implementing / Validating
1
Execute the implementation plan task by task
Most structure is already in place from planning. Execution is about building, not exploring. 💡 At this point the AI has the context it needs — the plan already contains most of the structure required.
2
Validate the produced code — Trust but verify
Let the AI run, but actively check that it:
  • Uses its tools correctly
  • Is reading/editing the right files
  • Is managing its tasks properly
  • Produces "thinking" tokens that show it understands the plan of attack
Validation Split — Human + AI
👤 Human Validation
Code Review
Manual Tests
Performs high-judgment validation — architectural sanity, edge case intuition, product correctness.
🤖 AI Coding Assistant Validation
Unit Tests
Integration Tests
Runs exhaustive automated tests. Better at systematic coverage than humans but needs human-defined scope.
⚠️ Critical — Iterate Until Complete

The PIV loop repeats per task, not just per feature. Don't batch multiple tasks into a single implementation + validation pass. Compounding unvalidated work is how AI projects lose coherence.

After Foundation — Set Up Regression Testing

Once the first phase is validated and committed, create a command that re-runs all end-to-end tests. This protects past work as you build future PIV loops.

🔁
Ask the agent to list all E2E tests it ran. Convert that list into a reusable /test-regression command so you can re-run the full suite after any future feature.
🐛
When a regression test fails, create a new test that captures the failure, fix the bug, then confirm the test passes. Now that failure class is permanently covered.
📈
Three parallel tracks compound together: as you build out your codebase, you're simultaneously evolving your test base and your AI layer. All three grow in parallel.

Global Rules + On-Demand Context

Rules are the AI's long-term memory between sessions. Global rules live in CLAUDE.md and load automatically. On-demand context loads only when relevant via @file imports.

Global Rules (CLAUDE.md / AGENTS.md)
⚡
Auto-loaded at the start of every conversation. Keep it concise — target under 250 lines. If it's too long, you're overloading context on every task.
🔗
Use as a pointer file — include an "on-demand context" section that tells the agent which reference files to load for which task types.
📌
Good global rules: tech stack summary, dev commands, project structure index, naming conventions, testing strategy, never-do-X constraints.
On-Demand Context (References)
📂
Files like references/components.md, references/api.md, references/styles.md — larger, deeper, loaded only when working on that domain.
🎯
Progressive disclosure. The agent discovers and loads these files as needed, guided by the pointers in CLAUDE.md. It's good at following these references when CLAUDE.md is concise.
📦
These files can be large — that's fine, because they only enter context when their domain is the active task.

⚡Encode a multi-step process you'd otherwise re-type every session.
⚡Examples: /create-prd, /create-rules, /plan-feature, /execute, /commit, /test.

Commands become your reusable interface to AI workflows. Once a command works well, it's free to use forever. Your workflow sophistication compounds across projects.

The Four Golden Rules

These are the highest-leverage operating principles — the ones that prevent the most common failure modes when building complex systems with AI assistants.

01
THE CONTEXT RESET
Planning and execution are SEPARATE conversations. Context window degradation is real — fresh starts matter.
PLAN
research, design
NEW CHAT
AI context
load PRD + plan
→
EXEC
clean context
⚠️ The Failure Mode

After many messages, agents get overwhelmed and make repeated mistakes. Context reset is the fix — not prompt tweaking.

02
COMMAND-IFY EVERYTHING
If you're doing something more than twice, make it a command (aka skill). Your workflows become reusable, shareable tools.
EXAMPLE COMMANDS
/commit
/review
/test
/fix-types
/add-feature
/optimize
One command can encode thousands of tokens of context and instructions. Running /commit beats typing the same instructions every session.
💡 Compounding

Commands accumulate across projects. Your toolkit grows. Each new project starts with a richer set of battle-tested workflows.

03
GIT LOG AS MEMORY
Your commit history is your long-term memory. Standardized commit messages communicate intent to future coding agent sessions.
GIT LOG — as AI context
a3f2d1 feat: add OAuth flow for GitHub
b9e1c4 fix: token refresh race condition
c7a0f2 refactor: split agent into subagents
d5e3b1 chore: update rules per session learnings
The prime command reads the git log to reconstruct what happened since the last session. Bad commit messages = lost context.
04
SYSTEM EVOLUTION MINDSET
Every bug, style mismatch, or hallucination is an opportunity to evolve your AI layer — not just fix the immediate issue. The question isn't "how do I fix this?" but "how do I prevent this class of mistake forever?"
bug / mismatch
→
ask: "why did AI do this?"
→
update rules / add reference file
→
+ rule in CLAUDE.md
Real example from the transcript: AI generated styles that didn't match the intended look. Root cause: no style guide in the AI layer. Fix: don't just restyle — ask the agent to reason about what should change in the rules. Then add references/styles.md (Tailwind conventions, component visual patterns) to prevent recurrence.
EVOLVING IN PARALLEL
CODEBASE

TEST BASE
regression coverage
AI LAYER

⚠️ Don't skip the meta-reasoning step

When something goes wrong: tell the agent "don't change anything yet — help me reason about what we should add to our rules or on-demand context to prevent this." Own the change to the AI layer yourself. Don't just let the agent patch it and move on.

Commands Quick Reference

Commands (skills) are the reusable unit of AI workflow. Each one encodes a repeatable process. Build your library incrementally — every project adds to it.

Core Commands
Command What It Does When to Use
/prime Reads PRD + CLAUDE.md, explores codebase structure, checks git log, outputs mental model summary Start of every new implementation conversation
/create-prd Converts the current conversation (brain dump + Q&A) into a structured PRD with MVP scope, out-of-scope, directory structure, and phases After brain dump + agent Q&A, before any code
/create-rules Reads PRD for tech stack context, researches testing/logging best practices, generates CLAUDE.md + reference files from your template After PRD is created, before first PIV loop
/plan-feature Converts vibe planning conversation into a structured plan doc: problem statement, task list, references, and validation strategy Planning phase of each PIV loop
/execute Takes the structured plan as its only context (fresh conversation), delegates all coding to the agent, runs validation automatically Implementation phase — always a new context window
/commit Stages changes and generates a standardized commit message following your project conventions After each validated task — git log = long-term memory
/review Reviews current diff for correctness, style, security, and compliance with project rules Before human code review or merge
/test Runs the appropriate test suite for the changed code and reports results Validation step of the PIV loop
Command Design Principles
✓ Good Commands
✅Encode a repeatable, multi-step process you'd otherwise re-explain every session.
✅Reference specific project files, conventions, or standards by path.
✅Have a clear output — a commit, a test report, a diff, a plan doc.
✅Are idempotent — running twice doesn't break things.
✗ Bad Commands
❌Vague instructions with no defined output ("be more helpful").
❌Commands that combine unrelated concerns in one shot.
❌Commands that assume context without loading it explicitly.
❌Commands you've only needed once — don't over-engineer your toolkit.
Prime Command — Full Anatomy

The prime command is the most important one to get right. Here's the full structure it follows when invoked:

# PRIME COMMAND — /plan
Step 1 — Read back context
  @PRD.md                     ← product requirements
  @CLAUDE.md                  ← global rules + pointers
  @.claude/references/*.md    ← relevant on-demand context

Step 2 — Explore structure
  list top-level directories
  read key entry points (main, index, app)
  map module relationships

Step 3 — Check git log
  git log --oneline -20        ← what changed recently
  git diff HEAD~3             ← recent diffs if needed

Step 4 — Build mental model
  Output written summary before acting:
  - Current state of the codebase
  - What was done recently
  - What the plan is for this session
  - Any risks or blockers identified
🚀 The Minimal Viable Starter Set

Start with: /prime, /create-prd, /create-rules, /plan-feature, /execute, and /commit. These six commands cover the entire greenfield loop. Everything else you build as you encounter repeated manual steps. The goal is a starter kit you can bring into any new project and evolve — not a framework you learn once and never touch.
