The flow of one placeholder, end-to-end

Click a section to see how $user participates. Notice it's the same name everywhere — that's the unification mechanic.
eventsBIND
matchGROUP
outcomeUSE
conditionGATE
events: $e.target.user.userid = $user — the placeholder is *created* by binding it to a field. Every event the engine sees now has a $user value attached.

Why this matters

The unification rule is what makes multi-event joins work. When you write the same placeholder name in two event variables' predicates, you're forcing the engine to consider only event combinations where the values agree. Same name = same value. This is the entire mechanic.
// MULTI-EVENT JOIN VIA SHARED PLACEHOLDERevents: $fail.target.user.userid = $user // $user bound here... $fail.security_result.action = "BLOCK" $ok.target.user.userid = $user // ...must equal here $ok.security_result.action = "ALLOW" match: $user over 10m // group joined results by $user

Resulting condition options

Each aggregation unlocks a different shape of predicate. Pick which question your query is asking.
// THRESHOLDoutcome: $n = count($e) condition: $e and $n >= 5
// FAN-OUToutcome: $h = count_distinct($host) condition: $e and $h > 3
// COMPOSITE RISKoutcome: $s = max( if($priv, 80, 0) + if($night, 30, 0)) condition: $e and $s >= 80

Cause & effect

REMEMBER. Aggregations only behave as you expect when match: is present. Without it, every group has size 1, so count($e) = 1, max(...) equals the single value, array_distinct is a one-element array. The query/rule still compiles, but the threshold logic in condition: never trips. This is the silent-bug class to watch out for.
