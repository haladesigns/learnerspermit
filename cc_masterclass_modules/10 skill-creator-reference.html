Skill-Creator

The meta-skill for building skills. Install it first — it covers the full lifecycle from creation → testing → optimization → shipping. Before Skill-Creator, you guessed at descriptions and hoped things worked. Now you measure.

What It Does
🏗️
Creates new skills — generates folder structure, SKILL.md, frontmatter, and optimized description from a plain-language brief.
🧪
Evals existing skills — define test prompts + expected outputs, get pass/fail rates, timing, and token usage.
⚙️
Improves skills — analyzes failures, proposes targeted fixes, iterates automatically up to 5 rounds.
📊
Benchmarks — runs standardized assessment across multiple runs with variance analysis. Tracks pass rate, time, tokens.
Four Internal Agents
Executor
Runs skills against eval prompts in parallel, each in a clean isolated context with its own token and timing metrics.
Grader
Evaluates each output against the defined expectations — produces pass/fail with reasoning.
Comparator
Blind A/B judge — compares two skill versions, or skill vs. no skill, without knowing which is which.
Analyzer
Reads graded failures, generalizes patterns, and suggests targeted improvements to the skill body or description.
Two Types of Skills (Matters for Evals)
⬆️
Capability uplift — encodes techniques the base model can't do consistently. Evals tell you when the model has caught up and the skill is no longer needed.
📋
Encoded preference — sequences Claude through your team's specific process. More durable, but evals verify it still matches the actual workflow.

Four Modes

Skill-Creator is a single skill with four operating modes. Invoke with /skill-creator and state your intent — it figures out which mode applies. You can also be explicit.

🏗️
Create
mode 1
Generates a complete new skill from a plain-language description. Produces folder structure, SKILL.md with frontmatter, body instructions, and an initial optimized description field.
"Create a new skill that reviews PRs for security issues"
"Create a skill that generates blog posts purely in emojis"
🧪
Eval
mode 2
Runs structured tests against an existing skill. You provide test prompts and describe what "good" looks like — the Grader agent scores each run. Reports pass rate, timing, tokens.
"Run evals on my code-review skill"
"Test my deploy skill against these 5 scenarios"
⚙️
Improve
mode 3
Takes existing evals and iterates on the skill. The Analyzer reads failures, proposes changes, re-runs — up to 5 iterations. Also runs the description optimizer separately to tune trigger accuracy.
"Improve my deploy skill based on these test cases"
"Optimize the description for my frontend-design skill"
📊
Benchmark
mode 4
Runs a standardized assessment across multiple runs. Tracks variance — so you know if a "95% pass rate" is stable or luck. Comparator agents do blind A/B: skill vs. no skill, or v1 vs. v2.
"Benchmark my skill across 10 runs and show variance"
"A/B test my new review skill against the old one"
💡 Flexible Order

The modes don't have to run in sequence. If you just want to vibe and iterate without formal evals, that's fine — tell Skill-Creator you don't need evals and it'll work more loosely. The structure is there when you want rigor, not as a gate.

Example Prompts per Mode
ModeExample PromptWhat Happens

Eval "Evaluate my code-review skill and suggest improvements" Generates test cases, runs them, shows eval viewer

The Eval System

Evals turn a skill that seems to work into one you know works. There are two entirely separate eval types — a skill can pass one and fail the other. You need both.

📋 Quality Eval — Does it do the job?

Define test prompts + files (if needed) and describe what good output looks like. The Grader scores each run against your expectations.

Catches: wrong format, missing sections, logic errors, regressions after a model update.

Target: 90%+ for critical skills · 70–80% for preference skills (some variation is natural)

🎯 Trigger Eval — Does it fire when it should?

Tests whether the description causes the skill to activate at the right times. Generates both trigger queries (should fire) and non-trigger queries (should not).

Catches: false positives (triggers on wrong requests), false negatives (never fires).

Run this after quality evals pass — no point optimizing trigger accuracy if output is bad.

How the Optimization Loop Works
1
Split eval set 60 / 40
60% train — used during optimization. 40% held out — used for final validation only. Prevents overfitting to the test set.
2
Evaluate current description
Each query runs 3 times to get a reliable trigger rate. Parallel agents run simultaneously — no context bleed between runs.
3
Analyze failures → propose improvement
The Analyzer sees which queries failed and generalizes to broader categories of intent. Instruction: "Don't list specific cases — that overfits. Stay under 200 words."
4
Re-evaluate on both train + test
The new description is scored on both sets. Best description is selected by test score, not train score — specifically to avoid overfitting. 💡 Repeats up to 5 iterations automatically.
5
Generate eval viewer
An HTML report showing outputs, qualitative feedback, and metrics — built for a human to review before accepting changes. ⚠ Always generate the eval viewer before making manual corrections yourself. See the results first.
Two Reasons to Eval
🔁
Catch regressions. Models and infrastructure evolve. A skill that worked last month may behave differently today. Run evals after any model update.
📈
Know when the skill is obsolete. If the base model passes your quality evals without the skill loaded, the techniques may have been absorbed into the model. The skill isn't broken — it's done its job.
Writing Good Eval Queries
✅
Substantive queries: Claude only consults skills for tasks it can't handle on its own. Simple one-step queries like "read this file" won't trigger skills regardless of description quality — don't use these as test cases.
✅
Include edge cases: minimal projects, unusual inputs, boundary conditions. These reveal blind spots in the skill instructions.
❌
Don't just use happy-path examples. A skill that only passes easy cases is a false sense of security.
⚠ Critical Failure Mode

A skill can have great output quality but never trigger — or perfect trigger accuracy but produce bad output. These are independent failure modes. Both evals must pass before shipping.

Description Optimization

The description field is not metadata — it's a learnable parameter. It determines when your skill fires. Skill-Creator has a dedicated optimizer for this, separate from the quality eval loop.

The Undertrigger Problem
⚠️
Claude currently has a built-in tendency to undertrigger — to not use a skill even when it would help. This is the most common failure mode in production skill libraries.
🔧
The fix: Make descriptions "pushy." Explicitly list the conditions that should trigger it, even if the user doesn't name the skill directly.
✓ Pushy (Works)
✅"Reviews code for bugs and security issues. Make sure to use this skill whenever the user asks to review, audit, check, or look over any code — even if they don't use the word 'review'."
✅Names synonyms: "deploy, ship it, push to staging, release to prod"
✅Covers implicit intent: "even if they don't explicitly ask for a dashboard"
✗ Passive (Undertriggers)
❌"Helps with deployment" — too vague, no trigger signals
❌"A code review skill" — passive noun phrase, not a trigger condition
❌No explicit "use when" clause
The 250-Character Budget
📏
Each skill description is capped at 250 characters in the index — regardless of how much you write. With many skills loaded, descriptions are truncated to fit a total budget.
⬆️
Front-load the key use case. Put the most important trigger condition first — that's what survives truncation. Don't bury the main trigger after a long preamble.
🔧
Total budget scales at 1% of context window (fallback: 8,000 chars total). To raise the per-session limit: set env var SLASH_COMMAND_TOOL_CHAR_BUDGET.
# Bad — key info buried after 200 chars of preamble
description: "This skill is designed to assist developers who are working on
  frontend web development projects and need help with design. It covers
  many aspects including layout, typography, color schemes, and UX patterns.
  Use when creating web interfaces."

# Good — key trigger front-loaded within first 100 chars
description: "Create distinctive frontend UIs. Use when building web pages,
  components, or any visual interface. Avoids generic AI aesthetics."
Running the Description Optimizer
1
Tell Skill-Creator which skill to optimize
"Optimize the description for my code-review skill" or "Run the description improver on my deploy skill"
2
It generates trigger + non-trigger test queries
Trigger queries = things that should activate the skill. Non-trigger queries = things that shouldn't. Balance is important — over-triggering is as bad as under.
3
Run the optimization loop
run_loop.py — 60/40 split, up to 5 iterations, test score selects winner. Returns best_description as JSON. 💡 Anthropic ran this across 6 public document-creation skills and saw improved triggering on 5 of 6.
4
Paste best_description into SKILL.md frontmatter
Skill-Creator shows you before/after and reports scores. Verify the new description still makes sense to a human before committing.

Bundled Skills

Skills that ship with Claude Code — available in every session with no install required. Unlike built-in commands (which run fixed logic), bundled skills are prompt-based: they give Claude a playbook and let it orchestrate using its tools, spawn parallel agents, and adapt to your codebase.

📌 Note: Commands → Skills

Custom commands (.claude/commands/deploy.md) and skills (.claude/skills/deploy/SKILL.md) are now unified. Both create /deploy and work identically. Existing commands keep working. Skills add optional extras: supporting files, invocation control, subagent execution. If a skill and command share the same name, the skill wins.

Built-in Bundled Skills
SkillWhat It DoesNotes
/batch <instruction> Orchestrates large-scale codebase changes in parallel. Decomposes work into 5–30 independent units, presents a plan, then spawns one background agent per unit in an isolated git worktree. Requires a git repo. Each agent implements, tests, and opens a PR.
/claude-api Loads Claude API reference for your project's language (Python, TS, Java, Go, Ruby, C#, PHP, cURL) plus Agent SDK docs. Covers tool use, streaming, batches, structured outputs. Also auto-activates when your code imports the Anthropic SDK.
/debug [description] Enables debug logging for the current session and analyzes the session log. Debug logging is off by default — running /debug mid-session starts capturing from that point. Optionally describe the issue to focus the analysis.
/loop [interval] <prompt> Runs a prompt repeatedly on an interval while the session stays open. Useful for polling deployments, babysitting PRs, or periodically re-running another skill. Example: /loop 5m check if the deploy finished
/simplify [focus] Reviews recently changed files for code reuse, quality, and efficiency. Spawns three review agents in parallel, aggregates findings, applies fixes. Pass text to focus: /simplify focus on memory efficiency
/skill-creator The meta-skill. Create, eval, improve, and benchmark other skills. Install via /plugin marketplace if not present.
Monorepo Support
📁
When editing files in subdirectories, Claude Code automatically discovers skills from nested .claude/skills/ directories. Editing packages/frontend/App.tsx → also loads skills from packages/frontend/.claude/skills/.
🔄
Skills inside --add-dir directories are also loaded automatically and support live change detection — edit them during a session without restarting.

End-to-End Workflow

The complete pipeline from idea to production-ready skill. Both eval tracks must be green before shipping. Maintain skills the same way you maintain code — re-run evals after model updates.

Full Lifecycle
1
Create with Skill-Creator
"Create a skill that [does X]" — generates the folder, SKILL.md, frontmatter, initial description. Or write manually and use Skill-Creator only for optimization.
2
Write quality eval test cases
Define 10–20 substantive prompts with expected output descriptions. Include edge cases, minimal inputs, and adversarial examples — not just happy path. 💡 Aim for 13+ queries. The eval set is the skill's contract — it defines what the skill is supposed to do.
3
Run quality evals
Executor runs prompts in parallel (clean context per run). Grader scores each. View the eval viewer before making any manual changes. ⚠ Always generate the eval viewer first. Don't self-correct before seeing the results.
4
Iterate on skill body until quality evals pass
Use Improve mode — Analyzer reads failures, proposes targeted edits, re-evals. Up to 5 automatic iterations. Then human review. ✓ Target: 90%+ for critical skills, 70–80% for preference skills.
5
Run trigger eval — optimize description
Only after quality evals pass. Run the description optimizer loop. Check the 250-char budget. Front-load trigger conditions. Use "pushy" language.
6
Benchmark
Run 10 passes for variance analysis. Run blind A/B vs. baseline Claude — if the base model matches, the skill may be redundant.
7
Ship — then maintain
Package with package_skill.py → .skill file for distribution. Re-run evals after model updates. A skill that worked last month may need re-tuning after a new model release.
Packaging & Distribution
📦
Package: python -m scripts.package_skill <path> → produces a .skill file for distribution.
🔗
Share via GitHub, team shared drive, or upload to the Anthropic skills marketplace. The awesome-claude-skills repo on GitHub curates community contributions.
📥
Install: place .skill file in skills directory, or use Claude's install command. Team members get the same skill pinned to the same version.
✓ Maintenance Checklist
✅Re-run quality evals after any model update
✅Re-run trigger evals if skill count in library changes
✅Check if base model now passes evals without the skill — may be obsolete
✅Update eval set when your workflow changes
✗ Common Mistakes
❌Skipping test cases — even subjective skills need at least a few qualitative checks
❌Optimizing descriptions before quality evals pass
❌Using only 2–3 eval queries — too few to detect patterns
❌Ignoring timing data — optimize for efficiency, not just correctness
🔭 Looking Ahead

As models improve, the line between "skill" and "specification" may blur. A SKILL.md is currently an implementation plan — detailed instructions telling Claude how to do something. Over time, a natural-language description of what the skill should do may be enough. The eval framework is a step in that direction: evals already describe the "what." Eventually, that description may be the skill itself.
