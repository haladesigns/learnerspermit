/loop — Recurring Prompts on Autopilot

Schedule any Claude Code prompt to run automatically on a repeating interval — no external cron jobs, no scripts, no alt-tabbing. Claude polls in the background and reports back when something changes. Works with any prompt or custom skill.

What It Does
🔄
Runs any prompt on a recurring interval while your terminal stays open — no manual re-prompting.
🛠️
Can invoke other slash commands on a schedule — /review-pr, /commit, any custom skill.
⏰
Handles one-shot reminders too — "remind me at 3pm to push the release branch."
🆔
Each task gets a unique 8-character ID for easy management and cancellation.
Key Limits at a Glance
72h
Auto-expires after 3 days
50
Max tasks per session
10%
Max timing jitter
Tasks are session-scoped — close the terminal and all loops stop. Requires Claude Code v2.1.72+.
✓ No Catch-Up

If Claude is busy when a task fires, it runs once when idle — not once per missed interval. No duplicate-run floods.

↗ Low Priority

Scheduled tasks run between turns. Active conversations take priority — you won't get interrupted mid-chat.

🌍 Timezone Aware

When you say "at 3pm", it means 3pm in your local timezone. Natural language timing just works.

⚠ Session-Scoped — This Is Not a Cron Job

Tasks die when you close the terminal. For persistent overnight or cross-restart automation, use /schedule in the Claude Desktop app or GitHub Actions instead.

Step-by-Step Usage
1
Schedule a recurring task
Interval first, then the prompt. The interval can use s, m, h, or d. 
/loop 5m check if the deployment finished and tell me what happened
/loop 20m /review-pr 1234
/loop 1h run the test suite and report any failures
💡 The interval can also go at the end: /loop check the build every 2 hours
2
Omit the interval — defaults to every 10 minutes
Claude infers a sensible default when no timing is given. 
/loop check the build     ← runs every 10 min
3
Schedule a one-time reminder
Natural language — fires once, then self-deletes. 
remind me at 3pm to push the release branch
in 45 minutes, check whether the integration tests passed
4
List all scheduled tasks
Ask Claude in plain English — shows IDs, schedules, and descriptions. 
what tasks do I have scheduled?
5
Cancel a task
Describe it naturally, or use the 8-char ID from the task list. 
cancel the deployment monitoring task
cancel task abc12345
⚠ Tasks also auto-expire after 72 hours — you don't have to cancel them manually.

🔧 Disable Entirely

Set the environment variable CLAUDE_CODE_DISABLE_CRON=1 to turn off scheduled tasks completely across a project or environment.

💡 Power Move

Combine /loop with a custom skill to turn any repeatable workflow into automated monitoring. If you built /check-status, just run /loop 10m /check-status.

📦 Pattern 1 — Deployment Monitor
You just deployed. Walk away and come back to a status report.
/loop 5m check if the Vercel deployment is finished
and whether the health check passes
Claude polls every 5 min, reports live status, flags errors.
🔍 Pattern 2 — PR Babysitter
Keep an eye on CI checks and review comments without refreshing GitHub.
/loop 20m /review-pr 1234
Every 20 min: new comments, CI status, merge conflicts.
🧪 Pattern 3 — Test Suite Watchdog
Continuous feedback while refactoring — only bothers you if something breaks.
/loop 10m run npm test and tell me if anything broke
Runs tests every 10 min; silent on green, loud on red.
🚨 Pattern 4 — Autonomous Error Monitor
Scan logs and automatically fix what's fixable.
/loop 15m scan the error logs and create a PR
if anything is fixable
Aggressive autonomy — Claude acts, not just reports.
⏰ Pattern 5 — End-of-Day Reminder
Simple but saves you. Fires once, done.
remind me at 5pm to commit and push my changes
One-shot. Timezone-aware. Self-deletes after firing.
Gotchas & Edge Cases
☠️
Dies on terminal close. /loop is session-scoped. Close the window, kill the process, shut down the machine — all loops stop instantly. No persistence, no recovery.
⏳
72-hour hard cap. Even with the terminal open, tasks auto-expire after 3 days. They fire one final time before deletion. This is a safety net, not a bug.
🕐
Timing jitter up to 10% (capped at 15 min). A 5-minute loop might fire at 5:30. One-shot tasks at round times (:00 or :30) can fire up to 90 seconds early.
🚫
No catch-up runs. If Claude missed 6 fires of a 5-minute loop because it was busy, it runs exactly once when idle. Intentional — prevents duplicate-run floods.
📊
Max 50 tasks per session. Unlikely to hit in normal use, but worth knowing. Each task consumes a slot regardless of type (recurring or one-shot).
⚡
Tasks queue, not interrupt. If you're actively chatting when a task fires, it waits for Claude to finish your request. You will never be interrupted mid-conversation.
🔧
CLAUDE_CODE_DISABLE_CRON=1 — set this env var to disable /loop entirely. Useful for CI environments or projects where scheduled tasks shouldn't run.

✅Monitoring a deploy during a work session
✅Babysitting CI while you work on something else
✅Running a custom skill on a short interval
✅One-time reminders during the day
✅Anything short-lived, temporary, session-scoped

❌Overnight or cross-restart automation
❌"Every morning at 9am" recurring jobs
❌Tasks that need to survive a laptop sleep/wake
❌Any automation requiring catch-up on missed runs
❌Tasks that must run even without an open terminal

Choosing the Right Tool

All three options can run Claude automatically on a schedule. The difference is persistence — how long they survive without a terminal, and whether they need a session open to run.

Feature Comparison
Feature /loop (CLI) /schedule (Desktop) GitHub Actions
Survives terminal close ✗ No ✓ Yes ✓ Yes
Survives system sleep/restart ✗ No ~ Partial ✓ Yes
Requires open session ✗ Yes (must stay open) ✓ No (creates new session) ✓ No
Auto-expires 3 days ✓ Persistent ✓ Persistent
Catch-up on missed runs ✗ No ✓ Yes ✓ Yes
Zero setup ✓ One command ~ Desktop app required ✗ YAML config needed
Works with custom skills ✓ Yes ✓ Yes ~ Indirectly
Best for Active session monitoring Daily recurring automation CI/CD workflows
/loop — Use When

You're in a working session and want temporary monitoring that dies when you're done. Quickest to set up.

/schedule — Use When

You want something that runs every morning or persists between sessions. Needs Claude Desktop app open, not terminal.

GitHub Actions — Use When

The task needs to run regardless of whether anyone has a session open. True CI/CD, no dependency on a running machine.
