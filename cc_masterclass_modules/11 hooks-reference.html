Hooks — Event-Driven Automation

Hooks fire automatically when specific events occur in Claude Code — not based on natural language. Think of them as triggers: "when X happens, always do Y." No prompting required, no skill description to match.

What Hooks Do
🔔
Fire unconditionally when an event occurs — not based on intent or language matching.
⚙️
Run shell commands, scripts, or external programs as side effects at key lifecycle points.
🔁
Attach to Stop, PreToolUse, PostToolUse, OnToolError, or SessionStart events.
🎯
Live in settings.json — not in a separate skill folder.

🔔 Primary Use Case (from video)

Audio/visual notifications when Claude finishes a task — so you can tab away and get a heads-up when Claude needs your attention again. The most common hook is a Stop hook that plays a sound or sends a system notification.

💡 Easiest Way to Create Hooks

Just ask Claude Code in plain English: "Create a hook that plays an audio sound whenever you finish a task." Claude will research the docs and write the hook config for you. Restart the Claude Code session for changes to take effect.

The Five Hook Events
Stop
Fires when Claude finishes responding and returns control to you.

Best for: audio chimes, desktop notifications, logging session end, auto-committing changes.
PreToolUse
Fires before Claude executes any tool call (Bash, Read, Write, etc.).

Best for: safety checks, confirmation prompts before destructive ops, pre-flight validation.
PostToolUse
Fires after a tool call completes successfully.

Best for: logging what was run, formatting output, triggering follow-up scripts, auditing file writes.
OnToolError
Fires when a tool call fails.

Best for: alerting on errors, fallback actions, error logging to external systems, Slack/email on failure.
SessionStart
Fires when a new Claude Code session begins.

Best for: loading env context, displaying project status, running `git status`, announcing active branch.
Example: Stop Hook (Audio Notification)

This is the exact hook created in the video — plays an ascending 3-tone chime when Claude finishes responding.

// ~/.claude/settings.json  (personal)  or  .claude/settings.json  (project)
{
  "hooks": {
    "Stop": [
      {
        "command": "python3 -c \"
import subprocess, sys
notes = [523, 659, 784]
  # C5, E5, G5
for freq in notes:
  subprocess.run(['play', '-n', 'synth', '0.15', 'sine', str(freq)], ...
\""
      }
    ]
  }
}
Example: SessionStart Hook (Git Status)
// Show current branch and status at session start
{
  "hooks": {
    "SessionStart": [
      {
        "command": "echo '=== Session Start ===' && git branch --show-current && git status --short"
      }
    ]
  }
}
Example: PreToolUse Hook (Guard Destructive Commands)
// Warn before any rm command is run
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": { "tool": "Bash", "command_pattern": "^rm " },
        "command": "echo '⚠️  rm detected — confirm in chat'"
      }
    ]
  }
}
Two Ways to Set Up Hooks
A
Ask Claude Code (easiest — shown in video)
Type exactly what you want in plain English. Claude researches the hooks documentation and writes the config. 
# Just say:
"Create a hook that plays an audio sound whenever you finish a task."
"Add a hook that runs git status at the start of every session."
"Add a hook that logs all Bash commands to ~/claude-audit.log."
⚠ Restart your Claude Code session after creation for the hook to take effect.
B
Edit settings.json manually
Open the settings file directly and add a hooks key. 
# Personal (all projects):
~/.claude/settings.json

# Project-level (this project only):
.claude/settings.json
💡 Use personal for notification preferences, project-level for team-shared guards and audit hooks.
settings.json Structure
// Full structure — hooks key sits alongside other settings
{
  "hooks": {
    "Stop": [                         // Array of hook objects
      {
        "command": "your-shell-command"   // Required: shell command to run
      }
    ],
    "PreToolUse": [
      {
        "matcher": {                     // Optional: filter by tool/pattern
          "tool": "Bash",
          "command_pattern": "^rm -rf"   // Regex pattern to match
        },
        "command": "notify-send 'Danger!' 'rm -rf detected'"
      }
    ],
    "PostToolUse": [ ... ],
    "OnToolError": [ ... ],
    "SessionStart": [ ... ]
  }
}

✓ Good Uses for Hooks
✅Audio/visual notifications when Claude finishes (Stop hook).
✅Logging all tool calls to an audit file for compliance.
✅Pre-flight guards before destructive Bash commands.
✅Auto-format or lint after every file write (PostToolUse).
✅Display git status or env info at session start.
✅Send Slack/webhook alert on tool errors.
✗ Use Skills Instead (not Hooks)
❌Workflows you want to trigger with a command or description.
❌Multi-step processes that need Claude's reasoning.
❌Anything requiring arguments or dynamic inputs at call time.
❌Code review, deployment, doc generation — use skills for these.
❌Actions that should only run sometimes (hooks always fire).
Gotchas
⚠️
Restart required: After adding or editing a hook in settings.json, you must restart your Claude Code session for the hook to take effect. Edits are not hot-reloaded.
⚠️
Hooks always fire: Unlike skills, there is no "when to use" filter. A Stop hook fires on every response. Keep hook commands fast and non-blocking, or they'll delay every response.
⚠️
Shell command context: Hook commands run in the shell — they do not have access to Claude's conversation context. You cannot inject text into Claude's response from a hook.
⚠️
Side effects on every event: Use hooks for notifications and logging. Do not use hooks to auto-deploy, auto-commit, or auto-send messages — these are irreversible and hooks fire unconditionally.
⚠️
Dependency availability: Your hook command must be available in the shell PATH. If you use say (macOS) or notify-send (Linux), verify the binary exists first. Ask Claude to check with a simple which say.
💡
Tip: Hooks and skills are complementary — a Stop hook can notify you that Claude finished, while the skill that ran the task handles all the logic. Layer them intentionally.
