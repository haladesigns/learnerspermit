Godot 4 — Level Design

Building a level in Godot means combining three things: TileMapLayer nodes (the grid-painted terrain), PackedScene instancing (reusable objects like enemies, pickups, platforms), and a scene hierarchy that keeps everything organised. This tool is your end-to-end reference — from first tileset import to scripting dynamic tile changes.

The Three Pillars of a Godot Level
🗺️
TileMapLayer — the painted grid. Terrain, walls, floors, decoration. Efficient, physics-aware, supports autotiling. One node per visual/logical layer.
🧩
Scene Instancing — every interactive object (enemy, coin, door, chest) lives in its own .tscn file and gets placed as an instance. Change once, update everywhere.
🏗️
Scene Hierarchy — a clean, ordered scene tree keeps your level readable, makes debugging easier, and prevents node-access bugs. Structure intentionally.
Recommended Level Scene Tree
Level1 (Node2D) ← scene root, holds level script ├─ World (Node2D) ← groups all tile layers │ ├─ BG (TileMapLayer) ← decoration, no collision │ ├─ Ground (TileMapLayer) ← solid terrain, has physics │ └─ Foreground (TileMapLayer) ← top-layer decoration ├─ Entities (Node2D) ← all spawned / placed objects │ ├─ Player (instance of player.tscn) │ ├─ Enemies (Node2D) ← container node │ │ ├─ Goblin1 (instance of goblin.tscn) │ │ └─ Goblin2 (instance of goblin.tscn) │ └─ Pickups (Node2D) ← container node │ └─ Coin1 (instance of coin.tscn) ├─ SpawnPoints (Node2D) ← Marker2D nodes, invisible └─ Camera2D
⚠️ TileMap is Deprecated in Godot 4.3+

The old TileMap node (with built-in layers) is deprecated as of Godot 4.3. Use TileMapLayer nodes instead — one node per layer. They share the same TileSet resource and work identically, just with a cleaner API. If you open old projects, Godot will offer to auto-convert.

💡 TileSet vs TileMapLayer

TileSet is the resource — the collection of tiles, their physics, terrains, and custom data. It's a .tres file you define once and reuse across multiple TileMapLayer nodes and multiple levels. TileMapLayer is the node — it paints cells from that TileSet onto a grid.

TileMapLayer Setup — Step by Step

This is the first thing you do when starting a level. Get this right once and painting tiles is instant for every future level.

Phase 1 — Create the TileSet Resource
1
Add a TileMapLayer node to your scene. In the Inspector, click the Tile Set property → New TileSet. This creates a shared resource — save it as a .tres file so other levels can reuse it. 💡 One TileSet can hold multiple atlases (sprite sheets). Add all your tile graphics into the same TileSet.
2
Set Tile Size in the TileSet inspector. This must match your sprite sheet tile size — common sizes are 16×16, 32×32, 48×48. Set this first before adding any tiles. ⚠️ Getting tile size wrong causes misaligned tiles and broken physics shapes. Set it before importing art.
3
Import your sprite sheet — drag the PNG from the FileSystem panel into the TileSet bottom panel → Tiles section. Godot will ask if you want to auto-create tiles from the grid. Click Yes. 💡 You can add multiple atlases (different PNGs) to one TileSet. Drag them in one by one.
4
Paint your level — switch to the TileMap bottom panel, select a tile, and left-click/drag on the canvas. Eraser tool removes tiles. Right-click to pick a tile from the canvas. 💡 Use the Rect and Fill tools for large areas. Hold Shift to draw a line.
Phase 2 — Layer Strategy
1
Add one TileMapLayer per logical layer. They all share the same TileSet resource. A typical 2D level uses: BG (sky, background), Ground (solid terrain), Decoration (foreground details), Hazards (spikes, lava). 💡 In the Inspector, each TileMapLayer has a Z Index property — use this to control draw order without changing the scene tree.
2
Only add Physics to layers that need collision. Background and decoration layers should never have physics layers — it wastes CPU and causes invisible collision bugs.
3
Rename layers descriptively — Ground, Walls, Hazards. You'll be selecting them from a dropdown in the TileMap panel; good names save confusion.

💡 Sharing a TileSet across levels

Save your TileSet as a .tres file (Inspector → right-click TileSet → Save). Then in every new level, drag that .tres file onto the Tile Set property of each TileMapLayer node. All physics shapes and terrains defined in the TileSet come along — zero repeated setup.

Tile Physics Setup

Tiles don't collide by default. You have to add a Physics Layer to the TileSet resource and draw shapes on tiles. Do this once in the TileSet, and every TileMapLayer that uses it inherits the collision.

Phase 1 — Add a Physics Layer to TileSet
1
Select your TileMapLayer node → click the TileSet resource in Inspector → expand Physics Layers → click Add Element. 💡 Name it something like "world" — it helps when you have multiple physics layers for different collision purposes (e.g. "world" for terrain, "hazards" for damage zones).
2
Set Collision Layer and Collision Mask. For a basic game: terrain on Layer 1, player on Layer 2, enemies on Layer 3. Terrain mask = 0 (nothing), Player mask = 1 (collides with terrain). ⚠️ The physics layer on the TileSet is separate from the layer/mask on the TileMapLayer node itself. Both need to be configured.
Phase 2 — Draw Shapes on Tiles
1
Open the TileSet panel (bottom of editor) → switch to Select mode → click a tile from your atlas. In the right panel, expand Physics Layer 0.
2
For rectangular tiles (most terrain): Switch to Paint mode → select Physics Layer 0 in the paint properties → paint across all your solid tiles at once. Godot fills each tile with a full rectangle automatically. 💡 Paint mode is the fastest way to bulk-assign rectangular collision. Use Select mode only for custom polygon shapes.
3
For irregular shapes (slopes, half-tiles): In Select mode, click the tile → click the + polygon icon in the physics panel → draw the collision polygon by clicking points. Click the first point again to close the shape. 💡 Use the snap-to-grid toggle when drawing polygons for crisp edges. Hold Ctrl to temporarily disable snap.
4
Verify in-game: Run the scene → open Debug → Visible Collision Shapes. You should see orange outlines on every solid tile. Gaps mean missing physics shapes. ⚠️ The most common reason a player falls through tiles is a missing or incorrectly sized collision shape on that tile.
One-Way Platforms
1
Add a second Physics Layer in TileSet → name it "one_way". Set its Collision Layer to match your terrain layer.
2
In the TileSet panel, Select your platform tile → expand Physics Layer 1 (the one-way layer) → draw a thin rectangle collision shape at the very top edge of the tile only.
3
In the physics shape panel, enable "One Way" on that polygon. Set one_way_collision_margin to a small value like 1.0. 💡 Drop-through logic (hold Down + Jump) requires a small script on the player — see the GDScript tab for the pattern.
⚠️ Ghost Collisions Between Tiles

If your player snags on invisible edges between adjacent tiles, it's because the physics shapes slightly overlap at seams. Fix: use simple, full-tile rectangles rather than polygons for flat terrain. The engine merges adjacent rectangles into a single surface automatically.

⚠️ Collision Layer vs TileSet Physics Layer

There are two separate layer settings: the Physics Layer defined inside the TileSet resource (which layer/mask the shapes use), and the collision_layer / collision_mask on the TileMapLayer node itself. Both must be set correctly. The TileSet physics layer inherits the node's settings.

Terrain & Autotile

Terrains let you paint with a brush instead of individual tiles — Godot picks the correct tile variant (corner, edge, inner corner) automatically based on neighbours. This is the workflow for organic-looking levels.

Terrain Setup — Step by Step
1
In TileSet Inspector → Terrain Sets → Add Element. Give it a name (e.g. "Grass") and choose a Mode: 💡 Match Corners and Sides — most common, gives full 47-tile autotile set. Match Corners — 16 tiles needed. Match Sides — 16 tiles needed, simpler.
2
Assign tiles to the terrain. In the TileSet panel → Select mode → click a tile → expand Terrain → set Terrain Set to 0, then set each peering bit (the 8 surrounding squares) to your terrain type. Tiles with all 8 bits set = full surrounded tile. Tiles with only top bit set = bottom edge, etc. ⚠️ This is the most tedious part of Godot 4 terrain. Use Paint mode to mass-assign terrain bits to multiple tiles at once.
3
Paint with Terrain mode. In the TileMap panel → switch to Terrains tab → select your terrain → paint. Godot automatically chooses the right tile variant as you paint. 💡 You can mix manual tile painting and terrain painting in the same layer.
Terrain Modes Explained

Custom Data Layers
🏷️
Add Custom Data Layers to TileSet (Inspector → Custom Data Layers → Add Element). Name: tile_type, Type: String.
🎨
In TileSet panel → Select a tile → assign its custom data value: "grass", "ice", "lava". Read it in GDScript to apply different movement friction.

💡 Better Terrain Plugin

Godot 4's built-in terrain system requires precise peering bit setup which is tedious. The free Better Terrain plugin (search Godot Asset Library) provides a simpler autotile workflow closer to Godot 3 and has a much cleaner runtime API. Worth installing for serious 2D level work.

Scene Architecture

In Godot, every meaningful object in your level is its own .tscn file. Scenes are blueprints — you place instances of them. Editing the blueprint updates every instance in every level. This is the core workflow that separates maintainable projects from spaghetti.

What Gets Its Own Scene File?
✅
Always its own scene: Player, any enemy type, any pickup type, moving platform, door, chest, projectile, UI overlay, boss.
✅
Usually its own scene: Spawn trigger, cutscene trigger, checkpoint, area-effect zones (damage, heal, water).
⚠️
Maybe inline: A one-off decorative static body that only appears in one level. Fine to keep inline but harder to edit later.
❌
Don't separate: Individual TileMapLayer nodes — they belong in the level scene because they hold per-level painted data.
Level Scene Structure (Platformer)
Level1.tscn (Node2D) ├─ BG (TileMapLayer) ├─ Ground (TileMapLayer) ├─ Player (player.tscn) ⬅ instance ├─ Enemies (Node2D) │ ├─ Goblin1 (goblin.tscn) ⬅ instance │ └─ Goblin2 (goblin.tscn) ⬅ instance ├─ Pickups (Node2D) │ └─ Coin1 (coin.tscn) ⬅ instance ├─ Spawns (Node2D) ← Marker2D children └─ Camera2D
Level Scene Structure (Top-Down RPG)
Room_Forest.tscn (Node2D) ├─ BG_Ground (TileMapLayer) ├─ Objects (TileMapLayer) ← trees, rocks ├─ Shadows (TileMapLayer) ← shadow overlay ├─ Player (player.tscn) ⬅ instance ├─ NPCs (Node2D) │ └─ OldMan (npc.tscn) ⬅ instance ├─ Enemies (Node2D) ├─ InteractZones (Node2D) ← Area2D triggers ├─ LevelBounds (StaticBody2D) ← invisible walls └─ Camera2D
The Scene Instancing Workflow in the Editor
1
Build the object's scene separately — e.g. open a new scene, create CharacterBody2D with sprite, collision, and script. Save as goblin.tscn.
2
In your level scene: In the Scene panel (top-left), click the chain link icon ("Instance Child Scene") → select goblin.tscn. Or drag the .tscn file directly from the FileSystem panel into the scene canvas.
3
Position the instance by dragging it in the canvas. Each instance is independent — move one without affecting others.
4
Override properties per-instance — click an instanced node, find any @export property in the Inspector, and change it. That change applies only to this instance. The scene file (blueprint) is untouched. 💡 This is how you set a Goblin's patrol_speed to 80 in one room and 140 in a harder room — same scene, different @export values.
5
Edit the original: Double-click an instance in the scene tree → click "Open in Editor" button at the top. Changes here propagate to every instance in every level. ⚠️ Never edit an instance expecting those changes to persist — they're overridden by the original scene. Edit the .tscn file to change all instances.

Instancing & Spawning in Code

Static instancing (placed in the editor) is fine for known objects. Dynamic spawning — creating enemies, projectiles, pickups at runtime — requires GDScript. The pattern is always: preload → instantiate → configure → add_child.

The Core Pattern — preload + instantiate
# ── preload at the top of the script ──────────────────────────
# preload() happens at parse time — file must exist.
# Use load() if the path is dynamic (determined at runtime).
const EnemyScene: PackedScene = preload("res://entities/goblin.tscn")
const CoinScene:  PackedScene = preload("res://entities/coin.tscn")
const BulletScene:PackedScene = preload("res://entities/bullet.tscn")

# ── spawn at a position ──────────────────────────────────────
func spawn_enemy(at: Vector2) -> void:
    var enemy: Node2D = EnemyScene.instantiate()  # create instance
    enemy.global_position = at                         # set position BEFORE add_child
    add_child(enemy)                                   # add to scene tree

# ── spawn at a Marker2D spawn point ─────────────────────────
@onready var spawn_points := $SpawnPoints.get_children()

func spawn_wave() -> void:
    for pt: Marker2D in spawn_points:
        var e: Node2D = EnemyScene.instantiate()
        e.global_position = pt.global_position
        $Enemies.add_child(e)     # add to container, not root

# ── spawn at random position ─────────────────────────────────
func spawn_random_coins(count: int, bounds: Rect2) -> void:
    for _i in count:
        var coin: Node2D = CoinScene.instantiate()
        coin.global_position = Vector2(
            randf_range(bounds.position.x, bounds.end.x),
            randf_range(bounds.position.y, bounds.end.y)
        )
        $Pickups.add_child(coin)
Projectile Spawning (Bullets)
# In player.gd — shoot toward mouse
const BulletScene = preload("res://bullet.tscn")

# Reference to the level root — so bullet exists
# independently of the player (doesn't get deleted
# if player is freed)
@onready var level := get_parent()

func shoot() -> void:
    var bullet := BulletScene.instantiate()
    # Set position at the gun muzzle Marker2D
    bullet.global_position = $Muzzle.global_position
    # Set direction toward mouse
    bullet.direction = (get_global_mouse_position()
                       - global_position).normalized()
    # Add to level, not player — survives player movement
    level.add_child(bullet)

func _input(event: InputEvent) -> void:
    if event.is_action_pressed("shoot"):
        shoot()
Scene Changing — Load Next Level
# In level.gd — transition to next scene
func go_to_next_level() -> void:
    # Immediate change
    get_tree().change_scene_to_file(
        "res://levels/level2.tscn")

# Or: load a PackedScene variable
@export var next_level: PackedScene
func go_to_next() -> void:
    get_tree().change_scene_to_packed(next_level)

# Reload current scene (restart level)
func restart() -> void:
    get_tree().reload_current_scene()

# Deferred — safer mid-physics-frame
func safe_change() -> void:
    get_tree().change_scene_to_file.call_deferred(
        "res://levels/level2.tscn")
Timed Wave Spawner — Full Pattern
# Attach to a Node2D "EnemySpawner" in your level
extends Node2D

const EnemyScene: PackedScene = preload("res://entities/goblin.tscn")

@export var max_enemies: int   = 10
@export var spawn_interval: float = 3.0   # seconds between spawns

@onready var spawn_points := $SpawnPoints.get_children()
@onready var timer: Timer = $Timer
@onready var container: Node2D = $Enemies

func _ready() -> void:
    timer.wait_time = spawn_interval
    timer.timeout.connect(_try_spawn)
    timer.start()

func _try_spawn() -> void:
    # Don't exceed max
    if container.get_child_count() >= max_enemies:
        return

    # Pick a random spawn point
    var pt: Marker2D = spawn_points[randi() % spawn_points.size()]

    var enemy := EnemyScene.instantiate()
    enemy.global_position = pt.global_position
    container.add_child(enemy)
    # Listen for death signal to decrement count
    enemy.died.connect(_on_enemy_died.bind(enemy))
⚠️ instance() is Godot 3 — use instantiate() in Godot 4

If you find tutorials using enemyScene.instance(), that's Godot 3 syntax. Godot 4 uses enemyScene.instantiate(). Same idea, different method name. This is one of the most common copy-paste errors when following old tutorials.

GDScript ↔ TileMapLayer

Reading and modifying tiles at runtime unlocks destructible environments, procedural generation, dynamic hazards, and runtime collision updates. The API is clean once you learn the cell coordinate system.

Cell Coordinates — The Basics
@onready var ground: TileMapLayer = $Ground

# ── World position ↔ cell coordinate conversion ──────────────
func world_to_cell(world_pos: Vector2) -> Vector2i:
    return ground.local_to_map(ground.to_local(world_pos))

func cell_to_world(cell: Vector2i) -> Vector2:
    return ground.to_global(ground.map_to_local(cell))

# ── Check if a cell has a tile ────────────────────────────────
func has_tile(cell: Vector2i) -> bool:
    return ground.get_cell_source_id(cell) != -1
    # source_id == -1 means empty cell

# ── Place a tile ──────────────────────────────────────────────
func place_tile(cell: Vector2i, atlas_coords: Vector2i) -> void:
    ground.set_cell(cell,
        0,             # source_id (which atlas in TileSet)
        atlas_coords   # which tile within the atlas
    )

# ── Erase a tile ──────────────────────────────────────────────
func erase_tile(cell: Vector2i) -> void:
    ground.erase_cell(cell)

# ── Get all painted cells ─────────────────────────────────────
func _ready() -> void:
    var all_cells: Array[Vector2i] = ground.get_used_cells()
    print("total tiles: ", all_cells.size())
Reading Custom Data from Tiles
# Read custom tile data at player's feet
# (e.g. surface type for footstep audio)
@onready var ground: TileMapLayer = $"../Ground"

func get_surface_type() -> String:
    var cell := ground.local_to_map(
        ground.to_local(global_position))
    var data := ground.get_cell_tile_data(cell)
    if data == null:
        return "air"
    return data.get_custom_data("surface_type")

# Apply movement modifiers based on tile
func get_friction() -> float:
    match get_surface_type():
        "ice":   return 0.05
        "mud":   return 0.7
        "grass": return 0.3
        _:       return 1.0
Destructible Tiles
# Breakable block — called when a projectile hits it
extends Node2D

@onready var ground: TileMapLayer = $"../Ground"
const DebrisScene = preload("res://fx/debris.tscn")

func break_tile_at(world_pos: Vector2) -> void:
    var cell := ground.local_to_map(
        ground.to_local(world_pos))

    # Only break tiles tagged as "breakable"
    var data := ground.get_cell_tile_data(cell)
    if data == null: return
    if not data.get_custom_data("breakable"): return

    # Spawn debris effect at tile center
    var centre := ground.to_global(
        ground.map_to_local(cell))
    var debris := DebrisScene.instantiate()
    debris.global_position = centre
    add_child(debris)

    # Erase the tile
    ground.erase_cell(cell)
One-Way Platform Drop-Through (GDScript)
# In player.gd — allow player to drop through one-way platforms
extends CharacterBody2D

func _physics_process(delta: float) -> void:
    _handle_drop_through()
    # ... rest of movement code ...
    move_and_slide()

func _handle_drop_through() -> void:
    # Hold Down + jump input to drop through one-way platform
    if Input.is_action_pressed("ui_down") and \
       Input.is_action_just_pressed("ui_accept") and \
       is_on_floor():
        _drop_through_platform()

func _drop_through_platform() -> void:
    # Temporarily disable one-way collision for a short time
    position.y += 2    # nudge below the platform surface
    var col: KinematicCollision2D
    # Disable one-way collision for 0.15 seconds
    set_collision_mask_value(2, false)   # layer 2 = one-way
    await get_tree().create_timer(0.15).timeout
    set_collision_mask_value(2, true)    # re-enable

Level Design Gotchas

These are the specific mistakes that waste hours. Most of them are invisible until you run the game.

TileMap / TileSet Gotchas
CRITICALTile size mismatch — wrong collision positions
▼

If you set the TileSet's tile size to 32×32 but your sprite sheet tiles are 16×16, all collision shapes will be misaligned. Set tile size first, before importing any art. If you need to change it later, you'll have to redraw all physics shapes.

CRITICALNo physics layer → no collision
▼

Adding a TileMapLayer does not automatically give tiles collision. You must: 1) add a Physics Layer to the TileSet resource, 2) draw shapes on individual tiles. If you skip either step, physics bodies fall through the floor with no error or warning.

GOTCHAGhost collisions between flat tiles
▼

If your player snags or stutters on flat ground, the physics shapes on adjacent tiles are slightly overlapping at their shared edge. Fix: use full-tile rectangles for all flat tiles. Godot 4 merges adjacent full-tile rectangles at runtime, eliminating seam collisions. Avoid polygons on flat terrain.

GOTCHAEditing instanced tiles expecting changes to persist
▼

If you change a tile's physics shape in TileSet editor while a level scene is open and then undo something in the level editor, your TileSet edits may be reverted. Save the TileSet resource explicitly (right-click it in Inspector → Save) after any changes to physics shapes or custom data.

NOTETileMap is deprecated — use TileMapLayer
▼

As of Godot 4.3, the TileMap node with its built-in layer system is deprecated. New projects should use individual TileMapLayer nodes instead. The API changes slightly: get_cell_source_id(layer, cell) becomes get_cell_source_id(cell) — no layer argument needed since each node is one layer.

Instancing & Spawning Gotchas
CRITICALinstance() vs instantiate() — Godot 3 vs 4
▼

Godot 3: scene.instance(). Godot 4: scene.instantiate(). Many tutorials and forum answers are for Godot 3. If you get "Nonexistent function 'instance'", you're using the wrong version's syntax.

GOTCHASet position BEFORE add_child()
▼

If you set global_position after add_child(), the node will briefly flicker at world origin (0,0) before snapping to the target position — because global_position only becomes valid once a node is in the scene tree. Set position (local) before add_child(), or set global_position immediately after.

Safe pattern: var e = Scene.instantiate() → e.position = target_pos → add_child(e)

GOTCHABullet parented to player gets wrong velocity
▼

If you add_child a projectile to the player node, its position is in the player's local coordinate space. When the player moves, the bullet moves with it. Add projectiles to the level root or a top-level container instead — never to a moving parent.

GOTCHAForgetting to queue_free() spawned objects
▼

Every enemy or bullet you spawn stays in the scene tree consuming memory and CPU until you explicitly call queue_free() on it. Bullets that fly offscreen forever are a common memory leak. Connect a VisibleOnScreenNotifier2D signal or use a Timer to auto-free offscreen projectiles.

NOTEpreload() vs load() — when to use each
▼

preload() runs at parse time — the path must be a literal string that exists at compile time. It's faster because the asset is ready before the scene runs. load() runs at runtime and accepts dynamic paths. Use preload() for all known scenes. Use load() only when the scene path is determined by data (e.g. loading a level from a save file string).

Scene Architecture Gotchas
GOTCHANode paths break when you rename or move nodes
▼

If you access a node via a hardcoded path like $Level/Entities/Enemies/Goblin1 and then reorganise the tree, the path breaks silently at runtime — returning null and causing null-reference errors. Prefer @onready var references from direct parents, or use groups to find nodes by role instead of path: get_tree().get_nodes_in_group("enemies").

GOTCHAEditing an instance, not the blueprint
▼

Changes made to an instanced node in the level editor (moving a child node, editing its script) are overrides on that instance only. They don't change the original .tscn file. If you want all instances to update, open the original scene file and edit there. Accidental instance-level edits are a major source of "why did my change not work?"

NOTEPlayer should usually be in the level scene, not a child of TileMapLayer
▼

The Player scene instance should be a direct child of the level root (or an Entities container), never a child of TileMapLayer. Making it a child of a TileMapLayer means its position is relative to the tile layer — if you ever move or offset the layer, the player moves with it.

Quick Debug Checklist
🔲
Player falls through floor → Debug → Visible Collision Shapes. Are there orange shapes on your tiles? If not: missing physics layer or missing shapes on tiles.
🔲
Player snags on flat ground → Physics shapes overlap at tile seams. Switch all flat tiles to full-tile rectangles using Paint mode.
🔲
Spawned enemy appears at 0,0 → You set global_position after add_child(). Set position before add_child(), or set global_position immediately after.
🔲
Enemy spawns but doesn't collide → Check collision_layer and collision_mask on both the enemy and the terrain TileMapLayer. Both must be configured.
🔲
Bullet moves with player → Bullet is parented to the player node. Add it to the level root instead: get_parent().add_child(bullet).
🔲
Terrain tiles don't autotile correctly → Peering bits not set, or not enough tile variants in the atlas. Check the Terrain editor — tiles should be highlighted with matching colour patches.
🔲
Changed an enemy in a level but it didn't update → You edited an instance override. Open the original .tscn file and edit there to update all instances.
