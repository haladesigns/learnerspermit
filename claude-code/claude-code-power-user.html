Claude Code — Agentic CLI Engineer

Claude Code is a command-line AI agent that reads, writes, and runs code directly on your system. It's not just a chatbot with a file viewer — it has real tool access (bash, filesystem, git) and a layered extension system: CLAUDE.md → Skills → MCP → Plugins → Sub-agents → Teams. Knowing which layer to reach for is the difference between power use and toy use.

Full Feature Map

/model, /rewind, /resume, /usage, /stats, /config, /insights, /plugins — built-in control plane
📄

🔧

Specialist workers spawned by the master agent. Each gets its own model, tools, and tool restrictions.

Model Selection — The Rule
🔥
Opus 4.6 — default for all serious work: refactoring, new features in large codebases, anything where correctness matters. 2× usage cost vs Sonnet
⚡
Sonnet 4.6 — default model in Claude Code. Use for faster/cheaper tasks. Switch with /model.
🏎️
Haiku — trivial tasks, quick lookups, high-volume sub-agent work. Very cheap, fast.
⚠️
Never use Sonnet by default for solid coding — the presenter always switches to Opus for anything serious.

💡 Mental Model

Think of Claude Code as a senior engineer on your terminal. You give it context (CLAUDE.md / Skills), tools (MCP), and then either work solo, spawn specialists (sub-agents), or run a full team. The bigger the task, the higher up this stack you climb.

Setup & Basic Usage

Installation is platform-specific. Core usage is: run claude in a directory, authenticate, and start prompting. A few non-obvious workflow tricks make daily use much smoother.

Installation
1
Check the official docs first
Installation method depends on your OS. The docs at docs.anthropic.com are the canonical source. Do not guess package names. 💡 Arch Linux: use the AUR package claude-code via yay or your preferred AUR helper — don't use the generic instructions from the main docs page.
2
Available installation modes
Terminal application, desktop application, VS Code extension. Choose what fits your workflow. 💡 The instructor uses the terminal version in all demos — this is the most powerful mode.
3
Authenticate on first run
Run claude in any directory. On first launch it will ask you to authenticate with your Anthropic/Claude Code subscription account.
4
Trust the folder
Claude Code asks you to confirm you trust the current directory before operating in it. This is a safety gate.
Core Workflow
①
Start in a project directory
cd my-project && claude — either an existing project or an empty folder for greenfield work.
②
Give your first instruction
Plain English: create a basic hello world Flask application using UV. Claude will run UV init, modify files, and ask permission for file writes.
③
Approve / reject tool use
For each file edit or command: y (yes), n (no), or "allow all edits this session" to skip prompts. ⚠ While rejecting, press Tab to provide context on what you want differently — without interrupting the full session.
④
Add context mid-task
You don't have to wait. While Claude is working, just type your additional info (e.g. by the way I want it to be red). Claude incorporates it into the current run without stopping. ✅ This is a major UX superpower — no need to interrupt and restart.
Session Management
⏹️
End session: Ctrl+C — Claude shows the session ID so you can resume later.
▶️
Resume (choose): claude --resume — presents a list of past sessions to pick from.
▶️
Resume (by ID): claude -r <id> — jump directly to a known session.
🆕
Fresh session: Just run claude without --resume for a clean start.
↩️
Rewind: /rewind — undo to a conversation checkpoint. Reverses code changes. No redo command exists — use git as your safety net.
🔑 By-the-Way Pattern

The "by the way" mid-task injection is one of Claude Code's most underused features. You do not need to stop the agent. Simply type your clarification or new constraint into the prompt while it's working and it will fold it in. This preserves flow and avoids wasted tokens from interrupts + restarts.

Slash Commands

Type / at any prompt to see the full list. These are Claude Code's built-in control plane — model switching, session management, analytics, configuration, and plugin discovery.

Command Reference
Command What It Does Notes
/model Switch the active LLM — Opus 4.6, Sonnet 4.6, Haiku, etc. Default is Sonnet 4.6. Always switch to Opus for serious coding work.
/rewind Undo to a previous conversation checkpoint. Reverses file changes. Restore code, conversation, or both. No /redo exists. Use git stash/commits as backup before big tasks.
/resume Pick up a previous session from an interactive list. Equivalent to claude --resume from the shell.
/usage Shows session usage %, weekly usage %, reset times, and your extra-usage balance. Useful for knowing when to throttle expensive Opus calls.
/stats Shows per-model usage breakdown, frequency, and session history. Good for auditing which model you actually spend time in.
/config Full configuration UI: fast mode, thinking mode, verbose output, progress bar, and more. Persistent settings for the Claude Code application itself.
/insights Analyzes all your past sessions and generates an HTML report — what you build, your patterns, what's slowing you down, token efficiency. Opens in browser. Great for retrospecting on your workflow habits.
/plugins Discover, install, and manage plugins from the official marketplace (and custom marketplaces). Browse by capability. Install scopes: user-wide, project-wide, or user+project only.
/agents View, create, and configure sub-agents for the current project. Assign models, restrict tools, set colors. See Agents tab.
/mcp List and manage connected MCP servers. Shows status of local + remote connections. Use claude mcp add from the shell to register new servers.
Usage Limits — How They Work
⏱️
Session limit: resets on a rolling basis (e.g. every few hours). /usage shows time to reset.
📅
Weekly limit: resets on a fixed day each week. Shows % consumed for the period.
💳
Extra usage: pre-loaded account balance. If you hit both limits, Claude Code falls back to billing from this balance rather than going offline.
📊
Opus costs 2× the usage vs Sonnet per token. Budget accordingly — use Haiku for sub-agent grunt work.
/insights — What It Reveals
📈
Session breakdown: frontend, backend, full-stack, Linux configs, etc. — what you actually spend time on.
🚧
Friction analysis: what Claude does wrong vs. what you do wrong (e.g. lightweight prompts triggering heavy tool use).
⏱️
Response time distribution: how long your requests typically take.
❌
Error + reject rates: how many commands fail or get manually rejected — a signal for prompt quality.
🕐
Time-of-day activity: when you tend to work with Claude Code.
💡 Explore More

Just type / in any session to see the full command list — the ones above are the most useful ones covered in the crash course, but there are more. The /config command in particular has settings worth exploring for thinking mode and fast mode if you haven't already.

The Context System

Claude Code has a four-layer context architecture. Each layer serves a different purpose. Loading everything at once kills the context window — the system is designed for precision loading. Skills are the key innovation here: they're like CLAUDE.md with built-in RAG.

Layer Hierarchy — What Goes Where
Layer
Purpose & Content
Scope
①
CLAUDE.md
(global)
Universal preferences that apply across every project — your default coding style, personal preferences, global toolchain choices. Stored in ~/.claude/CLAUDE.md.
GLOBAL
②
CLAUDE.md
(project root)
Project-specific instructions: directory structure, key file locations, package manager rules (always use uv add not pip install), Docker/migration conventions, code style, things Claude should never do in this repo.
PROJECT
③
CLAUDE.md
(subdirectory)
Fine-grained overrides for a specific feature or module. Can override parent-level instructions for that subtree. Useful for feature-specific constraints or alternate coding styles in a monorepo.
DIRECTORY
④
Skills
(.claude/skills/)
Lazy-loaded instruction fragments. Each skill is a directory with a SKILL.md containing a name, description, and instructions. Claude reads the description header and loads the full content only when that skill is relevant to the current task.
ON-DEMAND
⑤
MCP Servers
External APIs exposed as callable tools. Not static text — live connections to services (Google Calendar, Slack, DeepWiki, databases). Claude can call these tools during a session to retrieve real-time data.
API/LIVE
⑥
Plugins
Marketplace packages that bundle Skills + MCP servers together. Install via /plugins. Can be scoped to: all users, all projects, or this user in this project only.
BUNDLED

Plugins — Install & Scope
1
Open the plugin browser
Type /plugins → choose Discover to browse the official marketplace. By default only Anthropic's official plugin repo is listed.
2
Choose a plugin
Example: frontend-design by Anthropic — packages the frontend design skill (creates production-grade HTML/CSS/JS UIs). You can read its description before installing.
3
Choose install scope
• All users, all projects — global install
• All users, this project — project-wide
• This user, this project — most isolated, recommended for testing
4
Reload plugins
Run /plugins reload (or restart Claude Code) before the plugin's Skills and MCPs become active in your session. ⚠ Skipping the reload is a common gotcha — skills from new plugins won't be available until Claude reloads.

⚠ Context Window Trap

If you dump everything — Docker, UV, your CLI docs, migration notes, all your tools — into a single CLAUDE.md, you consume the context window before Claude even reads your actual code files. Skills exist precisely to avoid this: they're loaded lazily, only when needed. Think of CLAUDE.md as the always-on layer, and Skills as the on-demand layer.

Sub-agents & Agent Teams

Claude Code supports two distinct multi-agent modes. Sub-agents are specialists spawned and managed by your master session. Agent Teams EXPERIMENTAL are fully decentralized — agents communicate peer-to-peer without a central controller. They are architecturally different and not interchangeable.

Sub-agents — Centralized Architecture
🧠 Master Agent
(your claude session)
🎨 UI Agent
Sonnet / CSS+JS only
← tools restricted
🗄️ DB Agent
Opus / backend only
← separate model
all communication flows ↑↓ through master
How to create sub-agents
1
Open agent manager: /agents → Library → New Agent
2
Describe the specialist: e.g. "UI designer — HTML, CSS, JS animations only, no backend". Claude Code auto-generates the system prompt.
3
Configure per-agent settings:
• Model: Opus for architecture/DB, Sonnet for UI/design work
• Tool restrictions: disable specific MCP servers or tools for this agent
• Color: visual ID in the multi-agent UI (e.g. purple=UI, green=backend)
• Scope: project-level or global 💡 "Show advanced options" to see all tool toggles. Press Enter on a tool to disable it for that agent.
4
Use them: Simply describe a task involving both domains. Claude automatically recognizes it can use both specialists in parallel.
e.g. "implement a to-do app in Flask. Front-end: bluish style with animations. Back-end: Flask." ✅ Claude will say "let me use both specialized agents in parallel" — you don't need to explicitly invoke them.
Agent Teams EXPERIMENTAL — Decentralized
NO MASTER — agents communicate peer-to-peer
🎨 UX Agent
🏗️ Arch Agent
😈 Devil's Advocate
shared to-do list as coordination mechanism
How to enable + invoke
# Enable experimental teams (env var required)
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
claude
📋
Teams use a shared to-do list as their coordination mechanism — each agent picks tasks and updates the list as it works.
🗣️
Must explicitly instruct Claude Code to create a team — it won't happen automatically. Use phrasing like "create an agent team to explore X from three different angles."
👁️
Use Ctrl+T inside a running team session to inspect token usage and tool calls per agent in real time.
💸
Very token-expensive. Three agents working in parallel = 3× the usage. Use selectively for brainstorming/exploration tasks, not routine implementation.
⚗️
Results arrive as a combined summary once all agents finish. Currently not part of the default Claude Code install.
Sub-agents vs Agent Teams — Side by Side
Dimension Sub-agents Agent Teams (Experimental)
Architecture Centralized — master controls everything Decentralized — agents talk to each other
Communication All messages flow through the master agent Peer-to-peer via shared to-do list
Activation Automatic — Claude detects multi-domain tasks Must explicitly instruct team creation
Model per agent ✅ Yes — each agent can use a different model ✅ Yes
Tool restrictions ✅ Granular per-agent tool disabling ✅ Yes
Token cost Medium — parallel but coordinated High — 3+ agents running independently
Best use case Parallel implementation (front-end + back-end) Multi-angle brainstorming / exploration
Stability Stable — part of default install Experimental — interface may be buggy
✅ When Sub-agents Shine

Full-stack features where front-end and back-end work is genuinely independent. Give each agent the right model (Opus for DB schema design, Sonnet for CSS animations) and restrict their tool access so they can't accidentally overwrite each other's domain. The master handles coordination; you handle the prompt.

Git Worktrees + Claude Code

Worktrees are a git feature, not a Claude Code feature — but Claude Code integrates with them via claude --worktree to let you run N parallel Claude Code instances on the same codebase, each isolated in its own branch and directory. No interference, no merge conflicts mid-session.

How It Works — Conceptual Diagram
📦 git repo
(main branch)
→
This is your source of truth. Must be committed + pushed to git before spawning worktrees.
🌳 branch: fancy-amber
dir: /project-fancy-amber
→
Claude instance #1: "change entire app style to blue"
🌳 branch: feature-one
dir: /project-feature-one
→
Claude instance #2: "change entire app style to green"
🌳 branch: db-redesign
dir: /project-db-redesign
→
Claude instance #3: "redesign the database schema"
Each branch starts from the same committed state. They diverge independently. You merge / compare / delete when done.
Usage Commands
# Navigate to your project (must have git + commits)
cd my-project

# Spawn a new worktree with auto-generated name
claude --worktree
# → creates branch + dir: fancy-coalescing-amber

# Spawn with a specific branch name
claude --worktree feature-one
# → creates branch: feature-one
# → checks it out in: /project-feature-one/

# Run as many as you need — each is a separate terminal
# + separate Claude Code session + separate directory
⚠ Your changes must be committed and pushed to git before using worktrees — the new branch is checked out from the latest committed state.
Use Cases
🅰️
A/B testing implementations: Blue redesign vs. Green redesign — compare outputs before committing to one approach.
🔀
Parallel feature development: Frontend redesign + database schema change + API refactor — all happening simultaneously in isolation.
⚡
Maximizing throughput: Instead of waiting for one task to finish before starting the next, run 5–7 Claude Code instances in parallel.
🛡️
Safe experimentation: Each worktree is isolated — a failed experiment in branch #3 has zero effect on the others. Delete the branch and it's gone.
🔁
After completion: Push each branch, then use standard git merge or compare PRs. Merge conflict resolution is handled the usual git way.
⚠ Git Prerequisite

Worktrees require the project to already be a git repo with committed changes pushed to remote (or at least committed locally). The --worktree flag creates a new branch from HEAD. If your working directory has uncommitted changes, commit them first or they won't appear in the new worktree.

Worktree Lifecycle
1
Commit current state
Make sure your working directory is clean (or at a known-good commit). This is the starting point for all worktrees.
2
Spawn worktrees
Run claude --worktree [name] in multiple terminals. Each gets its own branch + directory + Claude session.
3
Work in parallel
Each Claude instance operates completely independently. No file locking, no shared state. They each get a full copy of the codebase at the branch start.
4
Review + merge
Inspect each worktree's result. Push branches to remote, open PRs, or merge directly. Delete worktrees you don't need: git worktree remove <path>. 💡 For A/B style experiments, open both in a browser side-by-side before deciding which to merge.

Memory System

Claude Code has a persistent memory layer separate from CLAUDE.md. When you tell Claude to "remember" something, it writes that fact to a memory store that is automatically loaded in all future sessions — no file editing required. This is for user-level preferences and facts, not project instructions.

How to Use Memory
1
Tell Claude to remember something
Just say it in natural language: 
Please remember for future sessions
that my name is Florian.
Claude will respond with "Recalling 1 memory... Writing 1 memory" — confirming the write.
2
Start any new session
Close the current session with Ctrl+C. Start a fresh claude session anywhere.
3
The memory is already loaded
Ask: "what is my name?" — Claude will answer from memory without needing you to tell it again. ✅ Works across projects, directories, and sessions — truly global user context.
Memory vs CLAUDE.md — When to Use Which
Dimension Memory CLAUDE.md
How to set Natural language during session Edit markdown file manually
Scope Global (all projects, all sessions) Global file or per-project file
Best for Personal facts (name, preferences, global style rules) Project structure, tool commands, repo-specific conventions
Persistence Automatic — survives restarts File on disk — must be in the repo
Discoverability Opaque — stored by Claude internally Transparent — you can read and edit the file
Good Things to Store in Memory
👤
Identity: your name, role, the team you work on
🎨
Global style preferences: always use Oxford commas, prefer functional over OOP, always use type hints in Python
🛠️
Toolchain preferences: always use UV for Python, always use pnpm not npm, prefer Bun for JS runtimes
💬
Communication style: prefer concise responses, always explain the "why" behind code choices
🚫
Global never-dos: never use any type in TypeScript, never suggest jQuery, never use subprocess.shell=True
📚
Context about you: I work primarily in Python + Lua, I'm learning LISP and Clojure, I use WSL on Windows
✓ Use Memory For
✅Personal identity and preferences that cross all projects
✅Global coding habits that never change
✅Things you're tired of repeating every session
✅Fast conversational storage — no file editing needed
✗ Don't Rely on Memory For
❌Project structure and file locations (use CLAUDE.md)
❌Tool workflows and how-tos (use Skills)
❌Anything you need to version-control or share with teammates
❌Secrets, API keys, or credentials
✅ The Combined Setup

Power users layer all three: Memory for personal context → global CLAUDE.md for cross-project conventions → project CLAUDE.md for repo specifics → Skills for tool how-tos. Each layer stays lean because nothing overlaps. Result: Claude always has the right context loaded without burning the entire window on boilerplate.
