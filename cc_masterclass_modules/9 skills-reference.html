Claude Code Skills

Skills are saved prompts that tell Claude Code how to do a specific thing in a specific way — loaded on demand or automatically. They eliminate repetitive instructions and encode your best workflows permanently.

Two Types of Skills
🔄
Workflow
Multi-step automation. "When I say /content, fetch recent articles, repurpose to blog, post to site." Bundles a sequence of actions into one command.
⚡
Enhancement
Skill multiplier. Loads conventions and rules that improve Claude's output quality — e.g. the front-end design skill that avoids AI slop aesthetics.
Two Ways to Trigger
⌨️
Manual
Type /skill-name (with optional args). Guaranteed trigger — use this for anything with side effects (deploy, commit, send).
🤖
Automatic
Claude matches your request against skill descriptions. Use this for enhancement skills (code review, design conventions). Relies on good description quality.
Installing Built-in Skills
1️⃣
Run /plugin inside Claude Code — "plugin" and "skill" are used interchangeably.
2️⃣
Choose Marketplace to browse and install community skills (e.g. front-end design, UI/UX Pro, skill creator).
3️⃣
Ask Claude Code "What skills are available?" to verify what's installed and discoverable in your project.
⭐ Most Important Skill

Install the Skill Creator first. It creates new skills, optimizes descriptions for better trigger accuracy, runs A/B evals, and benchmarks performance — all things you can't do manually.

Context-Efficient Loading

Skills don't clog your context window. Claude Code holds a short description of each skill (~100 tokens each), not the full content. The full prompt is loaded only when the skill is actually needed.

How Lazy-Loading Works
1
Skills sit in another "room"
Full skill prompts are not in the context window at startup — they're in a folder waiting.
2
Claude holds a short index
~100 tokens per skill: name + description. This is what's always loaded — very cheap.
3
Match triggers full load
When your request matches a description (or you use /skill-name), the full skill prompt is loaded into context. 💡 Result: ~100 skills can be "pre-loaded" with minimal cost.
The Sweet Spot Problem
⚠️
With 5 skills it's easy to know which does what. With 20+ skills, Claude may pick the wrong one or miss triggering entirely.
🔧
Fix 1: Avoid overlap. Don't load "frontend design", "UX/UI Pro", and "hero page tool" simultaneously — consolidate.
📝
Fix 2: Write better descriptions. The description field is how Claude decides to auto-trigger — make it specific with clear trigger conditions.

📁 Folder Structure

The folder name becomes the slash command. A skill named review-code needs to live at .claude/skills/review-code/SKILL.md — not .claude/skills/review-code.md.

Step-by-Step: Your First Skill
1
Create the skill folder
Project-level: mkdir -p .claude/skills/review-code
Personal (all projects): mkdir -p ~/.claude/skills/review-code
2
Write SKILL.md
Add YAML frontmatter between --- markers at the top, then your instructions below. 
---
name: review-code
description: Reviews code for bugs, performance, and style.
  Use when the user asks to review, audit, or check code.
---

# Your instruction prompt goes here...
Format findings as CRITICAL / WARNING / SUGGESTION.
End with: "X critical, Y warnings, Z suggestions found."
3
Test both trigger modes
Automatic: "Review the auth middleware" — Claude sees the description match.
Manual: /review-code src/middleware/auth.ts — guaranteed trigger. ⚠ Check: ask "What skills are available?" if your skill doesn't appear.
4
Iterate
Edit SKILL.md — Claude picks up changes immediately. Use the Skill Creator to run evals and compare with/without the skill. 💡 Run A/B tests: "with skill" vs "without skill" to verify it actually improves output.

Frontmatter Reference

YAML settings at the top of every SKILL.md, between --- markers. Controls naming, triggering, tool access, and isolation. Everything below the second --- is your instruction prompt.

All Fields
Field What It Does Notes
name The /slash-command name. Becomes how you call the skill. Lowercase, hyphens only. No spaces.
description Tells Claude when to auto-load this skill. This is the index entry. Most important field. Be specific — list trigger phrases.
argument-hint Shown during /autocomplete to guide usage. Example: [filename] [format]
disable-model-invocation Set true → Claude can NEVER auto-load. Manual /name only. Use for any skill with side effects: deploy, commit, send.
user-invocable Set false → hidden from / menu. Only Claude can load it. For background knowledge: legacy context, domain rules.
allowed-tools Restricts which tools Claude may use while skill is active. Bash(pattern) for specific commands only.
context Set fork to run in an isolated subagent with separate memory. Make forked skills fully self-contained — no chat history access.
Trigger Matrix
⚡
Both defaults
You can /skill-name AND Claude auto-loads. Good for general-purpose skills like code review.
🔐
disable-model-invocation: true
Manual only — Claude will never load it automatically. Use for anything with side effects.
🤫
user-invocable: false
Claude only — Won't appear in / menu. Use for passive background knowledge.
Full Frontmatter Example
---
name: my-skill
description: What this does and when.
  Use when user says "deploy", "ship it".
argument-hint: "[env: staging|production]"
disable-model-invocation: true
allowed-tools: Bash(git *), Bash(npm run*), Read
context: fork
---

# Instructions follow here...
⚠ Strings with colons need quotes in YAML
⚠ Formatting Is Strict

The frontmatter must be between exactly two --- markers at the very top. No whitespace before the first ---. No tabs (use spaces). Missing closing --- breaks the skill silently.

disable-model-invocation: true

Auto + Manual /run-tests
Run tests with coverage, watch mode, or CI mode
▼
---
name: run-tests
description: Run tests — single file, coverage, watch mode, or CI mode.
  Use when user asks to run tests, check coverage, or verify tests pass.
argument-hint: "[file-or-pattern] [--coverage] [--watch] [--ci]"
allowed-tools: Bash(npx jest *), Bash(npx vitest*), Bash(npm test *), Read, Grep, Glob
---
🧪
Auto-detects jest vs vitest from package.json. Flags coverage below 80%. On failure, reads source and suggests fixes.

Manual Only /db-migrate
Create, validate, and apply Supabase database migrations safely
▼
---
name: db-migrate
description: Create and manage Supabase database migrations.
disable-model-invocation: true
argument-hint: "[create|apply|status] [migration-name]"
allowed-tools: Bash(npx supabase *), Read, Write, Edit, Grep, Glob

⚠️
NEVER applies without user confirmation. Flags all destructive operations (DROP TABLE, DROP COLUMN) with a warning.
🗃️
Subcommands: create <name> · apply · status
✓ Good Descriptions
✅"Deploy the app to staging or production. Use when user says deploy, ship it, or push to staging."
✅Specific action + explicit trigger phrases
✅Include synonyms people actually type
✗ Bad Descriptions
❌"Helps with deployment" — too vague, won't trigger reliably
❌Generic phrasing with no trigger conditions
❌Overlapping descriptions across multiple skills
Gotchas
⚠️
YAML is strict: no whitespace before the first ---, no tabs (use spaces), strings with colons must be quoted: "Deploy: push to production"
⚠️
Wrong folder structure: skill must be at skill-name/SKILL.md, NOT skill-name.md — Claude won't find a bare markdown file.
⚠️
Forgot Read in allowed-tools: when you set allowed-tools, Claude can ONLY use those tools. Omitting Read means it can't read files at all.
⚠️
Forked skills have no memory: context: fork runs in an isolated subagent with zero access to chat history. Make forked skills fully self-contained.
⚠️
Side-effect skills without disable flag: any skill that deploys, commits, sends messages, or modifies databases MUST have disable-model-invocation: true. Otherwise Claude may decide to run it unprompted.
⚠️
Skill not discovered? Ask: "What skills are available?". Check: correct folder path, valid YAML frontmatter, correct working directory.
Quick Checks Before Shipping a Skill
✅
Description specificity: Does it list exact phrases that should trigger it?
✅
Side effect check: Does this deploy/commit/send anything? Add disable-model-invocation: true.
✅
Tool list complete: If you set allowed-tools, confirm Read is included if the skill reads files.
✅
Run an eval: Use Skill Creator to test with/without the skill. Confirm it actually improves output.
✅
Overlap audit: Do you have multiple skills doing the same thing? Consolidate — don't load more than ~20 total.
