Prompting Claude Code

Claude Code is an agent with real power — it reads files, runs commands, edits code. Vagueness doesn't get you a longer answer; it gets wrong assumptions baked into your codebase. This reference covers every pattern from the video and the full course module.

The 3-Part Foundation
🛡️
Plan Mode first — bumper rails for your prompt. Shift+Tab to enter. Claude asks clarifying questions before touching a single file.
❓
Open-ended questions — fill the gaps Plan Mode leaves. Ask: "What am I not thinking about?" or "What would an expert do here?"
🖼️
Context + examples — screenshots, existing files, sample writing. If you know what good looks like, show it.
Chatbot vs. Code Agent
💬
Chatbot: vagueness → longer, more detailed answer. Fine.
⚙️
Claude Code: vagueness → wrong assumptions baked into your actual codebase. You now have code to undo.
🧠
Golden rule: Be as specific as explaining a task to a brilliant colleague who just joined your team — smart but doesn't know your norms.
Specificity Spectrum
Exploration
Intentionally open-ended. Invite Claude to surface things you haven't thought about.
"What would you improve in the hero section?"
Guidance
Set direction, leave room for Claude's judgment. Good for UI and structure.
"Create a services section with cards. Use the same animation as the hero."
Specification
Define exact behavior. Use whenever you'd be annoyed to see it done differently.
"Create RLS policies: anyone can SELECT where published=true…"
Rule of thumb: if you'd be frustrated by Claude choosing wrong, specify it. If you'd be delighted by a surprise, leave it open.
💡 Key Takeaway from the Video

That 99% of prompting Claude Code correctly is just three things: Plan Mode, open-ended questions ("what am I not thinking about?"), and providing examples. Don't overthink it.

7-Step Prompting Workflow
1
Context before task (the WHY before the WHAT)
Tell Claude why the feature matters and what failure looks like, not just what to build.
✗ Weak
"Add validation to the contact form."
✓ Strong
"Users submit with invalid emails — our Resend integration fails silently. Add client-side validation: check email format, require name field, show inline errors below each field."
2
Reference existing code (don't re-describe, point)
Claude Code can read your files. Use that. Four ways to reference: 💡 By path: /app/api/apply/route.ts · By name: "the hero section component" · By @ syntax: @components/ServiceCard.tsx · By concept: "our other API routes"
✓ Strong
"Create /api/subscribe. Follow the same pattern as /api/apply — same error handling, same response format, same Supabase client setup."
3
Define what NOT to do (constraints)
Claude's latest models over-engineer. Telling it what to avoid is often more valuable than what to build. 💡 "Keep it simple — don't add features beyond what I described." · "Don't create a separate utility file." · "Don't add comments explaining each line." ⚠ Without constraints on UI work, Claude will add loading spinners, toast notifications, and error boundaries you never asked for.
4
Communicate design taste
Vague = generic. Use design vocabulary. 💡 Density: "spacious" vs "compact" · Tone: "editorial" vs "brutalist" · Inspiration: "think Linear or Vercel" · Anti-ref: "not Bootstrap-looking" · Specifics: "dark bg #1a1a1a, generous whitespace"
✗ Weak
"Make it look good."
✓ Strong
"Keep the design minimal and premium-feeling — not generic SaaS template vibes."
5
Iterate in conversation (the 80/20 loop)
No prompt nails it first try. First prompt → 80% → refinement → polish. ⚠ After 2+ corrections on the same issue, the conversation is polluted with failed approaches. Run /clear and write a better initial prompt. A clean session with a refined prompt beats a long session every time.
6
Plan Mode for complex features
Anything touching multiple files: separate thinking from doing. Shift+Tab → Claude reads and plans, no changes made. 💡 In Plan Mode: "Read the existing /api/apply route. What files need to change? What's the data flow?" Then switch back and say: "Implement the email flow from your plan."
7
Give Claude a way to verify
Per Anthropic's own docs: "the single highest-leverage thing you can do." Claude performs dramatically better with a feedback loop. 💡 For functions: include test cases in the prompt. For UI: paste screenshots or use Claude Chrome extension. For APIs: tell Claude to run curl or write a quick test.
✓ Strong
"Create validateEmail. Test: user@example.com → true, 'invalid' → false, user@.com → false. Run tests after implementing."

CLAUDE.md
📄
Persistent context that loads automatically at the start of every conversation. Use for project conventions, tech stack, preferences Claude should always know.
✂️
Keep it under 60–100 lines. For each line, ask: "Would removing this cause Claude to make mistakes?" If not, cut it. Over-stuffed CLAUDE.md buries important rules in noise.
👻
Ghost context fix: Claude has no memory between sessions unless you --continue. Put persistent rules in CLAUDE.md, not in your head.
ℹ️ About These Examples

All five examples come directly from the course module. Each note explains why the prompt works — not just what it says.

Example 1 — Project Scaffolding
# Basic
"Create a new Next.js project with TypeScript and Tailwind CSS.
Use the app router. Set up /components, /lib, /types directories."

# Even stronger — adds constraints
"Create a new Next.js 14+ project with TypeScript and Tailwind CSS.
Use the app router. Set up /components, /lib, /types directories.
Don't install extra packages beyond what's needed.
Use the default Next.js config — no custom webpack or turbopack."
Names exact tech stack, specifies router type (a critical fork), defines folder structure upfront. Constraints stop Claude adding Prettier configs, ESLint rules, or other extras you didn't ask for.
Example 2 — UI Component with Design Direction
"Create a hero section for an AI consulting agency. It should have:
- Large headline with gradient text effect
- Subheadline explaining value prop (2 lines max)
- Primary CTA button linking to booking calendar
- Subtle animated background (grid or particles)
- Fully responsive (mobile-first)

Use Framer Motion for entrance animations (fade up, staggered).
Keep the design minimal and premium-feeling —
not generic SaaS template vibes."
Structured as a checklist (Claude parses these cleanly). Anti-reference ("not generic SaaS template vibes") eliminates an entire class of default output. Mix of specific requirements + open design judgment.
Example 3 — Pattern Replication
"Create a testimonials section. Look at how the services cards
are built in @components/ServicesSection.tsx and follow the same
component structure, animation pattern, and spacing.
Each testimonial: quote, author name, company."
Most efficient way to maintain consistency. Instead of re-describing conventions, point to an example. Claude reads the file, replicates the pattern.
Example 4 — Debugging with Symptoms
"The mentorship form submits successfully but the confirmation
email never arrives. The Resend dashboard shows no API calls
after form submission.

Check /api/apply — I think the email send is happening before
the Supabase insert completes and failing silently.
Look at the error handling."
Describes the symptom, provides external evidence (Resend dashboard), offers a hypothesis (timing issue), points to a specific file. Give Claude the same info you'd give a colleague.
Example 5 — Multi-Step Feature with Constraints
"Add a pre-qualification step to the application form.
Before the user sees the full form:
1. Show 3 qualifying questions (budget, timeline, AI usage)
2. Based on answers: show full form OR 'not a fit' message
   with alternative resources
3. Store pre-qual answers alongside the application in Supabase

Keep UI consistent with existing form styling.
Don't add any new npm packages.
Use the existing form validation pattern."
Breaks complex feature into numbered steps. Defines both branches of the flow. Three explicit constraints prevent over-engineering.
✗ 7 Mistakes to Avoid
❌
The Mega-Prompt — 5 features in one message. Quality drops across all of them. Fix: one feature per prompt, commit between.
❌
Kitchen Sink Session — mixing unrelated tasks in one conversation. Fix: /clear between tasks.
❌
Correcting over and over — each correction adds failed approaches to context. Fix: after 2 fails, /clear and write a better initial prompt.
❌
Ghost context — assuming Claude remembers a previous session. Fix: put persistent context in CLAUDE.md.
❌
Trust gap — accepting Claude's output without verifying. Fix: always test. Tests, linter, screenshots, curl. Don't ship unverified code.
❌
Infinite exploration — "investigate this" without scope. Fills context with noise. Fix: scope it: "Look at the auth flow in /src/auth/, specifically token refresh."
❌
Over-stuffed CLAUDE.md — 200 lines of noise buries what matters. Fix: keep under 60–100 lines. Cut anything Claude already knows.
✓ 7 Pro Tips
✅
Read before edit — always let Claude read a file before changing it. Use @filename or "read the file first" to force a read.
✅
Commit often — after each successful feature, tell Claude to commit. Clean git history + rollback checkpoints.
✅
Use /clear aggressively — most people under-use it. Conversation memory pollution is the #1 cause of degrading output quality.
✅
Pipe in data — npm run build 2>&1 | claude -p "Fix these build errors" sends logs directly.
✅
Let Claude interview you — for complex features: "I want to build X. Interview me about implementation, edge cases, and tradeoffs before we start."
✅
Name your sessions — /rename oauth-migration so claude --resume shows useful names, not timestamps.
✅
Test the sad path — after Claude builds a feature: "Try submitting with invalid email, blank name, 10,000-char message. What breaks?"
🌐 Open-Ended Questions (from the Video)

When you don't know what you don't know, ask Claude directly. These questions course-correct even after a bad starting prompt: "What am I missing?" · "What am I not thinking about?" · "What would a top-tier [expert] be thinking about right now?" · "Can you search for current best practices on this?" · "Are there unintended consequences we aren't considering?"

⚠️ The Two-Correction Rule

If you've corrected Claude more than twice on the same issue, stop. Run /clear. Write a new initial prompt that includes everything you learned from those corrections. The second attempt with a better prompt almost always beats the fifth correction in a polluted session.
