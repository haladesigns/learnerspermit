YARA-L 2.0

A rule language for Google Chronicle / SecOps that detects threats in UDM-normalised security telemetry. Write once, run against petabytes of ingested logs — either interactively (UDM Search) or as persistent automated detection rules.

The Two Modes — Know Before You Write
UDM Search — Interactive
SyntaxBare events: block — no wrapper
RunsOnce, on demand
OutputResults in search UI
Use forTriage, investigation, hunting
Has condition:No — throws an error
Detection Engine — Persistent
SyntaxFull rule name { } block
RunsContinuously, 24/7
OutputAlerts in findings queue
Use forAutomated alerting, SOAR triggers
Has condition:Yes — required
⚠ Most Common Beginner Mistake

Pasting a Detection Engine rule into the UDM Search bar (or vice versa). The parsers are completely different — a condition: in UDM Search throws a compilation error.

Core Concepts
📌
UDM — Unified Data Model. Chronicle normalises all ingested logs into a common schema. YARA-L queries this normalised form.
$
Placeholders — Variables prefixed with $ bind field values. $e binds an event, $host = $e.principal.asset.hostname extracts a value.
#
Count shorthand — #e means "count of events matching $e". Use in condition: as #e > 5.
%
Reference lists — Named IOC/allowlists managed in Chronicle UI. Referenced with %list_name. Updates auto-propagate to all rules.
Rule Naming Rules
✅
Allowed: letters a-z A-Z, digits 0-9, underscores _
❌
Not allowed: spaces, hyphens -, dots ., any special characters
❌
Cannot start with a digit — 1_bad_name fails; bad_name_1 is fine
📐
Convention: tactic_behaviour_specificity e.g. lateral_movement_smb_new_host
💡 Golden Rule

Investigating right now? → UDM Search. Want to be alerted automatically going forward? → Detection Engine rule. These are not interchangeable.

Section Order — Parser-Enforced
meta: → events: → match: → outcome: → condition:
■ Required ■ Optional Sections cannot be reordered. Omit optional sections freely; do not swap them.
Fully Annotated Skeleton
rule rule_name_here {              // required — opens the rule block

  meta:                            // optional — always include severity
    author      = "your_name"
    severity    = "HIGH"           // LOW | MEDIUM | HIGH | CRITICAL
    priority    = "HIGH"
    description = "What this rule detects and why"
    assumption  = "Assumes EDR on all endpoints"
    confidence  = "HIGH"
    tactic      = "TA0002"         // MITRE ATT&CK tactic ID
    technique   = "T1059.001"      // MITRE ATT&CK technique ID

  events:                          // required — all filter conditions
    $e.metadata.log_type = "PROCESS_LAUNCH"
    $host = $e.principal.asset.hostname

  match:                           // optional — grouping + time window
    $host over 1h

  outcome:                         // optional — computed values for alert
    $hits = count($e.metadata.id)

  condition:                       // required — the firing threshold
    #e > 5

}                                  // required — closes the rule block
Minimal Valid Rule
rule minimal_rule {
  events:
    $e.metadata.log_type = "NETWORK_HTTP"
  condition:
    $e
}
Only events: and condition: are strictly required. Omitting meta: severity means alerts have no triage priority.

⚠ Parser Strictness

Section order is not a style preference — it is enforced at compile time. outcome: before match: throws a compilation error.

events: — How It Works
∧
Every line is an implicit AND. All conditions must be true simultaneously for an event to match. There is no explicit and keyword between lines.
$
Placeholder binding. $e.metadata.log_type = "USER_LOGIN" — left side is the UDM field path, right side is the value. The $e prefix binds the event variable.
↔
Cross-event join. Assigning the same placeholder in two events ($ip = $e1.principal.ip and $ip = $e2.principal.ip) forces Chronicle to match only events sharing that value.
Comparison Operators
OperatorMeaningExample

>Greater than$e.network.sent_bytes > 1000000
>=Greater than or equal$e.security_result.risk_score >= 80

event_type vs log_type
🔖
log_type — identifies the parser/vendor: "CROWDSTRIKE_EDR", "OKTA", "AWS_CLOUDTRAIL". Vendor-specific. One EDR's process launch ≠ another's.
🔀
event_type — Chronicle's normalised action taxonomy: "PROCESS_LAUNCH", "USER_LOGIN", "FILE_CREATION". Vendor-agnostic — works across all sources.
🎯
Use both for maximum precision: log_type scopes to a vendor, event_type scopes to an action. Use only event_type for cross-vendor detections.

Multi-Event Correlation Example
// Recon scan followed by successful login from same IP
rule recon_then_lateral_movement {
  meta:
    severity = "CRITICAL"
    tactic   = "TA0008"
  events:
    $scan.metadata.log_type = "NETWORK_SCAN"
    $ip = $scan.principal.ip            // bind $ip to scan source

    $login.metadata.log_type = "USER_LOGIN"
    $login.security_result.action = "ALLOW"
    $ip = $login.principal.ip           // join: same IP in both events

    // Enforce ordering — login MUST happen after scan
    $login.metadata.event_timestamp.seconds >
      $scan.metadata.event_timestamp.seconds

  match:
    $ip over 2h after $scan

  condition:
    $scan and $login
}
OR in events:
// OR on single field — inline
events:
  $e.security_result.action = "BLOCK" or
  $e.security_result.action = "UNKNOWN"

// OR across fields — use parentheses
events:
  $e.metadata.log_type = "PROCESS_LAUNCH"
  (
    $e.target.process.file.full_path = "C:\\...\\powershell.exe" or
    re.regex($e.target.process.command_line, `-enc`) nocase
  )
NOT in events:
// NOT on field value
NOT $e.principal.ip = "10.0.0.1"

// NOT on function
NOT net.ip_in_range_cidr($e.target.ip, "10.0.0.0/8")

// NOT on grouped OR
NOT (
  $e.target.process.file.full_path = "C:\\Windows\\...\\svchost.exe" or
  $e.target.process.file.full_path = "C:\\Windows\\...\\lsass.exe"
)
OR in condition:
condition:
  $scan or  $login          // either event found
  $scan and $login           // both must be present
  $scan and not $exfil       // scan seen but no exfil (incomplete attack chain)
  #e > 5 or $total_bytes > 1000000
any / all Quantifiers
// any — field matches AT LEAST ONE value (clean multi-value OR)
any $e.network.http.method = ("PUT", "DELETE", "PATCH")

// inline set with in operator (equivalent)
$e.target.process.file.name in ("certutil.exe", "regsvr32.exe", "mshta.exe")

// Reference list version
$e.target.hostname in %threat_intel_domains

// all — EVERY element in a repeated field must satisfy the condition
// (detect all DNS answers returning private IPs)
all $e.network.dns.answers.data in %internal_ip_ranges

// not in — field value absent from set
$e.target.process.file.sha256 not in %approved_software_hashes
$e.network.http.method not in ("GET", "POST", "HEAD", "OPTIONS")
Reference Lists — %list_name
📋
What they are: Named, managed value collections stored in Chronicle. Reference with %list_name. Updates auto-propagate — zero rule edits, zero redeploy.
🏗
Create: Chronicle UI → Rules → Reference Lists → New List. Or via API: POST /v2/lists with "syntax_type": "PLAIN_TEXT" | "CIDR" | "REGEX".
TypeUse ForUsage Syntax
PLAIN_TEXTDomains, usernames, hashes — exact match$e.field in %list
CIDRIP range matchingnet.ip_in_range_cidr($e.ip, %list)
REGEXMultiple patterns against one fieldre.regex($e.field, %list) nocase
💡 Anti-pattern

Hardcoding IOCs inside rule bodies. When a new C2 domain appears, you'd edit and redeploy the rule. With a reference list, the threat intel team updates the list; all rules referencing it pick up the change automatically.

OR-Rule Pattern — When Logic Gets Complex
// Split complex OR into separate rules. Each fires independently.
rule ps_execution_by_path {
  meta: severity = "MEDIUM"
  events:
    $e.metadata.log_type = "PROCESS_LAUNCH"
    $e.target.process.file.full_path = "C:\\...\\powershell.exe"
    re.regex($e.target.process.command_line, `-enc`) nocase
  condition: $e
}

rule ps_spawned_from_office {
  meta: severity = "HIGH"
  events:
    $e.metadata.log_type = "PROCESS_LAUNCH"
    $e.target.process.file.name = "powershell.exe"
    re.regex($e.principal.process.file.name, `(winword|excel|outlook)\.exe`) nocase
  condition: $e
}
match: — Grouping & Time Windows
⏱
Single variable: match: $host over 1h — groups all matching events by $host, evaluates the condition once per window per unique host value. One alert per host per hour max.
⊕
Multi-variable: match: $user, $src_ip over 10m — compound GROUP BY. Fires only when the same pair appears together. Far fewer false positives than single-variable grouping.
⚓
Anchored window: match: $ip over 30m after $scan — window starts at $scan's timestamp. Enforces ordering. Floating windows (no after) can match in any order.
// Multi-variable — brute force from specific source
match:
  $user, $src_ip over 10m

// Anchored — correlated events in sequence
match:
  $ip over 2h after $scan
Deduplication & Window Behaviour
📊
With match: — Chronicle waits until the window closes, evaluates the condition once, fires at most one alert per unique group value per window. 500 login failures → 1 alert.
⚡
Without match: — condition is evaluated for every single matching event. One alert per event. If you expect volume, always add a match: window.
// Noisy — one alert per event
rule brute_force_noisy {
  events:
    $e.metadata.log_type = "USER_LOGIN"
    $e.security_result.action = "BLOCK"
  condition: $e    // alert storm guaranteed
}

// Tuned — one alert per user per 10 minutes
rule brute_force_tuned {
  events:
    $e.metadata.log_type = "USER_LOGIN"
    $e.security_result.action = "BLOCK"
    $user = $e.principal.user.userid
  match:
    $user over 10m
  condition:
    #e > 10
}
outcome: — Aggregate Functions (Complete Table)
FunctionPurposeUse in condition?
count(field)Count of non-null field valuesVia $var > N or #e shorthand
count_distinct(field)Count of unique valuesVia $var > N
sum(field)Numeric sumVia $var > N
max(field)Maximum valueVia $var > N
min(field)Minimum valueVia $var > N
array(field)Collect all values (includes duplicates)No — alert context only
array_distinct(field)Collect unique valuesNo — alert context only
 
// count vs #e shorthand — equivalent
outcome:
  $hits = count($e.metadata.id)
condition:
  $hits > 10
// same as:
condition:
  #e > 10   // #e shorthand — use this for brevity

// Arithmetic in outcome:
outcome:
  $byte_range  = max($e.network.sent_bytes) - min($e.network.sent_bytes)
  $avg_bytes   = sum($e.network.sent_bytes) / count($e.metadata.id)
  $risk_score  = count($e.metadata.id) * max($e.security_result.risk_score)
  $log_count   = math.log(count($e.metadata.id))
  $dest_sample = array_distinct($e.target.hostname)
Arithmetic operators: + - * / — watch for divide-by-zero. Everything in outcome: appears in the Chronicle alert detail view — design for analyst triage readability.

// Enforce that event B happened after event A

// Detect two auth events within 60 seconds (impossible travel)

re.capture — The Silent Exclusion Gotcha
☠
When re.capture finds no match: the placeholder is unset (not null — unset), and the event is silently excluded from all results. No error. Rule compiles and runs. Zero results with no explanation.
🔢
Exactly one capture group required. Zero groups or two+ groups cause a compile-time error. re.capture($e.url, `^https?://([^/]+)`) is valid. re.capture($e.url, `^(https?)://([^/]+)`) is not.
// Safe debugging pattern — stage in UDM Search

// Stage 1: verify base pattern matches (no capture)
events:
  re.regex($e.network.dns.questions.name, `^[^.]+\..+`)

// Stage 2: add capture — verify count doesn't drop
events:
  re.regex($e.network.dns.questions.name, `^[^.]+\..+`)
  $tld = re.capture($e.network.dns.questions.name, `^[^.]+\.(.+)`)

// Stage 3: add constraint
events:
  $tld = re.capture($e.network.dns.questions.name, `^[^.]+\.(.+)`)
  $tld = "ru"
strings.coalesce — Cross-Vendor Normalisation
// Different vendors populate different hostname fields
$host = strings.coalesce(
  $e.principal.asset.hostname,
  $e.principal.hostname,
  $e.principal.ip          // fallback to IP if no hostname
)

// Normalise parent process field across EDR vendors
$parent = strings.coalesce(
  $e.principal.process.file.name,
  re.capture($e.principal.process.file.full_path, `([^\\]+)$`)
)
re.regex($parent, `(winword|excel|powerpnt|outlook)\.exe`) nocase

Saved Queries vs Persistent Rules
Saved Queries (UDM Search)
📍Location: Search bar → Save → "Saved Searches"
🖱Execution: Manual only — click to run
👁Output: Results in the search UI
📂Version control: None built-in
✅Use for recurring investigation queries, incident triage, bookmarked searches
Persistent Rules (Detection Engine)
📍Location: Detection → Rules → Rules Editor
⚡Execution: Automatic, continuous, 24/7
🔔Output: Alerts in the findings queue
📝Version control: Chronicle tracks every edit as new version
✅Use for automated detection; triggers SOAR playbooks

✓ Rule Promotion Workflow

Write → Test state → Retrohunt → Evaluate → Promote to Alerting. Never deploy directly to Live/Alerting. Start every new rule in Test.

Testing Rules Before Deployment
1
Prototype in UDM Search first
Paste your events: conditions without the rule wrapper. Visual, fast feedback on whether field conditions match real data. 💡 If UDM Search returns results → field paths and values are correct. Wrap in a rule.
2
Create rule in Test state
Detection → Rules → New Rule → set state to "Test" before saving. Production alert queues are not affected.
3
Run a retrohunt
Select rule → Run Retrohunt → specify time range (minimum 7 days; use 30+ days for rare attacks). Chronicle evaluates the rule against historical data. ⚠ Wrong log_type or empty fields cause zero retrohunt results. Verify field existence in UDM Search first.
4
Evaluate retrohunt results
Did it fire? Did it fire on the right things? Too noisy? Compare against known-bad events from past incidents.
5
Promote to Alerting
Rule state → "Alerting" → Chronicle now generates real production alerts.
Testing Gotchas
⚠️
Wrong log_type string: Rule never fires because "WINDOWS_SYSMON" vs "WINDOWS_SYSMON_8_0". Always verify the exact string via UDM Search first.
⚠️
Empty field: Filtering on a field your log source doesn't populate. Check: $e.target.asset.hostname != "" in UDM Search before using it in a rule.
⚠️
Retrohunt time range too short: Some attacks are rare. Run retrohunt over 30+ days for meaningful signal-to-noise data.
⚠️
re.capture silent exclusion: A broken capture pattern silently drops all events. Prototype with re.regex first, then add re.capture.
Rule Versioning & rule_id
🔑
rule_id — immutable UUID assigned at creation: ru_a1b2c3d4-e5f6-7890-abcd-ef1234567890. Never changes even if you rename or edit the rule. Use in SOAR playbooks and API calls.
📌
Version ID — every edit creates a new version: ru_...@v_12345678_000000. Findings reference which version generated them. Roll back via the API.
