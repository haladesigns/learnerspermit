Git — Without the Friction

Claude Code handles the entire version control lifecycle through plain English. This isn't about replacing git knowledge — it's about removing the gap between "I finished the code" and "the PR is open for review."

What Claude Code Does with Git
✍️
Analyzes your changes and writes commit messages automatically — reads diffs, not just filenames.
🌿
Creates branches with proper type/description naming from a single instruction.
🔀
Opens pull requests with structured summaries and test plans.
⚔️
Resolves merge conflicts by understanding what both branches intended.
🔍
Reviews code for bugs, security issues, and style problems before you push.
⚡
Chains the entire workflow — branch → commit → push → PR — in one command.
Built-in Skills
/commit
The fastest path to a clean commit. Claude reads staged and unstaged changes, understands what changed, and proposes a message. You always review and approve first.
/commit-push-pr
Full lifecycle in one shot: checks branch → creates new branch if on main → commits → pushes → opens PR on GitHub. No back-and-forth.
📋 Prerequisite

Basic git knowledge (commit, branch, push, pull) and GitHub CLI (gh) installed and authenticated for PR creation.

💡 Core Insight

Claude reads the actual diff — not just filenames. It understands what functions changed, what bugs were fixed, what features were added. Commit quality reflects change quality.

📖 Scenario

You've just built a notification preferences page and want to get it merged. Here's the complete flow.

Complete Feature Workflow
1
Check What Changed
Get a sanity check before committing. Claude gives you a plain-English summary of your diff.
"Summarize the changes I've made since my last commit."
2
Create a Commit
Three options depending on how much control you want:
Option A — Slash command: /commit → Claude proposes, you approve with Enter
Option B — Natural language: "Commit my changes with a descriptive message."
Option C — Directed: "Commit these changes. The main change is fixing the race condition, not the UI tweaks."
💡 Option C lets you tell Claude what matters most in the message.
3
Create a Branch and Push
If you're still on main, one prompt handles everything:
"Create a feature branch for these notification changes, commit, and push."
Claude creates feature/notification-preferences, commits, and pushes. Branch type is inferred from your description: feature/ fix/ hotfix/ chore/
4
Open a Pull Request
Claude creates a structured PR body with Summary and Test Plan sections, or use the full-lifecycle command:
/commit-push-pr
handles branch + commit + push + PR in one shot
💡 For complex PRs, prompt with what problem it solves, what approach you took, and risks to watch for.
5
Address Review Feedback
After your teammate leaves review comments:
"Look at the PR comments on my current branch and address them."
Claude reads the comments, implements the changes, and creates a new commit. ⚠ Review Claude's resolutions on business logic — don't blindly accept.

💡 Patterns vs Commands

These are repeatable workflows — not one-off prompts. Set them up once (especially CLAUDE.md conventions) and they pay dividends across every commit.

7 Common Patterns
P1
Set Up Commit Conventions in CLAUDE.md
Define your team's rules once and Claude follows them on every /commit. Put this in CLAUDE.md and commit it to the repo so the whole team benefits automatically. 
## Git Conventions
Commits: Conventional Commits format
Types: feat, fix, docs, style, refactor, perf, test, chore
Format: type(scope): imperative description
Max subject line: 72 characters
Always include a body explaining WHY, not just WHAT
Branch naming: type/short-description (e.g., feat/user-auth)
PRs: always target "develop", never directly to "main"
P2
Split Changes into Atomic Commits
Been working a while with changes across many files? Instead of one massive commit, let Claude sort it out:
"I've made several unrelated changes. Split them into logical commits: one for the API changes, one for the UI components, and one for the test updates."
Claude groups changes by purpose and commits each separately with appropriate messages.
P3
Detailed PR for Reviewers
Give reviewers everything they need in one shot:
"Create a PR that explains: what problem this solves, what approach I took, what risks reviewers should watch for, and how to test locally."
P4
Resolve Merge Conflicts
After git rebase main hits conflicts:
"There are merge conflicts. Analyze each conflicting file, understand what both sides intended, and resolve them. Keep both modifications when possible."
Simple conflicts (imports, non-overlapping additions) → resolved automatically. Complex conflicts (business logic) → Claude explains both sides and asks you to decide.
P5
Pre-Push Code Review
Use Claude as a reviewer before you push:
"Review the changes on this branch compared to main. Look for: bugs or logic errors, security issues, missing error handling, and inconsistencies with the rest of the codebase."
P6
Work on Multiple Tasks with Worktrees
When you need to work on two things at once without changes colliding, open separate sessions: 
claude --worktree feature-auth
# In another terminal:
claude --worktree bugfix-123
Each session gets its own working directory and branch.
P7
Resume a PR Session
When you come back to a PR after time away: 
claude --from-pr 42
Picks up with full context of the PR — including any review comments that came in since you last worked on it.
Gotchas & Tips
🔒
Claude Never Commits Without Permission: The /commit skill requires your approval by default. Claude never decides on its own to commit your code.
🎯
Be Specific About What to Commit: "Commit my changes" looks at everything — staged and unstaged. To commit only specific files: "Stage only src/api/notifications.ts and commit."
⚠️
The gh CLI Must Be Installed for PRs: Claude can do everything up to git push without it, but can't open PRs. See install table below.
⚠️
Don't Blindly Accept Conflict Resolutions: Claude handles simple conflicts (imports, non-overlapping additions) well. For business logic conflicts, Claude explains both sides and asks you to decide. Review carefully on critical code.
📏
Commit Messages Depend on Clean Changes: A 2,000-line dump of unrelated work produces vague messages. Keep changes focused — or use Pattern 2 (atomic commits) to split them first.
📖
CLAUDE.md Is Your Team's Git Playbook: Put your git conventions in CLAUDE.md and commit it. Every team member's Claude Code reads it automatically — no more "please follow commit conventions" review comments.

## Git Rules

Never commit directly to main
All PRs require at least one approval
Squash merge only
Delete branches after merge
Commit messages: conventional commits, English, imperative mood
✓ Good Habits
✅Define commit conventions in CLAUDE.md once, let them run forever.
✅Use atomic commits (P2) when changes span unrelated concerns.
✅Run a pre-push review (P5) on anything touching auth or payments.
✅Use --worktree for parallel tasks to avoid context collisions.
✗ Things to Avoid
❌Letting 2,000 lines of mixed changes accumulate before committing.
❌Blindly accepting conflict resolutions on business-critical files.
❌Using PR creation without gh installed — push will work but PR won't open.
❌Skipping the Step 1 sanity check before committing large changes.
