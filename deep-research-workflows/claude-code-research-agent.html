Claude Code as Research Agent

Move beyond "super Google." This deep dive shows how to build a modular skill pipeline that searches YouTube, selects the best videos, feeds them into NotebookLM, runs deep analysis, and delivers a research report — fully autonomously. The same philosophy applies to any research niche.

What Claude Code Can Do Out-of-Box
🌐
Web search — basic Google-style search, good for text sources. Sometimes requires explicit instruction: use web search to find…
🧠
Training data — has a cutoff (~9 months). Zero knowledge of anything after that. Always verify recent topics with web search.
🚫
Can't handle video — no LLM reliably ingests full-length YouTube content. This is the core problem the pipeline solves.
Why YouTube Needs a Skill
📡
Fountainhead problem — relying on Medium / news articles means you're already late. Real signal lives on YouTube, Twitter, and GitHub first.
🔁
Repetition = skill — any time you find yourself giving Claude the same specialized instructions repeatedly, that process should become a skill.
🎯
Specificity required — plain language can't reliably search YouTube, scrape metadata, filter by engagement, and route to NotebookLM. A skill can.
The Full Research System at a Glance
🔍
YT Search subskill — uses yt-dlp CLI to search YouTube and return structured metadata (title, channel, subs, views, ratio, URL).
🎛️
Auto-select — Claude picks the best 5–8 videos by relevance, engagement ratio, recency, depth, and source diversity. No user input needed.
📓
NotebookLM stack — creates a notebook, adds video URLs as sources, and runs analysis questions. Runs on Google's servers — doesn't cost tokens.
📄
Obsidian integration — the report is dumped directly into an Obsidian vault as a structured markdown note with date, summaries, gaps, and opportunities.
⚡
Extensible in-prompt — skills aren't rigid automations. You can say /yt-pipeline and add: "also search GitHub and Twitter for this topic."
💡 Core Takeaway

Most people use Claude Code as super Google. Skills let you use it as a research system — one that searches primary sources, filters by signal quality, offloads heavy analysis, and delivers structured output, all from a single sentence.

7-Step YT Pipeline

One skill, five sub-skills, fully autonomous. Invoke with a single natural language sentence and receive a structured research report. No user input between steps.

YT Search Skill — Internals

The foundational sub-skill. Searches YouTube as if a human typed into the search bar, then returns structured metadata including the engagement ratio (views ÷ subs) that identifies outlier videos. Only 55 lines.

When skills are loaded, Claude sees only the name + description first. The skill body activates only once the skill is invoked. This is why description quality is critical for correct triggering.
Dependency
📦
yt-dlp — CLI tool that powers the search. Acts as a YouTube client; handles the actual HTTP requests to YouTube search. Must be installed: pip install yt-dlp
🐍
search.py — Python wrapper around yt-dlp. Status messages go to stderr; clean formatted results go to stdout. Fetches 2× count to compensate for date filtering.
📍
Path: ~/.claude/skills/yt-search/scripts/search.py (or full Windows path per skill file)
Parameters
Parameter Default Description
<query> required Search terms. Claude crafts this from user intent — may differ from exact user words for better results.
--count N 20 Number of results to return. Pipeline skill uses 10 by default for focused sets.
--months N 6 Only return videos from the last N months. Use 12 for evergreen topics.
--no-date-filter off Include videos of any age. Auto-retried if initial filtered search returns no results.
Output Fields Per Video
📌
Title — used for relevance filtering in step 3
📣
Channel name + subscriber count — baseline for ratio calculation. May show N/A if unavailable.
👁️
View count — raw views
⏱️
Duration — pipeline prefers 10+ min videos for research depth; shorts/clips are deprioritized
📅
Upload date — used for recency scoring in step 3
📈
Engagement ratio (views ÷ subs) — the key signal. A high ratio means the video outperformed the channel's baseline = outlier content worth investigating.
🔗
Direct URL — passed to NotebookLM as a source in step 4

# YouTube Search

Search YouTube using yt-dlp and return structured, formatted results with engagement metrics.

## How to use

Run the search script:

python "C:/Users/Chase/.claude/skills/yt-search/scripts/search.py" <query> [--count N] [--months N] [--no-date-filter]

### Parameters
- `<query>` — search terms (required)
- `--count N` — number of results to return (default: 20)
- `--months N` — only show videos from the last N months (default: 6)
- `--no-date-filter` — include videos of any age

### Examples

# Basic search (top 20, last 6 months)
python "C:/Users/Chase/.claude/skills/yt-search/scripts/search.py" claude code tutorial

# Fewer results, shorter window
python "C:/Users/Chase/.claude/skills/yt-search/scripts/search.py" obsidian workflow --count 5 --months 3

# All time, no date filter
python "C:/Users/Chase/.claude/skills/yt-search/scripts/search.py" AI agents --count 10 --no-date-filter

## Output format

Each result includes:
- Title
- Channel name and subscriber count
- View count
- Duration
- Upload date
- Engagement ratio (views / subscribers) — higher means the video outperformed the channel's base
- Direct URL

Results are separated by dividers for easy scanning. Status messages (searching, filtering) go to stderr; the formatted results go to stdout.

## Notes

- yt-dlp must be installed (`pip install yt-dlp`)
- The script fetches 2x the requested count to compensate for date filtering
- Subscriber counts come from yt-dlp metadata and may occasionally be unavailable (shown as N/A)

# YouTube Research Pipeline

Fully automated pipeline: YouTube search → smart video selection → NotebookLM notebook → analysis → deliverable. Runs end-to-end without asking for input mid-flow unless the request is genuinely too vague.

## When This Skill Activates

**Intent detection — activate when the user wants to:**
- Research a topic using YouTube as a source
- Find and analyze YouTube videos on a topic
- Create a deliverable (podcast, report, quiz, etc.) from YouTube content
- Understand what YouTubers are saying about something

**Example triggers:**
- "Research what YouTubers are saying about Claude Code"
- "Find YouTube videos about AI agents and make a podcast"
- "What's the YouTube consensus on [topic]?"
- "Search YouTube for [topic] and summarize the findings"
- "Create a study guide from YouTube videos about [topic]"

**Do NOT activate for:**
- Simple YouTube searches with no analysis intent (use `/yt-search` instead)
- NotebookLM tasks that don't involve YouTube (use `/notebooklm` instead)

## Pipeline Steps

### Step 0 — Pre-flight Check

Before starting the pipeline, verify NotebookLM auth is working:
notebooklm status
If it fails or shows no authentication, tell the user to run `notebooklm login` and stop. This catches auth issues early instead of discovering them mid-pipeline after the search and selection work is already done.

### Step 1 — Understand the Request

Extract from the user's message:
- **Search query** — the topic to search for
- **Analysis goal** — what they want to learn (optional)
- **Deliverable type** — podcast, report, quiz, etc. (optional)

**If the request is too vague** to form a useful YouTube search query (e.g. "research stuff"), ask ONE short clarifying question, then proceed.
**Otherwise, proceed immediately.** Do not ask unnecessary questions. Infer reasonable defaults.

### Step 2 — Search YouTube

Run the yt-search script:
python ~/.claude/skills/yt-search/scripts/search.py <query> --count 10 --months <M>

- Craft the best search query from the user's intent — this may differ from their exact words
- Default to `--months 6` unless the topic warrants a wider window (evergreen content → `--months 12` or `--no-date-filter`)
- **Show the full results to the user** — present all videos with their complete metadata

**If no results:** Automatically retry with a broader query or `--no-date-filter` before reporting failure.

### Step 3 — Auto-Select Videos

Immediately after showing results, auto-select the best videos. **Do not wait for user input.**

**Selection criteria (in priority order):**
1. **Relevance** — does the title/channel match the user's intent?
2. **Engagement** — views-to-subs ratio outliers signal high-quality content
3. **Recency** — prefer newer content when quality is comparable
4. **Depth** — prefer longer videos (10+ min) over shorts/clips for research purposes
5. **Source diversity** — prefer videos from different channels over multiple from the same creator

**Drop obvious mismatches:** wrong topic, unrelated shorts, clickbait with low engagement.
**Target 5-8 videos.** NotebookLM works best with focused source sets.

### Step 4 — Create Notebook & Add Sources

1. Create notebook:
notebooklm create "YT Research: <topic>" --json
   Parse the notebook ID from the JSON output.

2. Set context:
notebooklm use <notebook_id>

3. Add each selected video as a source:
notebooklm source add "<youtube_url>" --json
   Collect source IDs from the JSON output.

4. Wait for all sources to process:
notebooklm source wait <source_id>
   Run these sequentially. Each takes 10-60 seconds.

**Error handling:**
- Auth failure → tell user to run `notebooklm login`
- Individual source failure → skip it, continue with others, report skipped sources at the end
- If all sources fail → stop and report the error

### Step 5 — Analysis

Run analytical questions via `notebooklm ask "..."` tailored to the user's goal.

**Default questions (no specific goal):**
1. `notebooklm ask "What are the key themes and main points across these videos?" --new`
2. `notebooklm ask "Where do the creators agree and where do they have different perspectives?"`
3. `notebooklm ask "What are the most actionable takeaways someone could apply immediately?"`

Use `--new` on the first question to start a fresh conversation. Subsequent questions are follow-ups in the same thread (no `--new` flag).

**If in an Obsidian vault:** Offer to save the analysis as a research note to `/research/` following vault conventions.

### Step 6 — Generate Deliverable (If Requested)

Only if the user asked for a specific deliverable (podcast, report, quiz, study guide, etc.).

**Map user intent to NotebookLM generation commands:**

| User says              | Command                                              |
|------------------------|------------------------------------------------------|
| "podcast" / "audio"   | `notebooklm generate audio "Focus on [goal]" --json` |
| "video"               | `notebooklm generate video "Focus on [goal]" --json` |
| "report" / "summary"  | `notebooklm generate report --format briefing-doc`   |
| "study guide"         | `notebooklm generate report --format study-guide`    |
| "quiz"                | `notebooklm generate quiz --json`                    |
| "flashcards"          | `notebooklm generate flashcards --json`              |
| "mind map"            | `notebooklm generate mind-map`                       |
| "slide deck"          | `notebooklm generate slide-deck --json`              |
| "infographic"         | `notebooklm generate infographic --json`             |

**For long-running artifacts (audio, video, quiz, flashcards, slide-deck, infographic):**
- Parse the `task_id` / `artifact_id` from the `--json` output
- Spawn a background agent to wait and download
- Tell the user generation has started and they'll be notified when it's ready

**For instant/fast artifacts (report, mind map, data table):**
- Generate and present immediately
- Offer to save to the vault if applicable

## Autonomy Rules

**Run the full pipeline automatically.** The entire point of this skill is end-to-end automation. Do not pause for confirmation between steps unless something fails.

**The only acceptable pause points:**
- Step 1: Request is too vague to search → ask ONE clarifying question
- Step 4: Auth failure → tell user to run `notebooklm login`
- Step 6: Before generating a deliverable the user didn't explicitly ask for

**Everything else flows automatically:** search → show results → select → notebook → sources → analysis → deliverable.

## Output Style

- Show YouTube search results in full (the script output is well-formatted)
- Keep selection reasoning to 1-2 sentences
- Present analysis findings in bullet points with a `## Key Takeaways` section
- Use progress updates between steps: "Searching YouTube...", "Creating notebook...", "Adding 6 sources...", "Analyzing content..."

Skill Design Philosophy

Mastering skills is the value unlock of Claude Code. Here's the mental model and process for turning any workflow into a modular, testable, composable skill system.

The Modular Skill Mindset
1
Break the large task into discrete phases
Before writing any skill, map out your workflow from step zero to the end deliverable. If you can't articulate it, Claude Code can help — stream-of-consciousness explain what you do, then ask it to identify phases.
2
Turn each phase into its own skill
One skill = one well-defined job. Keep them narrow. The YT search skill doesn't know about NotebookLM. The pipeline skill doesn't contain the search logic. 💡 Each skill should be independently invokable and testable.
3
Test each skill in isolation first
Before building the "super skill," verify every sub-skill works. Troubleshooting a 7-step pipeline is exponentially harder than troubleshooting a 55-line search script. ⚠ Don't build the pipeline until all sub-skills are locked in.
4
Combine into a pipeline skill
Once sub-skills are proven, create a higher-order skill that calls them in sequence. The pipeline skill is essentially: step 0 calls X, step 1 calls Y, step 2 calls Z.
Why Modular > Monolithic
🧩
Use sub-skills independently — you don't always want the full pipeline. Sometimes just /yt-search is enough.
🔬
Easier to debug — isolate which step is failing without guessing across 7 steps simultaneously.
♻️
Reusable across pipelines — the NotebookLM skill could be called from a Twitter pipeline, a GitHub pipeline, etc.
⚡
Piecemeal improvement — improve the search skill without touching the pipeline skill. Changes are contained.
Skill Creation Process
A

The Skill Creator skill (from Anthropic) tests and benchmarks skills as you build them.
B
Enable Plan Mode
Use plan mode so Claude thinks before acting. You're designing, not executing yet.

Talk for 2–5 minutes. Include deliverables, examples, edge cases, what you don't want.

Claude will identify logical boundaries and propose a modular structure.
What to Think About When Designing a Skill
🎯
End result / deliverable — what does success look like? A markdown file? A list of URLs? A report? Define this first.
⚙️
Parameters — what inputs does this skill need? What are sensible defaults? (e.g. --count 10, --months 6)
🚦
Trigger conditions — when should this skill activate? When should it explicitly NOT activate? Write both into the frontmatter description.
🤖
Autonomy rules — where should Claude pause and ask? Where should it proceed without asking? Be explicit. Silence = Claude uses judgment.
💥
Error handling — what happens when a step fails? Skip and continue? Stop entirely? Report at end? Define for each possible failure point.
📤
Output format — bullet points? Markdown file? JSON? Terminal summary + long-form report? Specify exactly.
🚦 The Trigger Signal

If you find yourself giving Claude Code the same specialized instructions more than once, that's your signal. Stop and create a skill. The goal is to never repeat yourself — any repeated, specialized workflow is a skill waiting to be written.

Beyond Manual — Automation

Skills require Claude Code to be open and running. But some research tasks can be fully automated — running on your machine every morning without you doing anything.

Two Automation Paths
🖥️
Claude Code Desktop — Scheduled Tasks

⚡
Direct API / Task Scheduler
If the research is a simple API request (no LLM needed), have Claude Code write a task scheduler script that runs independently on your machine.
Real Example: GitHub Trending
📅
GitHub publishes a daily trending report via their API. Pure API call — no LLM required.
🤖
Claude Code wrote a task scheduler script — runs on the machine itself at 8am every day. Claude doesn't need to be running.
🌅
Every morning: wake up to a report of trending AI repos — weekly and monthly. Ground-floor signal before anyone else is talking about it.
How to Find Your Own Automated Source
1
Identify your "fountainhead"
What is the primary source of signal in your niche? GitHub repos? arXiv papers? A specific API? Twitter/X? Product changelogs? Wherever the information originates is where you want to tap.
2
Ask Claude Code if an API exists
"Is there an API for [source] that gives me [type of data]?" If yes, Claude can write the retrieval + formatting code for you. 💡 You don't need to know if an API exists — that's Claude's job.
3
Have Claude write the task scheduler
Specify: what to fetch, how to format it, where to save it (Obsidian vault, email, etc.), and what time to run. ⚠ Claude writes the code; your machine runs it. Claude doesn't need to be involved at runtime.
4
Wake up to a report
That's it. You now have a passive intelligence feed on your niche that requires zero manual effort after setup.
