Building Complex Codebases with AI

A framework for using AI coding assistants on real, production-scale projects — not just toy scripts. The key insight: AI amplifies existing craft. You still own planning, architecture decisions, and validation. The framework gives AI the context it needs to execute reliably.

Each feature/task runs its own PIV loop. The loop is not optional — skipping validation is how AI-assisted projects accumulate hidden debt.

📐
Modular Rules: Split rules by domain — agents, references, components. Load only what's relevant per conversation.
🔁

System Evolution Mindset
🐛
Every bug is an opportunity to evolve your system rules — not just fix the immediate issue.
📝
When AI makes a mistake, ask: "How do I prevent this class of mistake forever?" Then encode the answer into rules.
📈
Your SYSTEM improves over time. Each project leaves you with better rules, commands, and context that compound on the next.
💡 Core Principle

AI doesn't replace engineering judgment — it amplifies it. The framework exists to give AI enough structured context that it can act reliably, while keeping humans in control of architecture, goals, and validation.

Greenfield Project Planning

Before writing any code, build the AI layer for your codebase. This means creating the PRD, rules architecture, and prime command that will guide every subsequent conversation.

PRD-First Development
📋
Start every project with a PRD, not code. The PRD becomes the source of truth for every AI conversation and each granular feature breakdown.
🆕
New projects: Fart the PRD with features, then break it into phases. Use AI to help flesh it out from your vibe plan.
🔍
Existing codebases: Document existing code first. AI makes assumptions — the PRD corrects them and keeps all conversations aligned.
⚠️
Context drift is real. Without a PRD, each new conversation forgets prior decisions. The PRD is persistent memory across sessions.
Modular Rules Architecture

Stop dumping everything into one massive rules file. Split by content, load only what's relevant per conversation via @path/to/file imports.

.claude/ ├── agents/ ← agent-specific rules (e.g. AGENTS.md) ├── references/ ← stable docs, API specs, architecture notes ├── components.md ← UI component conventions and patterns └── api.md ← API integration rules and gotchas CLAUDE.md ← auto-loaded; pointers to other files
✂️
Working on frontend? Load components.md + relevant references. Don't load API rules — they add noise.
🤖
Working with agents? Load agents/AGENTS.md with your main context. Context pollution is a real failure mode.
Subagents for Context Isolation

Delegate research to subagents. They explore everything (files, web, git), but only summarize results back to your main context. Keeps your main conversation focused.

MAIN AGENT
Owns high-level task. Stays focused.
→
Subagent
Explores codebase / web / git
→
Subagent

→
Summary back
Distilled insights only — not raw findings
🎯
Subagent outputs are summaries, not dumps. The main agent only sees distilled, recommendation-ready context.
🔒
Context isolation prevents hallucination cascade. Unrelated explorations that pollute the main context cause compounding errors.

The PIV Loop — Core Feature Development

Every feature runs through Plan → Implement → Validate. Planning and execution are separate conversations (see Golden Rules). The goal of planning is to front-load context so implementation is execution, not exploration.

PLANstructured context
IMPLEMENTexecute task by task
VALIDATEtrust but verify
ITERATEnext task / bugfix
Steps to Planning
1
"Vibe plan" — explore before structuring
Explore ideas, architecture, concepts, tech stack, etc. with the AI assistant. Not structured yet. For new projects: research online resources, previous projects.
For existing projects: research/analyze the existing codebase.
2
Create a structured plan using Context Engineering
Transform the vibe plan into a formal plan of attack. This is what you hand to the AI for implementation. 💡 The goal: produce a plan so complete that the AI has the context it needs — most of the structure is already in place before execution starts.
Context Engineering — What Goes in the Plan

Context Engineering = RAG + Memory + Task Management + Prompt Engineering. A complete plan covers all four zones.

RAG

External knowledge pulled in: docs, API specs, existing codebase analysis, architecture references.

Memory

What the AI should know about prior decisions, project history, constraints already established.

Task Management

Ordered task list, dependencies, what "done" looks like, validation criteria per task.

Prompt Engineering

How you frame the problem, persona, output format, constraints on behavior during execution.

Plan Components (Can Include)
🎯
Goal(s) — what success looks like for this feature
✅
Success criteria — measurable, testable outcomes
📚
Documentation to reference — which files, specs, or APIs are in scope
📋
Task list — ordered steps the AI will execute
🔬
Validation strategy — how each task gets verified
🏗️
Desired codebase structure — file layout, naming conventions, patterns
Two Layers of Planning
Layer 1 — Project Planning
Done once, updated rarely
Tech stack decisions
Architecture patterns
Constraints & conventions
Creating stable resources (PRD, global rules)
Layer 2 — Task Planning
Done per feature/task
Codebase analysis for this feature
Documentation analysis
Integration points
Task-specific rules
Steps to Implementing / Validating
1
Execute the implementation plan task by task
Most structure is already in place from planning. Execution is about building, not exploring. 💡 At this point the AI has the context it needs — the plan already contains most of the structure required.
2
Validate the produced code — Trust but verify
Let the AI run, but actively check that it:
  • Uses its tools correctly
  • Is reading/editing the right files
  • Is managing its tasks properly
  • Produces "thinking" tokens that show it understands the plan of attack
Validation Split — Human + AI
👤 Human Validation
Code Review
Manual Tests
Performs high-judgment validation — architectural sanity, edge case intuition, product correctness.
🤖 AI Coding Assistant Validation
Unit Tests
Integration Tests
Runs exhaustive automated tests. Better at systematic coverage than humans but needs human-defined scope.
⚠️ Critical — Iterate Until Complete

The PIV loop repeats per task, not just per feature. Don't batch multiple tasks into a single implementation + validation pass. Compounding unvalidated work is how AI projects lose coherence.

Global Rules + On-Demand Context

Rules are the AI's long-term memory between sessions. Global rules live in CLAUDE.md and load automatically. On-demand context loads only when relevant via @file imports.

Global Rules (CLAUDE.md)
⚡
Auto-loaded at the start of every conversation. Don't put volatile or feature-specific rules here.
🔗
Use CLAUDE.md as a pointer file — reference other rule files rather than stuffing everything into one place.
📌
Good global rules: coding conventions, project structure, test patterns, commit message format, never-do-X constraints.
On-Demand Context
📂
Load context explicitly with @.claude/references/api.md when working on API features.
🎯
Load only what's relevant. Every extra file in context is noise that degrades precision.
📦
Good on-demand files: component libraries, API specs, agent rules, feature-specific conventions.

📐 Architecture principle

Commands become your reusable interface to AI workflows. Once a command works well, it's free to use forever. Your workflow sophistication compounds across projects.

The Four Golden Rules

These are the highest-leverage operating principles — the ones that prevent the most common failure modes when building complex systems with AI assistants.

01
THE CONTEXT RESET
Planning and execution are SEPARATE conversations. Context window degradation is real — fresh starts matter.
PLAN
research, design
NEW CHAT
AI context
load PRD + plan
→
EXEC
clean context
⚠️ The Failure Mode

After many messages, agents get overwhelmed and make repeated mistakes. Context reset is the fix — not prompt tweaking.

02
COMMAND-IFY EVERYTHING
If you're doing something more than twice, make it a command (aka skill). Your workflows become reusable, shareable tools.
EXAMPLE COMMANDS
/commit
/review
/test
/fix-types
/add-feature
/optimize
One command can encode thousands of tokens of context and instructions. Running /commit beats typing the same instructions every session.
💡 Compounding

Commands accumulate across projects. Your toolkit grows. Each new project starts with a richer set of battle-tested workflows.

03
GIT LOG AS MEMORY
Your commit history is your long-term memory. Standardized commit messages communicate intent to future coding agent sessions.
GIT LOG — as AI context
a3f2d1 feat: add OAuth flow for GitHub
b9e1c4 fix: token refresh race condition
c7a0f2 refactor: split agent into subagents
d5e3b1 chore: update rules per session learnings
The prime command reads the git log to reconstruct what happened since the last session. Bad commit messages = lost context.
04
SYSTEM EVOLUTION MINDSET
Every bug is an opportunity to evolve your SYSTEM for AI coding. Not just fix the bug — improve the rules that prevent the bug class.
BUG / mistake
→
Ask: "Why did AI do this?"
→
Add rule to prevent it
→
+ RULE in CLAUDE.md
Example: AI keeps ignoring type safety → add rule: "Always run tsc before committing". Now that error never recurs.
📈 Flywheel Effect

More rules → fewer mistakes → more trust → bigger delegation → faster shipping. The system compounds.

Commands Quick Reference

Commands (skills) are the reusable unit of AI workflow. Each one encodes a repeatable process. Build your library incrementally — every project adds to it.

Core Commands
Command What It Does When to Use
/plan Prime command — reads back PRD, explores structure, checks git log, builds mental model Start of every implementation conversation
/commit Stages changes, generates a standardized commit message, pushes — following your git conventions After each validated task
/review Reviews current diff for correctness, style, security, and compliance with project rules Before human code review or merge
/test Runs the appropriate test suite for the changed code and reports results Validation step of the PIV loop
/fix-types Runs TypeScript compiler, identifies type errors, applies targeted fixes After any TS refactor or new API integration
/add-feature Kicks off Layer 2 planning for a new feature — codebase analysis, task breakdown, context setup Start of any new feature PIV cycle
/optimize Profiles and refactors a specified module for performance, following project optimization rules Post-validation performance pass
Command Design Principles
✓ Good Commands
✅Encode a repeatable, multi-step process you'd otherwise re-explain every session.
✅Reference specific project files, conventions, or standards by path.
✅Have a clear output — a commit, a test report, a diff, a plan doc.
✅Are idempotent — running twice doesn't break things.
✗ Bad Commands
❌Vague instructions with no defined output ("be more helpful").
❌Commands that combine unrelated concerns in one shot.
❌Commands that assume context without loading it explicitly.
❌Commands you've only needed once — don't over-engineer your toolkit.
Prime Command — Full Anatomy

The prime command is the most important one to get right. Here's the full structure it follows when invoked:

# PRIME COMMAND — /plan
Step 1 — Read back context
  @PRD.md                     ← product requirements
  @CLAUDE.md                  ← global rules + pointers
  @.claude/references/*.md    ← relevant on-demand context

Step 2 — Explore structure
  list top-level directories
  read key entry points (main, index, app)
  map module relationships

Step 3 — Check git log
  git log --oneline -20        ← what changed recently
  git diff HEAD~3             ← recent diffs if needed

Step 4 — Build mental model
  Output written summary before acting:
  - Current state of the codebase
  - What was done recently
  - What the plan is for this session
  - Any risks or blockers identified
🚀 Getting Started

Start with just three commands: /plan, /commit, and /test. These cover the core PIV loop. Add more only when you notice yourself re-explaining the same thing twice.
