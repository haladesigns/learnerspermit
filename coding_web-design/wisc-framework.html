WISC Framework

A battle-tested system for keeping your Claude Code context window lean and high-signal. Context rot — the measurable degradation in LLM output quality as input length grows — is responsible for roughly 80% of agent failures in real codebases. WISC is the engineering response: Write, Isolate, Select, Compress.

The Four Pillars
W
Write
Externalize agent memory so future sessions start fast
I
Isolate
Use sub-agents to research without polluting main context
S
Select
Load context just-in-time, not just-in-case
C
Compress
Last resort only — handoff or /compact when bloated
Why Context Rot Matters
~80%
of agent failures attributed to poor context management
30%+
accuracy drop for info buried in the middle of context RESEARCH
The Chroma technical report (2025) evaluated 18 frontier LLMs — GPT-4.1, Claude 4, Gemini 2.5, Qwen3 — and found that every single one degrades as context length increases. A 1M-token window still rots at 50K tokens. The issue is signal-to-noise ratio, not capacity.
Core Principle
🎯
The goal: keep the context window as lean as possible while giving the agent everything it genuinely needs.
🧲
Distractor problem: Similar-but-wrong code in a large context causes confident wrong answers — especially in codebases with repeated patterns.
📐
Bigger windows don't fix this. 1M token limit = more room to accumulate noise, not immunity to rot.
💡 Mental Model

Context is your most precious resource — not tokens you're allowed to fill, but signal you must engineer. Every token in the window competes for the model's attention. Unnecessary tokens don't sit quietly; they actively dilute the important ones.

🔁 The Loop

Repeat Steps 4–12 for every feature. Each /commit feeds the git log that the next session's /prime reads. The system compounds — the longer you use it, the faster each new session starts.

W Write — Externalize Agent Memory

Capture decisions and progress outside the context window so future sessions spend fewer tokens getting up to speed. Three strategies: git log as long-term memory, separate planning and implementation sessions, and progress/handoff files.

W1 — Git Log as Long-Term Memory
💾
Use standardized commit messages as a persistent, queryable log of what the agent has worked on. Everyone's already using Git — no extra tooling needed.
📋
A /commit command enforces the format: what was built + how the AI layer (rules/commands) was improved. Both halves matter.
🔍
At session start, the /prime command reads the git log --oneline to orient the agent on recent work before any planning begins.
⚠️
Vague commit messages break this entirely. If messages aren't structured and detailed, the log becomes useless as memory.
# Example /commit command output format
feat(cli): standardize test runner output formatting

Changes:
- Added colored output for pass/fail states
- Normalized exit codes across test suites

AI layer improvements:
- Updated /plan command to include validation step
- Added edge-case section to test writing rules
W2 — Separate Planning & Implementation Sessions
1
Planning session — run /prime, load context, discuss the task. The agent explores, asks questions, does research. 💡 This session gets messy. That's fine — it's disposable.
2
Create a structured plan — run /plan to produce a single markdown document. This is your hand-off artifact. It must contain everything the agent needs: what to build, how to validate, relevant file paths. ⚠ If the spec is incomplete, the implementation session will re-research and pollute its context.
3
Fresh context window — start a brand new session, run /execute path/to/plan.md. No other context loaded.
4
Implementation only — the agent is maximally focused. No planning residue, no exploratory dead-ends cluttering the window. 💡 The spec is doing the work that context would otherwise do.
The PRP — Your Structured Plan Format

A PRP (Product Requirements Prompt) is the formal artifact produced by the planning session. It's the handoff between planning and implementation — PRD + codebase intelligence + runbook. The minimum viable packet an AI needs to ship production-ready code on the first pass.

INITIAL.md template — feature request format
## FEATURE:
Build an async web scraper using BeautifulSoup
that extracts product data, handles rate limiting,
and stores results in PostgreSQL.

## EXAMPLES:
- Auth flow: examples/auth-flow.js
- Similar endpoint: src/api/users.js
- DB schema pattern: src/models/base-model.js

## DOCUMENTATION:
- JWT library: https://github.com/auth0/...
- Our API standards: docs/api-guidelines.md

## OTHER CONSIDERATIONS:
- Use existing error handling patterns
- Follow standard response format
- Include rate limiting
PRP workflow
1
Fill INITIAL.md with feature req, examples, docs, gotchas
2
/generate-prp INITIAL.md — agent researches codebase, fetches docs, produces PRPs/feature-name.md
3
/execute-prp PRPs/feature-name.md — fresh session, implements + validates until all criteria pass💡 The PRP scores its own confidence (1–10) before execution begins.
Key insight

Examples folder is critical — place real code patterns in examples/ and reference them in INITIAL.md. The AI follows patterns it can see far better than rules it must infer.

W3 — Progress Files & Decision Logs
📝
Use a handoff.md or todo.md to communicate between agent sessions — or between sub-agents in multi-agent workflows.
📊
When you hit ~200K tokens, performance starts to visibly degrade (confirmed by /context usage). A handoff file lets you start fresh without losing progress.
🔄
The /handoff command creates a structured summary: what was completed, what's in progress, known issues, next steps.
⚠️
If you've needed 2+ compactions in one session, that session is too bloated. Use /handoff to start fresh instead of compacting again.
✓ Write Summary

Log key decisions in git. Split planning from implementation into separate context windows. Create handoff docs when sessions get long. Every token saved at session start is a token that stays available for actual work.

I Isolate — Sub-agents for Clean Context

Run research in isolated context windows. Sub-agents can burn through hundreds of thousands of tokens exploring codebases and the web — only the distilled summary returns to your main session.

The Research Sub-agent Pattern
90.2%
performance improvement vs single-agent on Anthropic's internal research eval ANTHROPIC
~4%
of context window used — vs hundreds of thousands tokens if done inline
80%
of performance variance explained by token budget alone RESEARCH
Anthropic's own multi-agent research system (Claude Opus 4 lead + Claude Sonnet 4 sub-agents) outperformed single-agent Opus 4 by 90.2% on complex research tasks. Token usage alone explained 80% of performance variance — sub-agents let you spend more total tokens without degrading the main context window.
Use Sub-agents For
🔎
Codebase research — "explore the entire auth module and summarize how sessions are managed"
🌐
Web research — best practices, library comparisons, relevant docs
⚡
Parallel exploration — spin up 2–5 sub-agents simultaneously; they run concurrently, not sequentially
📁
Documentation scouting — decide whether to load a large doc before loading it (the Scout Pattern)
Don't Use Sub-agents For
❌
Implementation work — you want full context of what was done during execution, not a summary
❌
Tightly interdependent tasks — sub-agents work in isolation; they can't share real-time state mid-task
⚠️
Cost warning: multi-agent systems consume ~15× more tokens than standard chat. Only worthwhile when value outweighs cost.
The Scout Pattern
1
Identify candidate context — docs in Confluence, Google Drive, .claude/docs/, markdown deep-dives. Files you might want but aren't sure about.
2
Send a scout — "spin up a sub-agent to read everything in .claude/docs/ and tell me which files are relevant to planning a workflow builder"
3
Scout decides — returns a recommendation: load this doc, skip that one. Main context only grows by ~500 tokens for the scout call.
4
Load selectively — only bring in what the scout confirmed is relevant. 💡 Use this before any planning session involving large or infrequently-needed documentation.
✓ Isolate Summary

Sub-agents for research, always. They give you the benefit of broad exploration while returning only the concentrated insight. Use them heavily in planning — not at all in implementation.

S Select — Just-in-Time Context Loading

A four-layer model for what gets loaded, when, and how. The goal: every token in context is there because it's needed right now — not just in case.

The Four Layers
#
Layer
What / When
1
Global RulesAlways loaded
CLAUDE.md — architecture, conventions, testing strategy, logging patterns, commands to run. Applies to every session, every task.
Target: 500–700 lines. Some advocate even less. Keep it lean — these tokens are always present.
2

Rules that apply only to specific subsystems — frontend conventions, API endpoint patterns, workflow YAML spec. Load manually when you know you're working in that area.
Not in global rules because you're not always working on the frontend / that module.

A skill description (small) is in global rules. The full SKILL.md is only read when the agent decides it needs that capability. Skills can also reference scripts and additional docs.
Example: agent browser skill for end-to-end testing — loaded only when you're actually running tests.

Dynamic codebase exploration at the start of a planning session. Sub-agents explore live files so the agent's understanding is current, not stale.

Loading Decision Rule
The Test

If you are not 100% confident that a piece of information is important to the agent right now — don't load it.

✅
Load it — you are about to work on the frontend and have frontend rules
❌
Skip it — "it might be relevant" or "just in case" — leave it out
🔍
Unsure? Use a scout sub-agent to check relevance before loading
Global Rules Sizing & "Right Altitude"
📏
Target 500–700 lines for global CLAUDE.md. These tokens are loaded into every single session.
✂️
Things that belong here: architecture overview, conventions, commands, testing + logging strategy.
🚫
Things that don't: deep-dives into specific modules, rarely-needed reference docs, subsystem specifics.
The "Right Altitude" principle ANTHROPIC
✗ Too Low (Brittle)
❌Hardcoded if/else logic in prompts
❌Exhaustive edge-case lists for every scenario
❌Exact step-by-step scripts with no flexibility
✗ Too High (Vague)
❌Generic guidance that assumes shared context
❌"Write good code" without conventions
❌Prompts with no concrete behavioral signals
✓ Goldilocks Zone

Specific enough to guide behavior, flexible enough to provide strong heuristics. Use diverse, canonical examples rather than exhaustive rules — for an LLM, examples are pictures worth a thousand words.

✓ Select Summary

Global rules always. On-demand context when you know the task domain. Skills lazy-load themselves. Prime commands orient the agent at session start. Every layer adds tokens — load each one only when it earns its place.

C Compress — Last Resort Only

If W, I, and S are working well, you should rarely need this. Compression is a signal that something upstream broke down. When you must compress: two options — /handoff or /compact.

⚠ The Best Compression Strategy

Not needing compression. If you're compressing regularly, you have a context hygiene problem, not a compression problem. Fix W, I, and S first.

/handoff — Structured Summary
📤
Produces a structured markdown doc: what was completed, what's in progress, known issues, next steps. You define the format.
🔄
Works for: agent-to-agent handoffs, same-agent fresh sessions, sub-agent team coordination.
✅
Preferred when you've hit 2+ compactions — start a fully fresh session with this doc instead.
⚡
You control the format — document what actually matters for continuity, not just a generic summary.
/compact — Built-in Compaction
1
Run with instructions — always provide focus guidance: /compact focus on the edge cases we just tested ⚠ Without instructions, the summary may emphasize the wrong things.
2
Conversation is wiped — summary goes to top of new context window. Tool calls and intermediate results are cleared.
3
Verify retention — immediately ask the agent to summarize what it remembers. Don't assume compaction preserved the right things. 💡 "What do you remember about what we just did?" is a mandatory post-compact step.
Compression Decision Tree
?
Context is getting bloated — which to use?
→
First compaction of this session: use /compact with focus instructions, then verify retention.
→
Second+ compaction: the session is too bloated. Run /handoff, start a fresh context window, paste the handoff doc as your only input.
→
Compacting regularly: stop and fix upstream. Are you loading too much in global rules? Skipping the planning/implementation split? Not using sub-agents for research?
⚠ Compaction Caveat

Overly aggressive compaction loses subtle context — edge cases, constraints, failure modes that seemed minor but matter later. This is why /handoff (where you control what's preserved) is preferred over /compact when sessions get truly bloated.

generate-prp.md — Create a PRP from a feature file
Run as: /generate-prp INITIAL.md · Saves to: PRPs/{feature-name}.md
# Create PRP

## Feature file: $ARGUMENTS

Generate a complete PRP for general feature implementation with thorough research.
Ensure context is passed to the AI agent to enable self-validation and iterative
refinement. Read the feature file first to understand what needs to be created,
how the examples provided help, and any other considerations.

The AI agent only gets the context you append to the PRP and training data.
Assume the AI agent has access to the codebase and the same knowledge cutoff
as you, so it's important that your research findings are included or referenced
in the PRP. The Agent has WebSearch capabilities, so pass URLs to documentation
and examples.

## Research Process

1. **Codebase Analysis**
   - Search for similar features/patterns in the codebase
   - Identify files to reference in PRP
   - Note existing conventions to follow
   - Check test patterns for validation approach

2. **External Research**
   - Search for similar features/patterns online
   - Library documentation (include specific URLs)
   - Implementation examples (GitHub/StackOverflow/blogs)
   - Best practices and common pitfalls

3. **User Clarification** (if needed)
   - Specific patterns to mirror and where to find them?
   - Integration requirements and where to find them?

## PRP Generation

Using PRPs/templates/prp_base.md as template:

### Critical Context to Include

- **Documentation**: URLs with specific sections
- **Code Examples**: Real snippets from codebase
- **Gotchas**: Library quirks, version issues
- **Patterns**: Existing approaches to follow

### Implementation Blueprint

- Start with pseudocode showing approach
- Reference real files for patterns
- Include error handling strategy
- List tasks in the order they should be completed

### Validation Gates (Must be Executable)

# Syntax/Style
ruff check --fix && mypy .
# Unit Tests
uv run pytest tests/ -v

*** CRITICAL: BEFORE WRITING THE PRP — ULTRATHINK ABOUT YOUR APPROACH ***

## Output

Save as: PRPs/{feature-name}.md

## Quality Checklist

- [ ] All necessary context included
- [ ] Validation gates are executable by AI
- [ ] References existing patterns
- [ ] Clear implementation path
- [ ] Error handling documented

Score the PRP on a scale of 1-10 (confidence level to succeed in one-pass
implementation). Goal: one-pass implementation success through comprehensive context.
execute-prp.md — Implement a feature from a PRP
Run as: /execute-prp PRPs/feature-name.md · Always run in a fresh context window
# Execute PRP

Implement a feature using the PRP file.

## PRP File: $ARGUMENTS

## Execution Process

1. **Load PRP**
   - Read the specified PRP file
   - Understand all context and requirements
   - Follow all instructions in the PRP and extend the research if needed
   - Ensure you have all needed context to implement the PRP fully
   - Do more web searches and codebase exploration as needed

2. **ULTRATHINK**
   - Think hard before you execute the plan.
   - Create a comprehensive plan addressing all requirements.
   - Break down complex tasks into smaller, manageable steps using todos tools.
   - Use the TodoWrite tool to create and track your implementation plan.
   - Identify implementation patterns from existing code to follow.

3. **Execute the plan**
   - Execute the PRP
   - Implement all the code

4. **Validate**
   - Run each validation command
   - Fix any failures
   - Re-run until all pass

5. **Complete**
   - Ensure all checklist items done
   - Run final validation suite
   - Report completion status
   - Read the PRP again to ensure you have implemented everything

6. **Reference the PRP**
   - You can always reference the PRP again if needed

Note: If validation fails, use error patterns in PRP to fix and retry.

2. Run: git diff --stat HEAD (capture uncommitted changes)

[What was the goal of this session]

## Completed
[What was finished — be specific about files changed]

## Key Decisions
[Why things were done the way they were — critical for next session]

## Dead Ends
[What was tried and didn't work — so next session doesn't repeat it]

## Next Action
[Specific recommended first step for the next session]

commit.md 💡 What it does: creates an enriched commit with conventional tags, a WHY-focused body, and a Context: section that logs changes to rules, commands, or docs alongside code changes. 
# Create Enriched Commit

1. Run: git diff --staged (review what's staged)
   If nothing staged, run: git add -p  (interactive staging)
2. Analyze changes and write commit message in this format:

Format:
{type}({scope}): {what was done}

{WHY this was built — not what, but why it matters}
{Key implementation decisions made}

Context:
- Rules: {any changes to .claude/rules/ or CLAUDE.md}
- Commands: {any new or updated slash commands}
- Docs: {any new on-demand context files added}

# Types: feat, fix, refactor, test, docs, chore
# Context section is omitted if no AI layer changes
# This message becomes long-term memory via /prime git log

3. Run: git commit -m "{message}"
📁 Command File Template Structure

Every command file follows the same pattern: title → $ARGUMENTS declaration → numbered steps → output/format spec. The content is plain markdown — Claude reads it as its instruction set when you invoke the slash command. Keep instructions concrete and reference real file paths from your project.

Context Engineering vs Prompt Engineering
Prompt Engineering
📌Focuses on clever wording and specific phrasing
📌One-shot: optimize how you phrase a task
📌Like giving someone a sticky note
📌Works well for simple classification and text generation
Context Engineering
✅A complete system: docs, examples, rules, patterns, validation
✅Iterative: curated at every inference step, not just once
✅Like writing a full screenplay with all the details
✅Required for multi-step agentic workflows
Context Engineering is 10× better than prompt engineering · 100× better than vibe coding
