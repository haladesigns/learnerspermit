Big-O Notation

Big-O is the speedometer of scaling — it measures how an algorithm's execution time (or memory) grows as the input size n approaches infinity. It does not measure actual seconds; it measures the shape of the growth curve. We always analyze the worst-case scenario.

The Core Question
📈
Big-O asks: as n → ∞, how does runtime grow? This is called asymptotic or "tail" behavior.
🎯
Always worst-case. An unsorted search might finish on element 1, but Big-O assumes you checked all n elements.
✂️
Constants are dropped. O(2n) = O(n). O(n² + n) = O(n²). We care about growth class, not steepness.
🔑
Define n first. Is n the array length? Number of tree nodes? String length? Without a defined n, Big-O is meaningless.

💡 Core Mindset

Don't memorize the shape of code. Understand what your algorithm is actually doing. Two nested loops can be O(n) (sliding window, monotonic stack) if each element is only processed once total. Know the mechanism, not the pattern.

O(1) — Constant Time
⚡
Definition: Runtime does not change regardless of input size. A flat line on the chart.
📦
Examples: arr[i] index lookup, dict[key] hash map get/insert/delete, stack.push() / stack.pop(), grabbing the first item on a list, set.add().
⚠️
Caveat: Hash maps are O(1) average case. In worst case (all keys collide), they degrade to O(n). Always note average vs worst.
O(log n) — Logarithmic Time
✂️
The halving principle: Each step eliminates half the remaining search space. Ask: "How many times can I divide n by 2 until I reach 1?" That count is log₂(n).
🔢
The math: 2ˣ = n → x = log₂(n). For n = 4,000,000,000, log₂(4B) ≈ 32 steps. This is why O(log n) is practically a flat line at scale.
📦
Examples: Binary search on a sorted array, balanced BST search/insert/delete, heap push/pop. Requires an invariant — a property (like sortedness) that stays true and allows halving.
⚠️
BST warning: If the BST becomes skewed (inserting 1, 2, 3, 4... in order), it degenerates to a linked list and BST operations become O(n).
O(√n) — Square Root Time
🔍
When it appears: Finding all factors of n. Factors always come in pairs — one ≤ √n and one ≥ √n. So you only need to iterate up to √n to find both elements of every pair.
📦
Example: for i in range(1, int(n**0.5)+1): if n % i == 0: factors.add(i); factors.add(n//i) — adds both members of each factor pair in one iteration.
O(n) — Linear Time
➡️
Definition: Runtime grows 1:1 with input. Single pass through all elements.
📦
Examples: Sum an array, search unsorted list, iterate a linked list, sliding window algorithms, monotonic stack algorithms.
⚡
Heapify exception: Building a heap from scratch (heapify) is O(n) — even though it looks like it should be O(n log n). This is a known optimization. Individually pushing n elements into a heap is O(n log n).
⚠️
Array insertion trap: Inserting at the end of an array is O(1). Inserting at the front or middle is O(n) because every subsequent element must shift right.
⚠️
Nested ≠ n²: Sliding windows and monotonic stacks can have nested loops but still be O(n) — each element is pushed and popped from the stack exactly once, for O(n) total work.
O(n log n) — Linearithmic Time
🔀
The pattern: Perform O(log n) work for each of the n elements. Visualize merge sort: the array is split in half log n times (log n levels), and at each level, all n elements are merged — so n × log n total operations.
📦
Examples: Merge sort, quicksort (avg), heap sort, all built-in sort functions (sorted() in Python, .sort() in JS). Assume built-in sorts are O(n log n) in interviews.
💡
Heap sort = n log n: Popping the root from a heap is O(log n). Pop n elements → n × log n. Note: building the heap first is only O(n), so O(n) + O(n log n) = O(n log n) total.
O(n²) — Quadratic Time
🤝
The handshake pattern: Every element interacts with every other element. 10 students → 100 handshakes; 100 students → 10,000 handshakes.
📦
Examples: Nested loops over the same array, iterating a 2D grid (n×n), bubble sort, insertion sort, selection sort.
⚠️
Half-square trap: If you iterate n+(n-1)+(n-2)+...+1, that's a triangle — which is half of an n×n square. n²/2 drops its constant → still O(n²). Don't be fooled.
💡
N×M variant: If two arrays have different lengths n and m, nested loops give O(n×m). If sequential (not nested), it's O(n+m). If they're the same array, it's O(n²).
O(2ⁿ) — Exponential
🌳
The fork pattern: Each recursive call creates 2 new calls. A decision tree of height n and 2 branches per node = 2ⁿ total nodes.
📦
Examples: Naïve recursive Fibonacci, generating all subsets/combinations, backtracking problems.
💡
Generalizes to O(cⁿ): If each call creates 3 branches, it's O(3ⁿ). The base is the branching factor, the exponent is the tree height.
O(n!) — Factorial
🔄
The permutation pattern: The decision tree shrinks at each level. For "CAT": 3 choices → 2 choices → 1 choice = 3×2×1 = 6 permutations = 3!
📦
Examples: Generating all permutations, Traveling Salesman Problem (TSP).
⚠️
Scale reality: 20! = 2.4×10¹⁸. At 20 elements this is already computationally impossible for most problems. n=30 is unthinkable.

The 3 Golden Rules of Big-O

These three rules let you simplify any complex expression to its Big-O class in seconds. Master these and you can analyze any algorithm on the fly.

The 3 Golden Rules
1
Drop Constants
O(2n), O(50n), and O(n/2) are all just O(n). Constants only change the steepness of a line, not its shape. As n → ∞, a 2× constant becomes irrelevant relative to the growth class. 
# All of these are O(n):
2n + 5   →  O(n)
n/2      →  O(n)
100n + 999 → O(n)
2
Keep Only the Dominant Term
In O(n² + n), as n grows large, n becomes insignificant compared to n². Keep only the fastest-growing term. Exception: if the variables are different (n vs m), keep both. 
n² + n        →  O(n²)    # same variable, drop smaller
n³ + n² + n    →  O(n³)    # dominant term wins
m + n log n    →  O(m + n log n)  # different vars: keep both
3
Worst-Case Focus
Always analyze the worst-case scenario. Even if you find the item on the first try sometimes, Big-O assumes you might have to check every element. Use the worst-case to design systems that won't fail under bad inputs. ⚠ An interviewer might also ask about best-case (Ω) and average-case (Θ). Be ready to distinguish: "That's O(n) worst-case, but O(1) best-case if the element is first."
3-Step Fast Track: Finding the Tightest Bound
1
Remove standalone constants — strip all fixed numerical additions. 
5n² + 10  →  5n²
2
Identify the dominant growth term — keep only the fastest-growing factor. 
5n² + n log n  →  5n²
3
Discard leading coefficients — drop all multipliers. 
5n²  →  O(n²)
Addition Rule (Sequential)
➕
Two loops that run one after the other (not nested): add the complexities, then apply the dominant term rule.
# Loop 1: O(n), then Loop 2: O(m)
for i in range(n): ...  # O(n)
for j in range(m): ...  # O(m)
# Total: O(n + m)
# If n == m: O(2n) → O(n)
Multiplication Rule (Nested)
✖️
Two loops nested inside each other: multiply the complexities.
# Loop over n, inner loop over m
for i in range(n):
  for j in range(m):
    ...
# Total: O(n × m)
# If n == m: O(n²)

⚠️ Common Proof Technique — Upper Bound Validation

To prove f(n) = O(g(n)), find constants c and n₀ such that f(n) ≤ c·g(n) for all n ≥ n₀. For 2n+3 = O(n): choose c=5 and n₀=1. Then 2(1)+3 = 5 ≤ 5·1. Valid. To disprove O(1) for 2n+3: assume bounded by constant c. For n > c/2, we have 2n > c, so 2n+3 > c — contradiction.

Array — Row of Lockers
✅
O(1) index lookup: Contiguous memory means the CPU calculates the exact address and jumps there instantly. Excellent cache locality.
⚠️
O(n) front/middle insert: Adding a locker in position 1 forces every other locker to shift right. Worst case: insert at position 0.
⚠️
O(n) resizing: When capacity is exceeded, allocate a new larger block and copy every element. This O(n) spike is amortized O(1) over many operations.
💡
Best for: Random access by index, fixed-size data, cache-friendly iteration.
Linked List — Train Cars
✅
O(1) insert/delete (with reference): Just redirect pointers — no shifting. No resizing wall ever.
⚠️
O(n) access: No index jumps. You must traverse from the head car-by-car to reach node #50. Poor cache locality (pointer chasing).
⚠️
O(n) to find before insert: Insert itself is O(1), but finding the node to insert next to first requires traversal.
💡
Best for: Frequent insertions/deletions where you hold references, flexible sizing.
Hash Map — Office Mailroom
✅
O(1) average for all ops: Hash function maps key → index. Direct lookup without scanning.
⚠️
Collision degrades to O(n): If many keys hash to the same bucket, lookups degrade to scanning a list within that bucket.
💡
Resolution: Chaining (linked list per bucket) or linear probing (find next empty slot). Both restore average O(1) with good load factors.
💡
Space-time trade: Using a hash map trades O(n) extra space to reduce O(n²) search to O(n). Classic two-sum pattern.
Heap — Pyramid of Boxes
✅
O(1) peek at min/max: Root is always the smallest (min heap) or largest (max heap). Instant access to the "most important" item.
💡
O(log n) insert/delete: New element is added, then "bubbles up" by swapping with parents until heap property is restored. Tree height = log n.
⚡
O(n) heapify: Building a heap from an unsorted array is O(n), not O(n log n). Use heapq.heapify() in Python — not individual pushes.
⚠️
No arbitrary search: Heap only guarantees root access. Finding a specific non-root element is O(n).

Space Complexity

Space complexity asks the same question as time complexity — but for memory: how does memory usage grow as n grows? Same complexity classes apply (O(1), O(n), O(n²), etc.). The key distinction is auxiliary space — memory your algorithm allocates beyond the input itself.

Auxiliary vs Total Space
📐
Total space = input space + auxiliary space. Big-O usually refers to auxiliary (extra) space.
✅
O(1) auxiliary: Algorithm modifies input in-place. No new arrays, no proportional allocations.
💡
O(n) auxiliary: Creating one new array of size n. Common and often acceptable.
⚠️
O(n²) auxiliary: Creating an array of n elements for every n elements in input. Memory exhaustion risk.
The Trade-off Principle
💰
"We can buy memory, but we can't buy time." In production, it's almost always worth spending extra O(n) space to reduce time from O(n²) to O(n).
📦
Classic example: Two-sum problem. Brute force (nested loops) = O(n²) time, O(1) space. Hash map approach = O(n) time, O(n) space. The trade is almost always worth it.
✅
Interview rule: Only approve O(n) extra space if it yields a measurable reduction in time complexity class (e.g., n² → n).
Recursion — The Invisible Stack
📚
Every recursive call occupies a stack frame. The call stack is space you're using even though you didn't explicitly allocate it. Interviewers may ask you to include or exclude recursive stack space — ask first.
🌳
The max-depth rule: Space complexity of recursion = maximum depth of the call stack at any one time. Not total nodes visited — just how deep the stack gets before backtracking.
💡
Fibonacci example: The call tree for fib(n) has O(2ⁿ) total calls (time), but the call stack is only ever n levels deep at maximum → O(n) space. Backtracking removes completed calls before new branches are explored.
⚠️
Stack overflow risk: If recursion depth is O(n) and n is very large (millions), this is an architectural risk. Consider iteration + explicit stack, or tail recursion where supported.
# Fibonacci recursion tree — fib(4)
# fib(4)
#   fib(3)          fib(2)
#     fib(2) fib(1)   fib(1) fib(0)
#
# Time:  O(2^n) — exponential total nodes
# Space: O(n)   — max stack depth at any moment = n levels

✅ Memory Audit Checklist

Before submitting: Do you create any new data structures proportional to n? Count each new array, dict, or set. Is your recursion depth bounded? Does your algorithm have any O(n) resizing events (dynamic array overflow)? Can you modify the input in-place instead?

Critical Gotchas
⚠️
Nested ≠ always n²: Sliding window and monotonic stack algorithms have nested loops but are O(n) because each element is pushed/popped at most once. Count total work, not loop nesting.
⚠️
Array delete/insert at front is O(n): Even though it looks like one operation. Shifting all n elements is the hidden cost.
⚠️
Hash map worst case is O(n): Always distinguish average O(1) from worst-case O(n) when collisions cause degenerate chains. State "O(1) amortized average."
⚠️
Unbalanced BST degenerates to O(n): Insert 1, 2, 3, 4 in order and you get a linked list. BST guarantees only hold when balanced.
⚠️
Heapify vs n pushes: Building heap from array = O(n). Pushing n items one at a time = O(n log n). The difference matters. Always prefer heapify when you have the full array upfront.
⚠️
Triangular iteration is still O(n²): n + (n-1) + (n-2)...+1 = n(n+1)/2 = n²/2. The constant ½ drops → O(n²). A triangle is half a square.
⚠️
Two different input variables: If your algorithm uses array A (size n) and array B (size m), sequential loops = O(n+m). Nested loops = O(n×m). Don't collapse both to "O(n)".

JR

Identifies simple O(n) and O(n²) loops. Guesses complexity based on loop count. Misses hidden costs like array resizing, recursive stack depth, or the heapify distinction.

🎯 Interview Opening Move

When given a problem, immediately state the time and space complexities of your proposed solution before writing a line of code. It shows you're thinking about scalability from the start — and if your complexity claim hints at a known pattern (e.g., O(log n) suggests binary search), it can guide your implementation.

Common Python Patterns + Their Complexity
# ─── TWO SUM — O(n) time, O(n) space ───────────────────────────
def two_sum(nums, target):
    seen = {}                          # hash map: O(n) space
    for i, n in enumerate(nums):      # single pass: O(n) time
        diff = target - n
        if diff in seen:               # O(1) lookup
            return [seen[diff], i]
        seen[n] = i

# ─── BINARY SEARCH — O(log n) time, O(1) space ─────────────────
def binary_search(arr, target):
    lo, hi = 0, len(arr) - 1
    while lo <= hi:                    # halves search space: O(log n)
        mid = (lo + hi) // 2
        if arr[mid] == target: return mid
        elif arr[mid] < target: lo = mid + 1
        else: hi = mid - 1
    return -1

# ─── SLIDING WINDOW — O(n) time, O(1) space ────────────────────
# Nested loop that is still O(n): each element added/removed once
def max_sum_subarray(nums, k):
    window = sum(nums[:k])
    best = window
    for i in range(k, len(nums)):     # O(n) total
        window += nums[i] - nums[i-k]
        best = max(best, window)
    return best

# ─── HEAPIFY — O(n) vs n pushes O(n log n) ─────────────────────
import heapq
nums = [3, 1, 4, 1, 5, 9]

# FAST: heapify existing list in-place — O(n)
heapq.heapify(nums)

# SLOW: push one at a time — O(n log n)
h = []
for n in nums:
    heapq.heappush(h, n)           # each push: O(log n)
✅ Python-Specific Rules

Use deque not list when doing queue operations (BFS). Use set for membership tests, not list. Use heapq.heapify() not a loop of heappush. sorted() and .sort() are both O(n log n) Timsort. Value lookup in dict is O(n) — only key lookup is O(1).
