MCPs & CLIs

MCP (Model Context Protocol) is an open standard — like USB-C for AI — that connects Claude Code to external tools: databases, APIs, version control, design tools, and more. All via plain English in your terminal.

What MCPs Enable
🔌
One standard, any tool. Notion, GitHub, Figma, Supabase, Slack — add them all via a single command pattern.
📋
Team-shareable configs. Drop a .mcp.json in your project root and every teammate gets the same integrations on day one.
⚡
Lazy loading. Tools are no longer all pre-loaded — they're fetched on demand like skills, keeping your context window lean.
🌐
No app switching. "Find the top 5 Sentry errors and open a GitHub issue for the worst one" — all in one conversation.
MCP vs CLI
🔧
MCP: Extra structure that lives between Claude Code and the program. Includes tool descriptions + auth overhead built-in. Good for cloud services.
💻
CLI: Everything happens in the terminal. No middleware overhead. Tailor-made for AI agents. Faster, leaner.
📖
CLI needs a skill. Unlike MCPs, CLIs don't carry self-description — create a companion skill file so Claude knows how to use the CLI.
How It Works — The Three Pieces
MCP HOST
Claude Code — coordinates everything and decides which tools to call.
MCP CLIENT
Connector inside Claude Code — maintains a dedicated link to one MCP server.
MCP SERVER
Small program wrapping a tool or data source — exposes actions to Claude.
📡 Transport: stdio

Claude starts a local program on your machine. Best for local tools. No network setup needed.

🌍 Transport: HTTP

Claude connects to a remote service over the internet. Best for cloud services. Supports full OAuth/login.

🚀 The Trend

The space is moving from MCPs → CLIs. When a CLI exists with the same functionality, prefer it. Pair every CLI with a companion skill file. Ask Claude Code if you're unsure which to use — it can research it for you.

Supabase MCP Setup

A practical walkthrough using Supabase — the capstone project database. The same pattern applies to any MCP server.

Step-by-Step: Add Your First MCP Server
1
Add the Supabase MCP server (HTTP, recommended)
Run this in your terminal. Claude Code registers the server location. 
claude mcp add --transport http supabase https://mcp.supabase.com
2
Authenticate inside Claude Code
Type /mcp inside Claude Code, select Supabase → Authenticate. A browser window opens — log in and grant access. 
/mcp
⚠ You need an existing Supabase account. Claude Code cannot create one for you.
3
Verify the connection
Check server status from the terminal or inside Claude Code: 
claude mcp list         # terminal
/mcp                    # inside Claude Code
💡 Look for "supabase" with a healthy status indicator.
4
Use it in plain English
Claude calls MCP tools automatically behind the scenes: 
# Try these prompts:
"List all tables in my Supabase project [name]"
"Show me the schema for the posts table"
"Create a migration that adds a 'views' column to posts"
Alternative: Local / Offline Setup
—
stdio transport (runs locally on your machine) 
# Windows — requires cmd /c wrapper for npx
claude mcp add --transport stdio supabase-local -- cmd /c npx -y @supabase/mcp-server-supabase@latest --read-only --project-ref=YOUR_REF
# macOS / Linux
claude mcp add --transport stdio supabase-local -- npx -y @supabase/mcp-server-supabase@latest --read-only --project-ref=YOUR_REF
⚠ Windows: The cmd /c wrapper is mandatory. Skipping it causes "Connection closed" errors.
Adding a CLI Tool
1
Install the CLI
Ask Claude Code in natural language, or point it at the install URL. Example: Playwright CLI for browser automation. 
# Claude Code can install it — just ask:
"Install the Playwright CLI and show me how to use it"
2
Create a companion skill file
CLIs don't self-describe like MCPs do. Add a skill markdown file that teaches Claude how to use the CLI. 💡 Most well-known CLIs (like Playwright) already have community skill files — check their docs for a "Claude Skills" section.
3
Confirm with Claude Code
Ask Claude if a CLI exists before defaulting to an MCP: 
"Does Supabase have a CLI I can use instead of the MCP?"
Team Sharing via .mcp.json

Commit this file to your project root. Every developer gets the same MCP tools automatically.

{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    },
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp"
    },
    "database": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@bytebase/dbhub", "--dsn", "${DATABASE_URL}"],
      "env": { "DATABASE_URL": "${DATABASE_URL}" }
    }
  }
}
The ${VAR} syntax pulls from environment variables — sensitive values like passwords stay out of your code.

Managing Servers
Command What It Does Notes
claude mcp list List all configured servers Run from terminal
claude mcp get <name> Details for a specific server Shows transport, URL, scope
claude mcp remove <name> Remove a server —
/mcp View status, auth, manage — inside Claude Code Also shows auth state
claude mcp serve Run Claude Code as an MCP server Lets other tools use Claude Code's file abilities
Scope Flags
Scope Flag Who Sees It Stored In
local DEFAULT (no flag needed) Only you, this project only Local config
project TEAM --scope project All teammates via .mcp.json .mcp.json (commit it)
user GLOBAL --scope user You, across all your projects User config
Environment / Debug Flags
Command What It Does Default
MAX_MCP_OUTPUT_TOKENS=50000 claude Increase MCP output token limit 25,000
MCP_TIMEOUT=15000 claude Increase server startup timeout (ms) 10,000
ENABLE_TOOL_SEARCH=auto:5 Control lazy tool search threshold auto
⚠ Flag Order Rule

All flags (--transport, --env, --scope, --header) must come before the server name. The -- double-dash separates Claude's options from the server launch command. Flags after the server name are silently ignored.

✓ correct:  claude mcp add --transport stdio --env KEY=val myserver -- npx server
✗ wrong:    claude mcp add myserver --env KEY=val -- npx server
@ Mention Syntax

MCP servers can expose data you reference directly in prompts. Type @ to see suggestions from connected servers:

"Can you analyze @github:issue://123 and suggest a fix?"

✓ Best Practices
✅Use CLI over MCP when the CLI has equivalent functionality — it's lighter and faster.
✅Always pair a CLI install with a companion skill file.
✅Use --scope project + commit .mcp.json for team setups.
✅Use /mcp inside Claude Code to check status, re-auth, and troubleshoot.
✅Ask Claude Code to research whether a CLI or MCP is better for your use case.
✅Use ${VAR} in .mcp.json to keep secrets out of source control.
✗ Common Mistakes
❌Putting flags after the server name — they'll be silently ignored.
❌Skipping cmd /c on Windows with npx-based servers.
❌Connecting MCP to a production database.
❌Using unverified third-party MCP servers without reviewing their source.
❌Adding a CLI without a skill file — Claude won't know how to use it.
❌Ignoring MAX_MCP_OUTPUT_TOKENS warnings — large outputs eat context fast.
Gotchas Reference
🪟
Windows + npx = cmd /c: On native Windows (not WSL), wrap npx with cmd /c or you'll get "Connection closed" errors every time.
🚩
Flag order matters: All flags must come before the server name. The -- separator divides Claude's options from the server's launch command. Wrong order = silently broken.
🔒
Never connect to production DBs: Use dev or staging. MCP lets Claude run queries — you don't want experimental operations against real user data.
🕵️
Review third-party servers: Not all community MCP servers are verified by Anthropic. Stick to the official repository or known publishers.
📏
MCP output eats context: Tool responses count against your conversation window. If you see warnings, raise the limit: MAX_MCP_OUTPUT_TOKENS=50000 claude (default is 25k).
⏱
Slow server startup: If a server times out on start, increase the timeout: MCP_TIMEOUT=15000 claude.
🔑
Re-authenticate after credential reset: If you reset Supabase (or any service) credentials, the MCP shows as unauthenticated. Run /mcp → Authenticate to restore it.
🛠
Debugging deep issues: Use the MCP Inspector tool (github.com/modelcontextprotocol/inspector) for interactive debugging when /mcp isn't enough.
✅ Quick Sanity Check

Before shipping: run /mcp to verify all servers are connected and authenticated. Run claude mcp list from the terminal to confirm scopes are set correctly for local vs team use.
